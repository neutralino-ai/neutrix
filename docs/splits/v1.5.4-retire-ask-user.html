v1.5.4 splits — retire AskUserQuestion (defer to v3)

Remove the v1.4.8 AskUserQuestion tool and its entire interactive-input apparatus from v2. The concept defers to v3 (interactive human-feedback fits the RL-environment vision).

Why (user-directed 2026-05-30): "it's not stable. remove the askuser from all and defer to v3. v2 shall not take 'askuser'. so remove ask user tool etc." The asking machinery has been a recurring instability source — the v1.4.8 park-forever regression and the v1.5.1 prompt-hang both lived in the interactive-prompt path. An industrial agent does not pause for the human.
CC survey: CC keeps an AskUserQuestion-style interactive tool (`canUseTool`/AskUserQuestion). neutrix diverges (User-directed different) — it removes its own. After v1.5.3 nothing else uses the channel (permission was already decoupled), so this is a clean excision, not a refactor-with-a-replacement.

Split points

1.Scope of removal — the tool only, or the whole apparatusdecided
The fork
(a) drop just the tool registration (leave the channel/ports dormant); (b) remove the entire apparatus.
Decision
User-directed — (b) remove ALL. "remove the askuser from all … remove ask user tool etc." Gone: the AskUserQuestion tool + _ask_user_question backstop (tools.py); the whole prompts.py module; the Executor's AskUserQuestion branch + _needs_input/needs_user_input event; the CM ask_user port + AskUserPort import + the needs_user_input handler; terminal_chat._ask_user + PROMPT_TIMEOUT_S + the pending-answer Future machinery + the "answer needed" indicator; the cli.py wiring.
2.dispatch_all — keep the bidirectional generator, or revert to plaindecided
Now
v1.4.8 made Executor.dispatch_all a bidirectional async generator (answer = yield …; CM drives it with gen.asend()). AskUserQuestion was its only input consumer (permission stopped using it in v1.5.3).
Decision
Alternative — revert to a plain event-emitting async generator. With no input consumer, dispatch_all goes back to plain yield ToolEvent(...) and the CM's _dispatch_tools back to a plain async for event in gen (no asend/send loop, no aclose-on-park). Simpler and removes the class of "parked at a yield" bugs. (Keeping the scaffold "for v3" is YAGNI — v3 will design its own RL feedback channel.)
3.prompts.py — delete the module or keep the data typesdecided
Decision
Alternative — delete the module. Option/Question/QuestionSpec/Answer/QuestionAnswer, the parse/render helpers, ASK_NOT_AVAILABLE, and AskUserPort are used only by AskUserQuestion (permission's use went in v1.5.3). Nothing else imports them → delete src/neutrix/prompts.py and tests/test_prompts.py.
4.The subagent tool setdecided
Now
subagent_tool_names() = BUILTIN_TOOLS − {Agent, AskUserQuestion} (v1.4.8 split #7 kept a humanless subagent from deadlocking on a question).
Decision
Follow — mechanical. With the tool gone it's simply BUILTIN_TOOLS − {Agent}; the deadlock guard is moot (no question tool exists).
5.Deferring the concept to v3decided
Decision
No CC analog (a deferral) — record it on the v3 line. The roadmap's v3 (RL-environment) section gains a note: interactive human-feedback / ask-the-user belongs to v3, where a tunable, seedable env can model human input as part of the loop — not v2. v2 ships with no human-in-the-loop tool; the agent always proceeds autonomously (degrading exactly as headless runs already did).
