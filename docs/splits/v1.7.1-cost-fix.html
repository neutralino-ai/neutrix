v1.7.1 splits — config pricing · 3-number cache · first-run bootstrap

Fix the three issues live use of v1.7.0's cost feature surfaced: hand-estimated prices, cache tokens hidden behind ↑↓, and a broken interactive onboard.

Agent-decided under the user's standing "you make decision yourself" delegation — the user steered by direct instruction across the design conversation and rejected a turn-by-turn AskUserQuestion, so these are recorded (not surfaced one-by-one). The price numbers and cache field shapes are verified against the live IHEP gateway and the real LiteLLM JSON, not assumed. See docs/PRDs/v1.7.1-cost-fix.md.

Split points

1.Where prices livedecided
v1.7.0 bug
pricing.py shipped a hand-estimated USD table whose docstring overclaimed "seeded from LiteLLM" — and was 3× off (claude-opus-4-7 is really $5/$25 per Mtok, not the guessed $15/$75).
Options
(a) vendor a LiteLLM JSON subset in the package + a pricing_model name-map; (b) put prices in the config YAML, keyed by the exact model string; zero price data in code; (c) add the litellm dependency.
Decision
(b) — config YAML, user-directed ("write the price in yaml directly, no fixed code in neutrix"). A pricing: block: a currency symbol + a models map of model → {input, output, cache_read, cache_write} (per million tokens). pricing.py becomes pure mechanism (cost(usage, price)); no vendored data, no litellm, no name-mapping (the key is the slot's model string). Missing model → (cost unknown). Reprice = edit the YAML ([[project-v2-v3-direction]]: LiteLLM data not dep — taken further: not even the data ships).
2.Currencydecided
History
User first asked CNY, then — after we found neither LiteLLM nor a reliable CNY source covers these models — reversed to "use litellm and usd only".
Decision
A config display symbol (default $); the numbers are whatever the user writes. Pre-filled with real USD rates (from LiteLLM's public JSON), switchable to ¥ by editing the symbol + numbers. No currency logic in code — the CNY-vs-USD question dissolves into config.
3.Cache token model — 3 numbersdecided
v1.7.0
Status line + per-model fold cache into one ↑ (input+cache); user "only sees up and down".
Learned from
dr.sai-nano/notebooks/deepseek_cache_explained.ipynb: DeepSeek reports prompt_cache_hit_tokens / prompt_cache_miss_tokens (hit+miss = prompt), no cache-write tier.
Decision
Display hit · miss · output (user-directed "3 numbers, merge to 1"): hit = cache_read; miss = fresh input + cache_write; output. hit + miss = total input, matching DeepSeek's native model. The 4 normalized fields + raw stay internal (accurate pricing — cache_write still priced at its config rate — and no JSONL/ledger churn, v1.7.0-compatible); only the display collapses to 3.
4.Cache-capture robustness (the real v1.7.0 gap)decided
Verified live
IHEP gateway, DeepSeek warm call: prompt_tokens=1517, cached_tokens=1408 (gateway populates the OpenAI-standard field + a top-level cache_read_tokens). Direct api.deepseek.com: the native prompt_cache_hit_tokens instead. Anthropic gateway: cache_read_input_tokens (auto-caches a large prefix) / input_tokens / output_tokens — consistent across message_start + message_delta.
Decision
Follow the verified shapes. _usage_from_openai reads the hit from the first present of prompt_tokens_details.cached_tokens / cache_read_tokens / prompt_cache_hit_tokens, and the miss from prompt_cache_miss_tokens else prompt − hit. Anthropic unchanged. (v1.7.0's gateway path already worked; it missed only direct-DeepSeek's native fields.)
5.First-run bootstrap — replace the broken interactive onboarddecided
Research
opencode's "free model without registration" = its own funded zen gateway (provider id opencode) serving allowlisted models (kimi-k2-thinking, glm-4.6) free/anonymous, rate-limited, paid by opencode under their ToS — not a client-side key trick (the client stores only oauth/api credentials). Not replicable without a funded proxy; free-riding it violates their ToS.
Options
(a) fix the interactive onboard TUI; (b) an LLM-driven init skill (needs a model first → the bootstrap chicken-and-egg); (c) a static example config + a fill-the-key message, no interactivity, no model.
Decision
(c) — user-directed. Writing a config is deterministic, so no model is needed and the chicken-and-egg dissolves. Ship an IHEP-only example config (empty api_key, frozen USD pricing: block); cli.py already writes it when config is missing — extend the empty-key case (currently the broken run_onboarding fallback) to print "fill your key at <path> and re-run" + exit. Delete onboard.py + /onboard. Interactivity was the broken part; copying a file + a message can't break.
