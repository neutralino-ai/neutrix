v1.6.2 splits — tool_use execution timeout (harden Bash)

Not greenfield. neutrix's Bash tool already exposes a timeout schema parameter (integer seconds, default 30) and enforces it (src/neutrix/tools.py:404-457, :757-761). This release (a) fixes a latent deadlock in that timeout path, (b) calibrates the default/cap, and (c) keeps partial output. Scope is Bash-only, mirroring CC (user decision, Split #1).

The latent bug (likely the "stuck" the user hit): on timeout, _run_shell calls proc.kill() — SIGKILL to the direct child only, not the process group — then blocks on a bare proc.communicate() with no timeout (tools.py:437-438). If the command spawned a grandchild that inherited the stdout pipe (a backgrounded subprocess, a daemon), that second communicate() can block forever even though the timeout "fired". The fix reuses the _tree_kill helper that already exists for the cancel path (executor.py:185-212, killpg SIGTERM→SIGKILL).

CC survey (/datafs/users/dingxf/anthropic/cc2/src): CC's Bash tool exposes an LLM-settable timeout schema param — "Optional timeout in milliseconds (max …)" (tools/BashTool/BashTool.tsx:229); default 120 000 ms / max 600 000 ms, both env-overridable (BASH_DEFAULT_TIMEOUT_MS / BASH_MAX_TIMEOUT_MS, utils/timeouts.ts). Enforcement: setTimeout → SIGTERM → tree-kill SIGKILL after a 5 s grace (utils/ShellCommand.ts:135-140,323-343; ripgrep mirrors this, utils/ripgrep.ts:169-182). On timeout CC keeps partial output and prepends "Command timed out after <duration>" to stderr (ShellCommand.ts:323-327). Other tools have fixed internal timeouts not exposed in schema — ripgrep 20–60 s, WebFetch 60 s; file tools have none. There is no global per-tool-call timeout wrapper — each tool owns its own. CC also auto-backgrounds a Bash command on timeout and exposes run_in_background (BashTool.tsx:241,988-1010).

Status: Split #1 (scope) decided Bash-only by the user via AskUserQuestion (2026-05-31). Splits #2–#6, #8 are agent recommendations following from that scope (fix-the-bug / Follow-CC-default / keep-existing); the user saw them in chat and they are ratified at the PRD accept gate. Version v1.6.2 (reliability hardening, continues the v1.6.x sub-line, slots before v1.7.0 cost/timing).

Split points

1.Timeout SCOPE — per-call Bash schema param, or a global backstop over ALL tool_use, or both?user-directed
The user's Q
"for tool_use, need to add time out … pass it as scheme parameters?" — the core fork.
CC
Per-call schema param on Bash only; fixed internal timeouts on a few others; no global wrapper (BashTool.tsx:229).
neutrix today
Bash schema param (default 30 s); Grep-rg fixed 30 s (tools.py:354); pure-compute tools (Glob, Grep-python fallback, Read, Edit, Write) have NO timeout and run in asyncio.to_thread (executor.py:167) — a hung one can't be killed, only abandoned.
Options
(A) Bash-only harden — mirror CC, leave pure-compute unprotected; (B) Bash schema param + a global executor backstop wrapping every tool_use; (C) per-tool fixed timeouts everywhere + Bash settable.
Decision
(A) Bash-only — Follow CC, user-directed (AskUserQuestion, 2026-05-31; the user chose "Bash-only (mirror CC)" over the agent's backstop recommendation (B)). Harden the existing Bash timeout schema param; leave the pure-compute tools unprotected exactly as CC leaves its file tools. Rationale: fits the user's "keep it simple, otherwise we never ship" North Star — the pure-compute hang is judged too rare to engineer a thread-abandonment backstop for. Closes Split #7 as a consequence.
2.Units — seconds (neutrix today) or milliseconds (CC)?decided
CC
milliseconds (BashTool.tsx:229).
neutrix today
integer seconds, "Timeout in seconds." (tools.py:757-761).
Decision
Keep SECONDS (Alternative — diverge from CC on purpose). neutrix already ships seconds; switching to ms breaks the existing schema and the LLM's habit for zero user benefit, and seconds read better for humans. Update the description to state the new default + cap.
3.Default timeout — 30 s (today) or 120 s (CC)?decided
CC
120 000 ms = 120 s (utils/timeouts.ts).
neutrix today
30 s (tools.py:406,760).
Decision
Raise to 120 s (Follow CC). 30 s spuriously kills ordinary dev commands — pytest, pip install, a build, a git clone — and the LLM then misreads the timeout as a real failure and thrashes. Calibrate against the slow case, not the fast one ([[feedback_llm_timeout_calibration]]). Env-overridable via NEUTRIX_BASH_DEFAULT_TIMEOUT_S.
4.Maximum cap — none (today) or 600 s (CC)?decided
CC
600 000 ms = 600 s hard cap, env-overridable (utils/timeouts.ts).
neutrix today
No cap — the LLM can pass timeout: 999999 and the turn hangs ~11 days (only user-cancel or the LLM watchdog saves it).
Decision
Cap at 600 s, clamp with a note (Follow CC). A runaway value is exactly the liveness hole this release closes. Clamp and note "(clamped to 600s)" in the result so the LLM learns the ceiling. Env-overridable via NEUTRIX_BASH_MAX_TIMEOUT_S.
5.Kill scope on timeout — proc.kill() direct child (today, deadlock-prone) or _tree_kill process-group (CC)?decided
The bug
tools.py:437-438: proc.kill() (SIGKILL, direct child only) then a bare proc.communicate() with no timeout — a leaked grandchild holding the stdout pipe blocks it forever.
CC
SIGTERM → tree-kill SIGKILL after 5 s grace (ShellCommand.ts:135-140,323-343).
neutrix asset
_tree_kill already exists and is correct (executor.py:185-212: killpg SIGTERM → SIGKILL after 0.2 s grace) — but is wired only to the cancel path. _run_shell already uses start_new_session=True, so the process group is there to kill.
Decision
Route the timeout path through _tree_kill (Follow CC semantics; reuse the existing helper). Fixes the grandchild leak AND the latent forever-block — probably the actual "stuck" the user hit. Bound the post-kill output drain with a short timeout so the path can never re-block.
6.Partial output on timeout — discard (today) or keep + notice (CC)?decided
CC
Keeps partial stdout/stderr, prepends "Command timed out after <duration>" to stderr (ShellCommand.ts:323-327).
neutrix today
Discards everything, returns only "ERROR: command timed out after {timeout}s" (tools.py:443-444).
Decision
Keep partial output + the timeout notice (Follow CC). The partial stdout/stderr captured before the kill is often the most diagnostic signal (where did it hang, what was the last line). Keep the ERROR: prefix so the executor's ok=False still propagates (executor.py:173).
7.Non-killable pure-compute (thread) tools — protect them?decided
Hard constraint
Pure-compute tools run in asyncio.to_thread (executor.py:167). Python threads cannot be force-killed — a "timeout" can only abandon the wait; the thread keeps running.
CC
No analog — CC's file/glob tools are async Node I/O; it has no thread-pool-blocking failure mode.
Decision
Leave unprotected (Follow CC — consequence of Split #1 = Bash-only). No global backstop, no subprocess-ification. Accepted risk (documented, not silent): a pathological regex or a multi-GB rglob can still hang a turn unrecoverably (only user-cancel saves it); the user judged that hang rare enough to skip the abandon-and-report backstop. Reopen if real OMILREC use surfaces such a hang.
8.Auto-background / run_in_background on timeout — add or defer?decided
CC
Auto-backgrounds a long Bash command on timeout instead of killing; exposes a run_in_background schema param that bypasses the timeout (BashTool.tsx:241,988-1010).
neutrix
No background subsystem; background Bash is already DEFERRED post-v2 per the band scope ([[project_v2_v3_direction]] — "process registry + capture + poll + kill + lifecycle is a subsystem").
Decision
DEFER (Alternative — out of scope, already a deferred subsystem). v1.6.2 kills on timeout (with partial output); auto-background/run_in_background reopens only if real OMILREC use demands it, after v2. Keeps "simple, otherwise we never ship."
