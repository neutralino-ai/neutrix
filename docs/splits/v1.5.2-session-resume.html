v1.5.1 splits — session persist + resume

Auto-persist every turn to a CC-compatible session log; --continue the most recent, /resume a chosen one. Records per-turn timing (user-flagged gap).

Scope: conversation resume only (user split it from the roadmap's "/goal+resume" bundle; the native /goal loop is a separate later release). Builds on the existing transcript.py /save·/load (single pretty-JSON file). CC survey: sessions live at ~/.claude/projects/<sanitized-cwd>/<uuid>.jsonl, appended per message, each entry stamped with timestamp/cwd/sessionId; --continue = most-recent-by-mtime in the cwd dir; --resume = picker over head/tail-lite reads (utils/sessionStorage.ts, conversationRecovery.ts, listSessionsImpl.ts, sessionRestore.ts).

Split points

1.On-disk format + location — JSONL, but NOT in ~/.claudedecided
CC
One .jsonl per session under ~/.claude/projects/<sanitized-cwd>/<uuid>.jsonl, appended per message, never rewritten.
The fork
(a) write at CC's path (~/.claude/projects) for ccusage/CC---resume compatibility; (b) write in neutrix's own dir, out of ~/.claude.
Decision
User-directed different — (b): ~/.cache/neutrix/sessions/<sanitized-cwd>/<uuid>.jsonl, NOT ~/.claude (user, 2026-05-29: "don't write to ~/.claude folder"). Keep CC's line shape (append-per-message, {type, message, timestamp, …}) so it's familiar + future-portable, but the files live in neutrix's own cache. Tradeoff accepted: ccusage scans ~/.claude/projects so it won't natively see neutrix sessions — the user prefers a clean ~/.claude; resume value is fully preserved. Base honors $NEUTRIX_SESSION_HOME (override / test isolation). transcript.py /save·/load stays for explicit snapshots; this auto-log is additive. (Revises the earlier ".claude-write" inference: discovery of .claude config stays; writing INTO it does not.)
2.When to write — append-as-you-go vs on-exitdecided
CC
Buffered append per message, flushed async (survives a crash mid-session).
Decision
Follow CC — append-as-you-go. The CM is the sole message mutator; hook the session-writer to ChatStore.changes() (same subscription the renderer uses) and append each new MessageRecord as a JSONL line. A crash/kill mid-session (the user's recurring hang) then leaves a resumable log up to the last completed turn — exactly what was missing.
3.Per-entry timing record (user-flagged gap)decided
CC
Each entry carries a wall-clock timestamp.
Decision
Follow CC + Alternative add: per-turn durations. User: "LLM, tool time is not recorded, right?" — correct, the v1.5.0 elapsed is live-only. Each JSONL entry gets a timestamp; additionally, an assistant entry records its llm_ms (round wall-time, from phase_started_at) and a tool entry records tool_ms. So durations become queryable from the log — the seed for the later cost/timing view. Pure additive metadata (CC ignores unknown fields).
4.--continue — most recent in cwddecided
CC
Loads the highest-mtime .jsonl in the cwd's project dir, restores the message array (replayed to the model on the next turn).
Decision
Follow CC. New CLI flag --continue/-c: pick the most-recent session log for the cwd by mtime, rebuild messages + ChatStore + tasks via the existing ReplaceHistoryEvent path (already used by /load). No index file — just a dir scan + sort (CC does the same).
5./resume — pick a past sessiondecided
CC
Interactive picker listing sessions with first-prompt/title/branch/mtime, from head+tail lite reads.
Decision
Alternative — numbered in-flow list (reuse the AskUserQuestion channel), not a full-screen picker. /resume lists the cwd's sessions (newest first) as N. <first-prompt> · <n msgs> · <ago>; the user types a number (same numbered-answer model as v1.4.8, append-only-friendly). A bare CLI --resume <id> loads a specific session non-interactively. Lite head/tail read for the list metadata (don't parse whole files).
6.What's restoreddecided
CC
Full message array, tasks/todos, agent+model setting, worktree/cwd; NOT ephemeral UI/stream state.
Decision
Follow CC (scoped to what neutrix has): the full messages array (for the model), the typed MessageRecords (for the renderer), and tasks — exactly what ReplaceHistoryEvent already restores. Slot stays the launch slot (don't override the user's --fast/--strong choice on resume). Pending/streaming state is never restored (ephemeral). Orphan-tool_call repair is the existing pairing layer.
7.Session identity + interrupted-resumedecided
Decision
A session id is a UUID minted at launch; the live session writes to its own file. Resuming a session continues appending to the same file (no fork by default — matches "continue this conversation"). If the last entry is a user turn with no assistant reply (a killed/hung turn — the user's exact case), surface it normally; the model just answers it on the next send (no synthetic "continue" injection — neutrix already has cancel-as-steer). UUID minting needs a seed; Math.random/uuid4 is fine at launch (not in the replayable loop).
