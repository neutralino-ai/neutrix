Split points (skeleton open questions, now decided)

1.Where the summary lives in the payloaddecided
Decision
role:user wrapped in <system-summary>…</system-summary> (locked proposal). Avoids the "two system messages" ambiguity providers handle differently; consistent with the v0.8.0 <system-reminder> / v0.9.6 <system-compact> shapes, and reuses the existing marker-render + recall-exclusion machinery.
2.Tool round spanning the cut pointdecided
Decision
Cut only between rounds. Reuse v0.9.6 compact_messages's forward-snap so the kept tail never begins on an orphan role:tool (which the pairing layer does NOT repair — it only synthesizes orphan tool_use results). The summary segment ends on a clean boundary.
3.Summary-call failure modedecided
Decision
Leave messages unchanged + visible notice. If the summarize call raises or returns empty, smart_compact returns the original list with did_compact=False; the caller surfaces a dim error notice. The user can retry. The auto-trigger won't loop (it only fires above threshold; a failed summary leaves it above threshold, so guard against immediate re-fire — see #6).
4.Threshold: per-slot via max_context_tokensdecided
Decision
Per-slot. Add max_context_tokens: int | None = None to Slot. None ⇒ auto-compaction disabled for that slot (manual /compact still works). Auto-fires when estimate_tokens(messages) > 0.75 * max_context_tokens. Token estimate is the cheap len(json.dumps(messages))//4 heuristic (revisit only if it drifts).
5./compact upgraded — smart, with mechanical fallbackdecided
Decision
Smart when possible, mechanical when over-limit. /compact and the auto-trigger run summary-based compaction. But the summarize call itself sends the segment — impossible when already over the limit. So the under-limit token-budget cut (hardening #1) runs FIRST to get the payload safely sendable, then the summary runs on the (now-bounded) segment. v0.9.6's mechanical drop is the primitive the smart version stands on.
6.Re-compaction + auto-trigger re-fire guarddecided
Decision
Allow re-compaction; guard the loop. A summary is just a message; a later compaction may fold an older summary into a new one (single-level by definition). The auto-trigger fires at most once per turn boundary and only if the last compaction actually reduced tokens — a no-op/failed compaction does not immediately re-fire.

The >1M hardening (from the v0105-compaction-scope memory)

7.Three pieces v0.9.6 deliberately deferreddecided
Why here
Message-count halving doesn't guarantee getting under an arbitrary token limit (a huge recent message survives), and summary-based compaction can't run when already over-limit (the summarize call would itself exceed it). So the smart version needs these primitives.
Decision
#1 Token-budget cut — compact_to_token_budget(messages, budget): drop oldest (round-safe) until estimate_tokens < budget. The under-the-limit guarantee + the pre-summary primitive (#5).
#2 Reactive PTL retry — catch the provider's prompt-too-long error in _call_llm, run the token-budget cut, retry once (CC's truncateHeadForPTLRetry). One retry, then surface the error.
#3 Content-truncation — truncate_large_tool_results(messages, cap): truncate within an oversized role:tool body (CC's microCompact), the single-huge-message case message-dropping can't fix.
