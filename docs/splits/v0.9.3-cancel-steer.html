v0.9.3 splits — cancel as steer + ContextManager

Switch the cancel cleanup from rollback (v0.9.2) to append interruption signal; reshape the v0.9.2 Controller into a ContextManager state machine that owns the conversation loop.

Per .claude/rules/prd-compare-with-claude.md, this file is the durable record of the split-point discussion that happened before the PRD was drafted. CC source under /datafs/users/dingxf/anthropic/cc2/src. All splits below are decided; the PRD's ## Claude Code reference section summarizes them.

Architectural rule (user-directed)

★ ContextManager owns messages; only it mutates ChatStore user-directed
Rule
UI (TerminalChat / key bindings) emits events to the ContextManager. CM is the single layer that decides state, orchestrates LLM/Executor, and mutates messages / ChatStore. Executor's job narrows to subprocess + tool dispatch I/O. LLM's job narrows to API I/O + outgoing-payload validity (pairing layer). UI renders from ChatStore.changes() (already exists since v0.7); no UI-side store mutations.
Why
User: "don't mix responsibilities. ui don't manipulate the chatstore. only controller did. ... contoller => context manager, right? not agent loop. its respnsibilty is for building the messge sequence."

Split points

1. Marker wording & variant count decided
CC behavior
Two literal strings keyed off toolUse flag — cc2/src/utils/messages.ts:207-209, 545; callers at query.ts:1047, 1502.
Decision
Alternative — single unified marker '[interrupted by user]'. User: "use 'interrupted by user'". The orphan tool_result fill encodes "tool was active", so the user-message variant duplicates context.
2. Where orphan tool_results get filled in decided
CC behavior
Separate API-send-time pairing layer: ensureToolResultPairing() at cc2/src/utils/messages.ts:5133, called from claude.ts:1298. Forward pass inserts synthetic tool_result for any orphan tool_use; reverse pass strips orphan tool_results.
Decision
Follow CC — LLM-layer pairing in llm.py. Pure function on the outgoing payload: dedup by tool_call_id (first occurrence wins) + synthesize tool_result for any orphan tool_use. Does not mutate messages; operates on a copy of the outgoing payload only — preserves the ContextManager-as-sole-mutator rule. User: "tool_use must match tool_result and nothing in between. this is the key. but this is llm restriction. so llm take responsibility."
3. Synthetic tool_result content decided
CC behavior
Two literals depending on the path:
  • Defensive pairing: '[Tool result missing due to internal error]', is_error: true (messages.ts:246-247).
  • Intentional tool stop: CANCEL_MESSAGE ("The user doesn't want to take this action right now. STOP what you are doing and wait for the user to tell you how to proceed."), is_error: true (messages.ts:622-631).
OpenAI's tool role has no is_error; we drop that field.
Decision
Alternative — conditional content in the pairing layer:
  • If the message list contains a [interrupted by user] user message after the orphan → synthetic content = '[cancelled by user]'.
  • Otherwise → synthetic content = '[tool result missing]'.
User: "synthetic tool_result (missing) or tool_result (cancled by user) depending on if there is a 'interupted' message."
4. Skip marker when a queued user message follows decided
CC behavior
Skips marker when signal.reason === 'interrupt' (submit-interrupt). cc2/src/query.ts:1044-1051, 1499-1505.
Decision
Alternative — always append. User: "the cancel is an event. it's sent to controller. controller make decisions. now it's dumb: if two cancel in a row, ignore next." v0.9.3's ContextManager is explicitly dumb; no submit-while-busy-auto-cancels flow exists. Future: if added, CM can be taught to skip then.
5. Cancel routing: event-to-ContextManager user-directed
CC behavior
createUserInterruptionMessage yielded through the query generator (query.ts:1047, 1502). Single mutation path.
Decision
User-directed different — cancel is an event emitted by the UI to the ContextManager. CM is the single decision-maker and mutator. CM broadcasts to LLM.stop() + Executor.cancel(), then appends the marker.
6. Partial assistant text preservation + streaming user-directed
CC behavior
Streaming on; partial assistant content stays in history when cancel fires; partial tool_use blocks repaired by the pairing layer at next send.
Decision
Follow CC for the semantic; for v0.9.3, disable streaming. User: "follow CC. but for now disable stream. let's enable stream later." OpenAIChatLLM.stream_response reverts to stream=False (rolls back v0.9.2's switch). LLM.stop() cancels the awaiting HTTP request via task-cancellation. No partial text exists at cancel time; partial-text preservation re-enters scope when streaming re-enables in a later PRD.
7. View-side dim "interrupted" notice decided
CC behavior
No separate UI notice — the rendered [Request interrupted by user] message in the transcript IS the affordance.
Decision
Follow CC — drop the v0.9.2 dim view-side notice. The [interrupted by user] message renders as a regular user turn.
8. Idempotency of repeated cancel user-directed
CC behavior
Fire-once via AbortController architecture.
Decision
User-directed different — ContextManager handles dedup. "controller handle that. ui just provide input to controller." + "if two cancel in a row, ignore next." CM rejects a second cancel event while already in CANCELLING.
9. Fate of Agent.rollback_to decided
CC behavior
No analog.
Decision
Alternative — remove (YAGNI). Re-add cleanly if /undo materializes. Together with the Agent.stream_reply dismantling (see split #10), the entire Agent class folds into ContextManager.
10. Scope of ContextManager refactor user-directed
Context
The ContextManager-as-state-machine design dismantles Agent.stream_reply (the v0.9.2 async generator that drove the LLM/tool loop). Without the generator, AgentEvents and TerminalChat._mirror_new_agent_messages no longer have a producer. The UI rendering path must switch to subscribing to ChatStore.changes() (already exists since v0.7). So the "broader scope" is forced by the design, not optional.
Decision
User-directed different — bundle the broader refactor into v0.9.3. User: "bundle it." Concretely:
  • v0.9.2's Controller reshapes into ContextManager (substantial role expansion, not a rename).
  • Agent.stream_reply is dismantled; the loop becomes the CM state machine.
  • Agent class is dissolved into CM (messages, system-reminder injection, LLM/tool wiring all move to CM).
  • TerminalChat._mirror_new_agent_messages is removed; UI subscribes to ChatStore.changes() for rendering.
  • _send_message shrinks to: await ctx.handle_event(UserMessageEvent(text)).
11. Executor → ContextManager event surface decided
CC behavior
Each Tool is an async function* that yields progress events during execution, then a final result (cc2/src/query.ts / Tool.ts). CC needs progress for live output ("Bash running for 5s, output so far: …").
Decision
Alternative — minimal event surface for v0.9.3: tool_started(tool_call_id, tool_name, args) and tool_finished(tool_call_id, tool_name, content, ok). No progress events. No internal Executor state machine. Earned by the user: "for now, keep it simple. started / finished." Re-add progress when streaming re-enables (post-v0.9.3).
12. Fate of queued user messages on Esc cancel mid-impl
Context
Surfaced mid-implementation: the v0.9.3 worker_loop drains asyncio.Queue and dispatches each queued message as the next UserMessageEvent. So pressing Esc while a message is queued (typed during the busy turn) results in the queued message being sent after the [interrupted by user] marker is appended.
CC behavior
Bifurcated path (cc2/src/hooks/useCancelRequest.ts:87-122):
  • Path A — Esc while idle (queue exists but no LLM in flight): pops queued commands BACK into the editable input buffer via handleQueuedCommandOnCancel() (REPL.tsx:2166-2184). User re-edits.
  • Path B — Esc during active LLM response: aborts the LLM (signal.reason='user-cancel'), leaves the queue intact; queued messages get sent after the cancel marker.
Decision
Follow CC's Path B for both cases — keep the v0.9.3 current behavior (send queued messages after the marker). User: chose "Send queued messages after marker (current)". Optimizes for "I cancelled to steer with what I already typed"; the typed message becomes the steering prompt and the LLM sees the prior tool intent + marker + steering input in one go. CC's Path A (pop back to input draft on idle-cancel) requires injecting text into the prompt_toolkit DraftReader buffer the input loop owns — non-trivial and not earned-by v0.9.3's scope. If users find the auto-send surprising, the Path-A wiring can land in a later release.
13. ContextManager class name user-directed
Decision
User-directed different — class name ContextManager, file src/neutrix/context_manager.py, variable ctx (or ctx_mgr where two ctx's would collide). User: "use ContextManager." Minor Python-naming overload with the __enter__/__exit__ protocol is accepted as a tolerable cost for clarity of the term.
