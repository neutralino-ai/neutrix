v1.5.1 splits — interactive prompt can't hang the turn

Fix a v1.4.8 regression: an unanswered permission/AskUserQuestion prompt parked the turn forever. The agent must never hang on a prompt.

Regression (diagnosed live, advisor-confirmed): v1.4.8 turned permission-ask from block-with-notice (turn continues) into an interactive prompt that parks the turn awaiting _pending_answer with NO timeout/escape. The user's routine rm -rf build / git push --force trip auto-mode's danger heuristic → prompt → if unanswered (subtle indicator), the turn hangs forever (no network, no watchdog — it isn't in AWAITING_LLM). User: "the agent should handle it, not the user."

Split points

1.Prompt timeout → safe fallback (never park forever)decided
Decision
Bound the wait. _ask_user awaits asyncio.wait_for(self._pending_answer, PROMPT_TIMEOUT_S). On TimeoutError it returns a None answer (clears pending in finally). The Executor's existing answer is None branch already means "no interactive consumer" → permission-ask falls back to block-with-notice (deny, the turn CONTINUES — pre-v1.4.8 behavior), and AskUserQuestion returns the not-available result. So an unanswered prompt degrades to the safe non-hanging path — the agent handles it. PROMPT_TIMEOUT_S generous (default 300s; you have time to answer, but a walked-away session can't zombie).
2.Unmissable indicator + Esc (a required answer isn't an "idle hint")decided
Decision
The pending-prompt line in _above_input becomes prominent (bright/colored, with the call to action): ▶ answer needed: <header> — type a number, or Esc to skip — not the dim "answering:" that was missed. This is a required interaction, not an idle affordance, so it's exempt from the "no ugly hints" preference. Esc already cancels (cancel → drive-task cancel → CancelledError at the await); make sure it surfaces as a clean turn-continue, and document it in the line.
3.Cross-task safety of the timeoutdecided
Decision
On wait_for timeout the Future is cancelled (done) — the input loop's if pending is not None and not pending.done() guard already skips a done Future, and _ask_user's finally nulls _pending_question/_pending_answer, so a late keystroke is treated as a normal message. The multi-question loop re-arms a fresh Future per question; a timeout aborts the whole spec (returns None). No new state; reuses the v1.4.8 channel.
4.Should auto mode prompt at all for these?decided
Decision
Keep prompting (user wanted the v1.4.8 interactive upgrade), but it's now SAFE because of split #1 (can't hang). The danger heuristic still flags rm -rf/--force → a prompt; answered → proceed/deny; unanswered → safe block-with-notice. NOT reverting to silent block (the user asked for the interactive prompt). The fix is making the prompt non-fatal, not removing it.
