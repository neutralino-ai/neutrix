CC survey — what Claude Code does (one block)

★CC rewind, summarizedsurvey
Up recall
Up-arrow recalls prior user prompts from a project history file, only on an empty buffer (or once nav is active); the pre-recall draft is saved and restored on Down; Esc resets the nav index and clears the hint. Recall never mutates conversation state — it only fills the input buffer until submit. hooks/useArrowKeyHistory.tsx:124-207.
Rewind
Destructive truncation: setMessages(prev.slice(0, messageIndex)) drops the suffix — no checkpoint, no retained-hidden list — then mints a fresh conversationId. screens/REPL.tsx:3661-3707.
/rewind
Opens an interactive message selector; unit is one user message; tool rounds drop atomically (adjacent array entries). commands/rewind/rewind.ts.
Esc
Esc is cancel-while-busy, not a rewind key; there is no double-Esc rewind menu. Rewind-on-interrupt is automatic, not a binding. hooks/useCancelRequest.ts:87-167.
Render
Rewound turns are removed from view — React re-renders a shorter virtual list (VirtualMessageList). Impossible in neutrix's append-only renderer, which cannot un-print or restyle past scrollback (every print_* is a one-way console.print). screens/REPL.tsx:4381, Messages.tsx.

Scope change forced by the renderer

!The skeleton's [rewound] strikethrough is infeasibleneeds user
Skeleton said
Goal #4: rewound turns stay in the transcript with a dim [rewound] strikethrough; "reuse the folding pattern."
Reality
neutrix prints to scrollback append-only; it cannot reach back and restyle or remove already-printed turns. CC only achieves this by re-rendering a virtual list, which neutrix has no analog for. So the retroactive marker cannot be drawn on the existing lines.
Recommendation
Replace it with a forward notice printed at the cursor on rewind — e.g. ↶ rewound 2 turns (back to turn 3) — exactly the pattern v0.9.6 /compact already uses ("compacted N turns"). The dropped turns remain visible above in scrollback as historical record; the notice demarcates the new branch point. This is a scope change from Goal #4 (workflow step 10) — needs an explicit OK.
Decision
— open (surfaced in chat) —

1.History shape — destructive trim vs non-destructive checkpointneeds user
CC behavior
Destructive trim (messages.slice(0, i)), fresh conversationId. REPL.tsx:3671.
Alternatives
  • Destructive trim — drop the suffix from cm.messages (Follow CC).
  • Non-destructive checkpoint — keep dropped turns in a retained list + a pointer, restorable later.
Recommendation
Follow CC — destructive trim. CC's rewind is destructive (distinct from CC's cancellation, which used an append-marker — the v0.9.2/v0.9.3 incident this rule cites). Multi-branch retention is an explicit non-goal, and neutrix's compact() / _handle_replace_history already trim-and-rebuild. Flagged because history-shape is the exact class the rule says not to pre-decide alone — confirm.
Decision
— open (recommend Follow CC) —
2.Recall ↔ rewind coupling — does submitting a recalled turn rewind?needs user
CC behavior
Decoupled. Recall fills the buffer from input history; submitting just appends a new turn. Rewind is a separate action (the /rewind selector). Recalling + submitting does NOT truncate.
Alternatives
  • Coupled (skeleton Goal #2): submit a recalled/edited prior turn → rewind cm.messages to that turn, then append. One gesture to "redo from here."
  • Decoupled (CC): Up-arrow recall only fills the buffer; submit always appends a new turn; rewind only via explicit /rewind N.
Recommendation
Lean decoupled (Follow CC) for v0.9.7: coupling makes a normal-looking submit silently destroy turns 3+ (surprising, and destructive-on-Enter). Keeping recall = "convenience paste" and rewind = explicit /rewind N is safer and simpler. But the skeleton explicitly wanted the coupling, so this is a genuine product fork — your call.
Decision
— open (recommend decoupled) —

8.Esc during recall vs cancelneeds user
CC behavior
Esc = cancel-while-busy; during nav it resets the history index. No rewind on Esc.
neutrix today
Esc (escape, eager) is the cancel hook + part of quit-arming. When IDLE it is effectively a no-op (nothing to cancel).
Alternatives
  • Esc clears recall + buffer when IDLE with recalled text; stays cancel when busy (state-overloaded).
  • Leave Esc as pure cancel; use a distinct key (e.g. c-g) to exit recall.
Recommendation
Esc clears recall when IDLE (matches CC + the skeleton), falling through to cancel when busy. Needs careful filter ordering against the existing eager Esc binding — flagged so the overloading is a conscious choice.
Decision
— open (recommend Esc-clears-when-idle) —
