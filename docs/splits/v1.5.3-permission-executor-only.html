v1.5.3 splits — permission is an Executor-only safety layer

Remove the interactive permission prompt that parks the turn. The Executor is a deterministic safety layer: a dangerous action is denied directly (returns a "denied" tool_result) — never asked. The ContextManager and Advisor never see permission.

Why now (urgent): v1.4.8 turned the permission ask verdict into an interactive prompt routed up to the CM — every auto-flagged destructive Bash (rm -rf, git reset --hard, force-push…) parked the turn awaiting a human (bounded to 300 s by v1.5.1, still blocking). User-directed: "drop the permission part… permission should be in executor part. controller, advisor does not need to know it"; and "if it detect dangerous action, deny directly. return tool_result denied… don't ask user question. let's be like industrial agent."
CC survey: CC's main loop (queryLoop, query.ts:241) is permission-agnostic (zero permission imports); canUseTool runs in the tool-execution layer (toolHooks.ts:361). neutrix Follows CC on the agnostic loop; it User-directed-diverges on the prompt — CC asks interactively, neutrix denies directly (industrial: no human in the loop).

Split points

1.Where the permission decision livesdecided
CC
canUseTool is called from the tool-execution layer (toolHooks.ts:361), never from queryLoop. The loop is permission-agnostic.
neutrix now
decide() already runs inside Executor.dispatch_all (executor.py:173) — good — but the ask branch then yields needs_user_input, which the CM drives (context_manager.py:864-868), dragging permission into the loop.
Decision
Follow CC + User-directed. Permission is decided AND fully resolved inside the Executor. The CM holds no permission semantics; decide()/policy/mode stay on the Executor where they already live.
2.The interactive ask prompt — remove it entirelydecided
CC
ask shows an approval dialog; the tool's execution awaits the human's answer (resolved Promise), then runs or refuses.
neutrix now
ask → yield _needs_input(permission_question(...)) (executor.py:178) → CM awaits ask_user up to PROMPT_TIMEOUT_S = 300 s. This park is the bug.
Decision
User-directed different — delete the ask verdict and its prompt. "don't ask user question. let's be like industrial agent." The entire ask branch in dispatch_all (executor.py:177-191) is removed; decide() returns only allow/deny. No permission path ever yields needs_user_input, so the loop never parks on a permission verdict.
3.What auto does with a detected-dangerous actiondecided
The fork
Once it no longer prompts: (a) run routine, block only catastrophic; (b) run everything; (c) block-with-notice "pending approval"; (d) deny directly.
Decision
User-directed different — (d) deny directly. "if it detect dangerous action, deny directly. return tool_result denied." The danger heuristic (and any settings ask rule) resolves to deny: the Executor returns a role:tool "[denied …]" result and the round continues — the LLM sees the denial and adapts. Deterministic safety, no human gate, no hang. Trade accepted: a routine rm -rf build is denied (not run) under auto; /allow (allow-all) or an explicit .claude/settings.json allow rule is the deliberate escape hatch. Explicit allow still overrides the heuristic; allow-all still bypasses everything.
4.The CM ask_user port / needs_user_input channeldecided + flag
Decision

The v1.4.8 needs_user_input handler + ask_user port stay, but after this release only the agent-initiated AskUserQuestion tool uses them (a question the agent chooses to ask, not an automatic gate). If the industrial-agent intent is that neutrix should never ask the user anything, retiring that tool is a clean separate follow-up (v1.5.4) — say the word and it's a one-PRD change.
