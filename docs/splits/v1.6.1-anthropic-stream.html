v1.6.1 splits — anthropic/* streaming + empty-turn history repair

A live-bug patch with TWO linked causes. Bug #1: switching the slot to anthropic/claude-opus-4-7 produced no output — the IHEP gateway streams the Anthropic Messages SSE protocol for anthropic/* models; neutrix's inbound parser only understood OpenAI chunks, so every byte was dropped, leaving content=None empty turns. Bug #2 (caused by #1, but independent): a content:null-no-tool_calls assistant turn anywhere in history silently makes the next request return EMPTY on strict backends (openai/gpt-5.5, the anthropic gateway) — deepseek/* tolerates it — which is why gpt-5.5 "also stopped working" once the opus empties polluted the session.

Decisions made by the agent under the user's "fix it" directive (2026-05-31), not a turn-by-turn split discussion — the diagnosis was confirmed empirically against the live gateway and the core approach is forced (parse the protocol the gateway actually sends). CC survey: CC consumes the Anthropic Messages stream natively — updateUsage() reads cumulative usage off message_start/message_delta (services/api/claude.ts:2916-2987); text arrives via content_block_delta. neutrix mirrors those semantics but reads the events off the AsyncOpenAI-parsed chunk's model_extra (the SDK transports the SSE and stashes unknown fields there) rather than adding a native Anthropic client. Empirical: raw SSE shows event: content_block_delta / data:{"delta":{"type":"text_delta","text":"…"}}; the SDK yields choices_len=0 with the event in chunk.model_extra; a pure-OpenAI request (system as a message, no extra_body) still returns Anthropic SSE.

Split points

1.Where/how to fix the inbound parsedecided
Options
(A) read the Anthropic event off the SDK chunk's model_extra in the existing loop; (B) bypass the OpenAI SDK and parse raw httpx SSE for the anthropic path; (C) add the anthropic SDK as a second client; (D) coax the gateway into OpenAI-shaped output.
Decision
Alternative — (A) parse off chunk.model_extra. Empirically the SDK already transports the Anthropic SSE and the full event survives in chunk.model_extra, so (A) needs no new dependency, no second transport, and leaves cancellation (stop() closing _active_stream) and the timeout watchdog unchanged. (D) is ruled out empirically — the gateway routes by model prefix and returns Anthropic SSE even for a pure-OpenAI request. (B)/(C) are heavier and reintroduce a parallel cancel/timeout surface for no gain. Keeps the openai-SDK-only decision intact ([[project-v2-v3-direction]]).
2.Shape detection — per-chunk vs config-predicatedecided
CC
N/A (CC talks one protocol).
Options
(a) per-chunk: if a usable OpenAI choices[0].delta is present use it, else if chunk.model_extra["type"] is an Anthropic event parse that; (b) gate on the existing _uses_anthropic_messages_gateway() predicate (provider ihep + model anthropic/).
Decision
(a) per-chunk shape detection. Robust: a pure-OpenAI backend never carries an Anthropic model_extra["type"], and a misconfigured/renamed provider can't reintroduce the dropped-output bug. The predicate still gates the outbound reshape (unchanged); inbound keys off what actually arrived.
3.Tool-use accumulationdecided
Anthropic shape
content_block_start{content_block:{type:"tool_use",id,name}} then content_block_delta{delta:{type:"input_json_delta",partial_json}}, ended by content_block_stop.
Decision
Map into the EXISTING tool_calls_by_index accumulator ({index:{id,type:"function",function:{name,arguments}}}): tool_use start sets id+name; each input_json_delta.partial_json appends to function.arguments. Reuses _finalize_tool_calls (sorts by index). The CM's _extract_tool_calls already keys off the assembled tool_calls field — no CM change needed.
4.Text vs tool block disambiguationdecided
Decision
Track a per-stream block_kind[index] map set on each content_block_start (text vs tool_use), so a content_block_delta routes by its block's kind. A text_delta → token + content; an input_json_delta → tool arguments. Index 0 is normally text; tool blocks take later indices.
5.finish_reason mappingdecided
Decision
Map Anthropic stop_reason → OpenAI finish_reason from message_delta.delta.stop_reason: end_turn/stop_sequence/max_tokens → "stop", tool_use → "tool_calls". Cosmetic (the CM keys off the tool_calls field, not finish_reason) but keeps LLMResponse honest.
6.Token-usage capture — now or v1.7?decided
Decision
Alternative — DEFER to v1.7.0 (cost/timing). The Anthropic events carry usage (message_start.message.usage input, message_delta.usage output) and the SDK even maps some onto chunk.usage — but there is no consumer (no ledger, no /cost) until v1.7. YAGNI: v1.6.1 is a minimal unbreak (text + tool_use + finish_reason). v1.7's usage capture will read BOTH the OpenAI and Anthropic usage shapes; this fix's parser is the seam it extends.
7.Outbound extra_body.system hoisting — touch it?decided
Decision
Leave the outbound exactly as-is (out of scope). The gateway accepts system as a normal message too (probe D), so the hoist isn't strictly required — but it works today and a bug-fix patch must not perturb a working outbound. Any outbound simplification is a separate, non-urgent change.

Bug #2 — the empty-turn poison (surfaced mid-diagnosis)

8.Empty-turn (null-content, no-tool_calls assistant) repairdecided
Empirical
A {"role":"assistant","content":null} turn with no tool_calls anywhere in history → openai/gpt-5.5 and the anthropic gateway return an EMPTY reply (tokens=0, no error); deepseek/* tolerates it. (A null-content assistant WITH tool_calls is a normal text-free tool call — NOT the bug.) User-confirmed: clearing the opus empties made gpt-5.5 work again.
Options
(a) repair pre-translation (on the canonical OpenAI-shaped message); (b) prevent creation at commit time only; (c) drop the empty turn; (d) filter post-translation (on the literal sent payload).
Decision
User-directed ("if message sent has empty block, filter it, in ensure_sdk_compliant — just like tool_use-result pairing"). Realized as an _ensure_sdk_compliant umbrella that composes _ensure_tool_result_pairing + _repair_empty_assistants (null/blank, no-tool_calls assistant → [no response]) — one named outbound-compliance seam, both repairs inside it. Placeholder, not (c) drop — drop creates consecutive user turns strict backends reject; placeholder preserves alternation and is API-valid everywhere (all 3 repairs verified live; only placeholder is alternation-safe). Pre-translation (a), not (d) post-translation — empirically the poison is content:null specifically (the OpenAI-compat path passes it through → gpt-5.5 returns empty); content:"" is tolerated, and the anthropic path's _outbound_prompt already coerces null→"", so post-translation filtering would only catch tolerated "" blocks AND mislabel every anthropic tool-call turn (its tool_calls already dropped) as [no response]. Pre-translation sees tool_calls, so it repairs only genuine empties. Outbound (not commit-time (b)) so it HEALS the user's existing 603-msg session, not just prevents new empties; Fix #1 independently stops new empties at the source.
