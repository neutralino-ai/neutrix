v0.10.1 splits — re-enable streaming with cancel-as-steer

Restore the v0.9.2 stream=True path (removed in v0.9.3) with CC's "keep partial text" semantic, so an Esc mid-response preserves whatever bytes arrived — closing the cancel-as-steer arc.

Durable record of the split discussion (per prd-compare-with-claude.md). Authored and decided autonomously under a /goal that delegated all human options. CC source under /datafs/users/dingxf/anthropic/cc2/src; the prior neutrix implementation recovered from git tag v0.9.2.

Reference — what CC + v0.9.2 do

★Streaming + cancel, summarizedsurvey
CC
Streams via the SDK (services/api/claude.ts); accumulates content/tool_use deltas index-keyed; yields partial assistant messages as blocks complete. On abort: partial messages already yielded are kept; the stream is torn down eagerly (body?.cancel()).
neutrix v0.9.2
stream=True; stream_response yields LLMEvent("token", str) per content delta, accumulates tool_calls in an index-keyed dict, and yields ONE final LLMEvent("assistant", LLMResponse) carrying the full message. stop() closes the AsyncStream (eager teardown). No separate per-tool_call event.
What v0.9.3 added (must keep)
_ensure_tool_result_pairing (orphan tool_use repair) and _outbound_prompt (anthropic-gateway system split). These are NOT in v0.9.2 — the restore is a merge, not a revert.

Split points

1.Cancel mechanism — stream-close vs task-canceldecided
Options
  • v0.9.2: stop() closes the AsyncStream; the iterator exits.
  • v0.9.3: _do_cancel cancels the drive task; the await raises CancelledError.
  • Both.
Recommendation
Both — keep the tested v0.9.3 drive-task-cancel (it drives the unwind into _drive's except CancelledError), add stop()-closes-stream for eager network teardown.
Decision
Both. _do_cancel runs synchronously, so task.cancel() leaves a pending cancellation that raises CancelledError at the generator's next await — ahead of any StopAsyncIteration from the closed stream — landing in _drive's existing except CancelledError (caught before the generic except Exception). stop() closing the stream is just eager teardown so the HTTP body doesn't drain.
2.Where "keep partial text" lives — happy-path restructure vs cancel-time commitdecided
Options
  • Lane B: append a mutable assistant message to messages at first token, grow it per delta. Requires restructuring _drive's tested happy-path append (regression risk on the core loop) and only pays off if tokens render live.
  • Lane A: _call_llm stashes the partial text on self as tokens arrive; _do_cancel commits it (to messages + store) before the [interrupted by user] marker. Happy path untouched.
Recommendation
Lane A — near-zero regression on the shipped drive loop; the incremental-into-messages cost of Lane B only earns its keep if tokens render live (split #3), which we're not doing.
Decision
Lane A. _call_llm resets self._streaming_partial = "" each round and appends each "token" delta to it; the happy path still returns the final assistant message and _drive appends it once, unchanged. _do_cancel, before its marker, commits {"role":"assistant","content": self._streaming_partial} (if non-empty) via _append_assistant_message so it lands in both messages and the store — keep-partial needs no happy-path edit.
3.Live token rendering — does the user see tokens arrive?decided
CC behavior
Renders streamed blocks incrementally in a live region, commits on finish.
The constraint
neutrix's transcript is append-only scrollback (console.print, v0.6.8 standing non-goal to rework) — it can't grow a line in place. The _above_input live region could show a bounded preview, but it can't hold a long growing response.
Options
  • Bounded live preview (last N lines / a char counter) in _above_input; commit to scrollback on finish.
  • No live preview — the heartbeat is the liveness signal; the message commits to scrollback at finish (and on cancel, the partial commits).
Recommendation
No live preview in v0.10.1. The v0.9.4/v0.9.8 blink heartbeat is the liveness answer the user already converged on (the whole heartbeat saga); streaming's unique job here is the cancel arc (split #2). A bounded preview is a clean v1.x add once it earns its keep.
Decision
Alternative — defer live token rendering to v1.x (NOT "infeasible"). A bounded preview in the _above_input region is feasible but the bottom region can't hold a long growing response, so full inline live rendering needs append-only rework (out of scope). The heartbeat remains the during-LLM liveness signal. terminal_chat.py is therefore unchanged this release; the user-visible win is that an Esc mid-response now keeps the partial assistant text in the transcript (cancel-as-steer with prior intent), where before it kept nothing.
4.Event granularity — restore the speculative per-tool_call event?decided
Options
  • Restore v0.9.2 exactly: "token" per content delta; accumulate tool_calls internally; one final "assistant" event carrying them.
  • Add the skeleton's proposed LLMEvent("tool_call", ToolCall) per completed call.
Recommendation
Restore v0.9.2 exactly — it's the proven shape, and the only consumer (_call_llm) just needs the final assembled message. A per-tool_call event has no consumer (YAGNI).
Decision
Follow v0.9.2. Two event kinds: "token" (content delta, drives the partial stash) and "assistant" (terminal, full message + finish_reason). The skeleton's "tool_call" event and ToolCall dataclass are dropped — no consumer.
5.Partial text on a hard LLM error / dropped connectiondecided
Options
  • Keep the partial (whatever arrived) + append [LLM error: …] → two assistant messages for one round.
  • Discard the partial; the [LLM error] message is the outcome → one assistant message.
Recommendation
Discard on hard error (one message); keep on user-cancel + timeout (the user/clock interrupted a healthy stream — the bytes are meaningful). Errors are rare and the notice is the signal.
Decision
Discard-on-error, keep-on-cancel/timeout. _drive's except Exception clears self._streaming_partial before appending [LLM error] (one assistant message — landmine #2 resolved). User-cancel and timeout both route through _do_cancel, which commits the partial first — so a timed-out round yields [partial text][LLM timeout after Ns] (two messages, both informative). A provider drop mid-stream surfaces as the error path.
6.Open Q#1 — hide the heartbeat once streaming returns?decided
Decision
No change — keep the heartbeat. Since v0.10.1 renders no live tokens (split #3), the heartbeat is still the only during-LLM liveness signal; there is nothing redundant to hide. Revisit if/when a live preview lands.
