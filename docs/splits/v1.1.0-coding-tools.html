Split points — your call

1.read-before-edit enforcement + where read-state livesneeds user
CC behavior
Edit (and Write on an existing file) require a prior Read of that path in the session; CC tracks readFileState. tools/FileEditTool/prompt.ts:4-6.
The neutrix wrinkle
tools.dispatch() is a pure function with no session state. Read-state needs a home. The clean one: the Executor (already per-session, already forwarded to tools that declare it) gains a read_paths: set[str]; Read adds to it, Edit/Write check it.
Alternatives
  • Enforce (Follow CC): Edit/Write on an unread file returns an ERROR: read the file first. Prevents blind edits.
  • Soft: warn but allow.
  • None: stateless, no check (simplest, riskiest).
Recommendation
Enforce, read-state on the Executor. Matches CC; cheap; the executor is the right session-scoped home (and subagents get their own executor → own read-state, correctly isolated).
Decision
— open (recommend Enforce via Executor) —
2.Grep backing — ripgrep binary vs pure Pythonneeds user
CC behavior
Grep shells out to rg (ripgrep): regex, glob/type filters, output_mode (content/files_with_matches/count), -A/-B/-C, -n/-i, multiline. tools/GrepTool/GrepTool.ts:33-89.
Alternatives
  • Require rg: shell out; fast, full feature set; hard dependency on the binary.
  • rg-if-present, Python fallback: use rg when on PATH, else a Python os.walk+re implementation (subset of flags).
  • Pure Python: no dependency, slower on big trees, reimplements flags.
Recommendation
rg-if-present with a Python fallback. Best of both: full speed/flags where rg exists (it's near-ubiquitous on dev boxes), still works without it. The fallback supports the common subset (pattern, glob, -i, -n, content/files modes).
Decision
— open (recommend rg-if-present + Python fallback) —
3.Bash — banned commands (push to dedicated tools)?needs user
CC behavior
Bash discourages/blocks sed, awk, cat, head, tail, grep, find, echo — the model must use Read/Grep/Glob/Edit instead. tools/BashTool/prompt.ts.
Alternatives
  • Description-only: tell the model to prefer the dedicated tools; don't hard-block (it can still run them if needed).
  • Hard-block: refuse those commands with an error pointing at the tool.
Recommendation
Description-only for v1.1.0. The guidance shapes behavior; a hard block is a permission concern that belongs in v1.4.0 (permissions/plan-mode), not the tool layer. Keep the existing tree-kill cancellation + timeout.
Decision
— open (recommend description-only; hard-block → v1.4.0) —
4.Read — format + what's in scope for v1.1.0needs user
CC behavior
cat -n line-numbered output; offset/limit (default cap ~2000 lines); images/PDF/notebooks via multimodal. tools/FileReadTool/FileReadTool.ts:206-300.
Recommendation
Follow CC for text: line-numbered output + offset/limit + a default line cap (keeps huge files from blowing the round — feeds nicely into v0.10.5 compaction). Defer images/PDF/notebooks to a v2.x stretch (multimodal is a separate surface).
Decision
— open (recommend text + ranges; multimodal deferred) —
5.Edit — uniqueness + diff displayneeds user
CC behavior
old_string must match exactly + be unique (else replace_all); new_string must differ; returns a structured patch and renders a diff. tools/FileEditTool/types.ts:6-19.
Recommendation
Follow CC semantics (exact unique match, replace_all, must-differ). For the append-only transcript: render a compact one-line edit notice (± file.py: −N/+M lines) rather than a full inline diff; a full diff is reachable later via the same expand-by-append idiom (v0.10.2). No live in-place diff (append-only).
Decision
— open (recommend Follow CC + one-line edit notice) —
6.Glob backing + dispatch-architecture knock-onneeds user
CC behavior
Glob: pattern + optional path, returns up to ~100 files sorted by mtime. tools/GlobTool/GlobTool.ts:26-50.
Recommendation
Pure Python (pathlib/glob + mtime sort + 100-cap) — no dependency. Knock-on: tools stay sync string-returning funcs in the existing registry; the only new plumbing is the Executor read-state (split #1). No change to the async dispatch path.
Decision
— open (recommend pure-Python Glob) —
7.Migration — what breaks when the four are retiredneeds user
Impact
The LLM-facing tool names change (read_file→Read, etc.). Saved transcripts with old tool_calls still render (the renderer is name-agnostic) but the model won't see the old tools. The subagent allowlist (subagent_tool_names()) and any tests referencing the old names update.
Recommendation
Clean replace (no aliases). neutrix has no external API consumers; the tool surface is internal-to-the-LLM, so renaming isn't a public break. Update the subagent allowlist + tests. Treat as a minor (v1.1.0), not a major — the v1.0 "stable surface" was user-facing behavior, not the LLM tool names.
Decision
— open (recommend clean replace, v1.1.0 minor) —
