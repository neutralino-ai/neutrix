v1.4.9 splits — turn resilience (no-progress timeout)

A slow-but-live LLM stream must never be killed mid-flight; a dead proxy connection must error out, not hang forever.

Triggered by a LIVE debug session: the user's neutrix hung. py-spy showed an idle event loop; ss showed two sockets to the local proxy 127.0.0.1:7890 in CLOSE-WAIT with unread bytes — a dead LLM connection the turn never recovered from. Root cause: the watchdog was a HARD wall from round start (context_manager.py:732, single sleep(llm_timeout_s)) and last_progress_at was never bumped per token (it was set once at round start). CC survey: services/api/claude.ts (STREAM_IDLE_TIMEOUT_MS=90s, reset per chunk; API_TIMEOUT_MS=600s initial; 10× retry + non-streaming fallback).

Split points

1.Hard wall vs inactivity (no-progress) timeoutdecided
CC
Inactivity-based: resetStreamIdleTimer() on every streamed chunk; aborts only after STREAM_IDLE_TIMEOUT_MS (90s) of silence — NOT a wall from request start (claude.ts:1877,1940-1941). A live stream is never killed even past 10 minutes.
Decision
Follow CC — inactivity. Bump self.last_progress_at = time.monotonic() on every token event in _call_llm, and rewrite _llm_timeout_watchdog from one sleep(llm_timeout_s) into a poll loop that cancels only when monotonic()-last_progress_at > llm_timeout_s. A steadily-streaming deepseek-v4-pro response (minutes) is never cancelled; a genuinely silent connection is caught after one quiet budget.
2.Transport timeout AND app watchdog (belt + suspenders)decided
CC
Hand-rolls the idle watchdog because the Node SDK's timeout only covers the initial fetch, NOT the streaming body iteration (claude.ts comment + client.ts:144).
Decision
Both — and httpx gives the transport half for free. Set AsyncOpenAI(timeout=httpx.Timeout(llm_timeout_s, connect=10.0)) (was: no timeout → relied on the 600s SDK default). httpx's read timeout applies to EACH streaming read and resets per chunk — i.e. it IS a transport-level inactivity cap (better than CC's Node situation, where the body wasn't covered). So a dead/half-closed socket raises ReadTimeout → APITimeoutError → a visible [LLM error] even if the app watchdog's cancel can't break the read. The app-level no-progress watchdog (split #1) is the suspenders + drives the UI stall hint.
3.One knob vs CC's two timeoutsdecided
CC
Separate STREAM_IDLE_TIMEOUT_MS (90s) and API_TIMEOUT_MS (600s initial).
Decision
Alternative — one knob. slot.llm_timeout_s (default 300s) is the single inactivity threshold for BOTH the httpx read timeout and the app watchdog; connect is a fixed short 10s. Rationale: the user's calibration rule ("one knob + derived thresholds, calibrated against the slowest model"). 300s of silence is deliberately more lenient than CC's 90s because deepseek-v4-pro's time-to-first-token (model thinking before any byte) can be minutes via the IHEP proxy.
4.Retry / non-streaming fallback on timeoutdecided
CC
On idle-fire: fall back to non-streaming + retry up to 10× with exp backoff (500ms→32s); also 3× on 529 (withRetry.ts).
Decision
Alternative — YAGNI for now. On timeout the turn ends with a visible [LLM timeout after Ns] marker and the user re-prompts (cancel-as-steer already exists). Keep max_retries=2 on the SDK for the INITIAL request only (transient connect/5xx). Auto non-streaming fallback + N-retry is a future enhancement, not this fix.
5.Timeout marker wording + watchdog poll intervaldecided
Decision
Keep the marker [LLM timeout after Ns] (now means "no progress for Ns") — smaller blast radius, existing assertions hold. Watchdog polls at min(llm_timeout_s, 2.0) so a 0.1s test timeout still fires inside its wait window while a 300s prod timeout only wakes every ~2s; a per-iteration state != AWAITING_LLM guard retires it the instant the round ends, and last_progress_at is None mid-poll is treated as "wrapping up" (no crash).
v1.4.9 turn resilience · root-caused live via py-spy + ss (CLOSE-WAIT on the local proxy) · inactivity watchdog (per-token bump + poll loop) + explicit httpx timeout · one knob (llm_timeout_s) · marker wording kept · matches CC's reset-per-chunk semantics, gets the transport half free from httpx.
