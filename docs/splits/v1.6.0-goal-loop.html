v1.6.0 splits — native /goal autonomous loop

Set a goal once; the agent keeps working it across turns without re-prompting, until it declares done or a step cap hits. The practical-autonomy headline for unattended OMILREC runs.

CC survey: CC ends a turn when the model emits no more tool_use (query.ts:1062); a Stop hook can block turn-end and re-inject a continue-reason as a message (stopHooks.ts:268-280, re-entry query.ts:1278-1306); a token-budget "continue" nudge is the closest built-in auto-continue (query.ts:1308-1355); bounded by maxTurns (query.ts:1704-1712); Esc aborts via the abort signal. neutrix has no hooks (cut from v2), so /goal must self-contain the continue + done-detection. It reuses neutrix's own pieces: the _worker_loop turn boundary, the v0.9.3 cancel-as-steer, and the v0.8 system-reminder injection.

Split points

1.Where the loop livesdecided
CC
In the query loop itself (the Stop hook fires at turn-end inside the generator).
Decision
Alternative — in the chat _worker_loop, not the ContextManager. The CM stays a pure single-turn driver (run one turn → IDLE); the worker owns the "wait for input vs auto-continue" decision. After handle_event returns, if a goal is active + not done + under the cap, the worker enqueues the next continuation instead of blocking on input. Keeps the CM's single-mutator + single-turn contract intact (the v3 headless loop reuses the CM, not this chat-side policy).
2.How "done" is detecteddecided
CC
A user-defined Stop hook decides; no built-in self-check.
Options
(a) sentinel — the model ends its reply with a marker (GOAL DONE); (b) a dedicated CompleteGoal tool; (c) reuse the Advisor to judge completion each turn; (d) no detect (run to the cap).
Decision
Alternative — (a) a distinctive sentinel + the step cap as the hard guarantee. Each continuation prompt instructs: keep working, and when the goal is fully achieved end your reply with a final line <<GOAL_COMPLETE>> (a distinctive token, not prose). The loop checks ONLY the committed assistant text (never tool results) and only its FINAL non-empty line — so an instruction-echo, or the token inside a code block / quote, can't false-trigger. The 25-step cap (#4), not the sentinel, is the hard termination guarantee — the sentinel is the soft early-exit. No new tool, no per-turn extra LLM call (rules out (b) a tool and (c) the Advisor, which judges the task list not goal-exit and would double LLM cost every turn; a CompleteGoal tool is a clean later hardening if the sentinel proves flaky).
3.The continuation injectiondecided
CC
Injects the hook's reason / token-budget nudge as a message, then loops.
Decision
Follow CC (neutrix-shaped) — a labelled <system-reminder> turn, not a fake user message. Reuse the v0.8 system-reminder injection: between turns the worker drives a CM turn seeded with <system-reminder>Continue toward the active goal: <goal>. When fully done, end your reply with a final line <<GOAL_COMPLETE>>.</system-reminder>. It renders with the reminder prefix (visibility-parity holds — it's not a typed user turn), and the model sees it as the latest input and continues. (Mechanism added: the CM gains a "run a turn from an injected reminder, no new user text" entry, mirroring inject_advisor_message — single-mutator preserved.)
4.Step cap + graceful stopdecided
CC
maxTurns → yields max_turns_reached and returns.
Decision
Follow CC. A goal runs at most GOAL_MAX_STEPS auto-continuations (default 25, matching the subagent cap). On hit: stop the loop, clear the goal, print a notice (↯ goal paused after N steps — say "/goal <text>" to resume or refine). Not a crash; the conversation stays usable.
5.Interrupt + user-steer during the loopdecided
Decision
Esc cancels the current turn AND clears the goal (an explicit human stop ends the autonomous run — the v0.9.3 cancel path, plus clear the goal so it doesn't immediately resume). A typed message during the loop steers: it's the user taking back control — the goal is cleared and the message handled as a normal turn (the user can re-set a goal after). /goal commands (#7) manage it explicitly. Rationale: the goal is a "walk-away" convenience; any human touch (Esc or typing) reasserts manual control — never fight the user for the loop.
6.Resumability across --continue (v1.5.2)decided
Decision
Alternative — in-memory only for v1.6; NOT persisted. The active goal + step count live on TerminalChat; a killed/resumed session restores the conversation (v1.5.2) but not a running goal — resuming into an autonomous loop unattended is surprising and risky. The user re-issues /goal if they want to continue. (Persisting the goal is a clean later add if real use wants it; YAGNI now.)
7.Command surfacedecided
Decision
No CC analog (neutrix command) — mirror the harness /goal the user already knows: /goal <text> sets/replaces the goal and immediately starts the loop; /goal (bare) shows the active goal + step count; /goal clear stops + clears it. A status-bar line shows ◎ goal · step N/25 while active (reuses the v1.5.0 live region). Slash-command, parsed chat-side like /allow.
