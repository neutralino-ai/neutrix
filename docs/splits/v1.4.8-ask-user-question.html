v1.4.8 splits — AskUserQuestion (interactive)

A tool the model calls to pause the turn and ask the human a structured question — and the same channel upgrades permission ask from block-with-notice to a real prompt.

Decided autonomously under the build-to-v2.0.0 /goal (user: "also add AskUserQuestion tool"). The concurrency-bearing splits (#1, #4, #5, #6) are advisor-gated before push. CC survey: tools/AskUserQuestionTool/AskUserQuestionTool.tsx (schema 14-73, 1-4 Q / 2-4 opts, answers keyed by question text, multiSelect joined ", ", implicit Other), hooks/toolPermission/handlers/interactiveHandler.ts (canUseTool, onAbort 137-152), FallbackPermissionRequest.tsx (yes / always / no), useCancelRequest.ts (Esc aborts the turn).

Split points

1.Channel — bidirectional async generator, port on the CM (not the Executor)decided
CC
Tool execution calls an injected canUseTool/interactive handler that awaits a UI promise — all in one JS event loop, no threads (interactiveHandler.ts).
The fork
The turn already runs on the main loop (the _worker_loop task); AskUserQuestion does zero compute, so no worker thread is involved either way. The real question is who holds the UI port: (a) inject it into the Executor (CC's canUseTool shape) and await it inside dispatch_all; or (b) keep the Executor a pure event leaf — it yields a needs_user_input event and receives the answer via the generator's .asend(), with the CM owning the port and driving the prompt.
Decision
Alternative — (b) bidirectional async generator; port on the ContextManager. Diverges from CC on purpose: (a) makes the Executor call up into the UI, a sideways edge that breaks this repo's UI→store→CM→{LLM,Executor,Advisor} layering (the Executor is supposed to only yield events, never call anyone). (b) keeps that invariant — dispatch_all does answer = yield ToolEvent("needs_user_input", {spec}); _dispatch_tools drives it with gen.asend(await self.ask_user(spec)) (or asend(None) when no port). The Executor still builds the question spec and interprets the reply (pure functions) but never touches the UI. This is the idiomatic two-way generator (not an asyncio.Queue MQ — we want lockstep request-response with backpressure, not a decoupled buffer). (User-surfaced mid-implementation: "did you do full layered separation?" + "async for + yield … MQ?" — the first design leaked, this is the fix.)
2.Selector UI — numbered in-flow, not an arrow-key listdecided
CC
Arrow-key selectable Select/SelectMulti components (Ink), Space toggles checkboxes, page up/down (use-select-navigation.ts).
Decision
Alternative — numbered options + single-line answer. Reason: neutrix's renderer is append-only console.print scrollback with a one-line prompt_toolkit input; a full-screen arrow selector (radiolist_dialog) fights both the append-only log and the output_patch context. We print 1. label — description lines and read one line: a number (single), comma-separated numbers (multi), or free text (Other). The semantics (pick one / pick many / Other) match CC exactly; only the input modality differs, justified by the architecture.
3.Render target — scrollback (durable), not the live regiondecided
CC
The question and the chosen answer live in the transcript.
Decision
Follow CC — durable scrollback. The question block prints to scrollback when ask_user fires (it is a real exchange, like a user/assistant message); the answer is recorded as the tool result, which renders through the existing role:tool path. Not the _above_input live region (that is for ephemeral heartbeat/preview). The Q&A becomes part of the permanent log — and the ~/.claude/projects JSONL, so ccusage/transcripts see it.
4.Input routing while pending — orthogonal flag, 4-state machine unchangeddecided
The fork
Add a 5th ContextManager state (AWAITING_USER_INPUT), or keep 4 states and route input by a separate flag.
Decision
No CC analog (CC has no equivalent state enum) — orthogonal flag, minimal-state. The turn is conceptually still AWAITING_EXECUTOR (a tool is in flight; it just happens to await a human), so the state machine stays 4-state. TerminalChat holds _pending_question + an asyncio.Future; _input_loop checks it: when set, the next input line resolves the future (answer) instead of enqueue_user. Worker task (the turn) and input task coordinate via the Future on the same loop — cross-task, not cross-thread. Growing the state machine would ripple into every state == guard (cancel, heartbeat, compaction); the flag is contained.
5.Cancel while a question is pendingdecided
CC
Esc fires onAbort → cancels the tool use AND aborts the whole turn (interactiveHandler.ts 137-152; useCancelRequest.ts).
Decision
Follow CC — Esc aborts the turn, reusing the existing cancel path with near-zero new machinery. ctx.cancel() → _do_cancel() → _drive_task.cancel() raises CancelledError at the await self.ask_user(spec) point inside the CM's drive loop; it unwinds exactly like a cancelled tool and is caught by _drive's existing except CancelledError. The AskUserQuestion tool_call is left without a role:tool result — but that orphan is already repaired by llm.py's pairing layer (synthesizes a placeholder; [interrupted by user] after it reads as cancelled). Two small additions: the UI _ask_user's finally clears _pending_question so the input loop stops routing to a dead Future, and _dispatch_tools' finally await gen.aclose()s the executor generator parked at its needs_user_input yield. No new cancel code, no orphan handling.
6.Permission ask unification — share the whole channeldecided
CC
Permission prompts and AskUserQuestion are separate components that share a Select widget and the same abort channel (FallbackPermissionRequest.tsx: yes / yes-don't-ask-again / no).
Decision
Alternative — share the entire ask_user channel, not just a widget. v1.4.0's ask verdict is block-with-notice (no interactivity yet). v1.4.8 upgrades it: when decide() returns ask and a port exists, dispatch_all calls ask_user with a yes/no/always question; "yes" runs the tool, "always" runs it and appends an allow rule to the session policy, "no" blocks with the model-readable notice. SOLID win: one interactive channel for both. Falls back to v1.4.0 block-with-notice when the port is None (split #7).
7.Headless / subagent fallback — ask_user is Nonedecided
CC
Always interactive (a human REPL); a subagent's tool set excludes interactive prompts.
Decision
No CC analog — explicit non-interactive fallback. When the port is None (tests, piped stdin, and crucially inside a subagent — a subagent has no terminal of its own): AskUserQuestion returns a tool error [AskUserQuestion needs an interactive terminal — not available here]; the permission ask path stays block-with-notice (v1.4.0). AskUserQuestion is excluded from subagent_tool_names() so a subagent never even sees it (matches CC excluding it from sub-agent tool sets). Rationale: a sub-turn that blocks forever on a human who can't answer is a deadlock; fail fast and visibly.
8.Schema + answer encodingdecided
CC
1-4 questions, 2-4 options each, unique question/option text, header ≤12 chars, multiSelect default false, implicit Other; returns {questions, answers:{questionText→label(s)}}, multiSelect joined ", " (AskUserQuestionTool.tsx 14-73).
Decision
Follow CC. Same bounds and validation; ask the questions sequentially (print Q1 options, read, Q2…). Return a readable result keyed by question text, multiSelect labels joined ", ", Other → the verbatim free text. Drop the preview field (a terminal can't render side-by-side markdown previews — a YAGNI omission, restorable later).
9.Answer parsing rule — Other detection & empty inputdecided
Decision
No CC analog (CC has an explicit Other affordance; numbered input needs a parse rule). Input that is entirely comma-separated integers all in [1..N] → those option labels (one for single-select; for single-select with multiple numbers, take the first). Anything else (non-numeric, out-of-range, mixed) → Other, recorded as the verbatim text. Empty input while a question is pending → re-prompt the same question (do not abort; abort is Esc, split #5). Whitespace trimmed.
