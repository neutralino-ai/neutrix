v0.9.4 splits — heartbeat liveness indicator

A single pulsing glyph + short label above the input while the ContextManager is in AWAITING_LLM or AWAITING_EXECUTOR. Reads the v0.9.3 state machine; no new store fields.

Per .claude/rules/prd-compare-with-claude.md, this file is the durable record of the split-point discussion that happens before the PRD is drafted. CC source under /datafs/users/dingxf/anthropic/cc2/src. Eleven original splits + one Phase-2 reopen (split #3 cadence, joined by new splits #12-#14 on cycle period, smoothing method, and tick jitter — see below) are now decided; the v0.9.4 PRD's ## Claude Code reference section embeds the summary. Decision labels follow the rule: Follow CC / Alternative / User-directed different / No CC analog.

Phase-2 reopen (2026-05-27): first implementation shipped at the Phase-3 gate with a 720 ms cycle (~83 BPM). User rejected: "beating too fast. it should give people feeling of rythem of breathing". The reject re-opened split #3 and surfaced three new split points (#12 cycle period, #13 smoothing method, #14 randomness) that the original survey missed. New decisions, recorded turn-by-turn below.

Split points

1. Phase set — how many distinct labels? decided
CC behavior
Five (requesting / responding / thinking / tool-input / tool-use) driven by streaming content-block events.
Alternatives
  • 1 — "working" only.
  • 2 — "LLM" / "tool: {name}" (gated on the CM state).
  • 3 — "LLM (waiting)" / "LLM (streaming)" / "tool: {name}".
  • 5 — match CC.
Recommendation
Alternative — 2-phase. v0.9.4 only has two CM busy states (AWAITING_LLM, AWAITING_EXECUTOR); v0.9.6 reintroduces streaming but neutrix has no thinking blocks. Don't import phase granularity for features we don't have. If v0.9.6 wants to split AWAITING_LLM into pre-first-token vs streaming, that's a v0.9.6 split point — not v0.9.4's.
Decision
Alternative — 2-phase + CANCELLING. Maps 1:1 to neutrix's ContextManager state machine (AWAITING_LLM → "LLM", AWAITING_EXECUTOR → "tool: {name}", CANCELLING → "cancelling…"). v0.9.x has no streaming or thinking blocks to justify CC's 5-phase granularity; if v0.9.6 needs streaming sub-phases, those become their own split point.
2. Glyph set user-directed
CC behavior
✻-family Unicode flower forms cycled forward+backward. Platform-aware (Windows substitutes *; xterm-ghostty drops ✽).
Alternatives
  • Braille spinner ⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏ (the prior PRD draft).
  • CC's ['·', '✢', '✳', '✶', '✻', '✽'] with forward+backward cycle.
  • Dots . .. ... (Codex-style).
  • Single static glyph + dim/bright pulse on a clock.
Recommendation
Follow CC — the ✻-family with the forward+backward cycle. The breathing rhythm reads as "alive-and-thinking" rather than "spinning frantically." Braille spinners blur at typical 14–16 px terminal sizes; CC's flowers are crisper. Accept the platform-substitution rule so Windows and xterm-ghostty don't render a hollow box.
Decision
User-directed different — single ● with brightness pulse over 6 frames, no shape rotation. User: "suggest more. what would Steven Jobs do?" In a redo round the agent surfaced four Jobs-mode alternatives (breathing dot, label-only fade, the CC ✻ family, 3-dot Codex). User chose the breathing dot — single character, calmer than CC's 10-frame ✻ rotation, reads as alive without ornamental motion. Diverges from CC.
3. Tick cadence (re-opened Phase 2) decided (revised)
CC behavior
120 ms (~8 Hz) per frame.
Alternatives
  • 100 ms (10 Hz).
  • 120 ms (CC).
  • 200 ms (the prior PRD draft, 5 Hz).
  • 250 ms (4 Hz, lower redraw cost).
Recommendation
Follow CC — 120 ms. Below ~150 ms the eye reads "alive"; above ~200 ms it reads "stuttering." Redraw cost is negligible because the heartbeat is a single line in an invalidation-driven render path.
Decision (Phase-1, since revised)
Follow CC — 120 ms / ~8 Hz per frame. Matches the agent's recommendation and CC byte-for-byte.
Decision (revised Phase 2, 2026-05-27)
Alternative — 100 ms nominal with ±10% jitter. Now coupled to split #12 (cycle period) and split #14 (randomness): with 40 frames × 100 ms = 4.0 s breathing cycle, the per-frame jitter (~90–110 ms) keeps the eye reading "alive" while desyncing from CPU clocks for an organic feel. Diverges from CC.
4. Label policy — static phase names vs verb personality decided
CC behavior
Random verb from a 200+ list (Pondering, Cogitating, …), sampled once per turn, stable across ticks. Tool runs override with task activeForm when set.
Alternatives
  • Static "LLM" / "tool: {name}".
  • Verbed but neutral: "thinking…" / "running {name}…".
  • CC's full random-verb list (200+).
  • A small curated 10-verb list with the same sample-once policy.
Recommendation
Alternative — static phase names: "LLM" when AWAITING_LLM; "tool: {name}" when AWAITING_EXECUTOR. The phase IS the information; personality is post-v1.0 polish. Static text means the heartbeat is also a debugging affordance — the user can see at a glance which state the CM is in. (Could revisit at v1.x as a small "personality" PR if users miss it.)
Decision
Alternative — static phase names. "LLM" / "tool: {name}" / "cancelling…". Phase IS the information; doubles as a debug affordance for inspecting CM state. Random-verb personality (CC's 200+ list) is a v1.x polish item if anyone misses it.
5. Tool-name detail in AWAITING_EXECUTOR decided
CC behavior
Custom activeForm when a task is set (e.g. "Running bash…"). Args not echoed inline.
Alternatives
  • tool: {name} only.
  • tool: {name} {first-N-bytes-of-args}.
  • tool generic, no name.
  • tool: {name} ({k} args).
Recommendation
Alternative — name only. The -> tool_use {name} {args} line (v0.8.2 colored keyword) printed when the call started already shows the args; the heartbeat duplicates nothing. Keeps the line short on narrow terminals.
Decision
Alternative — name only. "tool: {name}", no args. Args appear on the preceding -> tool_use {name} {args} transcript line (v0.8.2 colored keyword); the heartbeat duplicates nothing and fits narrow terminals.

  • Multi-row — one heartbeat line per pending tool.
Recommendation
Alternative — first-only. v0.9.x has no parallelism (one tool at a time per round). Multi-row / count belong to whichever release adds parallel tool dispatch — defer the split until then. YAGNI.
Decision
Alternative — first-only. Head of store.pending_tool_calls. v0.9.x has no parallel tool dispatch (one tool per round); multi-row / count splits belong to the release that adds parallelism.
7. Cancel transition state — render CANCELLING or vanish? decided
CC behavior
No transition. resetLoadingState() hides the spinner immediately on Esc. REPL.tsx:2106.
Alternatives
  • Follow CC — vanish immediately on Esc.
  • Render cancelling… while the CM is in CANCELLING; vanish on transition to IDLE.
  • Render cancelled briefly (~500 ms) then vanish.
Recommendation
Alternative — render cancelling… while in CANCELLING. v0.9.3 introduced CANCELLING as a real state in the CM state machine; the heartbeat is the natural surface that reveals it. neutrix's cancel window is measurable (200 ms grace + tree-kill); CC's cancel is instant so vanish-immediately reads fine for them. Showing the state aligns user-view with the project's core abstraction.
Decision
Alternative — render cancelling… while CM is in CANCELLING. Aligns user-view with the project's core state-machine abstraction. neutrix's cancel window is measurable (~200 ms tree-kill) so the state visibility carries information CC's vanish-immediately doesn't. Diverges from CC.

9. Position in the message-area stack user-directed
CC behavior
Two-row footprint above the input prompt.
Alternatives

  • Between task panel and queue (eye-line above cursor).
  • Below queue, directly above the input cursor.
  • On the input prompt line (replaces the prompt prefix).
Recommendation
Follow CC for the spirit (right above the cursor's eye-line) but slot it between task panel and queue — stack order: task panel → heartbeat → queued messages → cursor. Heartbeat is transient; task panel is durable; heartbeat below the panel reads as "current activity within the broader plan."

Phase-2 reopen: cycle period, smoothing, jitter

The first implementation set the cycle at 6 frames × 120 ms = 720 ms (~83 BPM). User feedback at the Phase-3 gate: "it's beating too fast … it should give people feeling of rythem of breathing". Research:

★ Cognitive-science / physiological reference survey
Resting calm
Adult resting respiratory rate: 12–20 breaths/min (3.0–5.0 s per full cycle). Below 12 is unusual; above 25 flags stress / illness. (Medical News Today, American Lung Association.)
Coherent breathing
Meditation / HRV-training rate: 5–6 breaths/min (10–12 s per cycle). Targets ~0.1 Hz where heart-rate oscillations resonate. Stronger calming effect, but too slow for an ambient liveness indicator — would read as "dead" between dim phases. (VagusVital review.)
CC actual
SPINNER_FRAMES is built as [...DEFAULT_CHARACTERS, ...[...DEFAULT_CHARACTERS].reverse()] = 12 frames at 120 ms each = 1.44 s per cycle (~42 BPM). Faster than resting calm. (cc2/src/components/Spinner/SpinnerGlyph.tsx:7.)
CC reduced-motion
Single ● at 2.0 s cycle (1 s visible, 1 s dim). Calmer than normal mode. (cc2/src/components/Spinner/SpinnerGlyph.tsx:9.)
12. Cycle period — how long is "one breath"? user-directed
CC behavior
1.44 s (normal); 2.0 s (reduced-motion). Both faster than calm resting breathing.
Alternatives
  • 2.0 s — CC reduced-motion parity (~30 BPM).
  • 3.0 s — resting calm middle (~20 BPM).
  • 4.0 s — slow calm (~15 BPM).
  • 6.0 s — coherent-breathing range (~10 BPM).
Recommendation
3.0 s — middle of the human resting-calm band. Calm without going meditation-slow; the dot's dim phase stays short enough that the user never wonders if the app stalled.
Decision
User-directed different — 4.0 s per cycle (~15 BPM). User picked the slow-calm tier over the agent's 3.0 s middle-tier recommendation. Locks split #3's tick to ≈100 ms with 40 frames for a smooth-feeling 4-second breath.
13. Smoothing — how to avoid stepwise feel at slow cycle? decided
CC behavior
Doesn't apply — CC achieves smoothness by changing the character shape (the ✻-family progression) rather than brightness. neutrix is locked to a single ● (split #2, user-directed), so brightness is the only channel.
Alternatives
  • Stay ANSI (3 shades: bright / medium / dim), few frames, slow tick — steps visibly every ≥1 s; risks looking broken.
  • Stay ANSI but introduce a 4th shade via the dim modifier on bright/normal — marginal improvement, still steps.
  • Truecolor gradient — interpolate through ~20–40 hex gray shades (e.g. #3c3c3c → #ffffff) at fast tick. Genuinely smooth breath; needs truecolor terminal.
Recommendation
Truecolor gradient. Every modern terminal (iTerm2, Alacritty, Kitty, Ghostty, gnome-terminal, Windows Terminal, etc.) supports 24-bit color; the few that don't degrade to ANSI via prompt_toolkit's color-depth fallback. With a 4-second cycle the smoothness is what makes the animation read as "breathing" vs "blinking."
Decision
Truecolor gradient. 40 hex gray shades interpolated along a raised-cosine curve from peak rgb(255,255,255) to trough rgb(60,60,60). Brightness curve is (1 - cos(2π · frame / 40)) / 2 scaled into the gray range. Label keeps its bold-bright ANSI style (split #4 / #7 unchanged).
14. Randomness — strict period or organic jitter? decided
CC behavior
Strict periodic. Uses setInterval at 120 ms.
Alternatives
  • None — strict period (CC parity).
  • Tick jitter — sleep interval randomized within ±X%.
  • Phase offset — randomize the starting frame each time the cycle re-activates.
  • Brightness jitter — random shade selection on each tick.
Recommendation
±10% tick jitter only. Cheapest and most organic — desyncs the cycle from CPU-clock-multiple periods, so two consecutive breaths look subtly different without "glitching." Phase offset adds delight at boot but the heartbeat re-activates every turn; brightness jitter looks buggy.
Decision
±10% tick jitter, no phase offset, no brightness jitter. Each frame sleeps for HEARTBEAT_TICK_MS · uniform(0.9, 1.1). Tests override the jitter with a fixed-seed RNG (or pass a custom sleep function) for determinism.
