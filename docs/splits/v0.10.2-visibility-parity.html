Split points

1.Fold model — in-place toggle vs expand-by-appenddecided
The collision
The skeleton's headline is a Ctrl+E fold/expand toggle. But the transcript is append-only scrollback (console.print, v0.6.8) — you cannot collapse or re-render a block already printed. A true in-place toggle would mean reversing the append-only architecture (full-screen TUI rework) — out of scope, arguably never.
Options
  • In-place Ctrl+E toggle (needs append-only rework — infeasible here).
  • Expand-by-append: render a one-line folded summary; a command re-prints the full content below on demand (the existing /tool N idiom).
Recommendation
Expand-by-append — forced by the architecture, and already the neutrix idiom (/tool N re-prints a result below). Parity = every channel has a folded summary rendered; full content reachable by re-printing.
Decision
Alternative — expand-by-append (no toggle). Generalize /tool N into a /show <what> command that re-prints the full content of a folded block (system, tools) below the cursor. No Ctrl+E binding — key-binding wiring has regressed twice; a command matches the idiom and meets parity (full content is command-reachable). "Collapse" is not offered (can't un-print). This also defines v0.10.0's deferred subagent fold (split #5).
2.Scope — the skeleton's 4 channels vs the real gapdecided
Finding
Ground-truthing (table above) shows 3 of 4 channels already satisfy parity. The skeleton assumed all 4 were hidden; only tool schemas genuinely is.
Decision
Scope to the real gap. Deliver: (a) the [tools] folded block at session start (the gap), (b) fold the system prompt when it exceeds a threshold (short prompts stay inline — preserves current behavior for the default), (c) .claude/rules/visibility-parity.md, (d) the invariant test, (e) a [subagent] label (split #5), (f) the /show expand command. Reminders + tool results are already compliant and left untouched. The PRD's Problem section states this honestly.
3.The single-source bundle + AgentEventdecided
The catch
The skeleton proposes emitting AgentEvent("llm_round_start", bundle). But v0.9.3 removed AgentEvents — the renderer subscribes to ChatStore.changes(). System prompt + tool schemas are session-static (invariant across rounds).
Decision
No event. Render the static system/tools channels once at session start (from ctx), which satisfies per-round parity since they don't change. Add a frozen LLMRoundBundle(system, messages, tools) snapshot in context_manager.py built on demand (ctx.round_bundle()), used by the invariant test to enumerate channels — not as an event.
4.Invariant test shapedecided
Decision
Capture both render phases; iterate bundle fields dynamically. The test spies on TerminalView.print_* across BOTH session-start rendering (system/tools) and the store.changes() walk (messages/results/reminders), then asserts every populated field of the LLMRoundBundle produced ≥1 render call. Iterate the bundle's fields (not a hardcoded (system, messages, tools) tuple) so a future 4th channel trips the test — that dynamic enumeration is what gives the rule teeth.
5.v0.10.0's deferred subagent fold — what's actually neededdecided
Finding
v0.10.0 renders the subagent result as a normal tool result (already folded + /tool N-expandable). So the "deferred fold machinery" is already there — the only gap is that it reads as a generic tool_result, not visibly a subagent.
Decision
Just a label, no new machinery. When the folded tool-result summary is for the Agent tool, render the keyword as subagent instead of tool_result. Reuses the entire existing fold/expand path. The v0.10.0 deferral is discharged by a one-line label change, not a fold subsystem.
