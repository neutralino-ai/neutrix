v0.9.6 splits — emergency mechanical /compact

A slash command that drops the oldest ~50 % of ContextManager.messages mechanically — no LLM summarization, no automatic trigger, no configurable ratio. The bridge that keeps users working until smart compaction lands at v0.10.5.

Per .claude/rules/prd-compare-with-claude.md, this file is the durable record of the split-point survey that happens before the PRD is drafted. CC source under /datafs/users/dingxf/anthropic/cc2/src. Decision labels follow the rule: Follow CC / Alternative / User-directed different / No CC analog.

Autonomous mode. The user set a /goal (“proceed for v0.9.6 all the way until you ship it”) and walked away, so the turn-by-turn discussion the rule normally requires could not happen live. Every decision below is therefore tagged autonomous — an architect-made call, not a user choice. They are surfaced as a block at the Phase 3 gate; the user can redirect any of them (search this file for the literal string Decision (autonomous): to review all 12 at once). Per CLAUDE.md autonomous-mode note, the release is not tagged without an explicit human accept gate.

Tally: 1 Follow CC (#2) · 8 Alternative (#1, #3–9) · 3 No CC analog (#10, #11, #12). The headline finding from the survey: CC has no “drop the oldest N messages” path at all. CC's compaction is summary-based (services/compact/compact.ts); its only mechanical moves are (a) micro-compact, which clears tool-result content in place rather than removing messages (microCompact.ts:36,470-492), and (b) the reactive prompt-too-long escape hatch truncateHeadForPTLRetry, which drops whole oldest API-round groups until the payload fits (compact.ts:231-241,257). So v0.9.6 borrows exactly one thing from CC — the API-round boundary as the safe cut granularity (split #2) — and is otherwise a deliberately small, neutrix-shaped stopgap.

Context — what CC does, in one paragraph

★ CC's compaction subsystem, summarized survey
Primary path
LLM summarization. compactConversation() streams a summary of the older turns into a fresh single block (services/compact/compact.ts:387-763); the summarizer prompt is a 9-section template (services/compact/prompt.ts:61-143). This is the explicit v0.10.5 goal, not v0.9.6.
Mechanical move A
micro-compact (content-clear). Keeps the most recent N tool results, replaces the content of older ones with the literal string "[Old tool result content cleared]" — it does not remove messages. microCompact.ts:36,461-492.
Mechanical move B
reactive head-truncation. Only on an API prompt-too-long error: drop whole oldest API-round groups until the payload fits. compact.ts:231-241,257-291. This is the closest CC analog to “drop the oldest half.”
Cut granularity
groupMessagesByApiRound() groups messages at assistant-message-id boundaries; the API contract guarantees every tool_use is resolved before the next assistant turn, so a round boundary is an API-safe split. grouping.ts:22-63 (note at :34-42).
Boundary marker
A typed CompactBoundaryMessage carrying metadata (manual/auto, preCompactTokenCount, parent uuid, preserved tool state). compact.ts:598-611. Post-compact order is [boundary, …summary, …kept] (:330-338).
User-facing report
The /compact command prints a dimmed line; what it reports is tokens (pre/post estimates), not a turn count. commands/compact/compact.ts:230-248; token math at compact.ts:637-642. The command accepts optional free-text custom summary instructions (commands/compact/compact.ts:52).
Auto trigger
Token-budget threshold contextWindow − 13K (autoCompact.ts:72-91). Explicit v0.9.6 non-goal; lands at v0.10.5.

Context — what neutrix has today (v0.9.5)

★ neutrix history surface, summarized survey
Message shape
OpenAI Chat Completions. messages[0] is always {"role":"system",…}, seeded in ContextManager.__post_init__ (context_manager.py:198-209). An assistant turn that calls tools is {"role":"assistant","content":null, "tool_calls":[…]} followed by one {"role":"tool","tool_call_id":…} per call. Task reminders and the cancel marker ("[interrupted by user]") ride as ordinary role:user messages.
Dual mutator
ContextManager is the sole writer of both self.messages (the LLM payload) and self.store (the typed render source); the two are kept in parallel by _append_* helpers (context_manager.py:560-566).
History-mutating commands
/clear → ClearEvent, /load → ReplaceHistoryEvent (context_manager.py:334-346). Each cancels any in-flight turn first, rebuilds messages + resets the store. cancel() is a direct sync method returning a bool, separate from the event surface (context_manager.py:263-278) — precedent for a command that needs a return value.
Pairing repair
_ensure_tool_result_pairing (llm.py:53-161) runs on a copy at API-send time. It synthesizes a placeholder for orphan tool_use in the latest assistant message ("[cancelled by user]" / "[tool result missing]") — but does not repair a leading orphan tool_result (a role:tool with no preceding tool_calls), which most providers reject. This is the hard constraint on where the cut may fall.
Render + save
The terminal is append-only; the render watcher prints store.messages[_rendered_message_count:]. _cmd_clear/_cmd_load reset _rendered_message_count by hand (terminal_chat.py:1064-1094). Task reminders are detected and shown as a dim one-line notice instead of raw (terminal_chat.py:894-901). /save writes store.messages to transcript v2; reminders round-trip as user messages with no dedicated key (transcript.py:1-10,39-48). A cheap approximate_token_count (whitespace word count) already exists (terminal_chat.py:169-171).

Split points

1. Compression method — drop messages vs clear content vs summarize autonomous
CC behavior
Primary path summarizes (compact.ts:387-763). The only mechanical moves are content-clear of old tool results (microCompact.ts:36) and reactive head-drop of whole round groups (compact.ts:231-241). CC never “drops the oldest half of all messages” proactively.
Alternatives
  • Drop oldest ~50 % of messages — the skeleton-PRD direction; simplest possible relief, zero LLM calls, instant.
  • Clear tool-result content only (CC micro-compact) — keeps the message skeleton, frees the bulk (tool output is usually the heavy part). More surgical but leaves every user/assistant turn in the payload.
  • Summarize now — that is v0.10.5; building it here collapses two releases and needs a summarizer prompt + an LLM round.
Recommendation
Alternative — drop oldest ~50 % of messages. The whole point of v0.9.6 is a no-LLM, instant escape hatch that ships in one release; content-clear leaves the turn count (and thus most providers' per-message overhead) untouched, and summarize is explicitly the next release.
Decision
Decision (autonomous): Alternative — drop oldest ~50 % of messages. No-LLM, mechanical, one-release stopgap. Content-clear and summarize are both deferred (the latter to v0.10.5). Re-uses CC's round boundary for safety (split #2) but not its compression strategy.
2. Cut granularity & direction — where exactly does the boundary fall? autonomous
CC behavior
Cuts at API-round boundaries (grouping.ts:22-63) and drops whole oldest groups (compact.ts:231-241). A round boundary is API-safe because every tool_use is resolved before the next assistant turn (grouping.ts:34-42).
Alternatives
  • Naive message-index cut at 1 + floor(L/2) — can land mid-round, leaving the kept tail starting on a role:tool orphan that the pairing layer does not repair → API rejection.
  • Round-safe, snap forward — compute the naive index, then advance past any leading role:tool messages so the first kept message is never an orphan tool_result. Drops the partial oldest round entirely.
  • Round-safe, snap backward — pull the cut back to the owning assistant so the partial round is kept whole. Keeps slightly more (<50 % dropped).
Recommendation
Follow CC — round-safe boundary, snap forward. Forward-snap gives the cleanest falsifiable invariant: the first kept message is never a role:tool. It matches CC's “drop whole oldest groups” semantics and sidesteps the orphan-tool_result gap in _ensure_tool_result_pairing entirely.
Decision
Decision (autonomous): Follow CC — round-safe, snap forward. Naive index 1 + floor(L/2) over the non-system body, then while messages[cut].role == "tool": cut += 1. Same effect as CC dropping whole oldest API-round groups. Invariant under test: the kept tail never begins with an orphan tool_result.
3. Placeholder — what (if anything) replaces the cut slice? autonomous
CC behavior
Inserts a typed CompactBoundaryMessage with metadata (compact.ts:598-611), then the summary, then the kept messages (:330-338).
Alternatives
  • Nothing — just splice. The LLM never learns context was dropped; may re-ask for things it “should remember.”
  • One role:user placeholder — <system-compact>{N} earlier turns removed by /compact (no summary)</system-compact>. Mirrors neutrix's existing “ride as a user message” convention for <system-reminder> and [interrupted by user].
  • Typed boundary message (CC) — neutrix has no message-metadata infrastructure; would need a store/transcript schema change.
Recommendation
Alternative — one role:user placeholder, text verbatim from the skeleton. Tells the model context was dropped, not summarized, so it asks rather than hallucinates continuity. role:user (not system) because a second system message is rejected by some providers; reminders already use this trick. No schema change.
Decision
Decision (autonomous): Alternative — single role:user <system-compact> placeholder. Text kept verbatim: <system-compact>{N} earlier turns removed by /compact (no summary)</system-compact>. CC's typed boundary is a No-CC-analog gap for neutrix (no metadata infra); the user-message convention is the established neutrix shape.
4. Head preservation — what survives the cut at the front? autonomous
CC behavior
Rebuilds the system prompt fresh from config post-compact (compact.ts:269-287); it is not “preserved” so much as regenerated.
Alternatives
  • Preserve the leading system message(s) verbatim — keep messages[:k] where k is the count of leading system-role messages (normally 1), cut only the body after it.
  • Regenerate the system prompt (CC) — neutrix has a static system_prompt; regeneration buys nothing and risks drift from a loaded transcript's saved prompt.
Recommendation
Alternative — preserve leading system message(s) verbatim. neutrix's system prompt is static and already anchored at messages[0]; computing k = leading system count is robust to a loaded transcript that carries its own saved system prompt.
Decision
Decision (autonomous): Alternative — preserve the leading system prefix verbatim. Cut the body after it. CC's regenerate-from-config has no neutrix analog and would diverge from a loaded session's stored prompt.
5. Report — turns vs tokens, and where does the token number come from? autonomous
CC behavior
Reports tokens (pre/post estimates from a real tokenizer), not a turn count. compact.ts:637-642; commands/compact/compact.ts:230-248.
Alternatives
  • Turns + approx tokens — skeleton wording: compacted 15 turns (~3.2k tokens dropped). Count is exact, token figure is a rough estimate.
  • Tokens only (CC) — needs a tokenizer; the dropped-count is the more honest number for a mechanical cut.
  • Messages instead of “turns” — arguably more precise for a message-level cut, but diverges from the skeleton/roadmap wording.
Recommendation
Alternative — compacted N turns (~K tokens dropped), token figure from the existing approximate_token_count word count. Keep the skeleton's “turns” wording verbatim (no unilateral taste-edit to “messages” in autonomous mode); the ~ already signals the token figure is approximate, so a tokenizer dependency is unwarranted for a stopgap.
Decision
Decision (autonomous): Alternative — compacted N turns (~K tokens dropped). “turns” kept verbatim from the skeleton; token estimate reuses the existing whitespace word-count helper rather than adding a tokenizer. (If the user prefers “messages” or tokens-only, this is a one-line change at the Phase 3 gate.)
6. Command arguments — does /compact take any? autonomous
CC behavior
Accepts optional free-text custom summary instructions (commands/compact/compact.ts:52), passed into the summarizer prompt (compact.ts:440).
Alternatives
  • No arguments — ratio hardcoded at 50 %, nothing to instruct (no summarizer).
  • /compact <ratio> — a configurable cut fraction. Explicit non-goal (the skeleton hardcodes 50 %; tuning is v0.10.5 territory).
  • Custom instructions (CC) — meaningless without a summarizer.
Recommendation
Alternative — no arguments. There is no summary to steer and the ratio is deliberately fixed. Extra args invite a configurable-ratio surface that the PRD calls a non-goal.
Decision
Decision (autonomous): Alternative — no arguments. /compact only; any trailing tokens are ignored. CC's custom-instructions arg has no analog without a summarizer; configurable ratio is a v0.10.5 non-goal.
7. Idempotency — what happens on a second /compact? autonomous
CC behavior
No idempotency guarantee; re-running summarizes the prior summary (summary-of-summary). Only partial compaction filters old boundaries (compact.ts:791-800).
Alternatives
  • Monotonic further-halve with a no-op floor — the old placeholder (now oldest) is dropped and replaced; when the body is too short to drop anything (floor(L/2) == 0), it's a clean no-op that prints “nothing to compact.”
  • Detect already-compacted & refuse — needs an “is compacted” flag; more state for little gain.
  • Stack placeholders — never; would accrete <system-compact> lines.
Recommendation
Alternative — monotonic further-halve, no-op floor. Safer than CC for a mechanical tool: each call shrinks or no-ops, never crashes, never stacks markers (the prior placeholder is the oldest body message, so it's the first thing dropped).
Decision
Decision (autonomous): Alternative — further-halve with a clean no-op floor. Second call drops the old placeholder + more body and inserts one fresh placeholder; when nothing can be dropped it prints “nothing to compact (conversation too short)” and leaves history untouched.
8. Busy behavior — /compact while the assistant is working autonomous
CC behavior
Compaction runs inside CC's own query loop (auto-compact fires between turns; manual compact operates on settled history). No direct “user typed /compact mid-turn” analog.
Alternatives
  • Refuse while busy — join the existing {fast, strong, save, load, onboard} guard set (terminal_chat.py:977); print “/compact waits for the assistant to finish.”
  • Cancel-and-compact — like /clear//load, cancel the in-flight turn then compact. Surprising: a compact silently aborts the turn.
Recommendation
Alternative — refuse while busy. Compacting mid-turn risks cutting a round whose tool results haven't landed; refusing is consistent with /save & /load and keeps the CM entry point simple (only called from IDLE).
Decision
Decision (autonomous): Alternative — refuse while busy. Add compact to the busy-guard set so the command waits for IDLE. The CM compaction routine therefore only runs against settled history.
9. Live rendering — reprint the compacted tail, or notice-only? autonomous
CC behavior
Full-screen Ink/React app re-renders the whole view including the boundary marker and summary on every compaction.
Alternatives
  • Notice-only + suppress reprint — print one dim compacted … line; set _rendered_message_count = len(store.messages) so the already-visible tail isn't duplicated and future appends stay index-aligned.
  • Reprint the tail (mirror /load, which sets _rendered_message_count = 0) — but the kept tail is already in scrollback, so reprinting duplicates content.
Recommendation
Alternative — notice-only, suppress reprint. The terminal is append-only and the kept messages are already on screen; reprinting them under a “compacted” notice is confusing. /load reprints because its content is unseen; compaction's tail is seen, so the cases differ.
Decision
Decision (autonomous): Alternative — notice-only + suppress reprint. After compaction, set _rendered_message_count = len(self.store.messages) (both prevents tail duplication and re-aligns the index for future appends) and print one dim notice. Diverges from /load on purpose.
10. Store + payload — compact both, and what about tasks? autonomous
CC behavior
Single message model; no neutrix-style split between an LLM payload list and a typed render store. No analog.
Alternatives
  • Compact both cm.messages and the store, preserve tasks — rebuild the store from the compacted messages so /save persists the compacted state (acceptance requirement), and re-seed store.replace_tasks(saved_tasks) because store.reset() clears tasks.
  • Compact payload only — leaves the store (and therefore /save and the render source) showing the full transcript: violates “/save preserves the compacted state.”
  • Drop tasks too (like /clear) — wrong: compaction trims old context, not the live work list.
Recommendation
No CC analog — compact both, preserve tasks. neutrix's dual store/payload structure forces this: rebuild the store from the compacted messages, then replace_tasks(saved_tasks). Tasks are the live work list and must survive a context trim.
Decision
Decision (autonomous): No CC analog — compact both, preserve tasks. Save tasks = store.tasks → store.reset() → append the compacted records → store.replace_tasks(tasks). (replace_tasks recomputes _next_task_id from max(id)+1 — a no-op when tasks are unchanged.) Covered by a dedicated task-preservation test.
11. Save/load round-trip & marker rendering autonomous
CC behavior
JSONL-on-disk transcript; the boundary message is a typed record. No neutrix-format analog.
Alternatives
  • No schema change; render marker raw for now — the <system-compact> message saves/loads as an ordinary role:user entry (transcript v2, no key needed). On reload it prints as a plain cyan user turn. Special dim labelling is deferred to v0.10.2 (visibility-parity).
  • Detect & dim now (like is_task_reminder) — add an is_compact_marker render branch. Small, but the skeleton explicitly punts marker styling to the visibility-parity release.
  • New transcript key for compaction — schema bump for a stopgap; unjustified.
Recommendation
No CC analog — no schema change; render raw, defer dim labelling to v0.10.2. The marker rides as a user message exactly like task reminders do; the live path suppresses reprint (split #9) so raw styling only ever shows on a reload, and v0.10.2 visibility-parity is the release that owns marker styling.
Decision
Decision (autonomous): No CC analog — transcript v2 unchanged; marker renders raw for v0.9.6. Round-trips as a role:user message. Dim/special rendering of <system-compact> is left to v0.10.2 visibility-parity, per the skeleton's stated intent.
12. Programmatic entry point — cm.compact() method vs CompactEvent autonomous
CC behavior
Compaction is a service called from the query loop and from the command; auto-compact fires it on a token threshold (autoCompact.ts). The trigger and the mechanism are separable.
Alternatives
  • Direct async cm.compact() -> CompactionOutcome — mirrors the existing direct cancel() method (returns a value the command needs; the event surface stays for UI-emitted, fire-and-forget events). v0.10.5's threshold watchdog calls the same method.
  • CompactEvent through handle_event — uniform with ClearEvent/ReplaceHistoryEvent, but handle_event returns None, so the command can't get the turns/tokens for its notice without extra plumbing.
  • Both — event wraps method. YAGNI for one caller.
Recommendation
No CC analog — direct async cm.compact() returning an outcome. The command needs the dropped counts back; cancel() is the precedent for a value-returning direct method alongside the event surface. It is also exactly the seam v0.10.5's auto-trigger will call, so the shape survives the next planned release.
Decision
Decision (autonomous): No CC analog — direct async cm.compact() -> CompactionOutcome. Returns (did_compact, turns_dropped, approx_tokens_dropped). The boundary/placeholder logic lives in a pure compaction.compact_messages() so v0.10.5 can reuse the cut computation and swap the placeholder step for a summarizer. No CompactEvent until a fire-and-forget caller needs one.
