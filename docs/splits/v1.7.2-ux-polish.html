v1.7.2 splits — user-message background & compaction heartbeat

Two transcript/liveness UX fixes: make user messages easy to spot (a non-black background) and stop /compact from looking dead (a heartbeat while it works).

Agent-decided under the user's standing "decide yourself" delegation ([[feedback-investigate-decide]]) — CC surveyed (/datafs/users/dingxf/anthropic/cc2/src), advisor-reviewed. Backlog spec: ships as v1.7.2 (the next ascending tag); the still-unbuilt arch doc bumped to v1.7.3 ([[feedback-versioning]]). See docs/PRDs/v1.7.2-ux-polish.md.

Split points

1.User-message visual treatment — a backgrounddecided
Now
User turns render with USER_STYLE = "cyan" (foreground only, via terminal_chat › ChatView.print_user_now); they blend into the assistant/system text in dense scrollback.
CC
CC styles user/command turns with a foreground color + prefix (Ink Text color=…, e.g. components/messages/UserCommandMessage.tsx color="subtle"), not a background.
Options
(a) span background — add on <color> to USER_STYLE (colors the message text); (b) full-width bar — pad each line to the console width so the bg is a clean rectangle; (c) just a brighter fg / a prefix glyph.
Decision
(a) span background — User-directed ("bkg color different than black"). Add a fg/bg pair to USER_STYLE (a readable combo like cyan on grey19 / white on grey23 — NOT a clashy on blue; exact pair is PRD-level + tweakable). Full-width bar (b) rejected: the renderer is append-only scrollback (v0.6.8) — padded lines don't reflow on resize, leaving ragged colored rectangles (the "ugly" the user rejects, [[feedback-ui-hints]]); for the common 1–2 line prompt the span bg looks identical anyway (YAGNI — revisit only if multi-line raggedness actually bothers). Applies to real user prompts only — the role=="user" → print_user branch sits below the task-reminder / goal / advisor / summary-marker special-cases, so it naturally excludes injected user-role turns.
2.Compaction liveness — a heartbeat while compactingdecided
Now
ContextManager.compact() runs smart_compact — an LLM call — but sets no state; it runs while the CM is IDLE (the /compact command refuses while busy). The status bar (format_heartbeat) returns [] when IDLE, so during the (slow) compaction the bar is empty → "looks dead".
CC
CC sets a dedicated status setSDKStatus('compacting') (commands/compact/compact.ts) and shows a "Compacting conversation" spinner (screens/REPL.tsx) — i.e. compaction is a recognized busy/liveness state.
Options
(a) a CM _compacting flag surfaced to the liveness UI; (b) a new State.COMPACTING in the state machine.
Decision
(a) a CM _compacting flag — Follow CC's semantics. Mirrors the existing _advisor_busy precedent (another "busy while CM is IDLE" liveness handled by a flag, not a State); a new State wouldn't unify anything (advisor stays a flag) and would force a cancel/guard audit. The serial input loop makes re-entrant /compact structurally impossible, so nothing needs is_busy() to flip during compaction → flag is safe. The flag must register as busy in THREE places or the heartbeat ships present-but-frozen (still dead-looking): the _heartbeat_ticker run/blink condition (today state != IDLE), format_heartbeat's early-return + an actor branch ("Compacting · Ns" off phase_started_at), and _above_input passing the signal. Set the flag around the smart_compact call on both paths: manual compact() and auto _maybe_auto_compact() (inside _drive). Verify with a live /compact watch — a format_heartbeat unit test won't catch a frozen ticker.
