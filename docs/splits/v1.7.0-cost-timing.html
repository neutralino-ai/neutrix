v1.7.0 splits — usage, cost & timing

The last PRD before the v2.0.0 lock. Capture per-turn + cumulative token usage and dollar cost, surface them via a /cost command and a compact status-line readout, building on v1.5.2's per-turn llm_ms/tool_ms timing and the v1.6.1 dual-protocol stream parser.

Decisions are AGENT-MADE as architect under the user's explicit delegation ("you should not wait for my input. you make decision yourself", 2026-05-31) — NOT a turn-by-turn discussion. They are documented here so the user can veto any by reading this file; the high-stakes one (Split #10, deferring parallel tools + background Bash) carries a veto badge. See [[feedback-autonomous-goal-ship-gate]] (refined: present + "decide yourself" = consent) and [[feedback-think-like-architect]].

CC survey (/datafs/users/dingxf/anthropic/cc2/src): CC reads input_tokens/output_tokens/cache_creation_input_tokens/cache_read_input_tokens off stream events in updateUsage() (services/api/claude.ts:2924-2987, cumulative-per-message with >0 guards), sums across turns in accumulateUsage() (:2993-3038, cache tokens kept separate). Cost = a hardcoded per-model table in per-million-token units, 4 token classes priced separately (utils/modelCost.ts:26-142). No /cost command — cost shows in a session-exit summary (cost-tracker.ts:228-244) + a status line of cumulative totals (StatusLine.tsx:83-97). Timing = two buckets, API-duration + wall (logging.ts:686-688, state.ts), no per-message granularity. Cumulative state = a single mutable global STATE accumulator with per-model nested usage, getters sum on demand, saved/restored per session. Subagent costs do not auto-roll-up; the parent accumulates.

Split points

1.Usage capture — both stream protocolsdecided
Reality
neutrix drives TWO inbound shapes (v1.6.1): OpenAI chunks carry usage only in a final chunk and only if stream_options.include_usage was sent; the anthropic gateway carries usage in message_start.message.usage (input + cache) and message_delta.usage (cumulative output).
Options
(a) capture both, off the existing v1.6.1 parser seam; (b) OpenAI-only; (c) a post-hoc /cost that re-tokenizes locally (no API usage).
Decision
(a) capture BOTH in stream_response: extend the OpenAI path to read the final-chunk usage, and _apply_anthropic_event to read message_start/message_delta usage. The assembled LLMResponse gains a normalized usage field (Split #3). The v1.6.1 inbound parser is exactly the seam this extends — mirrors CC's updateUsage semantics without a native client.
2.stream_options.include_usage — some backends 400 on itdecided
Risk
Advisor/user flagged: not every OpenAI-compatible IHEP backend accepts stream_options.include_usage — some return HTTP 400. Sending it unconditionally would break streaming on those slots.
Decision
Send-and-catch, probe-once-per-slot. Send stream_options={"include_usage":true} only on the OpenAI path (gate on _uses_anthropic_messages_gateway() being false — anthropic slots get usage free in the SSE). On a slot's first turn, treat ANY 400 while it was set as "unsupported" (do NOT parse the opaque gateway error body for a param name), retry once without it, and cache the capability on the slot so it is one risky request at session start, not per-turn. Tests must cover both: a genuine non-include_usage 400 still propagates as an error, and the retry cannot double-yield tokens. No usage → "(usage unavailable)", never an error. (This touches the streaming/cancel/watchdog path — the riskiest code — so the probe is deliberately one-shot and cached.)
3.Token-class normalization across providersdecided
Shapes
OpenAI: prompt_tokens, completion_tokens, prompt_tokens_details.cached_tokens. Anthropic: input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens.
Decision
Store the RAW per-provider token counts; normalize to a 4-field Usage (input, output, cache_read, cache_write = Anthropic cache_creation; OpenAI has no cache-write → 0) for display. Cache accounting is provider-dependent and settled EMPIRICALLY before any pricing (capture one real usage payload per backend, like the v1.6.2 wiring check — read whether input + cache_read equals the reported total): typically OpenAI's prompt_tokens includes cached_tokens → fresh input = prompt_tokens − cached; Anthropic reports cache_read_input_tokens separately → no subtraction. Getting this backwards ships a plausible-but-wrong dollar figure (worse than none), so it is verified against a real payload, not docs. Missing fields → 0.
4.Cost source — pricing tabledecided
CC
Hardcoded per-model table, per-million units, 4 classes (modelCost.ts).
Options
(a) add the litellm dep; (b) vendor LiteLLM's model_prices_and_context_window.json as DATA + a tiny lookup; (c) hand-rolled table.
Decision
(b) — LiteLLM's pricing JSON as data, not the dependency ([[project-v2-v3-direction]] locked this). Ship a small curated subset (the models neutrix's slots actually point at) seeded from that JSON, per-million units like CC; refreshable. Cost = Σ over 4 classes (tokens/1e6 × price). Unknown model → "(cost unknown)", tokens still shown. No network fetch at runtime.
5.Cumulative accounting — where the ledger lives (v3-RL constraint)decided
CC
A single mutable global STATE accumulator (state.ts).
Constraint
[[project-v2-v3-direction]]: keep the ContextManager loop headless & pure for the v3 RL-environment vision — no TUI/global coupling in the core loop.
Options
(a) mutable field on ContextManager (CC-style global-ish); (b) a session-scoped CostLedger that observes completed LLMResponses + persists per-message usage to the v1.5.2 JSONL; (c) derive cumulative purely on demand from stored messages (no accumulator).
Decision
(b) a session-scoped CostLedger. Persist the RAW 4 token counts as the source of truth — NOT the computed dollar amount — to the v1.5.2 session JSONL (line shape settled in Split #11): prices in the curated table get corrected, and storing tokens lets /cost reprice old sessions retroactively, whereas storing dollars would freeze a possibly-wrong price. The ledger accumulates as turns complete, computes $ lazily on read, and is rebuilt from the JSONL on resume. It observes the loop (does not drive it), so the CM core stays headless/pure — (a) would couple the loop to a mutable global; (c) is purest but needs a store for every read and can't feed a live status line cheaply. Mirrors CC's accumulate-and-persist without the global.
6.Timing — reuse v1.5.2, add cumulativedecided
CC
API-duration vs wall, no per-message (logging.ts, state.ts).
Decision
Reuse the v1.5.2 per-turn llm_ms (API) / tool_ms (tool) already in the JSONL; the CostLedger rolls them up. Surface API time, tool time, and wall (session-elapsed) — three buckets, slightly richer than CC's two, at no extra capture cost since the fields already exist.
7.Surface — a /cost command (CC has none)decided
CC
No /cost — a session-exit summary + status line only.
Decision
Alternative / No CC analog — ADD /cost. neutrix is slash-command-driven (/help, /model, …) and has no exit summary, so an on-demand /cost is the natural surface: session totals (tokens by 4 classes, cost $, API/tool/wall time) + a per-model breakdown. Reuses the visibility-parity fold convention (a compact summary, expandable). Renders from the CostLedger.
8.Status-line readoutdecided
CC
Cumulative cost/tokens in the status line (StatusLine.tsx).
Decision
Add a compact cumulative readout to the v1.5.0 status bar — e.g. $0.0123 · 12.4k↑ 3.1k↓ — terse, no inline hint lines ([[feedback-ui-hints]]: keep it quiet, not chatty). Shown only once usage exists; hidden when (cost unknown).
9.Subagent cost attributiondecided
CC
Subagent costs don't auto-roll-up; the parent accumulates (forkedAgent.ts).
Decision
Roll subagent usage into the same session ledger (the store is the single source of truth and a subagent drives the same CM against the same store). A subagent turn's usage is persisted like any other; /cost shows the session total. Per-subagent attribution is deferred (YAGNI — one session-level number is what budgeting needs).
10.Parallel tools + background Bash — DEFER (the high-stakes call)veto?
Old plan
The roadmap still encodes v1.7.0 = parallel tools, v1.8.0 = cost/timing. This split reshapes that: cost/timing becomes v1.7.0 (this PRD); parallel + background Bash are deferred.
Evidence
Parallel tools: executor.py's v0.9.3 docstring already calls concurrent dispatch "an unused optimization" — the LLMs neutrix drives rarely emit multiple tool_calls per round, so the wall-clock win is mostly theoretical. Background Bash: a process registry + output capture + poll-tool + kill + lifecycle is a whole subsystem, against the "keep it simple, otherwise we never ship" North Star.
Decision
DEFER both; cost/timing = v1.7.0 = the last PRD before v2 (user-delegated architect call). Reopen post-v2 only if real OMILREC use demands it. This is the vetoable one — if you want parallel tools or background Bash inside the pre-v2 band instead of (or before) cost/timing, say so and the band re-shapes. The roadmap reshape (drop the old v1.7.0=parallel / v1.8.0=cost entries) lands with the v1.7.0 PRD acceptance.
11.Usage persistence — line shape in the JSONL (decided mid-implementation, 2026-05-31)decided
Discovery
Split #5 originally read "persist as namespaced JSONL extras on the message line (extending llm_ms/tool_ms)". Implementing the ledger surfaced a real choice the draft glossed: a turn's tool_ms is not known when the assistant message line flushes (tools dispatch after the assistant append), and the message-flush cursor lives in terminal_chat while the timing/usage live in the ContextManager — smearing per-turn accounting across an assistant line + N tool lines would force the data through MessageRecord or a fragile side-channel and make from_jsonl correlate multiple lines to rebuild one entry.
CC
No analog — CC persists a single global usage accumulator blob per session (state.ts), not per-message JSONL.
Options
(a) per-message extras on the assistant line (the #5 draft): append_message(..., llm_ms, tool_ms, usage_raw); (b) a dedicated {"type":"usage", …} line written once per completed turn (mirrors the existing append_tasks typed-line precedent); (c) a separate sidecar file.
Decision
(b) — a dedicated per-turn usage line. The CostLedger is the clean channel from the CM (which owns usage + timing) to persistence, and a bundled per-turn line lets CostLedger.from_jsonl rebuild one LedgerEntry from one line — no correlating assistant + tool lines, no widening the core MessageRecord type. load_session already ignores unknown line types, so resume is unaffected. terminal_chat persists new ledger entries via its own cursor beside _persist_new_records, and flushes once explicitly at turn completion (not only on the store.changes() tick) so the final turn's usage can't be lost to a future await reordering between the assistant append and ledger.record(). (c) is overkill — one file per session is the resume unit. This is the v0.9.2→v0.9.3 lesson applied: a persistence-shape split, recorded in the same change that makes it true.
