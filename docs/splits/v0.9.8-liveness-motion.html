v0.9.8 splits — CC-style liveness motion

Replace the heartbeat's single-cell brightness fade with a palette-independent on/off blink (CC's tool-loader wink), so the liveness pulse is robust on the user's tmux-256color terminal (and every other) instead of banding. Outcome: after surveying CC's shape-bloom and shimmer and prototyping both, the user tested four dot candidates and chose the on/off presence wink — the simplest palette-independent option, with nothing to interpolate. Re-affirms v0.9.4 #2 (single calm ●), replaces #13 (truecolor gradient → presence toggle), and reverses the v0.9.5 "120 Hz" cadence (→ 600 ms toggle).

Per .claude/rules/prd-compare-with-claude.md, this file is the durable record of the split-point discussion that happens before the PRD is drafted. CC source under /datafs/users/dingxf/anthropic/cc2/src. All six decisions below are now decided (settled turn-by-turn in chat). Labels: Follow CC / Alternative / User-directed different / No CC analog.

Why we're reopening v0.9.4

★ The diagnosis changed — not the taste root cause
Measured
The user's terminal is tmux-256color with COLORTERM unset, so prompt_toolkit resolves to 8-bit / 256-color. The truecolor gray gradient (60,60,60)→(255,255,255) quantizes onto only ~22 distinct grays per cycle (truecolor would give ~178). Because the raised cosine sits flat at trough/peak, those 22 changes bunch into the mid-slope as large jumps → reads as ~5 Hz and feels steppy. Raising the tick 10→120 Hz at v0.9.5 changed nothing because the palette, not the clock, is the ceiling.
Implication
v0.9.4 #2 (calm single ●, no shape change) made brightness the only animation channel; v0.9.4 #13 drove that channel with a truecolor gradient and assumed non-truecolor terminals "degrade gracefully via prompt_toolkit's fallback." The degradation is the bug. #2 and #13 are retroactively incompatible. The taste behind #2 (calm, uncluttered) still stands — what must change is the channel: from brightness (quantization-bound) to shape/position (palette-independent).
How CC avoids it
CC never breathes brightness. Its spinner animates shape (the ✻-family bloom, · ✢ ✳ ✶ ✻ ✽ forward+reversed — components/Spinner/SpinnerGlyph.tsx:7, utils.ts:6-10) and position (a 2-colour shimmer band sweeping the verb text — bridge/bridgeStatusUtil.ts:61-111). It also clamps down to 256-colour inside tmux on purpose (ink/colorize.ts:47-57), because default tmux won't re-emit truecolor SGR to the outer terminal — so "just force truecolor" is a fix CC explicitly rejected.
★ The forcing constraint: the liveness-gap on short labels survey
CC shimmer geometry
A 3-column band sweeps right→left one column per SHIMMER_INTERVAL_MS = 150, cycling every cycleLength = width + 20 ticks (so ~10 empty columns of lead-in/out → a deliberate pause between sweeps). bridgeStatusUtil.ts:21,61-67.
On a 3-char label
label "LLM" → width = 3 cycleLength = 3 + 20 = 23 ticks × 150 ms = 3.45 s / cycle band overlaps cols [0,2] only when index ∈ [-1,3] = 5 ticks = 0.75 s ⇒ label is STATIC ● LLM for ~2.7 s — ~78% of every cycle
For a CC spinner the bloom glyph animates through that gap. Strip the bloom and a shimmer-only neutrix indicator is frozen 78% of the time — worse than today's bug. CC's verbs are 8–13 chars on purpose (30–45% animated); neutrix's LLM / tool: x / cancelling… can't match without becoming verbose, which v0.9.4 #4 rejected (the phase name IS the debug affordance).
Consequence
The motion-channel choice (split #1) is forced, not aesthetic: either something blooms on the glyph during the gap, or the shimmer must run continuously (no CC pause), or the label grows. Short labels make a pure CC-geometry shimmer a non-starter.

Split points

1. Motion channel — what carries the animation? (FORCED) decided
CC behavior
Both at once: a shape-blooming glyph and a positional shimmer band over a long verb. The bloom guarantees continuous motion; the shimmer adds texture.
Alternatives
  • (a) Shape-bloom on the glyph — the ● cell cycles through shapes/sizes. Carries motion by itself, independent of label width. Palette-independent.
  • (b) Continuous shimmer on the label — drop CC's +20 padding so the band sweeps without a gap. Keeps the calm static ●, but on a 3-char label the band covers most of the word most of the time → reads as a flicker, not a sweep.
  • (c) Both (full CC) — bloom glyph + shimmer on a (necessarily longer) label.
Recommendation
(a) Shape-bloom on the glyph. neutrix's labels are too short for a shimmer to carry the animation without the 78%-frozen gap, and lengthening them re-opens the #4 debug-affordance trade we already lost. A shape-bloom on the single glyph cell is width-independent, palette-independent, and stays calm and uncluttered (honoring #2's taste). Shimmer can return as polish in a later release if a long-text surface ever appears — not now (YAGNI).
Decision
User-directed different — (d) presence blink. A new option that surfaced mid-discussion: toggle the dot's visibility (● ↔ blank) rather than its shape or position. The user pointed to CC's own components/ToolUseLoader.tsx + hooks/useBlink.ts (BLINK_INTERVAL_MS = 600), which winks a BLACK_CIRCLE on/off for in-progress tools — so this is Follow CC on the tool loader, not the spinner. Chosen by eye after testing candidates A–D in the user's terminal. A 2-state presence toggle has nothing to interpolate, so the 256-color quantization that caused the bug cannot occur. Supersedes the agent's shape-bloom lean (kept above for the record).
2. Bloom style — which shapes? (reopens v0.9.4 #2) decided
CC behavior
['·','✢','✳','✶','✻','✽'] cycled forward then reversed (12 frames) — a flower "blooming" open and closed. SpinnerGlyph.tsx:7, utils.ts:6-10 (platform-aware: Windows/Ghostty substitute *).
v0.9.4 history
Split #2 was user-directed: you rejected the ✻ family as "spinning frantically" and chose a single ● — "what would Steven Jobs do?" The calm goal is unchanged; only the channel must move off brightness.
Alternatives
  • (a) Calm dot size-bloom · • ● • · (or ·→∙→ •→●→⬤ and back) — a single dot gently growing and shrinking. Shape motion, no rotation, the calm cousin of CC's bloom.
  • (b) CC's ✻ family · ✢ ✳ ✶ ✻ ✽ forward+reversed — literal CC, busier.
  • (c) Keep static ● — only viable if split #1 picks shimmer to carry motion (rejected for short labels).
Recommendation
(a) Calm dot size-bloom. It satisfies both constraints that pulled in opposite directions: palette-independent shape motion (fixes the bug) and the calm, single-glyph aesthetic you chose at #2 (no frantic rotation). The ✻ family stays available if you now want the literal CC look. Final glyph set to be pinned against terminal/font support.
Decision
Superseded → single ●, no shape cycle. The size-bloom was the agent's pick and was previewed, but the user then chose the presence-blink (#1), so the glyph does not change shape — it winks on/off. This re-affirms v0.9.4 #2 (a single calm ●); only the smoothing method (#13) changes. Glyph stays ● (BLACK_CIRCLE), width-1, no jiggle.
3. Cadence — per-frame step (reverses v0.9.5 120 Hz) decided
CC behavior
Bloom advances one frame per 120 ms (12 frames = 1.44 s/cycle, SpinnerGlyph.tsx); shimmer one column per 150 ms (bridgeStatusUtil.ts:21).
Why it changes
Brightness fade needed many sub-steps, so v0.9.5 cranked the tick to 120 Hz (8.33 ms). Shape motion is paced per visible frame by the eye — at 120 Hz a 5-frame bloom would complete in ~42 ms (invisible). The loop reverts to a slow per-frame tick, which also drops the heartbeat's CPU cost ~10–20×.
Alternatives
  • 120 ms/frame — CC bloom parity.
  • 150 ms/frame — CC shimmer parity / slightly calmer.
  • Derive from a breath period: pick frame-count × interval to hold a ~2–4 s cycle (echoes v0.9.4 #12's 4 s breath).
Recommendation
Pace per frame, ~120–150 ms. Choose the bloom's frame count so one open-close breath lands in the calm 2–4 s band (e.g. a 9-frame dot-bloom × ~140 ms ≈ 2.5 s). Follow CC's per-frame ballpark; retire HEARTBEAT_REFRESH_HZ = 120 and the 480-frame table.
Decision
Follow CC — 600 ms toggle (1.2 s cycle), strict. Matches useBlink's BLINK_INTERVAL_MS = 600; tested at this rate (candidate C) and accepted. Reverses v0.9.5's HEARTBEAT_REFRESH_HZ = 120 — the loop now sleeps ~600 ms and flips a boolean, retiring the 480-frame table and ~70× the heartbeat CPU. Exposed as one constant (HEARTBEAT_BLINK_INTERVAL_MS) so it's a one-line dial if it ever reads too fast. Divergence: the blink phase is anchored to busy-entry (tick reset to 0 each IDLE→busy) so the first frame of a turn is always a visible ● — CC's useBlink uses a shared global clock and can start blank, tolerable for its list of per-tool dots but not for neutrix's single liveness dot.
4. Color model — drop the gradient (reopens v0.9.4 #13) decided
CC behavior
Two named theme colours; in the 256/ansi themes they are literally ansi:blue + ansi:blueBright (utils/theme.ts:202-203). The 2-colour shimmer switch is a hard flashOpacity > 0.5 threshold (GlimmerMessage.tsx:162) — no interpolation needed.
Alternatives
  • (a) Single named ANSI colour, shape carries motion — the bloom needs no colour change at all; one calm colour (e.g. neutrix's existing white) with nothing to quantize.
  • (b) Two named ANSI styles (e.g. normal + bold/bright) if a little brightness texture on top of the shape is wanted.
  • (c) Keep truecolor hex gradient — the thing that quantizes. Rejected.
Recommendation
(a) Single named ANSI colour. Once motion lives in the shape, the truecolor-vs-256 question simply goes away — there's no gradient left to quantize, so depth-adaptivity stops being load-bearing and we don't carry two render paths. Keep neutrix's neutral/white identity (blue is available if you'd rather match CC).
Decision
Single colour — bright white, presence toggle. With an on/off wink the dot is either shown (one colour) or blank, so there is no second colour and no gradient at all — the truecolor-vs-256 question is fully moot. Drop the hex gray gradient and the 480-shade table; keep neutrix's bright-white identity.
5. Stalled state — keep the red signal decided
CC behavior
Interpolates the message colour toward ERROR_RED by a continuous stalledIntensity (GlimmerMessage.tsx:87-92).
Alternatives
  • Discrete swap to a red ANSI colour while the bloom continues (neutrix's current discrete model, minus the gradient).
  • CC-style continuous interpolation toward red (needs truecolor; reintroduces the quantization problem).
  • Bloom + red blink attribute for extra urgency.
Recommendation
Discrete red ANSI swap, bloom unchanged. Same intent as CC (red = stalled) but via a named ANSI colour, not interpolation — consistent with split #4 and the v0.9.5 stalled-palette decision. The label still reads LLM (stalled).
Decision
Red ●, same on/off wink. When stalled the dot winks red (maps to CC's error colour) and the label reads LLM (stalled); cadence unchanged. Discrete white→red swap, no interpolation — consistent with #4 and the v0.9.5 stalled-palette decision.
6. Jitter — strict period now (revisits v0.9.4 #14) decided
CC behavior
Strict periodic step (setInterval), no jitter.
Why it changes
v0.9.4 #14 added ±10% tick jitter for an "organic" brightness fade. On shape/positional motion, time-jitter makes the bloom visibly speed up and slow down (and would make a shimmer band skip backward/forward in time) — it reads as broken, not organic. The #14 rationale was specific to brightness and doesn't carry over.
Recommendation
Follow CC — strict period. Drop the ±10% jitter for the new motion model.
Decision
Follow CC — strict period, no jitter. useBlink derives state from a strict clock (floor(time / interval) % 2). Jitter on an on/off wink would look erratic; v0.9.4 #14's organic-jitter rationale was specific to the brightness fade and does not carry over. Retire jittered_sleep from the heartbeat path.
