v0.9.5 splits — LLM error surface (stalled / timeout / inline error)

Two-layer feedback when the LLM API is slow or hung: a soft UI hint at ~5 s of silence, a hard abort at ~60 s. Pure addition to the v0.9.4 heartbeat + the existing LLM_ERROR_PREFIX catch in ContextManager._drive.

Per .claude/rules/prd-compare-with-claude.md, this file is the durable record of the split-point discussion that happens before the PRD is drafted. CC source under /datafs/users/dingxf/anthropic/cc2/src. Twelve split points are all now decided as of 2026-05-27; the v0.9.5 PRD's ## Claude Code reference section embeds the summary. Decision labels follow the rule: Follow CC / Alternative / User-directed different / No CC analog.

Tally: 3 Follow CC (#10, #11, #12) · 9 Alternative (#1–9). The Alternatives cluster around the non-streaming v0.9.x state: where CC's design is tuned for per-token streaming (3 s stall, fade animation, env-var-only config, SDK-level timeout), we adapt for one-big-await semantics (5 s, pulse on red palette, YAML per-slot config, background watchdog through the v0.9.3 cancel infrastructure). The watchdog choice (split #7) is the architecturally significant one: it makes the cancel and timeout paths share machinery, so v0.10.1 streaming can extend the same field (last_progress_at) and the same watchdog without redesign.

Context — what CC does, in one paragraph

★ CC's two-layer error surface, summarized survey
Soft stall hint
3 s of no streaming tokens → isStalled flag flips; spinner intensity rises and fades over a further 2 s; clears on the next chunk. Hardcoded threshold; not user-tunable. cc2/src/components/Spinner/useStalledAnimation.ts:42-44. Suppressed while a tool is active.
Streaming watchdog
Default 90 s of no SSE bytes → fetch controller aborted, APIConnectionTimeoutError("Request timed out") raised. Opt-in via CLAUDE_ENABLE_STREAM_WATCHDOG; threshold via CLAUDE_STREAM_IDLE_TIMEOUT_MS. cc2/src/services/api/claude.ts:1877-1878, :2334.
HTTP timeout
Default 600 s on the SDK client, set at construction. Configurable via API_TIMEOUT_MS env var. cc2/src/services/api/client.ts:144. Non-streaming fallback uses a tighter 120 s (remote) / 300 s (local) default. cc2/src/services/api/claude.ts:807-810.
Auto-retry
429 / 5xx / 529 → exponential backoff (BASE_DELAY_MS = 500; 2^attempt; capped at 32 s) with 25% random jitter. Retry-After header overrides. Timers reset per attempt; up to maxRetries attempts. cc2/src/services/api/withRetry.ts:55,543-547,696.
Error surface
A synthetic AssistantMessage with isApiErrorMessage: true is appended to the transcript — same render path as a real assistant message but flagged for analytics. Content is the literal string "Request timed out" for APIConnectionTimeoutError. cc2/src/services/api/errors.ts:169,435-442.
Cancel vs timeout
Same transcript path — both render as isApiErrorMessage synthetic messages — but different error classification ("aborted" vs "api_timeout") for analytics. User sees only the content text. cc2/src/services/api/errors.ts:967-977.

Context — what neutrix has today (v0.9.4)

★ neutrix LLM surface, summarized survey
LLM call
OpenAIChatLLM.stream_response at src/neutrix/llm.py:185 wraps self._client.chat.completions.create(...) in asyncio.ensure_future, awaits the task, yields one LLMEvent. stream=False (v0.9.3); no partial chunks.
No timeout
AsyncOpenAI(base_url=..., api_key=...) at llm.py:183 — no timeout=. Falls back to the openai SDK default ~600 s (10 min). This is the root cause of the user-observed "LLM dead, neutrix waiting forever" symptom.
Cancel path
OpenAIChatLLM.stop() at llm.py:226 calls task.cancel() on the in-flight create-task. The parked await in stream_response raises CancelledError; the ContextManager._drive loop at context_manager.py:358 catches it via _consume_cancel.
Error catch
context_manager.py:361-370 — any Exception from _call_llm becomes an assistant message "{LLM_ERROR_PREFIX}{err}]" with LLM_ERROR_PREFIX = "[LLM error: ". Returns from _drive; CM state → IDLE.
Heartbeat
format_heartbeat at src/neutrix/terminal_chat.py:220 reads state and renders a single breathing ● + a state-derived label (● LLM / ● tool: {name} / ● cancelling…). The breathing animation comes from HEARTBEAT_BRIGHTNESS_CYCLE; pure renderer, no new store fields. Today the same affordance shows whether the LLM is computing or silently hung.

Split points

1. Stall threshold — how long before the soft hint fires? decided
CC behavior
3 s of no streaming tokens. Hardcoded; calibrated for the streaming path where any silence longer than typical inter-token spacing is suspicious. useStalledAnimation.ts:42.
Alternatives
  • 3 s — mirror CC. Aggressive for our non-streaming path; ordinary fast-model responses (~2–4 s) may briefly flicker the stall affordance.
  • 5 s — skeleton PRD direction; a generous floor above typical fast-completion latency.
  • 8 s — only fires after most models would have answered. Lower false-positive rate but the user stares longer before they get a signal.
  • Configurable, default 5 s — env var NEUTRIX_LLM_STALL_HINT_MS for the slow-model power user; ships with a sensible default.
Recommendation
Alternative — 5 s, hardcoded for v0.9.5. CC's 3 s is calibrated for streaming; without streaming we'd flicker the hint on routine fast calls (DeepSeek-V3, gpt-4o-mini commonly answer in 2–4 s for short prompts). Keep it hardcoded for v0.9.5 to match CC's discipline; if usage shows a model regularly straddles the threshold, promote to an env var in a follow-up.
Decision
Alternative — 5 s, hardcoded. CC's 3 s was calibrated for streaming; our non-streaming path needs more headroom above routine fast-completion latency. Hardcoded to match CC's discipline; promote to env var only if usage shows the need.

Revised at acceptance (2026-05-28): the fixed 5 s flickered on healthy slow models (deepseek-v4-pro). Replaced by a derived threshold max(10, llm_timeout_s / 6) ≈ 50 s at the 300 s default — a single knob that scales the hint with the per-slot timeout. Code-only tiny tweak per user direction.
2. Stall hint visual — what changes when the hint fires? decided
CC behavior
Spinner intensity ramps over 2 s; subtle visual shift, no label change. The verb beneath the spinner stays the same. useStalledAnimation.ts:42-44.
Alternatives
  • Color shift only — heartbeat's truecolor gradient switches from amber to red; label unchanged. Mirrors CC most closely.
  • Label suffix only — ● LLM becomes ● LLM (stalled); color unchanged. Unambiguous in screen-readers / monochrome terminals.
  • Both — color shift AND label suffix. Glanceable + accessible.
  • Glyph swap — ● becomes something else (e.g. ○ or !). More intrusive than a color shift; risks reading as "broken".
Recommendation
Both. Color shift gives the glanceable signal ("something is different"); label suffix names what's happening ("it's stalled, not just slow"). Accessibility cost is near-zero since the label was already there. CC ships color-only because the spinner's verb text would be visually noisy to swap mid-cycle; our static LLM label has no such constraint.
Decision
Alternative — both color + label. Gradient shifts amber→red, label ● LLM becomes ● LLM (stalled). Glanceable + accessible. CC's color-only constraint (busy verb text) doesn't apply to our static label.
3. Stall hint smoothing — fade-in or hard switch? decided
CC behavior
Linear ramp over 2 s after the 3 s threshold. useStalledAnimation.ts:42-44. Gives a "creeping concern" feel rather than a startle.
Alternatives
  • Hard switch — at 5 s, the gradient palette is replaced atomically with the stalled palette. Simplest; matches v0.9.4's discrete state-driven approach.
  • Fade-in over 1–2 s — interpolate between the amber and red gradients. Mirrors CC's feel; one extra HEARTBEAT_STALLED_FADE_CYCLE table.
  • Pulse — keep the breathing animation but on a red gradient. Reads "stalled but still alive."
Recommendation
Alternative — pulse (continue breathing on red palette). The breathing rhythm reads as "we're still here, just waiting" and matches v0.9.4's Steve-Jobs-mode decision: the dot keeps breathing, only the palette changes. Mechanically: at stall, the tick lookup switches from HEARTBEAT_BRIGHTNESS_CYCLE to a parallel HEARTBEAT_STALLED_CYCLE. No fade, no new state machine. CC's fade is calibrated for streaming where the spinner already moves; for our discrete state-machine heartbeat, the palette swap is the clean analog.
Decision
Alternative — pulse on red palette. The breathing rhythm keeps going; only the palette swaps. No fade scaffolding. Matches v0.9.4's Steve-Jobs breathing-dot decision.
4. Stall detection mechanism — how do we know we're stalled? decided
CC behavior
A React effect compares Date.now() - lastTokenAt every render; the spinner already re-renders on a 50 ms animation clock, so the check is free. useStalledAnimation.ts:34-44.
Alternatives
  • Renderer-side math. Add awaiting_llm_since: float | None to ContextManager (or ChatStore); set on AWAITING_LLM entry, clear on exit. format_heartbeat reads it, computes time.monotonic() - since > 5.0, and chooses palette/label. No new tasks; the heartbeat already ticks at 100 ms.
  • Background timer task. asyncio.create_task(_stall_timer()) on state entry; sleeps 5 s; sets store.stalled = True; cancelled on state exit.
  • New lifecycle event. llm_stalled event with a reducer entry. Symmetric with v0.9.0's existing events.
Recommendation
Renderer-side math. The heartbeat already re-renders every ~100 ms; the check is one subtraction. No new background tasks (which means no new cancel surfaces). The timestamp lives on ContextManager next to state — set at the same point the state transitions to AWAITING_LLM — so the source of truth stays colocated. New events earn their keep when multiple consumers need them; here only the heartbeat reads the stall.
Decision
Alternative — renderer-side math, but with a forward-looking field name. The mechanism is the renderer-side computation as recommended, but the field is named last_progress_at: float | None (not awaiting_llm_since) so v0.10.1 streaming can bump it on each chunk arrival without renaming. Lives on ContextManager alongside state; heartbeat takes it as a parameter and computes monotonic() - last_progress_at > 5.0. v0.9.5 sets it once on AWAITING_LLM entry; v0.10.1 also bumps on stream chunks. Same field, same renderer, same math — semantics evolve. User instruction: "better align with future stream design \[…\] think like an Architect" (2026-05-27).
5. Hard timeout duration — how long before we abort? decided
CC behavior
90 s streaming-idle (opt-in); 120 s non-streaming remote / 300 s non-streaming local; 600 s HTTP-level upper bound. Multiple layers, all env-tunable.
Alternatives
  • 60 s — skeleton PRD direction. Aggressive vs CC; trades fast retry for occasional false-abort on legitimate slow calls.
  • 90 s — match CC streaming-idle.
  • 120 s — match CC non-streaming remote. Safer for slow models; longer wait under network failure.
  • Configurable, default 60 s — env var for the user who needs longer.
Recommendation
Configurable, default 60 s. The skeleton PRD's 60 s reflects daily-blocking pain from the openai SDK's 600 s default; this is the user-perceived "I'd rather see the error and resend than keep waiting" sweet spot. Slow models (large local quants, geographically distant inference) will need tuning, so ship the env var NEUTRIX_LLM_TIMEOUT_S at v0.9.5 rather than retrofitting later. CC's 90/120/300 layering needs to be replayed only if we re-enable streaming (v0.10.1).
Decision
60 s default (paired with split #6). Value is 60 s; configurability lives on the per-slot YAML field (split #6), not an env var. CC's 90/120/300 layering replays at v0.10.1 if streaming-idle gets its own threshold.

Revised at acceptance (2026-05-28): 60 s was too aggressive for slow hosted reasoning models — raised to 300 s default (CC's local-model tier), still aborting inside the SDK's 600 s read-timeout. Per-slot llm_timeout_s override unchanged. Code-only tiny tweak per user direction.
6. Configurability surface — env var, YAML, or both? decided
CC behavior
Env vars exclusively (API_TIMEOUT_MS, CLAUDE_STREAM_IDLE_TIMEOUT_MS, CLAUDE_ENABLE_STREAM_WATCHDOG); no settings.json keys for these.
Alternatives
  • Env var only — match CC. Fast to override per-session.
  • YAML config in our existing slot file — per-slot timeout (slow local model gets a longer timeout than a hosted-API slot).
  • Both — YAML for defaults, env var override. Matches the openai SDK's own pattern.
  • Neither — hardcoded — fewest surfaces, but doesn't survive the slow-model use case.
Recommendation
Env var only for v0.9.5. YAML per-slot timeout is the right long-term home but pulls in slot-schema migration (existing user configs would need a default fallback) — out of scope for a daily-blocking fix. Add YAML in a follow-up if the slow-model audience grows. Matches CC's discipline of "env vars for operator knobs; settings.json for product behavior."
Decision
Alternative — YAML config (per-slot). Add a llm_timeout_s field to the slot YAML schema; default 60 s when absent. Per-slot tuning is the natural home (a slow local model gets a different timeout than a hosted-API slot); env var deferred. Slot-schema migration is additive (existing configs fall back to default), so the "schema migration burden" the original recommendation feared is small. Diverges from CC.
7. Hard timeout enforcement path — where does the abort fire? decided
CC behavior
Two parallel mechanisms: (1) SDK client constructed with timeout: 600_000 (client.ts:144) — fires inside the SDK, raises APIConnectionTimeoutError; (2) explicit setTimeout watchdog on the stream (claude.ts:1877) aborts a fetch AbortController on idle. Both bubble up to the same error renderer.
Alternatives
  • SDK timeout= at create-call. Pass timeout=60.0 to self._client.chat.completions.create(...) at llm.py:204. SDK raises openai.APITimeoutError after 60 s; existing except Exception at context_manager.py:361 catches and renders via LLM_ERROR_PREFIX. Zero new code paths.
  • asyncio.wait_for(task, 60) in stream_response. Same outcome but the timer lives in our code, not the SDK's. Useful if the SDK timeout isn't honored by some backend; otherwise duplicate plumbing.
  • Background watchdog that calls self.stop(). A create_task(_timeout(60)) on AWAITING_LLM entry fires OpenAIChatLLM.stop() after 60 s, which the parked await reads as CancelledError. Hijacks the cancel path; would render as [interrupted] by accident unless we distinguish.
Recommendation
SDK timeout= at create-call. Pass the timeout per-request (not at client construction) so an env-var change takes effect on the next call without rebuilding the client. The SDK raises a typed exception which the existing except Exception catch surfaces as LLM_ERROR_PREFIX — clean reuse of v0.9.3's error path. Avoids hijacking the cancel pathway (split #9 stays semantically distinct from cancel).
Decision
Alternative — background watchdog that calls cm.request_cancel(reason='timeout'). On AWAITING_LLM entry, spawn asyncio.create_task(_llm_timeout_watchdog()) that sleeps slot.llm_timeout_s (default 60 s) then invokes the same cancel infrastructure that user-Esc uses, carrying a reason='timeout' flag. Watchdog is cancelled on AWAITING_LLM exit (success, error, or user-cancel). Forward-compatible with v0.10.1's streaming-idle: the watchdog gets bumped on each stream chunk arrival, so "no chunk for 90 s" reuses the same machinery. Reshapes split #9 — the transcript distinction must now ride the reason flag, since both paths converge on stop(). User instruction: align with future stream design (2026-05-27).
8. Error message wording — what does the transcript actually say? decided
CC behavior
Literal "Request timed out" as the synthetic AssistantMessage body. cc2/src/services/api/errors.ts:169. Terse, no duration, no recovery hint.
Alternatives
  • Match CC verbatim — [LLM error: Request timed out] via the existing LLM_ERROR_PREFIX. Symmetric with other LLM errors.
  • Include duration — [LLM timeout after 60s]. User knows exactly how long they waited and whether the threshold is too aggressive.
  • Include recovery hint — [LLM timeout after 60s. Resend your message or set NEUTRIX_LLM_TIMEOUT_S.]. More verbose but actionable.
  • Drop the prefix entirely — render as a plain assistant message. Cleanest but loses the visual marker that "this came from the system, not the LLM".
Recommendation
Alternative — [LLM timeout after Ns]. Uses the same [LLM…] bracketed-system-message shape as LLM_ERROR_PREFIX so the visual category is preserved, but distinguishes "timed out" from "errored" and states the duration. Skip the recovery hint — users who care about the env var find it in --help; the message should be glanceable, not a tutorial. CC's terse style without the duration would leave users wondering "was that 30 seconds or 10 minutes?"
Decision
Alternative — [LLM timeout after 60s]. Bracketed-system-message shape preserves visual category; duration baked in. No recovery tutorial in the body.
9. Cancel vs timeout — same transcript path or distinct markers? decided
CC behavior
Both render as isApiErrorMessage synthetic messages. Different error classification ("aborted" vs "api_timeout") for analytics, but the user sees only the content text — and the content text differs ("Request was aborted" vs "Request timed out"). Same rendering path, different bodies. errors.ts:967-977.
Alternatives
  • Distinct markers (Recommended) — cancel appends [interrupted by user] (v0.9.3, unchanged); timeout appends a fresh assistant message [LLM timeout after Ns]. User can scroll back and immediately see which path fired.
  • Unified [interrupted: reason] — [interrupted by user] stays; [interrupted: timeout] for the abort. More symmetric but loses the duration.
  • Same body — both write [LLM error: …]. Simplest; user reads the exception text to distinguish.
Recommendation
Distinct markers. Cancel and timeout are semantically different: cancel is the user's intentional steer (v0.9.3 keeps the partial in-flight tool intent in history for LLM re-read); timeout is "the LLM never spoke." The v0.9.3 cancel marker lives on the prior tool call or assistant message; the timeout marker is a fresh assistant message with the duration baked in. Different surfaces, different uses.
Decision
Alternative — reason flag on request_cancel(). cm.request_cancel(reason: Literal['user', 'timeout'] = 'user'). CM stores self.cancel_reason; the CancelledError catch in _drive reads it: user → v0.9.3's [interrupted by user] marker; timeout → fresh assistant message [LLM timeout after 60s] (split #8). Single broadcast path through llm.stop(); distinction is on the receiving side. Reshape consequence of split #7.
10. Auto-retry — do we retry transient failures? decided
CC behavior
Yes — withRetry wraps the API call: 429 / 5xx / 529 → exponential backoff with jitter; Retry-After honored; up to maxRetries attempts. cc2/src/services/api/withRetry.ts:696. The openai Python SDK has analogous max_retries=2 default on the client.
Alternatives
  • None for v0.9.5 — skeleton PRD non-goal. Hard abort fires on the first failure; user resends.
  • SDK default (2 retries) — don't override max_retries; let the openai SDK retry 429/5xx invisibly. Free, since the SDK already does it.
  • Custom retry policy — match CC's exponential + jitter + Retry-After.
Recommendation
SDK default (2 retries). The openai Python SDK already retries 429 / 5xx with backoff (AsyncOpenAI.max_retries = 2 by default). Leaving it on is free; the only adjustment is making sure our 60 s timeout is the total envelope (per-attempt 60 s × 3 attempts would be 180 s end-to-end, defeating the point). Set max_retries=0 if our 60 s should be the hard ceiling, or leave default if we want the retries inside the envelope. Either way, document the choice. Skeleton PRD's "None" is a defensible alternative; SDK-default is strictly less work.
Decision
Follow SDK default — 2 retries. Because the timeout enforcement (split #7) is our external watchdog, not the SDK's timeout=, the SDK retries happen inside our 60 s envelope. A fast 5xx-retry cycle finishes in <15 s; only true silent hangs consume the full envelope. Clean composition: SDK handles transient HTTP failures; our watchdog handles silent hangs.
11. Stall hint during AWAITING_EXECUTOR — show or suppress? decided
CC behavior
Suppressed — isStalled = timeSinceLastToken > 3000 && !hasActiveTools. Tools run on a different latency budget; a 30 s test run shouldn't paint the heartbeat red. useStalledAnimation.ts:42.
Alternatives
  • Suppress (mirror CC) — stall hint and hard timeout apply only in AWAITING_LLM. Tool execution has its own affordance (the executor's stdout line-streaming) and its own concept of "stuck."
  • Apply to AWAITING_EXECUTOR too — a long-running tool would also trigger the red heartbeat after 5 s. Unifies the affordance but creates false positives for legitimate slow tools.
Recommendation
Follow CC — suppress in AWAITING_EXECUTOR. The awaiting_llm_since timestamp is set only on AWAITING_LLM entry; AWAITING_EXECUTOR doesn't get a sibling timestamp. Tool stall is a different problem (covered by the v0.9.2 tool-cancel path + the tool's own progress reporting) and warrants its own design if a user reports the pain.
Decision
Follow CC. Stall hint and hard timeout apply only in AWAITING_LLM. last_progress_at and the timeout watchdog are set/spawned only on AWAITING_LLM entry. Tool stall is a future problem with its own design.
12. Logging — what does the operator see in the log? decided
CC behavior
Stalls log nothing; timeouts log at error via the central error reporter (logError). claude.ts:2334, errors.ts:169.
Alternatives
  • Silent stall, ERROR timeout. Mirrors CC. Stall is just UI noise; timeout is an operator-relevant event.
  • WARN stall, ERROR timeout. Easier forensics — "the user complained about a stall at 14:32" becomes greppable.
  • Silent both. Smallest log footprint; rely on the user's transcript for post-mortems.
Recommendation
WARN stall, ERROR timeout. The stall hint is rare enough (5 s threshold) that the log spam is bounded; making it greppable is worth the line cost when debugging slow-model setups. Use the project's loguru convention (logger.warning("LLM call stalled at {}s", elapsed) and logger.error("LLM call timed out after {}s", elapsed)).
Decision
Follow CC — silent stall, ERROR timeout. Stall is UI noise; timeout is operator-relevant. logger.error("LLM call timed out after {}s", elapsed) on watchdog fire; no log line on stall hint entry/exit.
