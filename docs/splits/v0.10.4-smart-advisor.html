v0.10.4 splits — Smart Advisor (third actor)

A third actor that watches the store, makes its own cheap LLM call on a trigger, and either revises the task list (via Task tools) or injects a judged <advisor> suggestion the main LLM sees next turn.

Durable record. Decided autonomously under a delegating /goal. CC analog: the auto-dream / extractMemories / promptSuggestion turn-end forked actors (cc2/src/services/autoDream/, extractMemories/, PromptSuggestion/) — same shape (turn-end hook, cheap gate, own LLM call, surface via inline message + state mutation, run between turns), different domain (task-list judgment).

Split points

1.Trigger mechanism + cadencedecided
Decision
Follow CC — turn-end hook. The Advisor is invoked from TerminalChat._worker_loop after a turn completes (the neutrix analog of CC's stop hook). Triggers: periodic (every N=5 completed turns) + explicit /advise; a run-lock drops re-entry; a session cap (20 auto-runs) then auto-trigger stops (/advise still works). Idle-timer trigger deferred (needs a background task coordinating with the lock + IDLE-state requirement) — record, not infeasible.
2.Awaited vs fire-and-forgetdecided
CC
Fire-and-forget + an apply-only-when-idle guard / input lock.
Decision
Alternative — awaited at turn-end (race-free in neutrix's single loop). _input_loop and _worker_loop are decoupled, so the user keeps typing into the queue during the advisor call; only the next turn's processing waits a beat (occasional, fast-slot). Awaited avoids the lock + idle-guard machinery fire-and-forget would need. The advisor call is wrapped like handle_event (its failure never kills the worker), and runs only after a turn that completed (skip after a cancelled turn — the user just interrupted).
3.Injection path — the single-mutator invariantdecided
The landmine
context_manager.py is "the only layer that mutates messages and ChatStore" — v0.10.3 just reinforced this. An Advisor doing ctx.messages.append(...) breaks it and desyncs messages/store/renderer.
Decision
Route injection through CM. Advisor.run_once(...) is side-effect-free — it makes the LLM call and returns an AdvisorOutcome(task_calls, suggestion). The orchestrator applies task mutations via tools.dispatch(store=...) (Task tools mutating the store IS the intended path) and injects the suggestion via a new CM method inject_advisor_message(text) that writes messages + store record in lockstep (mirrors _append_user_message), run while CM is IDLE (turn-end guarantees it). This also makes both channels trivially mock-testable.
4.Marker handling — recall exclusion + distinct renderdecided
Decision
Treat <advisor> like the reminder/compact markers. (a) Add is_advisor_message(content) and include it in _is_real_user_prompt so Up-arrow recall (v0.9.7) never surfaces advisor pseudo-turns as user input. (b) Add an <advisor> branch to _render_record (before the user branch) → an expanded dim ↳ advisor: notice with the advice body, distinct from the v0.8.0 folded reminder. Missing either site is a real bug (caught in review).
5.Advisor's LLM / slotdecided
Decision
Fresh client, fast slot (Follow CC intent). Build a new OpenAIChatLLM for the advisor (the v0.10.0 don't-mutate-the-shared-client lesson — never switch ctx.llm and switch back). Use the configured fast slot if present, else the active slot. run_once takes an injected llm so tests pass a stub.
6.Rendering advisor Task-tool callsdecided
Decision
Folded one-line audit (locked PRD proposal). An advisor task mutation renders as a dim ↳ advisor: marked #3 completed notice — an audit trail without the visual weight of full tool blocks. The suggestion message renders expanded (split #4).
7.Tag name + guardrailsdecided
Decision
<advisor>…</advisor> (locked; distinct from <system-reminder>). The advisor system prompt carries non-optional guardrails: review-and-suggest don't-execute; do not create/update/delete yourself or your scheduling; do not call /advise from within an advisor response; only Task tools.
