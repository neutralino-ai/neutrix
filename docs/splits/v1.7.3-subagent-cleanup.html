v1.7.3 splits — sub-agent HTTP-client cleanup

Stop the recurring RuntimeError: Event loop is closed ("Unhandled exception in event loop") that fires after each Agent sub-agent finishes.

Agent-decided under "decide yourself" ([[feedback-investigate-decide]]); CC-surveyed (/datafs/users/dingxf/anthropic/cc2/src), advisor-reviewed. Diagnosis (confirmed): each sub-agent runs on its own asyncio.run() loop on a worker thread (tools.py › run_subagent dispatch) with its own AsyncOpenAI client (tools.py:641) that is never closed. When the sub-agent's loop closes, the lingering proxied keep-alive connections in httpx's pool tear down and schedule their close (loop.call_soon) on the dead loop → the error (the http_proxy.py frames = the user's local proxy). Cosmetic (fires after the result returns) but noisy + recurs per sub-agent. See docs/PRDs/v1.7.3-subagent-cleanup.md.

Split points

1.Fix shape — match the pool's lifetime to its loopdecided
CC
CC (Node/Bun) has one process-wide event loop; sub-agents (utils/forkedAgent.ts) are async tasks on it, cancelled via AbortController — they never spin up / tear down their own loop or client. So CC is structurally immune; it never closes a loop mid-process.
Options
(a) re-architect to CC's single-loop model — run sub-agents as tasks on the main loop; (b) aclose() the sub-agent client inside its own loop (in run_subagent's finally); (c) leave it (cosmetic).
Decision
(b) — aclose within the loop. Add async def aclose(self): await self._client.close() to OpenAIChatLLM and call it in run_subagent's finally, so the httpx pool closes on the live sub-agent loop before asyncio.run() returns — nothing lingers to tear down on a dead loop. (a) rejected: neutrix deliberately isolates sub-agents on their own loop/thread (subagent.py) for clean cancellation; adopting CC's single-loop model would undo that isolation + rework cancellation, for a cosmetic error. Match semantics, not CC's runtime shape ([[prd-compare-with-claude]]). (c) rejected — recurs per sub-agent + leaks teardown.
2.Keep-alive — keep it or disable it on sub-agent clientsdecided
CC
CC embraces keep-alive: "Bun's fetch shares a keep-alive connection pool globally, so the real API request reuses the warmed connection" (utils/apiPreconnect.ts) — it even pre-warms the handshake. (It does skip the preconnect under a proxy/mTLS, since the SDK's custom dispatcher won't share the global pool — the proxy is the aggravating factor here too.)
Options
(a) keep keep-alive; (b) disable it on sub-agent clients (httpx.Limits(max_keepalive_connections=0)) so connections close within each request.
Decision
(a) keep keep-alive — Follow CC. It's a real latency win, and once Split #1's aclose closes the pool inside the loop there's nothing dangling to misfire. Disabling keep-alive (b) is a valid backstop but trades the perf for redundancy with the real fix; not adopted (revisit only if the error survives the aclose).
3.Main-session client teardown (hygiene)decided
Now
The long-lived main client also lives on a loop (the session's asyncio.run(run_async)) that closes at process exit — so the same cosmetic error can fire once at shutdown.
Decision
Also aclose the main client in terminal_chat.run_async's finally. Lower stakes (only at exit, not per-sub-agent) but the same one-liner; closes the loop-teardown error end-to-end.
