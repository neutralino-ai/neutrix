v0.10.0 splits — subagent framework (Agent tool)

An Agent tool that spawns a fresh-context worker (its own ContextManager + message list + tool set), runs the LLM/tool loop to completion, and returns only the final text — so the parent's context grows by one tool_result instead of by the whole sub-task. The structural answer to context-explode.

Per .claude/rules/prd-compare-with-claude.md, this is the durable record of the split discussion that precedes the PRD. CC source under /datafs/users/dingxf/anthropic/cc2/src. Authored and decided autonomously under a /goal that delegated all human options ("you make decisions for me, including whether or not to push"). Every Decision below is the agent's own call, recorded for later audit. Labels: Follow CC / Alternative / User-directed / No CC analog.

Split points

1.Tool schema — how widedecided
CC behavior
10-field schema (description, prompt, subagent_type, model, run_in_background, name, team_name, mode, isolation, cwd). AgentTool.tsx:82-138.
Alternatives
  • Mirror CC's full schema (carries fields for features we don't ship).
  • Minimal CC-shaped core: description, prompt, subagent_type="general-purpose".
Recommendation
Minimal CC-shaped core — the three fields that map onto v0.10.0 scope; model/run_in_background/isolation/teammate fields are non-goals (YAGNI).
Decision
Alternative — minimal CC-shaped core. Agent(description: str, prompt: str, subagent_type: str = "general-purpose"). Same field names/semantics as CC's first three; the rest are deliberately omitted as out-of-scope. subagent_type is accepted but only general-purpose is valid in v0.10.0 (others → error), so the field is forward-compatible without shipping specialization.
2.Execution model — own ContextManager vs bespoke loopdecided
CC behavior
A forked context drives the same query engine the parent uses; not a separate loop implementation. forkedAgent.ts.
Alternatives
  • Spin up a full ContextManager instance per subagent (reuse cancel/round/tool-dispatch).
  • Write a thin standalone LLM/tool loop in subagent.py (no CM).
Recommendation
Reuse ContextManager (DRY/SOLID): the LLM round, tool dispatch, cancel-and-wait, and tool-round-boundary logic already live there and are tested. A bespoke loop would duplicate that surface and drift.
Decision
Follow CC (semantics) — reuse ContextManager. run_subagent() builds a fresh ContextManager with its own ChatStore, the subagent system prompt, the filtered tool set, and the parent's llm/executor; submits the user prompt; drives the existing round loop until IDLE with a final assistant text + no pending tool calls; returns that text. No new loop is written.
3.Tool allowlist — what the subagent may calldecided
CC behavior
Inherits parent tools minus a disallow-set; Agent itself is blocked (no recursion). agentToolUtils.ts:70-116, recursion guard AgentTool.tsx:332.
Alternatives
  • Inherit the full parent tool set minus Agent.
  • Hand-pick a read-only subset (Read/Grep/Glob/…).
Recommendation
Inherit parent tools minus Agent — matches CC, keeps the worker capable, and the single removal enforces the single-level non-goal.
Decision
Follow CC, with a runtime backstop. The subagent's ContextManager gets a tool_names allowlist = parent builtin tools minus Agent; _call_llm advertises only those schemas, so the subagent never sees the tool that spawns another (recursion structurally impossible). Belt-and-suspenders for the rare model that calls an unlisted tool: a threading.local "inside a subagent" flag set across run_subagent and checked at the top of the Agent tool func — if already inside, return an ERROR: string (~5 lines, closes the single-level non-goal even if scoping is bypassed).
4.Return packaging + size capdecided
CC behavior
Final assistant text only, capped at 100_000 chars; metrics travel out-of-band, not in the tool_result text. agentToolUtils.ts:229,276.
Alternatives
  • Return final text only (PRD), uncapped.
  • Return final text only, capped to protect the parent's context (the whole point of subagents).
  • Return text + structured metrics inline.
Recommendation
Final text only, capped. An uncapped return defeats the context-budget goal if the subagent's answer is huge; metrics belong in the (separate) render summary, not the LLM-visible result.
Decision
Follow CC. SubagentResult(final_text, turn_count, cancelled, error) is the internal struct; the tool_result content the parent LLM sees is final_text only, truncated to SUBAGENT_MAX_RESULT_CHARS = 100_000 with a "…[truncated]" suffix when exceeded. turn_count/chars feed the render summary (split #5), not the LLM.
5.Rendering at v0.10.0 — before v0.10.2's Ctrl+E fold existsdecided
CC behavior
Folds the subagent's messages inline, default-collapsed, with a summary line and an expand affordance. AgentTool.tsx:1084-1122.
The constraint
The PRD wants a default-folded block with Ctrl+E expand "per visibility parity" — but that fold machinery is v0.10.2, which ships after this release (strict numeric tag order, so v0.10.0's tag must precede v0.10.2's). The fold widget does not exist yet.
Alternatives
  • Print the subagent's full transcript inline, unfolded (noisy; the opposite of the context-saving intent visually).
  • Print a one-line forward summary now; retrofit the Ctrl+E fold/expand when v0.10.2 lands.
  • Build a throwaway minimal fold now, replaced at v0.10.2.
Recommendation
One-line forward summary now (the v0.9.6 /compact / v0.9.7 /rewind notice pattern), fold retrofit at v0.10.2. Smallest correct surface; no throwaway widget.
Decision
Alternative — defer ALL bespoke subagent rendering to v0.10.2 (refined post-advisor). In v0.10.0 the Agent tool's final text is returned as the tool_result and renders through the existing tool-result path (foldable like any tool result) — zero terminal_chat.py changes this release. The dedicated folded block with turn count / Ctrl+E is v0.10.2 (visibility parity), which owns the fold machinery and will extend run_subagent's return shape as needed. The originally-planned ↳ subagent — N turns notice is dropped from v0.10.0 because turn-count is not carried on the append-only tool-result path and plumbing it now is throwaway. See split #12 for why the full per-round transcript is not retained.
6.Live progress vs batch while the subagent runsdecided
CC behavior
Sync agents stream progress live per message. AgentTool.tsx:1111.
Alternatives
  • Stream the subagent's rounds into the parent transcript live.
  • Show only a "working" liveness indicator while it runs; print the summary on completion.
Recommendation
Liveness indicator only. Live streaming needs the v0.10.1 streaming path and the v0.10.2 fold to not be a wall of text; both ship later. The PRD already lists "streaming subagent output" as a non-goal.
Decision
Follow CC intent, deferred mechanism. While the subagent runs, the parent's existing v0.9.8 heartbeat shows liveness (state stays AWAITING_EXECUTOR during the tool call, which already drives the dot). No live sub-transcript in v0.10.0. The summary notice (split #5) is the only new print. Matches the PRD non-goal "streaming subagent output to the parent."
7.Cancellation — Esc during a subagent rundecided
CC behavior
Sync agent shares the parent AbortController; ESC aborts the child; the tool call ends. AgentTool.tsx:1127.
Alternatives
  • Esc cancels the whole parent turn (subagent dies with it, no result).
  • Esc cancels only the subagent; the parent receives [cancelled by user] as the tool_result and continues under cancel-as-steer (v0.9.3).
Recommendation
Cancel-the-subagent-then-steer — matches the PRD Goal #4 and reuses the shipped v0.9.3 cancel-as-steer semantics, so a runaway subagent is interruptible without losing the parent turn.
Decision
Follow CC + reuse v0.9.3 (data flow corrected post-advisor). Because tool dispatch runs in a worker thread (asyncio.to_thread) with the subagent on its own event loop, an asyncio.Event can't cross loops — the token is a threading.Event registered with the parent Executor. On Esc, _do_cancel calls executor.cancel() (which sets the event) and cancels the parent drive task — so the parent's await to_thread(...) raises CancelledError and the Agent future is abandoned. The [cancelled by user] the parent LLM sees does NOT come from the Agent return — it comes from _do_cancel appending [interrupted by user] + _ensure_tool_result_pairing synthesizing the tool_result at next send (v0.9.3, free). The watcher's only job is to stop the subagent's token/subprocess burn: it polls the threading.Event and calls subagent_cm.cancel(). run_subagent detects cancel via the event after handle_event returns (the subagent CM swallows CancelledError internally), and returns SubagentResult(cancelled=True) — meaningful only on the non-Esc path; on Esc its return is discarded with the abandoned future. No parent-side cancelled-delivery plumbing is built (it would be dead code).
8.Runaway guard — max turnsdecided
CC behavior
Per-agent maxTurns; on limit the loop exits and finalizes a partial result. runAgent.ts:756,773.
Alternatives
  • No cap — trust the model to finish (a tool-loop bug burns tokens unboundedly).
  • A default max-turns cap; on hit, stop and return the best final text + a note.
Recommendation
A cap. A subagent is unattended-by-the-user delegation; an uncapped tool loop is the exact runaway the feature must not introduce. CC caps; we should too.
Decision
Follow CC. SUBAGENT_MAX_TURNS = 25 LLM rounds (a turn = one assistant message, whether text or tool_calls). On reaching it, stop the loop, take the last assistant text (or a "[subagent hit the {N}-turn limit before finishing]" note if none), and return it. Logged via loguru at warning.
9.Error handling inside the subagentdecided
CC behavior
Recover a partial text result if any assistant messages exist, else re-throw. AgentTool.tsx:1220-1234.
Alternatives
  • Let the exception propagate and fail the parent turn.
  • Catch, return SubagentResult(error=...), surface as a tool_result so the parent LLM can react.
Recommendation
Catch and surface. A subagent failure is data for the parent LLM ("the worker errored, try another approach"), not a reason to crash the parent turn.
Decision
Follow CC. run_subagent wraps the loop; on exception it returns SubagentResult(error=str(exc), final_text=<best-so-far or "">). The Agent tool maps a non-null error to a tool_result of [subagent error: {error}] (plus any partial text). The parent turn continues.
10.General-purpose subagent system promptdecided
CC behavior
Each built-in agent type has its own getSystemPrompt(); general-purpose is the catch-all. loadAgentsDir.ts.
Alternatives
  • Reuse the parent's full system prompt verbatim.
  • A dedicated, shorter general-purpose worker prompt.
Recommendation
A dedicated worker prompt. The parent prompt is full of UI/REPL framing irrelevant to a headless worker; a focused prompt that states "your final message IS the return value — return raw findings, no preamble" gets cleaner results (CC tells subagents exactly this).
Decision
Follow CC (semantics). A module-level SUBAGENT_SYSTEM_PROMPT constant in subagent.py: general-purpose worker, has tools, runs to completion, and is told its final assistant message is consumed verbatim as the tool result (so: return the answer/data directly, no "I'll now…" preamble, no questions back to the user — it cannot ask).
11.Which model/slot the subagent usesdecided
CC behavior
Inherits the parent model unless a model override is given. AgentTool.tsx schema.
Alternatives
  • Reuse the parent's active slot/LLM.
  • Accept a per-call model override (a "fast" slot for cheap delegation).
Recommendation
Reuse the parent's active slot. The model override is a CC field we dropped in split #1 (YAGNI); a fast-slot delegation is a clean v1.x follow-up once subagents earn their keep.
Decision
Alternative — reuse parent slot. run_subagent receives the parent's llm instance; the subagent's ContextManager drives the same active slot. No model selection in v0.10.0. (Recorded as a v1.x idea: a subagent_slot config for cheap delegation.)
12.Does the subagent's per-round transcript survive the call?decided
The catch
run_subagent builds a local ChatStore and discards it — only final_text escapes. So a later release cannot fold "the full subagent transcript" unless v0.10.0 persists it now. (Surfaced by the advisor: silently assuming retention while building discard is the trap.)
Alternatives
  • (a) Persist the subagent's messages now (side-band, e.g. on the tool result's MessageRecord.extra) so v0.10.2 can fold the real transcript.
  • (b) Retain nothing; v0.10.2 folds final-text (+ metrics it re-derives); full per-round transcript deferred to v1.x.
Recommendation
(b) — YAGNI. Side-band transcript persistence is speculative plumbing for a fold that doesn't exist yet; the final text is the contract that matters. v0.10.2 can re-shape run_subagent's return when it actually builds the fold.
Decision
Alternative (b) — retain nothing in v0.10.0. The subagent's ChatStore is local and discarded; only final_text (capped, split #4) is returned. Goal #3 is downgraded accordingly: v0.10.0 renders the result as a normal tool result; v0.10.2 owns rich subagent rendering and will extend the return shape if it needs turn-by-turn data; full transcript retention is a v1.x decision.
