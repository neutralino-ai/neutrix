# v0.3.1 â€” onboarding arrow-key nav + tap-to-confirm Ctrl+C quit

## Problem

In the v0.3.0 onboarding TUI:

1. Focus moves between api_key inputs and model rows only via `Tab` /
   `Shift+Tab`. The first user expected `Up` / `Down` arrow keys to work
   too â€” they don't, so navigation feels broken.
2. The only "leave without saving" key is `q`, but `q` only fires when
   focus is on a model row (Inputs swallow letter keys). Users typing in
   an Input can't quit without first tabbing out.
3. The terminal-native `Ctrl+C` muscle memory is the right answer for
   "get me out of here," but a single accidental `Ctrl+C` shouldn't drop
   a half-entered api_key on the floor. A two-tap confirm is the
   recognised pattern.

## Goal

- Bind `Up` / `Down` to the standard Textual `focus_previous` /
  `focus_next` actions on `OnboardApp`, so arrow keys traverse the same
  focusable widgets that Tab traverses.
- Bind `Ctrl+C` as the universal escape hatch, with two-tap confirm:
  - 1st `Ctrl+C` â†’ toast: "press Ctrl+C again to quit, Esc to cancel".
  - 2nd `Ctrl+C` within 5 s â†’ exit (result `False`).
- Bind `Esc` to cancel the pending quit while it's armed; no-op otherwise.

## Non-goals

- Up/down inside the `Input` widget (single-line; arrows are unused).
- A graphical modal dialog for the confirm â€” a toast is enough.
- Routing `Ctrl+C` through the chat TUI (out of scope for this patch).
- Persisting "you almost quit" state across sessions.

## Design

### Bindings on `OnboardApp`

```python
Binding("up",     "focus_previous", "Up",                show=False),
Binding("down",   "focus_next",     "Down",              show=False),
Binding("ctrl+c", "confirm_quit",   "Quit (Ctrl+C Ă—2)",  priority=True),
Binding("escape", "cancel_quit",    "Cancel quit",       show=False, priority=True),
```

`priority=True` on `Ctrl+C` and `Esc` overrides the App / Input defaults
(`Ctrl+C` is Textual's built-in quit; Input handlers may also touch it).

### State

```python
self._quit_pending: bool = False
self._quit_timer: Timer | None = None
```

### Actions

```python
def action_confirm_quit(self) -> None:
    if self._quit_pending:
        self.exit(False)
        return
    self._quit_pending = True
    self.notify("press Ctrl+C again to quit, Esc to cancel",
                severity="warning", timeout=5)
    if self._quit_timer is not None:
        self._quit_timer.stop()
    self._quit_timer = self.set_timer(5.0, self._reset_quit_pending)

def action_cancel_quit(self) -> None:
    if self._quit_pending:
        self._reset_quit_pending()
        self.notify("quit cancelled")

def _reset_quit_pending(self) -> None:
    self._quit_pending = False
    if self._quit_timer is not None:
        self._quit_timer.stop()
        self._quit_timer = None
```

Auto-expire after 5 s so an accidental tap doesn't stay armed forever.

The existing bare-`q` binding stays as a quick path when focus is on a
model row; `Ctrl+C` is the universal one.

## Acceptance

- `Down` from any focused widget moves focus to the next focusable
  widget; `Up` moves to the previous.
- One `Ctrl+C` triggers a toast and arms the pending state; a second
  `Ctrl+C` within 5 s exits with result `False`. `Esc` while armed clears
  the state and toasts "quit cancelled".
- An accidental `Ctrl+C` followed by 5 s of inactivity self-cancels; a
  third `Ctrl+C` then re-arms (not insta-quit).
- Headless `run_test()` pilot:
  - `await pilot.press("down")` advances focus from one `ModelRow` to
    the next.
  - `await pilot.press("ctrl+c")` arms; second press exits.
  - `await pilot.press("ctrl+c"); await pilot.press("escape")` keeps
    the app alive.
- `neutrix --version` reports `0.3.1`.
- `pytest` + ruff stay green.

## Out of scope

- PageUp/PageDown, Home/End â€” Tab already covers the common case.
- Arrow-key / Ctrl+C behavior inside the chat TUI.
- Configurable confirm-window length.
