# v0.6.7 - terminal mouse ownership and final-only LLM replies

## Problem

The main chat has two practical regressions:

- The user needs mouse wheel scrolling without losing normal terminal
  selection, copy, and paste behavior.
- LLM responses can fail to appear in the UI while the code still depends on
  token-delta streaming. For this release, correctness is more important than
  live token rendering.

The UI/backend controller split is a good next step, but it is larger than the
bug fix needed here.

## Goal

Keep the main chat simple and reliable:

- terminal owns mouse behavior, so drag selection, copy, paste, and wheel
  behavior remain terminal-native instead of being captured by Textual;
- the LLM client makes one non-streaming OpenAI-compatible chat completion
  request;
- the agent loop receives one final assistant event and the UI renders it as
  an assistant block.

## Non-goals

- Build a UI/controller/backend event bus.
- Rework tool execution or sandbox boundaries.
- Add token-delta streaming back in this release.
- Add a second mouse mode that captures wheel events in the app.

## Design

1. Launch the main Textual app with mouse reporting disabled.
2. Keep the main app in its existing fullscreen mode. Do not use Textual
   inline mode because it adds top/bottom terminal rules that pollute the
   copy-friendly chat surface.
3. Keep the TUI-to-agent interface unchanged: the UI still submits a message
   and listens for agent events.
4. Change `OpenAIChatLLM.stream_response()` to preserve the public async event
   interface but perform a final-only chat completion call with `stream=False`.
5. Parse the returned assistant message content and any full tool-call payloads
   from the final response.
6. Emit one `LLMEvent("assistant", LLMResponse(...))`; do not emit token
   events from the OpenAI adapter.
7. Keep `Agent.stream_reply()` compatible with token events so future delta
   streaming can be restored without changing the UI contract.
8. Do not send OpenAI Chat Completions `type: function` tool schemas to IHEP
   Anthropic models, because that route maps to an Anthropic tools surface
   that rejects the OpenAI function schema. The chat should answer instead of
   failing before content arrives.
9. For IHEP Anthropic models, keep the internal transcript unchanged but move
   outbound `system` content into the first user message, because that gateway
   rejects OpenAI-style `system` role messages.
10. Keep using the OpenAI SDK for IHEP Anthropic chat-completions requests.
    Pass the Anthropic-style top-level system prompt through SDK `extra_body`
    while keeping `system` out of `messages`.

## Acceptance

- The main CLI launches the Textual app with mouse reporting disabled.
- The main CLI does not launch inline, avoiding Textual inline top/bottom
  rules in the terminal output.
- Native terminal drag selection and copy/paste are not captured by Textual
  mouse reporting.
- Submitting a chat message produces exactly one final assistant UI block when
  the provider returns final content.
- The OpenAI adapter sends `stream=False`.
- The OpenAI adapter emits no token-delta events.
- Final assistant content is still appended to the agent transcript.
- Full non-streaming tool-call payloads still flow through the existing
  agent-loop tool continuation path for models that accept OpenAI tool schemas.
- IHEP Anthropic models omit OpenAI tool schemas and show
  `tools:unsupported` instead of failing the request.
- IHEP Anthropic model requests do not send an outbound `system` role.
- IHEP Anthropic model requests still use the OpenAI SDK.
- IHEP Anthropic model requests pass system prompt text through SDK
  `extra_body`.
- Existing draft submit and multiline shortcuts still pass.
- The controller/backend separation remains explicitly deferred.
