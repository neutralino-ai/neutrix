# v0.7.0 - chatstore + queued user message display

## Problem

Queued user messages are invisible. As of v0.6.8 the terminal chat already
queues user inputs typed while the assistant is working — see
`terminal_chat.py:287` (`self._input_queue = asyncio.Queue()`) — but the only
visible signal is a `queued:N` counter in the status line. The text the user
typed disappears until the assistant catches up.

There is no single owner of chat state. The messages list, the input queue,
the streaming "assistant in progress" text, and the pending tool call list
all live as private attributes on the `TerminalChatApp` class
(`terminal_chat.py:287–289` and around the streaming reducer). Two other
modules — `tui.py` and `agent_loop.py` — also know about pieces of this
state through ad-hoc patterns. There is no module a future renderer, test,
or persistence helper can read to learn "what is the conversation right
now."

Two near-term features compound this. v0.8.0 will add a heartbeat /
liveness indicator that needs to know whether the LLM stream and the
executor are currently active. v0.9.0 will add todos and reminders, which
are pure state with their own panel. Both will pile more private attributes
into `terminal_chat.py` if state has no home of its own.

`session.py` saves and loads chat history but takes `provider`, `model`,
and `messages` as separate arguments. Every new state field becomes another
positional argument, and the file's name implies a stateful object it does
not contain.

## Goal

Introduce **ChatStore**, the canonical in-memory record of a chat session,
and ship the first feature that exercises it: render queued user messages
as visible items in the chat surface.

- ChatStore owns the messages list, the queued user inputs, the
  in-progress assistant stream text, and the pending tool call list.
- ChatStore is a plain Python module. It imports nothing from `tui.py`,
  `terminal_chat.py`, `agent_loop.py`, `llm.py`, or `tools.py`.
- ChatStore exposes an async `changes()` iterator so any renderer can
  await store mutations without polling.
- The terminal chat reads queued messages from the store and renders each
  queued item below the live transcript, in a muted style, with the
  user's text visible. The `queued:N` counter goes away or becomes
  redundant.
- `session.py` is renamed `transcript.py` and rewritten to (de)serialize a
  ChatStore as a whole rather than discrete fields.

## Non-goals

- Refactoring `agent_loop.py` into a pure controller. Deferred to v0.8.0.
- Refactoring `tui.py` / `terminal_chat.py` into pure renderers. Deferred
  to v0.9.0.
- Adding the heartbeat / liveness animation. v0.8.0.
- Adding todos and reminders. v0.9.0.
- Web frontend. Not planned in this PRD arc.
- New persistence formats, multi-session UI, or resume-from-disk on the
  terminal chat start path. Persistence stays opt-in via the existing
  save/load surface.
- Onboarding flow refactor. `onboard.py` is untouched in v0.7.0.
- Changes to `tools.py`, `llm.py`, or the agent event protocol in
  `agent_loop.py`.
- Adding a CI grep / import-check job. Boundary discipline is enforced by
  review in v0.7.0; an automated check can come later.

### Boundary types

The types that cross the ChatStore boundary are defined in `store.py` and
imported by callers. No other module is allowed to define duplicates.

```python
# store.py
from dataclasses import dataclass, field
from datetime import datetime
from typing import AsyncIterator, Literal

Role = Literal["user", "assistant", "system", "tool"]

@dataclass(frozen=True)
class MessageRecord:
    role: Role
    content: str
    ts: datetime
    tool_name: str | None = None       # set when role == "tool"
    tool_call_id: str | None = None    # set when role == "tool"

@dataclass(frozen=True)
class QueuedUserMessage:
    text: str
    queued_at: datetime

@dataclass(frozen=True)
class PendingToolCall:
    name: str
    arguments: str
    started_at: datetime
```

### ChatStore API

```python
# store.py (continued)
class ChatStore:
    def __init__(self) -> None: ...

    # read-only views (return tuples / copies; never the internal list)
    @property
    def messages(self) -> tuple[MessageRecord, ...]: ...
    @property
    def queued_user_messages(self) -> tuple[QueuedUserMessage, ...]: ...
    @property
    def pending_assistant_text(self) -> str | None: ...
    @property
    def pending_tool_calls(self) -> tuple[PendingToolCall, ...]: ...

    # writes (the only mutators)
    def append_message(self, msg: MessageRecord) -> None: ...
    def enqueue_user(self, text: str) -> None: ...
    def dequeue_user(self) -> QueuedUserMessage | None: ...
    def start_assistant_stream(self) -> None: ...
    def extend_assistant_stream(self, delta: str) -> None: ...
    def finish_assistant_stream(self) -> MessageRecord: ...   # also appends
    def add_pending_tool_call(self, name: str, arguments: str) -> None: ...
    def clear_pending_tool_calls(self) -> None: ...

    # observation
    def changes(self) -> AsyncIterator[None]:
        """Async iterator that yields once per mutation since the last
        yield. Multiple consecutive mutations between yields coalesce
        into a single yield — the consumer re-reads the full store, so
        per-mutation granularity is unnecessary. Multiple subscribers
        are independent (each gets its own iterator). Closing the
        iterator (`aclose()` or letting it fall out of scope) removes
        the subscription."""
```

Notification mechanics: every mutator increments an internal "changed"
signal that wakes any awaiting `changes()` iterator. Each subscriber
holds its own `asyncio.Event`; the mutator calls `set()` on every active
subscriber's event. Mutators never await — they fire-and-forget the
wake and return immediately. Slow or stalled subscribers cannot block
mutations because there is no shared queue and `Event.set()` is
idempotent: if multiple mutations occur before a subscriber awakens,
they coalesce into the single next yield. The subscriber then re-reads
the store and renders whatever it sees, so coalescing loses no
information.

Implementation sketch (for clarity; the real version lives in
`store.py`):

```python
import asyncio

class ChatStore:
    def __init__(self) -> None:
        self._subscribers: set[asyncio.Event] = set()
        ...

    def _notify(self) -> None:
        for event in self._subscribers:
            event.set()

    async def changes(self) -> AsyncIterator[None]:
        event = asyncio.Event()
        self._subscribers.add(event)
        try:
            while True:
                await event.wait()
                event.clear()
                yield None
        finally:
            self._subscribers.discard(event)
```

Every mutator method ends with a call to `self._notify()`. There is no
synchronous-callback variant; if a renderer needs sync behavior, it
wraps `changes()` in its own pattern.

### transcript.py

```python
# transcript.py
def save(
    path: str | Path,
    store: ChatStore,
    *,
    provider: str,
    model: str,
) -> Path: ...

def load(path: str | Path) -> tuple[ChatStore, dict[str, str]]:
    """Returns (store, metadata) where metadata has provider / model /
    saved_at. Raises ValueError on version mismatch."""
```

The on-disk format keeps `version`, `saved_at`, `provider`, `model`, and a
`messages` array compatible with the v0.6.x `session.py` format, so files
saved by older versions load cleanly. Queued messages and pending tool
calls are intentionally NOT persisted — they are transient by definition.

### Terminal chat integration

`terminal_chat.py` keeps its `asyncio.Queue` for cross-coroutine input
hand-off but mirrors the queue contents into `ChatStore.queued_user_messages`
at enqueue and dequeue time. The renderer reads from the store, not from
the queue.

The pending tool call list (`_pending_tool_calls`, currently a list of
tuples at `terminal_chat.py:289`) is replaced with reads/writes to
`store.pending_tool_calls`.

The streaming "assistant in progress" text is mirrored through
`start_assistant_stream` / `extend_assistant_stream` /
`finish_assistant_stream`. The renderer reads `pending_assistant_text`.

Queued user messages render as one line each, **directly above the
input cursor** (via prompt_toolkit's `PromptSession.message` parameter,
which sits inside the input area and above where the user types).
Each line uses `fg:ansibrightblack` foreground on the input area's
natural background, so the queue and input share one visual region
rather than appearing on the differently-styled bottom toolbar. Each
line is prefixed with `› ` (a single right-angle quote U+203A plus a
space) so multiple queued items stack legibly. No section header —
the muted styling and prefix glyph carry the meaning. The existing
`queued:N` counter is removed from the status line; if real usage
later shows the queue routinely scrolling off screen, a count can be
re-added in a patch release.

Screen updates are **invalidation-driven**, not periodic.
`PromptSession` is constructed without `refresh_interval`; instead a
background coroutine subscribes to `store.changes()` and calls
`prompt_toolkit.application.current.get_app_or_none().invalidate()`
once per batch of mutations. The worker loop also invalidates around
its `_busy` toggle. Net effect: the screen only redraws when state
actually changed, so the prior rhythmic 0.5-s flash is gone.

The bottom status toolbar is removed entirely. In prompt_toolkit's
append-only mode, every stdout write triggers a hide-restore cycle of
the prompt area; the toolbar window vanished for one frame per
streamed token, producing a visible blink during the assistant's
response. No styling trick fixed that — the toolbar literally was
not on screen during the hide half of every cycle. The new `/status`
command (`_cmd_status`) prints slot, provider/model, tool state, and
message count to scrollback on demand; the "working" indicator comes
from the visible streaming tokens themselves. A non-flickering live
indicator is the right design problem for v0.8.0's heartbeat (it
needs a render region that does not get hidden by `patch_stdout`).

### Migration order inside this release

1. Add `store.py` with dataclasses + `ChatStore` + `subscribe`.
2. Add unit tests for the store in isolation.
3. Rename `session.py` to `transcript.py` (`git mv`); rewrite `save` /
   `load` against `ChatStore`. Adjust the CLI import. Update
   `test_smoke.py` if it references session paths.
4. In `terminal_chat.py`, replace the private messages list, pending tool
   call list, and streaming text holder with reads/writes against a
   single `ChatStore` instance owned by `TerminalChatApp`. Mirror the
   `asyncio.Queue` into `enqueue_user` / `dequeue_user`.
5. Add the queued-message render row. Remove the `queued:N` status text.
6. Manual acceptance run.
7. Update `CHANGELOG.md` and tag.

### What dies

- `session.py` as a name and as a `provider/model/messages`-argued API.
  Replaced by `transcript.py(store)`.
- The `queued:N` counter inline in the status text (replaced by the
  visible queue render).
- Ad-hoc message-list and pending-tool-call attributes on
  `TerminalChatApp` (folded into the owned `ChatStore`).

## Acceptance

### Automated (the bulk of the gate)

Steps 3–7 of the original script are fully automated and run in CI on
every `pytest` invocation:

- `tests/test_store.py` (NEW, 16 tests): append / enqueue / dequeue /
  stream start-extend-finish round-trip; `changes()` yields on every
  mutation; coalescing across mutations between yields; multiple
  independent subscribers; `aclose()` unsubscribes the iterator;
  OpenAI bridge round-trips lossless for user / assistant-with-tool-calls
  / tool / unknown-role / non-string-content.
- `tests/test_transcript.py` (NEW, 5 tests): save → load round-trip
  preserves messages and OpenAI extras; v0.6.x `session.py`-format
  files load without error; unknown version raises; transient state
  (queue, pending tool calls, in-progress assistant text) never
  persists.
- `tests/test_terminal_chat.py::test_terminal_chat_accepts_and_queues_input_while_agent_is_busy`
  (UPDATED): a single queued message appears in the store and toolbar;
  the `queued:N` counter is gone from the toolbar.
- `tests/test_terminal_chat.py::test_terminal_chat_renders_multiple_queued_messages_in_order`
  (NEW): covers steps 3 + 4 + 5 + 6 in one test — two messages

  toolbar fragments are `["› second\n", "› third\n"]` with
  `fg:ansibrightblack` styling and no `queued:` substring; both items

  (NEW): when nothing is queued, `_status_text` returns a plain str

- `tests/test_terminal_chat.py::test_terminal_chat_save_and_load_round_trip_via_commands`
  (NEW): covers step 7 — `/save` then a fresh chat `/load` reconstructs

- All existing tests in `test_terminal_chat.py`, `test_tui.py`, and
  `test_smoke.py` continue to pass.

Total automated coverage: 100 tests, ~26 s, no network, no real LLM.

### Manual (genuinely-human judgment only)

Three things tests can't decide; the rest of the acceptance gate
collapses into these:

1. **Toolbar legibility on a real terminal.** Launch `neutrix` with
   your own provider. Send a prompt. While the assistant is responding,
   type and submit one or two more messages. Confirm:
   - the `› …` queued lines are visibly dim against your terminal
     theme (not invisible, not the same color as the active turn),
   - their location (above the status line, below the input editor)
     does not confuse you.
2. **Refresh latency.** The toolbar's `refresh_interval` is 0.5 s, so
   a queued line can take up to half a second to appear after submit.
   Confirm that delay feels acceptable. If it doesn't, the fix is
   `app.invalidate()` on store change — a small follow-up patch, not a
   v0.7.0 blocker.
3. **Real-LLM end-to-end smoke.** Tests use a stub agent. One real
   round-trip with your live provider against the new code path
   confirms nothing structural broke for the actual chat path.

If any of those three flags an issue, say so before the tag.

### Boundary discipline (review-enforced in v0.7.0)

- `store.py` imports zero from `tui`, `terminal_chat`, `agent_loop`,
  `llm`, `tools`, or `onboard`.
- `transcript.py` imports from `store` only (plus stdlib).
- `terminal_chat.py` and `tui.py` may import from `store`; the converse
  is forbidden.

A CI grep job to enforce the above can land in a later release; for v0.7.0
the discipline is enforced by code review.
