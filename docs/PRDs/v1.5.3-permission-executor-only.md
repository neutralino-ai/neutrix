# v1.5.3 — permission is an Executor-only safety layer

> **Status: proposed (Phase-1 accept gate pending).** Urgent — the live
> permission prompt parks the turn. Splits:
> [docs/splits/v1.5.3-permission-executor-only.html](../splits/v1.5.3-permission-executor-only.html).

## Problem

v1.4.8 turned the permission `ask` verdict into an **interactive prompt routed up
to the ContextManager**: when `auto` mode flags a destructive Bash (`rm -rf`,
`git reset --hard`, force-push, `dd`, …), `Executor.dispatch_all` yields a
`needs_user_input` event (`executor.py:178`), and the **CM** — which holds the
`ask_user` port — awaits the human for up to `PROMPT_TIMEOUT_S` (300 s, v1.5.1).
That **parks the whole turn** on a routine command. Two faults:

1. **It blocks the agent.** The user runs destructive-looking commands as normal
   dev workflow; each one stalls the loop awaiting an answer.
2. **Permission is entangled with the loop.** The Controller (and, by reading the
   same channel, the conceptual surface the Advisor sits next to) has to know
   about permission. The user: "permission should be in executor part. controller,
   advisor does not need to know it."

The user's directive for the fix: **"if it detect dangerous action, deny directly.
return tool_result denied … don't ask user question. let's be like industrial
agent."**

## Goal

- **Permission is decided and fully resolved inside the Executor.** The
  ContextManager and Advisor carry **zero** permission semantics.
- **`auto` is a deterministic safety layer:** a detected-dangerous action is
  **denied directly** — the Executor returns a `role:tool` "[denied …]" result and
  the round **continues**; the LLM sees the denial and adapts. **No interactive
  permission prompt — ever.** No park, no hang.
- The CM `needs_user_input` / `ask_user` channel survives **only** for the
  agent-initiated `AskUserQuestion` tool.

## Non-goals

- **Removing the `AskUserQuestion` tool** (the agent's own clarifying-question
  tool, v1.4.8). Kept — see the open flag below. A separate release if you want it
  gone.
- Plan mode / interactive approval (already out, v1.4.0; this seals it).
- Changing the `.claude/settings.json` schema. `load_policy` still reads
  `allow`/`deny`/`ask`; the `ask` bucket now resolves to **deny** (neutrix never
  prompts).
- A configurable per-pattern danger policy or a sandbox (YAGNI; sandbox is v1.14.0).

## Claude Code reference

See [docs/splits/v1.5.3-permission-executor-only.html](../splits/v1.5.3-permission-executor-only.html)
for the full survey (6 split points). CC: `queryLoop` (`query.ts:241`) is
permission-agnostic — zero permission imports; `canUseTool` is invoked in the
tool-execution layer (`toolHooks.ts:361`); `ask`/`deny` both surface to the model
as a `tool_result {is_error:true}` (interactive prompt only on `ask`).

### Split point: where permission is decided
- **CC**: tool-execution layer (`toolHooks.ts:361`), loop is agnostic.
- **Decision**: Follow CC + User-directed — `decide()` stays on the `Executor`; the
  CM holds no permission logic.

### Split point: the interactive `ask` prompt
- **CC**: interactive approval dialog, tool execution awaits the human.
- **Decision**: User-directed different — **delete the `ask` verdict and its
  prompt** (`executor.py:177-191`). `decide()` returns only `allow`/`deny`; no
  permission path yields `needs_user_input`.

### Split point: what `auto` does with a dangerous action
- **CC**: prompts (`ask`).
- **Decision**: User-directed different — **deny directly** ("return tool_result
  denied"). Danger heuristic + settings `ask` rules → `deny`. Explicit `allow`
  overrides; `allow-all` (`/allow`) bypasses. Trade accepted: routine `rm -rf
  build` is denied under `auto`; `/allow` or an `allow` rule is the escape hatch.

### Split point: the CM `ask_user` port
- **Decision**: Follow CC — keep the port for the `AskUserQuestion` tool only;
  permission stops using it. **Open flag**: if "industrial / no ask user" means the
  agent should never ask the user *anything*, retiring `AskUserQuestion` is a clean
  v1.5.4 follow-up.

### Split point: Advisor
- **Decision**: No change — `advisor.py` already bypasses the Executor gate
  (`tools.dispatch(store=…)`); it is permission-agnostic. Preserve the invariant.

### Split point: dead interactive-permission code
- **Decision**: Alternative — delete `permission_question`, `verdict_from_answer`,
  `apply_always_rule`, `USER_DENIED` (YAGNI). Static `allow`/`deny` rules + `/allow`
  remain.

## Design

### `src/neutrix/permissions.py`
- `decide(...)` returns **`"allow"` | `"deny"`** only:
  - `allow-all`/`bypass` → `allow`; `deny` rule match → `deny`; `allow` rule match →
    `allow` (overrides the heuristic); **`auto` + `_is_dangerous` → `deny`** (was
    `ask`); settings `ask` rule match → `deny`; else `allow`.
- `block_reason(tool_name, args_json="")` → a clear denied message distinguishing
  the **safety layer** (dangerous Bash) from a **deny rule**, e.g.
  `"[denied by safety layer: Bash looks destructive — not run; /allow to override]"`.
- **Delete** `permission_question`, `verdict_from_answer`, `apply_always_rule`,
  `USER_DENIED`, and the now-unused `from neutrix.prompts import …` line.
- `load_policy`, `PermissionPolicy`, `_is_dangerous`, `_matches`, `_DANGEROUS_BASH_RE`
  unchanged. Module docstring updated to the industrial / deny-direct model.

### `src/neutrix/executor.py`
- In `dispatch_all`, the per-tool gate becomes:
  ```python
  verdict = decide(name, args, mode=self.permission_mode, policy=self.policy)
  if verdict == "deny":
      yield _finished(tcid, name, block_reason(name, args), ok=False)
      continue
  ```
  The whole `if verdict == "ask": …` block (177-191) is **removed**.
- Drop the now-unused imports (`apply_always_rule`, `permission_question`,
  `verdict_from_answer`, `USER_DENIED`); keep `decide`, `block_reason`,
  `PermissionPolicy`. `_needs_input` stays (AskUserQuestion at line ~202).

### `src/neutrix/context_manager.py`
- No code change to the loop. `_dispatch_tools` still handles `needs_user_input`,
  now reached **only** by `AskUserQuestion`. Update the docstring (839-851) and the
  `ask_user` field comment (240-241) to drop "permission ask" — permission is no
  longer a CM concern.

### `src/neutrix/terminal_chat.py` / `src/neutrix/prompts.py`
- Comment/docstring cleanup: the pending-prompt indicator + `_ask_user` now serve
  `AskUserQuestion` only (lines ~170, ~971, `_ask_user` docstring; `prompts.py`
  module docstring). `/allow` toggle and `PROMPT_TIMEOUT_S` stay.

### Tests
- `decide()`: each `_DANGEROUS_BASH_RE` pattern → `deny` under `auto`; explicit
  `allow` rule → `allow`; `allow-all` → `allow`; settings `ask` rule → `deny`.
- Executor: a dangerous Bash yields a **denied `tool_finished`** and **never** a
  `needs_user_input`. AskUserQuestion still round-trips via `needs_user_input`.
- Drop/repoint the permission-`ask`-round-trip tests; remove tests for the deleted
  helpers. CM: assert no permission import / no permission branch.

## Acceptance

- **Automated**: the suite above is green; full test suite green; `ruff` clean.
  `grep -n permission src/neutrix/context_manager.py` shows no permission logic
  (only the updated docstring).
- **Manual**: in `auto`, `rm -rf build` returns `[denied …]` **immediately** (no
  park) and the turn continues; `/allow` then runs it; a normal command runs
  unchanged; `AskUserQuestion` still prompts (it's the agent's own tool).
- Ship `v1.5.3`.

## Out of scope

- Retiring `AskUserQuestion` (flagged; v1.5.4 if wanted), sandbox (v1.14.0),
  per-pattern configurable danger policy.
