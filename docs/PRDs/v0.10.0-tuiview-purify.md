# v0.10.0 - TUIView as pure renderer + visible task panel + swap-test

> **Status: skeleton.** Detail solidifies after v0.8.0 (tasks),
> v0.9.0 (lifecycle events + store reducer), v0.9.1 (keyboard),
> v0.9.2 (cancellation + Executor / LLM / Controller decomposition),
> v0.9.3 (heartbeat), and v0.9.4 (visibility parity) land. This
> PRD closes the v0.7-arc onion refactor and ships the
> architectural proof.

## Problem

After v0.9.0–v0.9.4 the agent loop emits lifecycle events with a
cancellation-safe stream, the store owns the canonical "what's in
flight" state, the keyboard handles Codex-style two-tap exit and
suspend, Esc returns the loop to idle, and the heartbeat renders
the live state. `terminal_chat.py` and the legacy `tui.py` are
*mostly* readers of the store but still host:

- inline command-dispatch logic that decides whether `/save` /
  `/load` happen at all,
- transcript-rendering decisions (which lines fold, which messages
  to skip on load, how to print the assistant's streaming tokens),
- a few residual writes that bypass the store (e.g. `_tool_records`
  for the folded-result tray).

This is the last layer of the onion that isn't on the right side of
the boundary. As long as it stays, every new feature risks
re-tangling the renderer with state — exactly what v0.7.0 set out to
prevent.

A separate, near-term user-visible motivation: tasks shipped in
v0.8.0 are only visible via `/tasks`. Many users will want a
**persistent panel** that shows pending and in-progress tasks above
the input at all times. Doing that without re-introducing the blink
requires the view to be a clean reader of the store on the
invalidation-driven render path.

## Goal

- TUIView (the union of `tui.py` and `terminal_chat.py`) becomes a
  pure reader of `ChatStore`. No state mutation inside view code;
  all writes go through controller events or through the input-queue
  hand-off.
- Move any remaining residual state into the store
  (notably `_tool_records` for folded tool result display).
- Add a `TaskPanel`: a dim, compact rendering of pending +
  in_progress tasks in the message-area above the input (same
  invalidation-driven path as the queue and the heartbeat). Done
  tasks are not shown there — `/tasks` keeps the full list.
- Add a **`FakeView`** in `tests/`: a ~50-line stdout renderer that
  reads `ChatStore` and prints conversation, queue, heartbeat, and
  tasks. `tests/test_fake_view.py` drives a synthetic controller
  event stream and asserts the printed output matches a captured
  baseline. Passing this test is the architectural proof that the
  view layer is genuinely swappable — the v1.0 web-frontend bet
  (Path B strategy) only makes sense if this passes.

## Non-goals

- Web frontend. The `FakeView` swap test stays in `tests/`; it is
  the proof, not a product. Web is v1.0.
- Multi-session UI / transcript browser.
- Cross-session task aggregation.
- Persistent visible heartbeat history (e.g. recent latencies).
- Refactoring `onboard.py`. Continues to stay out of scope for the
  v0.7-arc.
- Removing the legacy `tui.py` Textual fullscreen path. It can stay
  as a parallel renderer; if the onion is right, both `tui.py` and
  `terminal_chat.py` are mere views.

## Design (skeleton — finalize after v0.9.0–v0.9.4)

> v0.9.0 deferred the `Agent` rename; v0.9.2 lands the
> decomposition (`Agent` stays as the stateful router, `Executor`
> and `LLM` and `Controller` get their own homes). v0.10.0
> inherits that layout — the FakeView swap-test gate is renderer
> purity against `ChatStore`, independent of the executor /
> controller split.


- Audit `tui.py` and `terminal_chat.py` for any state writes that
  bypass the store. Each one moves to either:
  (a) an input-queue hand-off (user actions),
  (b) a controller-emitted event (LLM- / tool-driven), or
  (c) a store mutator called from the reducer (state changes).
- Move `_tool_records` (folded-tool tray) onto the store as
  `folded_tool_results: tuple[ToolRecord, ...]`. `/tool [N]` reads
  from there.
- Add a `TaskPanel` block in the message-area supplier: dim
  foreground, one short line per pending + in_progress task. Hidden
  when no such tasks.
- Stack order in the message area (top → bottom):
  1. TaskPanel (pending + in_progress)
  2. HeartbeatLine (only while active)
  3. Queued user messages (`›` lines)
  4. Input cursor
- Add `tests/fake_view.py`: a `FakeView` class with a `render(store)
  -> str` method and a `subscribe(store)` async loop. Captures
  full snapshots after each store change.
- Add `tests/test_fake_view.py`: drives a sequence of synthetic
  events through a `ChatStore` and asserts that `FakeView` prints
  the expected baseline at each step.

## Acceptance (sketch)

- Manual: with v0.8.0 / v0.9.0 features intact, the TaskPanel
  appears above the input when there are pending or in_progress
  tasks, and disappears when all are done. The heartbeat shows
  during streaming / tool runs. The queue shows when typed during
  busy. No blinks.
- v0.7.0, v0.8.0, v0.9.0 acceptance scenarios still pass.
- `tests/test_fake_view.py` runs the swap test and asserts on
  captured output. This is the **view-is-swappable** gate.
- Boundary check (review-enforced unless we wire a CI grep job):
  `tui.py` and `terminal_chat.py` import only from `store` (plus
  Textual / Rich / prompt_toolkit / stdlib). No imports from
  `controller`, `llm`, `tools`.
- Human gate before tag.

## Open questions (to resolve before implementing)

- Should the TaskPanel cap at N lines and add "… and M more"? Pick
  N at design review. Initial guess: 5.
- Order of tasks in the panel: by id, by status, or by updated_at?
  Default proposal: in_progress first, then pending, both by id.
- Does the legacy `tui.py` Textual fullscreen path get the same
  TaskPanel, or only the append-only terminal chat? Default: yes,
  both — the panel is a view-layer concern, both views can render
  it.
- What does the `FakeView` baseline look like — exact bytes, or
  structured assertion (set of expected substrings)? Exact bytes is
  brittle; structured is friendlier. Decide during implementation.
