# v0.9.8 — liveness motion (blink, not brightness)

Status: **Phase-1 PRD — awaiting accept gate.**
Splits: [`docs/splits/v0.9.8-liveness-motion.html`](../splits/v0.9.8-liveness-motion.html)
(all six decided). Supersedes the v0.9.5 heartbeat smoothing approach;
re-affirms v0.9.4 #2.

## Problem

The v0.9.4 heartbeat animates a single `●` by **brightness**, and v0.9.5
cranked the refresh to `HEARTBEAT_REFRESH_HZ = 120` to make the fade look
smooth. It still reads as ~5 Hz and steppy, and the user finds it
uncomfortable.

Root cause (measured 2026-05-28): the user's terminal is
`tmux-256color` with `COLORTERM` unset, so prompt_toolkit resolves to
**8-bit / 256-color** and quantizes the truecolor gray gradient
`(60,60,60)→(255,255,255)` onto only **~22 distinct grays** (24-step
xterm gray ramp `#080808…#eeeeee`, step 10; 24-bit would give ~178).
The finest possible 256-color gray step (~4% brightness) is *above* the
eye's discrimination threshold, so the breath bands no matter how it is
paced. Raising the tick 10→120 Hz did nothing because **the palette, not
the clock, is the ceiling.** Empirically confirmed in the user's terminal
that cosine, temporal-dither, and even-linear-ramp all band or flicker;
`tmux` reports `RGB: missing`, so truecolor is unavailable without
reconfiguring the terminal stack.

## Goal

A liveness pulse that is **smooth/robust on the user's 256-color terminal
and every other**, by moving the animation off brightness (palette-bound)
onto the dot's **presence** (palette-independent): a single `●` that winks
on/off, exactly as Claude Code's tool-use loader does.

## Non-goals

- **No truecolor dependency.** Forcing 24-bit is fragile in `tmux`
  (RGB missing) and breaks over SSH/CI/plain terminals; CC clamps *down*
  to 256 for the same reason. Out of scope.
- **No shape-bloom or shimmer.** Both were surveyed and prototyped; the
  user chose the simpler presence blink. (Shimmer also fails on neutrix's
  short labels — see splits "liveness-gap" survey.)
- **No configurable glyph / verb personality / elapsed-time overlay.**
  Standing v0.9.4 non-goals unchanged.
- **No `tui.py` Textual app.** Standing v0.9.x non-goal.

## Claude Code reference

See [`docs/splits/v0.9.8-liveness-motion.html`](../splits/v0.9.8-liveness-motion.html)
for the full survey (root-cause + liveness-gap analysis + all six split
points). Summary, one sub-block per split:

### Split point: Motion channel (#1)
- **CC**: `components/ToolUseLoader.tsx` renders `BLACK_CIRCLE` when
  visible else `" "`; `hooks/useBlink.ts` (`BLINK_INTERVAL_MS = 600`)
  toggles visibility on a strict shared clock, paused when unfocused.
- **Decision**: User-directed different — **(d) presence blink**, a new
  option surfaced by the user mid-discussion (not in the agent's original
  shape/shimmer list). It is *Follow CC* on the **tool loader**. A 2-state
  presence toggle has nothing to interpolate → no 256-color quantization.

### Split point: Glyph (#2, re-affirms v0.9.4 #2)
- **CC**: tool loader uses `BLACK_CIRCLE` (`●`); spinner uses the
  `· ✢ ✳ ✶ ✻ ✽` bloom.
- **Decision**: Follow CC (tool loader) — single `●`, **no shape cycle**;
  it winks on/off. Re-affirms v0.9.4 #2's calm single dot; only the v0.9.4
  #13 smoothing method changes.

### Split point: Cadence (#3, reverses v0.9.5 120 Hz)
- **CC**: `BLINK_INTERVAL_MS = 600` (1.2 s cycle, 50% duty), strict clock.
- **Decision**: Follow CC — 600 ms toggle; tested (candidate C) and
  accepted. Retires `HEARTBEAT_REFRESH_HZ = 120` and the 480-frame table;
  the loop now sleeps ~600 ms and flips a boolean (~70× less CPU). One
  constant `HEARTBEAT_BLINK_INTERVAL_MS` for tuning.

### Split point: Color model (#4, replaces v0.9.4 #13)
- **CC**: tool loader colors the dot by *state* (dim in-progress / red
  error / green success), not by a gradient.
- **Decision**: Alternative — single named color (bright white), presence
  toggle. With on/off there is no second color and no gradient, so the
  truecolor-vs-256 question is moot. Drop the hex gradient + shade table.

### Split point: Stalled state (#5)
- **CC**: tool loader uses `error` color (red) for the error state.
- **Decision**: Follow CC (intent) — red `●`, same on/off wink, label
  `LLM (stalled)`. Discrete white→red swap, no interpolation. Consistent
  with the v0.9.5 stalled concept.

### Split point: Jitter (#6, revisits v0.9.4 #14)
- **CC**: strict period; `useBlink` derives state from
  `floor(time / interval) % 2`.
- **Decision**: Follow CC — strict period, no jitter. v0.9.4 #14's
  organic-jitter rationale was brightness-fade-specific; jitter on an
  on/off wink looks erratic. Retire `jittered_sleep` from the heartbeat
  path.

## Design

All in `src/neutrix/terminal_chat.py`.

**Remove** (v0.9.4/v0.9.5 brightness machinery):
`HEARTBEAT_BREATH_PERIOD_S`, `HEARTBEAT_REFRESH_HZ`,
`HEARTBEAT_CYCLE_FRAMES`, `HEARTBEAT_TICK_MS`, `HEARTBEAT_JITTER_RATIO`,
`HEARTBEAT_TROUGH_RGB`, `HEARTBEAT_PEAK_RGB`,
`HEARTBEAT_STALLED_TROUGH_RGB`, `HEARTBEAT_STALLED_PEAK_RGB`,
`_build_brightness_cycle`, `HEARTBEAT_BRIGHTNESS_CYCLE`,
`HEARTBEAT_STALLED_CYCLE`, `jittered_sleep`.

**Add / keep:**
- `HEARTBEAT_GLYPH = "●"` (kept).
- `HEARTBEAT_BLINK_INTERVAL_MS = 600` — one toggle per interval; 1.2 s cycle.
- `HEARTBEAT_GLYPH_STYLE = "fg:ansiwhite bold"` (visible, normal).
- `HEARTBEAT_STALLED_GLYPH_STYLE = "fg:ansired bold"` (visible, stalled).
- `HEARTBEAT_LABEL_STYLE` (kept); stall detection (`stall_threshold_for`,
  `HEARTBEAT_STALL_FRACTION`, `HEARTBEAT_STALL_FLOOR_S`) unchanged.

**`heartbeat_loop`**: default `sleep_seconds = HEARTBEAT_BLINK_INTERVAL_MS
/ 1000`; default `sleep_fn = asyncio.sleep` (no jitter). Loop structure
unchanged (idle waits on `store.changes()`, busy ticks + `on_tick`). Add a
new `on_enter_busy` callback invoked **once** each time the loop leaves the
IDLE wait (i.e. on every IDLE→busy transition, before the first busy tick);
the app wires it to reset `self._heartbeat_tick = 0`. `sleep_fn` /
`on_enter_busy` stay injectable for deterministic tests.

**First-frame-visible anchoring (the fix for the blank-start bug):**
`_heartbeat_tick` is monotonic, so without a reset the first frame of a
turn would land on an arbitrary parity → ~50% of turns would open with a
blank dot for 600 ms, reading as "stalled" at the worst moment. Resetting
the tick to 0 on busy-entry makes `visible = tick % 2 == 0` always start
**`●` visible**, then wink off after one interval. This is a deliberate
divergence from CC's `useBlink` (whose dots derive from a shared global
clock and can start blank) — justified because neutrix has a *single*
liveness dot, not CC's list of per-tool dots, so the first frame carries
real "did my turn start?" signal.

**`format_heartbeat(state, store, tick, …)`**: `visible = tick % 2 == 0`.
Pick `glyph_style = HEARTBEAT_STALLED_GLYPH_STYLE if is_stalled else
HEARTBEAT_GLYPH_STYLE` — so `AWAITING_LLM`, `AWAITING_EXECUTOR`, and
`CANCELLING` all use the normal **bright-white** dot; only the stalled
sub-state is red. Emit `f"{HEARTBEAT_GLYPH} "` when visible, else `"  "`
(two spaces — same width, so the label never shifts). Label fragment
unchanged (`"LLM"` / `"tool: {name}"` / `"cancelling…"` /
`"LLM (stalled)"`). `IDLE` still returns `[]`.

The monotonic `_heartbeat_tick` counter, `_invalidate_app`, and
`_above_input` wiring are unchanged.

## Acceptance

**Automated (pytest):**
- `format_heartbeat(..., tick=0)` (even) → glyph fragment is `"● "`;
  `tick=1` (odd) → `"  "`; both keep the correct label fragment.
- **First-frame-visible**: after an IDLE→busy transition the tick is reset
  to 0, so the first rendered frame is `"● "` (never blank). Test the
  `on_enter_busy` reset directly and assert tick starts at 0 each busy phase.
- `CANCELLING` (not stalled) → normal bright-white glyph style.
- Stalled (`AWAITING_LLM`, `last_progress_at` older than threshold) →
  glyph style is `HEARTBEAT_STALLED_GLYPH_STYLE` and label `LLM (stalled)`.
- `IDLE` → `[]`.
- `heartbeat_loop` with injected fast `sleep_fn` advances the tick once
  per interval while busy and blocks on `store.changes()` when idle.
- No remaining references to the removed brightness constants / functions
  (grep clean); existing suite green.

**User-based (manual, in the user's `tmux-256color`):**
- Trigger an LLM turn: the `●` winks on/off at ~600 ms, bright white,
  robust (no banding), label `LLM` steady.
- Force a tool call: label shows `tool: {name}`, dot keeps winking.
- Induce a stall (slow model > threshold): dot winks **red**, label
  `LLM (stalled)`.
- Confirm the line never shifts horizontally as the dot winks.
