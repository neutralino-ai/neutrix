# v0.6.3 - full-width chat blocks and onion agent loop

## Problem

The main chat screen still has two product issues and one architecture issue:

- blocks are constrained to a narrow column, so the UI feels cramped on a
  full terminal;
- committed user messages and the editable draft block are not visually the
  same shape;
- `agent.py` owns conversation state, the tool loop, and the OpenAI client,
  which makes the "type a message, stream a reply" path harder to debug.

Users have also seen the app appear stuck after submitting a simple message
such as `hi`. The UI should surface model errors in the transcript/notice
and reliably restore the draft block to an editable, focused state when a
turn ends.

## Goal

Make the main app feel like a full-width list of model-visible blocks and
split the agent internals into onion layers:

- view layer: Textual widgets, theme-aware block styling, input focus, and
  slash-command handling;
- agent-loop layer: append user turns, call the LLM layer, append assistant
  turns/tool results, and repeat only while tools require a follow-up model
  request;
- LLM layer: OpenAI-compatible streaming request parsing only.

## Non-goals

- Add multi-line editing, transcript editing, branching, subagents,
  compaction, or prompt-cache management.
- Change the wire protocol for saved sessions.
- Add direct hard-coded color palettes beyond semantic Textual theme tokens.
- Redesign onboarding.

## Reference Notes

The Claude Code `query.ts` loop keeps immutable query inputs separate from
mutable loop state, records assistant messages before running tools, appends
tool results, and continues only when follow-up sampling is required.

The dr.sai-nano notes on Codex/agent loops point to the same useful
minimal pattern for this app: append-only history in the loop, model client
as a separate sampling layer, and UI events as projections of loop events.
Neutrix should adopt that boundary without importing larger systems such as
auto-compaction, subagent queues, or deferred tool registries.

## Design

1. Add `llm.py` with an `OpenAIChatLLM` that owns `AsyncOpenAI`, performs one
   streamed chat-completions request, yields token events, and yields a final
   assistant message event containing content and tool calls.
2. Add `agent_loop.py` with `Agent` and `AgentEvent`. `Agent.stream_reply()`
   owns message mutation and tool continuation. It appends the user turn
   exactly once, appends each assistant/tool message in order, yields UI-safe
   events, and emits an error event on failures.
3. Keep `agent.py` as a compatibility wrapper that re-exports the public
   names, so existing imports and saved-session behavior do not churn.
4. Update `tui.py` to consume the loop facade and render only three primary
   transcript block types:
   - system prompt block;
   - user message/draft block;
   - LLM message block.
   Tool and error blocks remain available for operational feedback but do
   not drive the primary visual language.
5. Make `#blocks`, `Message`, and `#composer` full-width within the screen.
   Use theme tokens and role classes for styling instead of raw literal
   colors. The draft block should share the user block surface, with the
   only behavioral difference being that its inner input is focusable and
   editable.
6. When a normal message is submitted, clear and blur nothing permanently:
   disable the draft during the active turn, then re-enable and focus it in
   the turn finalizer even when the model returns an error.

## Acceptance

- Launching the app shows a full-width system block followed by a full-width
  editable user draft block.
- A committed user message and the user draft use the same outer block
  styling.
- Committed `Message` blocks cannot receive focus; the draft input remains
  the only editable/focusable chat block.
- Submitting `hi` posts a user block, starts the model turn, and eventually
  either streams an LLM block or surfaces an error while restoring the draft
  input.
- The agent loop and OpenAI streaming client are tested separately from the
  Textual view.
- Existing onboarding submit isolation remains intact.
