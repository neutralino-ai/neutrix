# v0.8.0 - controller + heartbeat liveness indicator

> **Status: skeleton.** Detail will solidify after v0.7.0 lands. This PRD
> exists so the v0.7.0 design choices can be checked against what v0.8.0
> will need.

## Problem

There is no visible signal of "we are alive but waiting." Users cannot
tell the difference between:

- the LLM API is still streaming (waiting on bytes from the provider),
- a tool is still running (e.g. a long `run_shell`),
- the app has hung.

The information required to render that signal is scattered. `llm.py`
knows when it is mid-stream. `tools.py` (via `agent_loop.py`) knows when
a tool call is in flight. The renderer has no clean event to subscribe
to.

`agent_loop.py` is also still organized as "a thing the terminal calls"
rather than a pure pipeline. As long as that is true, every new status
signal requires another ad-hoc wire from somewhere inside the loop up to
the UI.

## Goal

- Rename `agent_loop.py` to `controller.py` and tighten it into a pure
  async generator that yields a defined set of `AgentEvent` types,
  including new status events: `llm_request_start`, `llm_request_end`,
  `tool_start`, `tool_end`.
- Add `llm_active: bool` and `executor_active: PendingToolCall | None`
  (or equivalent) to `ChatStore`, updated by a reducer that consumes the
  new events.
- Render a heartbeat / liveness pulse in the terminal chat that animates
  while either is active and stops when both are quiet. Use a low-cost
  Textual interval timer or equivalent — no external dependency.
- The status field shows which side is currently busy (LLM vs tool name)
  rather than a single opaque "thinking" label.

## Non-goals

- Refactor TUIView into a pure renderer. Deferred to v0.9.0.
- Add todos / reminders. v0.9.0.
- Add provider-side retry, timeout policy, or cancellation UI for stuck
  requests. Out of scope.
- Make the heartbeat configurable (speed, glyph). Pick one good default.
- Surface stream-event-level deltas. The heartbeat is a coarse signal,
  not a token-rate meter.
- Introduce a separate event-bus library. Stick with async generators.

## Design (skeleton — finalize after v0.7.0)

- Extend the `AgentEvent` union (currently in `agent_loop.py`) with
  `LlmRequestStart`, `LlmRequestEnd`, `ToolStart`, `ToolEnd`. Existing
  event types are untouched.
- `controller.py` emits the new events around each LLM call and each
  tool dispatch.
- A small reducer module (`store_reducer.py`, or a method on `ChatStore`
  named `apply`) maps each event type to one or more `ChatStore`
  mutator calls. The terminal chat hosts one drain coroutine that does:
  `async for event in controller.run_turn(...): store.apply(event)`.
- A `HeartbeatTimer` Textual widget subscribes to the store and pulses
  while `store.llm_active or store.executor_active is not None`.
- All existing v0.7.0 surface stays compatible. The renderer still reads
  from the store; v0.8.0 adds two more fields it can read.

## Acceptance (sketch)

- Manual: while the LLM is streaming, the heartbeat pulses and the
  status line names the active model. When a tool (e.g. a long
  `run_shell`) runs, the heartbeat keeps pulsing and the status names
  the tool. When both are quiet, the heartbeat stops.
- v0.7.0 acceptance scenarios (queued message display, transcript
  save/load) still pass.
- Unit tests covering the reducer: each event maps to the right store
  mutation, idempotently.
- Human gate per `.claude/rules/release-workflow.md` before tag.

## Open questions (to resolve before implementing)

- Is the heartbeat a single glyph in the status line, a dedicated
  widget, or a pulse on the input border? Pick one in design review.
- Does cancelling a long tool cancel the heartbeat? Tied to the broader
  cancellation story not yet specified.
- Does `controller.py` keep `ask()` / `stream_reply()` as the public
  entry, or rename? Defer to implementation.
