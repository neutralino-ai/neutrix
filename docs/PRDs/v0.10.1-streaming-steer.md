# v0.10.1 — re-enable streaming with cancel-as-steer

> **Status: skeleton.** v0.9.3 disabled streaming and explicitly
> deferred the re-enable to "a later PRD with the CC-aligned
> 'keep partial text' semantic." This is that PRD. Detail
> solidifies after the [v0.9.4 heartbeat](v0.9.4-heartbeat.md)
> splits + PRD refresh land — the heartbeat is the only liveness
> signal until streaming returns, and shipping it first gives
> users immediate relief without depending on this PRD.
> Phase 1 splits HTML at `docs/splits/v0.9.6-streaming-steer.html`
> comes when implementation begins; until then this file enumerates
> the problem and the design budget.

## Problem

After v0.9.3:

- `OpenAIChatLLM.stream_response` runs with `stream=False`. One
  awaited Chat Completions call returns one
  `LLMEvent("assistant", LLMResponse(...))`. The user sees nothing
  for the full duration of an LLM round.
- v0.9.4 heartbeat (when it ships) renders `"✻ LLM"` while
  `AWAITING_LLM`, which signals liveness but not progress. The
  user still cannot tell whether the model is 1 % or 99 % of the
  way through a 10-second response.
- Cancel-as-steer (v0.9.3) appends `[interrupted by user]` to a
  history that contains no partial assistant text — because at
  cancel time no partial text exists. The user steering message
  ("instead, just `ls`") arrives without the prior assistant turn
  visible, only the user_turn that triggered it. The "steer with
  prior intent in context" pitch is half-realized.

Closing the cancel arc means re-enabling streaming with CC's
"keep partial text" semantic — so the assistant turn at cancel
time is preserved in history (whatever bytes arrived before Esc),
not dropped.

## Goal

1. Switch `OpenAIChatLLM.stream_response` back to `stream=True`.
2. Re-introduce the v0.9.2-era token accumulator (per-delta
   `LLMEvent("token", str)` events) and the `index`-keyed
   `tool_calls` rebuild across streaming deltas.
3. `OpenAIChatLLM.stop()` closes the underlying `AsyncStream`
   eagerly (v0.9.2 mechanism) so the iterator's next `__anext__`
   exits cleanly — same broadcast contract as v0.9.3, different
   mechanism aligned with `stream=True`.
4. On cancel, the partial assistant text already appended to
   `ContextManager.messages` **stays**. Orphan `tool_use` blocks
   are repaired at next API-send by the v0.9.3 pairing layer
   (`_ensure_tool_result_pairing`), which already handles this
   case. No new code needed there.
5. The transcript renderer (terminal_chat) renders token deltas
   incrementally on the invalidation-driven path. Same approach
   v0.7.0 introduced; the v0.9.3 `_mirror_new_agent_messages`
   removal stays in place — UI subscribes to
   `ChatStore.changes()` and walks new bytes per-tick.

## Non-goals

- **Token-by-token UI redesign.** The append-only terminal-chat
  surface (v0.6.8) is unchanged; tokens stream into the same
  assistant block.
- **A separate streaming mode for tool-supporting providers vs
  non-streaming providers.** Both go through one streaming code
  path.
- **Partial tool_use rendering UI.** A half-formed tool_call in
  the live stream renders nothing user-visible until the call
  completes (then renders as the v0.8.2 colored `-> tool_use`
  line). Cancel during a partial tool_use repairs via the
  pairing layer; no new transcript shape.
- **Heartbeat changes.** v0.9.4 ships first; v0.9.6 doesn't
  modify the heartbeat behavior — whether to hide it during
  `AWAITING_LLM` once tokens are arriving is tracked as Open
  Question #1 below, not silently inside this non-goal.

## Design (sketch — finalize when v0.9.4 lands)

Today `src/neutrix/llm.py:48` defines
`LLMEvent(kind: str, data: Any)` with `kind="assistant"` the only
in-use value (carrying `LLMResponse(message, finish_reason)`).
v0.9.6 extends the kind set:

```python
LLMEvent(kind="token",     data=str)             # one delta of content
LLMEvent(kind="tool_call", data=ToolCall)        # completed tool_call (index-keyed)
LLMEvent(kind="assistant", data=LLMResponse)     # terminal: full message+finish_reason

@dataclass(frozen=True)
class ToolCall:                                  # restored from v0.9.2
    id: str
    name: str
    arguments: str                               # JSON string per OpenAI shape
```

- `OpenAIChatLLM.stream_response` (`src/neutrix/llm.py`):
  `stream=True`. Per chunk, emit `LLMEvent("token", delta.content)`
  for content deltas and accumulate `delta.tool_calls` into an
  index-keyed dict; on `finish_reason in {"tool_calls", "stop"}`
  emit one `LLMEvent("tool_call", ToolCall(...))` per completed
  call, then the final `LLMEvent("assistant", LLMResponse(...))`.
- `OpenAIChatLLM.stop()`: hold a reference to the in-flight
  `openai.AsyncStream`; close it on `stop()`. Idempotent.
  Replaces the v0.9.3 task-cancellation mechanism.
- `ContextManager._await_llm_round`
  (`src/neutrix/context_manager.py`): on each `"token"` event,
  append `data` to the *current* assistant turn's `content` field
  in `self.messages` (creating the turn on the first token of a
  round). On `"tool_call"`, accumulate into the same turn's
  `tool_calls`. On `"assistant"`, finalize. Because the bytes are
  in `messages` before any cancel fires, the v0.9.3 cancel path
  (which only appends `[interrupted by user]`) automatically
  preserves them — "keep partial text" needs no extra branch.
- The streaming switch requires no changes to
  `src/neutrix/executor.py`, `src/neutrix/store.py` (`apply` reducer),
  or key bindings in `src/neutrix/terminal_chat.py`.

## Acceptance (sketch)

- Manual:
  - Send a long prompt; tokens stream visibly into the assistant
    block at ~roughly the rate the provider emits them.
  - Press Esc mid-stream. The partial assistant content stays
    visible. The `[interrupted by user]` user turn renders next.
    Steer with "instead, just `ls`". The next assistant turn
    references the partial context appropriately.
  - Send a prompt that triggers a tool call. Cancel during the
    LLM-streaming-tool-call phase. The half-formed `tool_use`
    block is repaired by the pairing layer; the next round
    receives a valid `messages` payload.
- Automated:
  - Mock-LLM test: drive a fake stream that yields N token deltas
    and a `tool_call` event; assert `ContextManager.messages` ends
    with one assistant turn whose `content` is the concatenated
    deltas and whose `tool_calls` matches.
  - Mock-LLM cancellation test: yield half the deltas, fire
    `stop()`, assert `messages` contains the partial-content
    assistant turn and the next outgoing payload (after a
    follow-up user turn) passes pairing-layer validation.
- Human gate before tag.

## Open questions (resolve before implementing)

- **Open Question #1 — Hide heartbeat during AWAITING_LLM once
  streaming returns?** Streaming tokens are themselves a liveness
  signal; the heartbeat's `AWAITING_LLM` label becomes redundant
  once the first token has arrived. Default proposal: in
  `format_heartbeat`, hide the `AWAITING_LLM` fragment when
  `cm.state == AWAITING_LLM and len(messages[-1].get("content", "")) > 0`
  (i.e., the assistant turn already has bytes). Keep the
  `AWAITING_EXECUTOR` and `CANCELLING` heartbeat unchanged. Decide
  during implementation; cheap to flip.
- **What if the provider drops the connection mid-stream?**
  v0.9.2 surfaced this as an error event. v0.9.6 should keep
  the partial text in history (it's already there) and append
  an error notice. Confirm during implementation.
- **Tool_call streaming for providers that don't support it.**
  IHEP Anthropic gateway emits `tools:unsupported` (v0.6.7). For
  those, `stream=True` still works for plain text; tool_calls
  don't fire. No new branch needed — existing code paths
  handle it.
- **Cost / token rate display while streaming.** Tempting but
  out of scope for v0.9.6. Defer to a v1.x patch alongside
  `/status` cost tracking.
