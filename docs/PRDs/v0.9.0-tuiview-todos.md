# v0.9.0 - tuiview as pure renderer + todos and reminders

> **Status: skeleton.** Detail will solidify after v0.7.0 and v0.8.0 land.
> The point of writing it now is to keep the architecture honest: v0.7.0
> and v0.8.0 are only worth doing if v0.9.0 actually rides on them.

## Problem

After v0.8.0, `tui.py` and `terminal_chat.py` are read-side citizens of
`ChatStore` but still host business logic — input parsing, command
dispatch, queue draining, transcript rendering decisions. Adding a
substantial new feature (todos and reminders) without first stripping
that logic risks repeating the v0.6.x tangle that v0.7.0 set out to fix.

Todos and reminders themselves are a near-term user need. Like in
Claude Code, the LLM should be able to write down "things to do later"
that the user can see at a glance, mark done, or re-order. Without a
clean place for state (a `todos` list on the store) and a clean view
boundary (a `TodoPanel` widget that only reads), the feature would have
to bleed into the same files all over again.

## Goal

- TUIView (the union of `tui.py` and `terminal_chat.py`) becomes a pure
  reader of `ChatStore`. No mutation of chat state happens inside view
  code; all writes go through controller events or through user-input
  hand-off into the input queue.
- Add a `todos: tuple[Todo, ...]` field to `ChatStore` plus mutators
  (`add_todo`, `update_todo`, `complete_todo`, `remove_todo`).
- Add LLM-callable tools `todo_write` and `todo_update` (names final
  during implementation) in `tools.py`, modeled on Claude Code's
  `TaskCreate` / `TaskUpdate`.
- Render a `TodoPanel` widget in the terminal chat that subscribes to
  `store.todos` and displays the current list with status glyphs.
- Ship a `FakeView` in `tests/` — a ~50-line stdout renderer that reads
  `ChatStore` and prints the conversation, the queued messages, the
  heartbeat status, and the todo list. The FakeView test is the swap
  test: if it passes, view-pluggability is real, not aspirational.

## Non-goals

- Web frontend. The FakeView swap test stays in `tests/`; it is the
  proof, not a product.
- Multi-session UI or transcript browser.
- Reminder notifications via out-of-band scheduling (cron, system
  notifications). Reminders here are in-conversation only.
- Persistence of todos across runs. Treat them as session-scoped first.
  Persistence can be added later by extending the existing
  `transcript.py` format with a version bump.
- Refactoring `onboard.py`. Still out of scope for the v0.7.x arc.
- Importing or depending on Claude Code's todo tool implementation
  directly. Match the user-facing shape, not the internals.

## Design (skeleton — finalize after v0.8.0)

- Add `Todo` dataclass with `id`, `text`, `status`
  (`pending` | `in_progress` | `done`), `created_at`, `updated_at`.
- Add `todos` field to `ChatStore` plus mutators.
- Add `todo_write(items: list[dict])` and `todo_update(id, status)`
  tools.
- Add `TodoPanel` Textual widget that subscribes to the store and
  renders the list. Place it in the chat surface in a location agreed
  during design review (probably above the input editor, below the
  queued-message row).
- Audit `tui.py` and `terminal_chat.py` for any remaining state writes
  that bypass the controller. Move each one through either:
  (a) an input-queue hand-off (user actions),
  (b) a controller-emitted event (LLM / tool driven), or
  (c) a store mutator called from the reducer (state changes).
- Add `tests/fake_view.py`: a script that constructs a `ChatStore`,
  attaches a print-on-change subscriber, drives a synthetic controller
  event stream, and asserts the printed output matches a captured
  baseline.

## Acceptance (sketch)

- Manual: prompt "track these as todos: ship onion refactor, add web
  frontend, write CHANGELOG" — three items appear in `TodoPanel` as
  `pending`. Then "mark the first one in progress" — it shifts to
  `in_progress`. Then "mark it done" — it shifts to `done` and stays
  visible.
- v0.7.0 and v0.8.0 acceptance scenarios still pass.
- `tests/test_fake_view.py` runs the FakeView swap test and asserts
  on captured output. This is the "view layer is swappable" gate.
- Boundary check: `tui.py` and `terminal_chat.py` import only from
  `store` (plus Textual / stdlib). No imports from `controller`,
  `llm`, `tools`, or `agent_loop`.
- Human gate before tag.

## Open questions (to resolve before implementing)

- Are todos transient or persisted? Default in this PRD is transient;
  could revisit if user feedback says otherwise.
- Does the LLM see existing todos in its context automatically, or only
  when the user asks? Affects whether the system prompt injects the
  list.
- Where does the `TodoPanel` live on screen relative to the
  queued-message row and the heartbeat? Pick during design review.
- Does cancelling a turn roll back any todo mutations the LLM made
  during that turn? Probably no; document the chosen behavior.
