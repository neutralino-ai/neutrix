# v0.9.4 — heartbeat liveness indicator

> **Status: skeleton.** Detail solidifies once
> [v0.9.0 lifecycle events](v0.9.0-lifecycle-events.md) lands. The
> store already carries `llm_active: bool` and `pending_tool_calls`
> after v0.9.0; v0.9.4 is the renderer that pulses next to them.

## Problem

After v0.9.0, the store carries the canonical "what's in flight" state:

- `store.llm_active` is `True` while the LLM is streaming a round.
- `store.pending_tool_calls` is non-empty while a tool runs.

But there is still no visible cue. Users cannot tell the difference
between:

- the LLM API is still streaming (waiting on bytes from the provider),
- a tool is still running (a long `run_shell`, slow tool dispatch),
- the app has hung.

A 5–30 s tool call looks indistinguishable from a hung process.

## Goal

Add a one-line heartbeat in the message-area above the input that:

- Pulses with a single rotating glyph while `llm_active or
  pending_tool_calls`.
- Labels which side is busy:
  - `"LLM …"` while `llm_active` and no pending tool calls.
  - `"tool: {name} …"` while there is at least one pending tool call
    (first by insertion order; tool name from the head of
    `pending_tool_calls`).
- Disappears (renders nothing) when both are quiet.

## Non-goals

- **No new store fields.** Reads only `llm_active` and
  `pending_tool_calls`. If both are insufficient, that surfaces as a
  v0.9.0 bug, not new v0.9.4 work.
- **No `patch_stdout`-based rendering.** v0.7.0's lesson stands. The
  heartbeat lives in the message-area supplier that `DraftReader`
  already draws on the invalidation-driven path.
- **No configurable glyph or cadence.** One good default.
- **No cancellation UI.** Pressing Ctrl-C still does what it does
  today.
- **No latency / token-rate display.** The heartbeat is a coarse
  liveness signal, not a meter.
- **No additional event kinds.** The v0.9.0 event union is enough.

## Design (sketch — finalize once v0.9.0 lands)

- New `format_heartbeat(store, tick: int) -> list[(style, text)]` pure
  function in `terminal_chat.py`, returning prompt_toolkit fragments
  (or `[]` when idle). Tick is `(monotonic_ms // 200) % len(glyphs)`.
- Glyph set: `["⠋", "⠙", "⠹", "⠸", "⠼", "⠴", "⠦", "⠧", "⠇", "⠏"]`
  (the Braille spinner already used by many CLIs). Single rotating
  character; same vertical position across pulses.
- `TerminalChat._above_input` stack order becomes:
  1. Task panel (`format_task_panel`)
  2. Heartbeat line (`format_heartbeat`)
  3. Queued user messages
- `TerminalChat` gains a background `_heartbeat_ticker` coroutine that
  wakes every 200 ms while `store.llm_active or
  store.pending_tool_calls`, increments an internal tick counter, and
  calls `self._invalidate_app()`. It sleeps without ticking when both
  are quiet (subscribes to `store.changes()` to know when to wake).
  Cancelled cleanly on `TerminalChat` shutdown alongside the existing
  `worker` and `invalidator` tasks.

## Acceptance (sketch)

- Manual:
  - During a streaming assistant reply, the heartbeat shows `"⠋ LLM"`
    and rotates the glyph ~5×/s.
  - During a deliberately-slow `run_shell` (e.g. `sleep 5`), the
    heartbeat shows `"⠋ tool: run_shell"` and rotates.
  - When both `llm_active` and `pending_tool_calls` go to `False/empty`,
    the heartbeat line disappears within ~200 ms.
  - No visible blinking of unrelated regions (transcript, task panel,
    queue) during the pulse.
- Automated:
  - Unit test `tests/test_heartbeat_format.py`: `format_heartbeat(store,
    tick)` returns `[]` when idle, returns a single glyph + label
    fragment when `llm_active`, and uses the tool-name label when
    `pending_tool_calls` is non-empty.
- Human gate before tag.

## Open questions (resolve before implementing)

- Does the tool-name label include the arguments preview? Default
  proposal: no — too long to fit on one line; the existing `-> tool_use`
  line already showed the args.
- Multi-pending case (parallel tool calls, future): show
  `"tools: {n} running"` or pick the first? Default: pick the first
  until parallelism arrives.
- Tick cadence: 5 Hz (200 ms) vs 10 Hz (100 ms). 5 Hz keeps redraw
  cost low while still feeling alive.
- Position above or below the task panel? Default: between task panel
  and queue, so it sits in the user's eye-line just above the cursor.

## Known cosmetic risk

Under the v0.9.0 reducer-then-renderer ordering, the store transitions
through `pending_tool_calls = []` momentarily between a `tool_result`
event and the next round's `llm_request_start` event. If the agent
loop yields both events back-to-back (the common case) the gap is
sub-millisecond and the heartbeat never sees it. If async scheduling
inserts other work between the two yields, the heartbeat could blink
to hidden for one tick (~200 ms) before reappearing as `"LLM …"`.
Acceptable for v0.9.4 — observe in manual testing; if it bites,
gate the hide on a small debounce (e.g. 250 ms of continuous idle)
in a follow-up.
