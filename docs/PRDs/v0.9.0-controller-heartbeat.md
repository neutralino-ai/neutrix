# v0.9.0 - controller + heartbeat liveness indicator

> **Status: skeleton.** Detail solidifies after v0.8.0 (tasks) lands.
> Heartbeat is the first feature that needs a non-blinking live UI
> region; the controller refactor is the architectural prerequisite.

## Problem

There is no visible signal of "we are alive but waiting." Users
cannot tell the difference between:

- the LLM API is still streaming (waiting on bytes from the provider),
- a tool is still running (a long `run_shell`, a slow tool dispatch),
- the app has hung.

After v0.8.0 ships tasks and their LLM tools, this gap is the next
most-felt one: tool calls that take 5–30 s look indistinguishable
from a hung process.

Architecturally, `agent_loop.py` is still organized as "a thing the
terminal calls" rather than a pure pipeline that any frontend (TUI
today, web tomorrow per the Path B strategy) can drive. Every new
status signal currently requires another ad-hoc wire from somewhere
inside the loop up to the UI.

v0.9.0 fixes both at once: refactor `agent_loop.py` into a pure
controller and emit structured status events that the store consumes
and the renderer surfaces as a heartbeat.

## Goal

- Rename `agent_loop.py` → `controller.py`. Public entry stays an
  `AsyncIterator[AgentEvent]` (`run_turn` or similar; final name
  decided during implementation).
- Extend the `AgentEvent` union with `llm_request_start`,
  `llm_request_end`, `tool_start`, `tool_end`. Existing events stay.
- Add `llm_active: bool` and `executor_active: PendingToolCall | None`
  fields to `ChatStore`, updated by a reducer that consumes the new
  events.
- Render a heartbeat in the terminal chat that pulses while either
  field is "active" and stops when both are quiet. Must NOT blink
  via `patch_stdout` (the v0.7.0 lesson). Likely lives in the
  message-area above the input — same region the queue uses — so it
  rides on the existing invalidation-driven redraw path.
- The visible label names *which* side is busy: `"LLM …"` while
  streaming, `"tool: {name} …"` while a tool runs.

## Non-goals

- Refactor TUIView into a pure renderer. v0.10.0.
- Configurable heartbeat (speed, glyph). One good default.
- Surface stream-event-level deltas. The heartbeat is a coarse
  signal, not a token-rate meter.
- Cancellation UI for stuck requests. Cancellation gets its own
  design discussion before being scoped.
- Provider-side retry / timeout policy. Out of scope.
- A separate event-bus library. Stay with async generators.

## Design (skeleton — finalize after v0.8.0)

- Extend `AgentEvent` with the four new event kinds.
- `controller.py` emits them around each LLM API call and each tool
  dispatch.
- A reducer (`ChatStore.apply(event)` or a free function in
  `store_reducer.py`) maps each event kind to one or more store
  mutators. The terminal chat hosts one drain coroutine:
  `async for event in controller.run_turn(...): store.apply(event)`.
- A `HeartbeatLine` rendered in the message area above the input,
  driven by `store.llm_active` and `store.executor_active`. Uses a
  small set of glyphs cycled by tick. Tick comes from a background
  task that calls `app.invalidate()` every 200–300 ms while the
  heartbeat is active, and stops cleanly when it isn't.
- All v0.7.0 + v0.8.0 surface stays compatible. The renderer still
  reads the store; v0.9.0 adds two more fields to read and one more
  line to draw.

## Acceptance (sketch)

- Manual: while the LLM is streaming, the heartbeat pulses next to
  `"LLM"`. When a long tool runs, the heartbeat pulses next to the
  tool name. When both are quiet, the heartbeat line disappears.
  No blinking of unrelated regions during the pulse.
- v0.7.0 + v0.8.0 manual acceptance scenarios still pass.
- Unit tests covering the reducer: each event maps to the right
  store mutation, idempotently. Heartbeat tick task starts and
  stops in step with `llm_active or executor_active`.
- Human gate before tag.

## Open questions (to resolve before implementing)

- Glyph set: single rotating character, three-dot ellipsis cycle, or
  a small spinner frames sequence? Pick during design review.
- Does cancelling a long tool clear `executor_active` immediately, or
  on the next `tool_end`? Tied to a broader cancellation discussion.
- Does `controller.py` keep `ask()` / `stream_reply()` as the public
  entry, or rename to `run_turn()`? Defer to implementation.
- Does `Agent` (the v0.8.0 dataclass) become an internal of the new
  `controller`, or stay a separate facade? Decide once the public
  API is sketched.
