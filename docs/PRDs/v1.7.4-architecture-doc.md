## Problem

There is no single place that shows **how neutrix actually works**. The wiring —
the UI, the typed `ChatStore`, the `ContextManager` state machine, and the
`LLM`/`Executor`/`Advisor` it drives — is real and well-commented, but it lives
scattered across module docstrings and a decade of version annotations. A new
contributor (or the user returning after a gap) has to reconstruct the component
graph and the control flow of the load-bearing operations from source.
`docs/roadmap.html` answers *where the project is going*; it deliberately does not
answer *how the current system is wired*. As the pre-v2 surface has settled
(v1.7.0 shipped), that "how it works now" picture is worth capturing once,
visibly, and keeping current.

## Goal

A single self-contained **`docs/architecture.html`** the user opens in a browser
(same CSS/palette as `roadmap.html`/the splits files) that documents the
**current** (v1.7.0) system:

1. **The component graph** — the real edges, not a shorthand chain: the UI sends
   typed *events* to the CM **and separately** subscribes to the store's change
   notifier to render; the CM is the only writer of conversation state into the
   store; the Executor also writes back into the store; the Advisor runs off a
   separate slot while the CM is idle. `ChatStore` is the **single source of
   truth**; `ChatStore.changes()` is the one render/persist trigger.
2. **The CM state machine** — `IDLE → AWAITING_LLM → AWAITING_EXECUTOR` (loop) →
   `IDLE`, with the `CANCELLING` short-circuit — and the per-round `_drive` loop.
3. **Four function walkthroughs**, each a short control-flow story with
   **symbol-level** code citations: **cancel**, **rewind**, **save/load**,
   **goal**.

## Non-goals

- **Not a behavior change.** This documents existing behavior; zero code changes
  to `src/neutrix/` (the only `src` touch, if any, is none).
- **Not auto-generated API docs** and not a replacement for the module docstrings
  (which stay the authoritative low-level reference).
- **Not the legacy `tui.py` Textual app** (standing v0.9.x non-goal).
- **Does not replace `roadmap.html`.** Roadmap stays *direction + per-release
  decisions*; `architecture.html` is the *how-it-works-now* companion. Roadmap
  links to it.
- **Not exhaustive** — the four named functions plus the spine, not every command
  or every subsystem (compaction, skills, onboarding, subagents get a one-line
  mention and a pointer, not a full walkthrough).

## Claude Code reference

**No CC analog → no CC survey, no splits HTML.** Per `prd-compare-with-claude.md`'s
own out-of-scope clause ("pure internal refactors with no behavior change, and
features that genuinely have no CC analog"), this task qualifies on both counts:
it is documentation of **neutrix's own internals** with **no behavior change**, so
there are no design split points to survey against CC and no decisions to gate
turn-by-turn. The only choices are doc-level presentation (single page, layered
hub-and-spoke diagram, per-function sequence stories, symbol-based citations),
which are recorded in **Design** below rather than as split points.

## Design

A single HTML file, `docs/architecture.html`, reusing `roadmap.html`'s CSS
variables and card/entry shape so it sits in the same visual family.

### Sections

1. **Header + one-paragraph orientation** — the layered model in a sentence:
   *UI ↔ store ↔ CM ↔ { LLM, Executor, Advisor }*, with the caveat that the real
   graph is hub-and-spoke, drawn next.
2. **Component graph** (an ASCII/HTML box-and-arrow diagram, no JS) showing the
   **verified** edges:
   - `TerminalChat` (UI) **→** `ContextManager`: typed events via
     `handle_event(...)` — `UserMessageEvent`, `CancelEvent`, `SlotSwitchEvent`,
     `ClearEvent`, `ReplaceHistoryEvent`.
   - `TerminalChat` (UI) **←** `ChatStore.changes()`: the async notifier the
     render watcher subscribes to, which also drives per-turn persistence
     (session JSONL writer + the v1.7.0 `CostLedger`).
   - `ContextManager` **→** `ChatStore`: the **only** writer of conversation
     records / pending stream text / pending tool calls / tasks (single source of
     truth).
   - `ContextManager` **→** `OpenAIChatLLM.stream_response(...)`: streamed tokens
     + a final `LLMResponse` (with normalized `usage`).
   - `ContextManager` **→** `Executor.dispatch_all(...)`; `Executor` **→**
     `ChatStore` (`executor.store`, so `Task*` tools on the dispatch thread reach
     the store).
   - `TerminalChat` **→** `Advisor` (turn-end review on a *separate* fast-slot
     client, run while the CM is `IDLE`).
   - `ContextManager` **→** `CostLedger` (observes each completed turn; never
     drives the loop — v1.7.0 / v3-headless).
   Each arrow is **verified against the code** while authoring (Acceptance #2).
3. **The state machine** — the four `State` values and the `_drive` round loop
   (LLM round → optional tool dispatch → loop), plus where compaction / reminder
   injection / the timeout watchdog sit.
4. **Four function walkthroughs** — each: a one-line "what the user does", then the
   control-flow path with symbol citations and the key design choice:
   - **cancel** — `Esc` (draft keybinding `cancel_hook`) → `ContextManager.cancel()`
     → `CancelEvent` → `_do_cancel()`: `executor.cancel()` (tree-kills the process
     group), `llm.stop()` (closes the active stream), **keep-partial** (commits
     `store.pending_assistant_text`), and appends `INTERRUPTED_BY_USER_MARKER` as a
     real turn (append-marker semantics, v0.9.2/v0.9.3). State → `CANCELLING`,
     consumed by `_drive`.
   - **rewind** — `/rewind [N]` → `ContextManager.rewind_to(index)` →
     `_safe_rewind_index()` snaps the cut to a **tool-round boundary** (never
     bisect a round / leave a dangling `tool_calls`) → rebuild via the same
     `ReplaceHistoryEvent` path as load. Dropped user prompts are recallable with
     **Up/Down** (`recall_provider` / `recallable_user_turns`). (v0.9.7)
   - **save/load** — the **two** distinct mechanisms, and why: (a) `transcript.save`
     / `transcript.load` — explicit single-file export via `/save` `/load`; (b)
     `session_store` — **automatic** append-per-turn JSONL under
     `~/.cache/neutrix/sessions/…` (never `~/.claude`), restored by `--continue`
     (most-recent) / `/resume` (picker) through `ReplaceHistoryEvent`, plus the
     v1.7.0 `CostLedger.from_jsonl` rebuild. Both restore through the same
     `ReplaceHistoryEvent` seam as rewind.
   - **goal** — `/goal <text>` sets `_goal` and kicks the idle worker
     (`_GOAL_KICK`); after each turn settles, `_run_goal_continuations()` drives
     the next turn autonomously until **done / step cap (`GOAL_MAX_STEPS`) /
     interrupt**; a user message typed mid-goal sets `_goal_interrupt` (takeover).
     In-memory, not resumed across sessions. (v1.6.0)
5. **Footer** — version/asof stamp, pointer back to `roadmap.html` and the PRDs.

### Citations are symbol-based, not line-numbered

This is a **living** doc; line numbers rot on the next edit (v1.7.0 already shifted
`_do_cancel` and everything below it). Every code reference is
`module.py › Class.method()` / `module.py › function()`, which survives edits.
(Acceptance #4.)

### Companion-doc scope (the living-docs invariant)

Creating a second architecture surface would make `CLAUDE.md`'s "Living documents"
note false — it currently calls `docs/roadmap.html` "the project's **single**
user-visible living document" holding "**the current architecture**". This PRD's
scope therefore **includes**:
- a one-line update to that `CLAUDE.md` note: roadmap = direction + per-release
  decisions; `docs/architecture.html` = how the current system is wired; roadmap
  links to it; both must stay true to reality.
- a link from `docs/roadmap.html` to `docs/architecture.html`.

## Acceptance

1. `docs/architecture.html` exists, opens standalone in a browser (no JS, no
   external assets), and reuses `roadmap.html`'s CSS palette/shape.
2. The component-graph diagram shows the **real** edges — at minimum the
   UI→CM-events path **and** the distinct UI←`store.changes()` render/persist path,
   plus `Executor`→store and CM→{LLM, Executor, Advisor, CostLedger}. Each arrow is
   checked against current code; no linear-chain misrepresentation.
3. Each of the four functions (**cancel, rewind, save/load, goal**) has a
   walkthrough naming its real entry point, control-flow path, and key design
   choice.
4. Every code citation is **symbol-based** (`file › Symbol`), not `file:line`.
5. `CLAUDE.md`'s "Living documents" note is updated so it no longer claims a single
   architecture doc, and `docs/roadmap.html` links to `architecture.html`.
6. CHANGELOG entry added; version handled per the gate decision (tag `v1.7.4`, or
   ride untagged).
