# v0.3.0 — first-run onboarding TUI

## Problem

v0.2.0 writes a YAML template on first run, prints the path to stderr, and
exits. The user then has to:

1. Find `~/.config/neutrix/config.yaml` in their shell.
2. Open it in an editor.
3. Figure out which provider they have keys for.
4. Paste the key, save, and re-run `neutrix`.
5. Discover via an API error that the model name they kept by default isn't
   actually one the provider serves (or the key works but they typoed it).

Two friction points: editing YAML by hand, and only learning the credentials
are bad once a real chat message fails. The onboarding step needs to live
inside the same TUI surface the user already opened.

## Goal

When **neither** `fast` nor `strong` resolves to a valid slot (api_key missing
on both bound providers), launch an onboarding TUI instead of erroring out.
The onboarding screen lets the user:

- Paste an api_key for any of the known providers, inline.
- Verify any single model from a curated list with one keystroke (sends a
  bounded, 1-token request to the provider).
- Assign a verified model to the `fast` slot and another (or the same) to the
  `strong` slot.
- Save the result back to `~/.config/neutrix/config.yaml` and drop straight
  into the chat TUI.

## Non-goals

- Preserving YAML comments on save. The fresh-bootstrap path is the only one
  that triggers onboarding; comments are expendable here.
- Auto-discovery of available models from `/v1/models`. Provider-specific
  quirks aren't worth the code.
- Auto-verifying every model after the key lands. On-demand only — the user
  decides what to spend tokens on.
- Re-running onboarding from inside the chat TUI (e.g. `/onboard`). If the
  config breaks later, the user edits the YAML or deletes it and re-runs.
- Persisting model verification status across runs. It is session-only state.
- Multi-key encryption / OS keychain integration. The YAML on disk is plain
  text, same as today.

## Design

### Trigger logic (`cli.py`)

```python
fast_slot = strong_slot = None
fast_err = strong_err = None
try: fast_slot = config.slot("fast")
except ConfigError as e: fast_err = e
try: strong_slot = config.slot("strong")
except ConfigError as e: strong_err = e

if fast_slot is None and strong_slot is None:
    OnboardApp(config).run()        # blocks; user fills key + assigns slots
    config = load_config()           # re-read post-onboard YAML
    fast_slot = config.slot("fast")  # must succeed now, or hard error
elif fast_slot is None:
    fast_slot = strong_slot          # graceful fall-through to whatever works
```

`fast` remains the default slot for the chat session.

### Hardcoded model catalog (`config.py`)

```python
PROVIDER_DEFAULT_MODELS: dict[str, list[str]] = {
    "ihep": [
        "anthropic/claude-haiku-4-5",
        "anthropic/claude-opus-4-7",
        "anthropic/claude-sonnet-4-6",
        "deepseek/deepseek-v3",
    ],
    "deepseek": ["deepseek-chat", "deepseek-reasoner"],
    "glm": ["glm-4.6", "glm-4-plus", "glm-4-flash"],
}
```

Lives in `config.py` because it is config-adjacent data, and a dedicated
`defaults.py` for one dict is the YAGNI failure mode.

### Onboarding screen (`onboard.py`, new module)

Layout per provider section, scrollable:

```
ihep
  base_url: https://aiapi.ihep.ac.cn/apiv2/
  api_key:  [********xyz4]                 ← editable Input, password-masked
  models:
    [?] anthropic/claude-haiku-4-5
    [✓] anthropic/claude-opus-4-7    fast
    [✗] anthropic/claude-sonnet-4-6
    [?] deepseek/deepseek-v3         strong
```

Per-row state:
- api_key Input: placeholder text `EMPTY` when unset; masked when set. On
  commit (Enter or blur), the YAML is rewritten immediately so the user
  doesn't lose the key by quitting early.
- model rows: status icon (`?` `✓` `✗`) + name + optional `fast`/`strong` tag.
  Each row is focusable.

Bindings:

| Key  | Action                                          |
|------|-------------------------------------------------|
| Tab / Shift+Tab | move focus between Inputs and model rows |
| Enter (on Input) | commit key, persist to YAML            |
| `v`  | verify the focused model (1-token API call)     |
| `f`  | assign the focused verified model to `fast`     |
| `g`  | assign the focused verified model to `strong`   |
| `s`  | save slot assignments and exit to chat          |
| `q`  | exit without launching chat                     |

Focus-conflict rule: action keys (`v` `f` `g` `s` `q`) only fire when focus
is on a model row, never on an Input. Letter typing inside an Input goes to
the Input.

### Verification (`onboard.py`)

```python
async def verify_model(base_url, api_key, model) -> bool:
    client = AsyncOpenAI(base_url=base_url, api_key=api_key)
    try:
        await asyncio.wait_for(
            client.chat.completions.create(
                model=model,
                messages=[{"role": "user", "content": "ping"}],
                max_tokens=1,
            ),
            timeout=8.0,
        )
        return True
    except Exception as e:
        logger.warning("verify {} failed: {}", model, e)
        return False
```

Bounded cost (1 token), bounded time (8 s), broad except — anything that
isn't a clean reply is `✗`.

### YAML write-back (`config.py`)

```python
def save_config(config: Config, fast: dict | None, strong: dict | None,
                path: Path = CONFIG_PATH) -> Path:
    data = {
        "providers": {n: dict(p) for n, p in config.providers.items()},
        "fast": fast or config.slots["fast"],
        "strong": strong or config.slots["strong"],
    }
    path.write_text(yaml.safe_dump(data, sort_keys=False), encoding="utf-8")
    return path
```

Round-trippable via `load_config`. Loses the template comments. Acceptable
per non-goals.

### Save mechanics

- API-key commits: written to YAML immediately on Enter/blur.
- Fast/strong assignments: held in memory; written to YAML on `s`.
- `q` writes nothing further (api_keys already persisted by then).

## Acceptance

1. Clean machine, no config file: `neutrix` writes the template and exits
   (unchanged from v0.2.0).
2. Config exists but both providers' `api_key` are `""`: `neutrix` launches
   the onboarding TUI instead of erroring.
3. In onboarding, the user pastes a working key on `ihep`, presses `v` on a
   model row → status flips to `✓` within 8 s.
4. User presses `f` then moves to another row, presses `g`, then `s` →
   YAML is updated; chat TUI opens on the assigned `fast` slot.
5. Reloading after onboarding: `neutrix` opens directly into the chat with
   the chosen fast slot active, no onboarding.
6. Editing the api_key Input and pressing Enter persists immediately;
   pressing `q` afterwards keeps the key for next time.
7. `pytest`: smoke tests pass; new tests cover `save_config` round-trip and
   the "both slots unresolvable" trigger predicate.
8. `ruff check`: clean.
9. `neutrix --version`: reports `0.3.0`.

## Out of scope (deferred)

- Re-launching onboarding from a slash command (`/onboard`).
- Comment-preserving YAML edits.
- Automated `/v1/models` discovery.
- Storing per-key verification cache between runs.
- Multi-config / profile support.
