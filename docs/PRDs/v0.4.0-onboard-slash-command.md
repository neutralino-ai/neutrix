# v0.4.0 — `/onboard` slash command (re-enter onboarding mid-chat)

## Problem

The onboarding TUI only opens on first run — when neither `fast` nor
`strong` slot resolves. Once a user has a working config, there is **no
way to re-enter onboarding** without leaving the chat TUI, hand-editing
`~/.config/neutrix/config.yaml` (deleting the api_keys to force the
trigger), and re-launching. Real-world cases this blocks:

- Adding a new provider (e.g. just got a deepseek key while ihep is set).
- Rotating an api_key.
- Re-verifying a model after the provider says "this should work now."

## Goal

Add a `/onboard` slash command to the chat TUI that opens the same
onboarding surface used on first run. After the user dismisses
(save-and-launch or quit), the chat resumes on the **same active slot**;
the YAML is reloaded so `/fast` and `/strong` see any updated bindings.

## Non-goals

- Auto-switching the live agent to a freshly bound slot after onboarding.
  User explicitly chose "stay on current slot, only update config" — they
  run `/fast` or `/strong` themselves to adopt new bindings.
- A CLI flag `--onboard`. Slash command in TUI is the only re-entry.
- Per-conversation onboarding state (history, drafts) — onboarding is
  stateless across invocations.
- Resetting / clearing chat history on re-onboard.

## Design

### Refactor: `OnboardApp` → `OnboardScreen` + thin `OnboardApp` wrapper

Textual's Screen stack is the right primitive for "open another full-page
view inside the same App." Pull the body of `OnboardApp` into
`OnboardScreen(Screen[bool])`. Keep `OnboardApp(App[bool])` as a 10-line
wrapper that pushes `OnboardScreen` as its initial screen — preserves
the existing first-run CLI path and `run_onboarding(config) -> bool` API.

Mechanical changes inside the screen:

| App version              | Screen version          |
|--------------------------|-------------------------|
| `class OnboardApp(App[bool])` | `class OnboardScreen(Screen[bool])` |
| `self.exit(True/False)`  | `self.dismiss(True/False)` |
| `self.set_timer(...)`    | `self.set_timer(...)` (unchanged — MessagePump) |
| `self.notify(...)`       | `self.notify(...)` (unchanged — Widget) |
| `self.focused`           | `self.focused` (Screen has it) |
| `App.CSS`                | `Screen.CSS` (same attribute name) |

### New `/onboard` slash command in `tui.py`

```python
async def _cmd_onboard(self, args: list[str]) -> None:
    from neutrix.config import ConfigError, load_config
    from neutrix.onboard import OnboardScreen

    saved = await self.push_screen(OnboardScreen(self.config),
                                   wait_for_dismiss=True)
    try:
        self.config = load_config()
    except ConfigError as e:
        self._post("error", f"config reload failed: {e}")
        return
    if saved:
        self._post(
            "system",
            "onboarding saved. Use /fast or /strong to switch to "
            "the new binding; current slot unchanged.",
        )
    else:
        self._post("system", "onboarding cancelled.")
```

`push_screen(..., wait_for_dismiss=True)` is the standard Textual idiom
for "open a sub-flow, wait for its result." The live agent's `slot` is
untouched.

Help text adds the new line.

### CLI path

`cli.py` keeps calling `run_onboarding(config)` for the first-run case,
which now goes via `OnboardApp` → `OnboardScreen`. Same return contract.

## Acceptance

- From a chat session with a working config, typing `/onboard` opens the
  onboarding screen (provider sections + api_key Inputs + model rows).
- All onboarding bindings (`v`, `f`, `g`, `s`, `q`, `up`, `down`,
  `ctrl+c×2`, `esc`) work the same as v0.3.1.
- Saving via `s` updates `~/.config/neutrix/config.yaml` and returns to
  chat on the **same slot**; toast says "use /fast or /strong to switch."
- Quitting (`q` or `ctrl+c×2`) returns to chat without any config change;
  toast says "onboarding cancelled."
- First-run behavior is unchanged: empty `api_key`s on both slot
  providers still triggers `run_onboarding(config)` from `cli.py`.
- Headless `run_test()` pilot can push `OnboardScreen` onto a test app
  and dismiss it.
- `neutrix --version` reports `0.4.0`.
- `pytest` + ruff stay green.

## Out of scope (deferred)

- A second `--onboard` CLI flag (might add later if there's a scripted
  use case).
- A `/keys` lighter-weight command for "just paste a new api_key, don't
  touch slots."
- Auto-switching to a verified model on dismiss.
- An "onboarding history" — verification status is still session-only.
