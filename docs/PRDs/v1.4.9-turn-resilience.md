# v1.4.9 — turn resilience (no-progress LLM timeout)

> **Status: finalized.** Bug-fix release, root-caused in a LIVE debug session.
> Decided autonomously under the `build-to-v2.0.0` /goal; **user-reported "neutrix
> get stuck again" + "LLM stalled often" + "for timeout, how claude handle it?"**.
> Splits: [docs/splits/v1.4.9-turn-resilience.html](../splits/v1.4.9-turn-resilience.html).

## Problem

Two live symptoms on the user's session (running deepseek-v4-pro via an IHEP
proxy, a *minutes*-scale model):

1. **"Stalls often"** — the UI stall hint (red dot, ~50s) fired on essentially
   every strong-model response.
2. **"Gets stuck" / "dies in the middle"** — turns hung indefinitely.

**Root cause (proven live with py-spy + `ss`):** the timeout watchdog
(`_llm_timeout_watchdog`) was a **hard wall from round start** — a single
`asyncio.sleep(llm_timeout_s)` then cancel — and `last_progress_at` was set
once at round start and **never bumped per streamed token**. So the stall clock
measured total round time, not inactivity (→ fires on every slow response), and
a long-but-live stream could be cancelled mid-flight. Separately, the LLM client
was built with **no explicit timeout** (`AsyncOpenAI(base_url, api_key)`), so a
dead/half-closed proxy connection (observed: two CLOSE-WAIT sockets to
127.0.0.1:7890 with unread bytes) left the streaming read parked far past any
sane budget.

(The frequent *silent* "turn just ends" the user also saw earlier was a separate
artifact — a **stale long-lived 0.9.6 process** hitting `max_rounds=3`, a value
current source doesn't contain; remedied by restarting on current source.)

## Goal

Make turn liveness **inactivity-based**, matching Claude Code's semantics:
- A slow-but-live stream is **never** cancelled, however long it runs.
- A genuinely silent/dead connection errors out within one quiet `llm_timeout_s`,
  with a **visible** marker — never an infinite hang.
- One knob (`slot.llm_timeout_s`) drives both the transport read timeout and the
  app watchdog; the UI stall hint stops false-firing.

## Non-goals

- Non-streaming fallback + N-retry on timeout (CC does this; YAGNI here — the
  turn errors visibly and the user re-prompts). `max_retries=2` covers only the
  initial request.
- A per-tool timeout (the watchdog covers AWAITING_LLM only; a hung Bash is out
  of scope — see the separate tool-liveness work).
- Changing the `[LLM timeout after Ns]` marker wording.

## Claude Code reference

See [docs/splits/v1.4.9-turn-resilience.html](../splits/v1.4.9-turn-resilience.html).
CC: `services/api/claude.ts` (`STREAM_IDLE_TIMEOUT_MS=90s`, reset per chunk via
`resetStreamIdleTimer()` at 1877/1940-1941; stall logging at 30s), `client.ts:144`
(`API_TIMEOUT_MS=600s` initial, covers connect+headers only), `withRetry.ts`
(10× exp backoff + non-streaming fallback).

### Split point: hard wall vs inactivity
- **CC**: inactivity, reset per chunk (claude.ts:1940-1941).
- **Decision**: **Follow CC** — per-token `last_progress_at` bump + a no-progress
  watchdog poll loop.

### Split point: transport timeout AND app watchdog
- **CC**: hand-rolls the idle watchdog (Node SDK timeout misses the body).
- **Decision**: **Both.** `AsyncOpenAI(timeout=httpx.Timeout(llm_timeout_s, connect=10))`
  — httpx's `read` timeout is itself a per-chunk inactivity cap, so a dead socket
  raises `ReadTimeout` even if the app cancel can't break it; the app watchdog is
  the suspenders + drives the UI.

### Split point: one knob vs two timeouts
- **CC**: separate 90s idle + 600s initial.
- **Decision**: **Alternative** — one knob `llm_timeout_s` (300s) for inactivity,
  fixed 10s connect. Calibrated for the slow IHEP models (time-to-first-token can
  be minutes); per the user's one-knob timeout-calibration rule.

### Split point: retry / fallback
- **CC**: non-streaming fallback + 10× retry.
- **Decision**: **Alternative (YAGNI)** — visible `[LLM timeout]`, user re-prompts;
  `max_retries=2` on the initial request only.

### Split point: marker wording + poll interval
- **Decision**: keep `[LLM timeout after Ns]`; poll at `min(llm_timeout_s, 2.0)`;
  per-iteration `state != AWAITING_LLM` guard + `None` mid-poll handled.

## Design

### `src/neutrix/llm.py`
- `_build_client`: `AsyncOpenAI(..., timeout=httpx.Timeout(slot.llm_timeout_s,
  connect=10.0), max_retries=2)` (was no timeout). `import httpx`.

### `src/neutrix/context_manager.py`
- `_call_llm` token branch: `self.last_progress_at = time.monotonic()` on each
  `token` event (before `store.extend_assistant_stream`).
- `_llm_timeout_watchdog`: poll loop — `poll = min(llm_timeout_s, _WATCHDOG_POLL_CEILING_S=2.0)`;
  each wake: return if state left `AWAITING_LLM`; `continue` if `last_progress_at is None`;
  cancel(reason="timeout") when `monotonic() - last_progress_at > llm_timeout_s`.

## Acceptance

- **Automated**: `test_steady_stream_survives_past_total_timeout` (a stream whose
  total time > `llm_timeout_s` but with no single inter-token gap > it is NOT
  cancelled and completes with full content); `test_mid_stream_stall_fires_after_inactivity`
  (tokens then silence > `llm_timeout_s` → watchdog fires, partial kept, marker lands);
  the existing 4 timeout tests stay green (silent stream still caught; user-cancel
  still wins; fast success doesn't false-fire). Full suite green (388); ruff clean.
- **Manual**: with a slow model, a multi-minute live response streams to completion
  without the red stall dot latching; pull the proxy mid-call → the turn errors with
  `[LLM error]`/`[LLM timeout]` within `llm_timeout_s` instead of hanging.
- **Operational**: the user must reinstall current source into the run venv and
  restart (the stuck process was a stale 0.9.6 install).
- Ship `v1.4.9`.

## Out of scope

- Non-streaming fallback/retry, per-tool timeout, the LLM/executor/advisor status
  bar (separate follow-up the user requested).
