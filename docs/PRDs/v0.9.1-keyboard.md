# v0.9.1 — keyboard ergonomics (Codex parity + backslash continuation)

## Problem

The `prompt_toolkit` draft editor in `src/neutrix/terminal_chat.py` has
four keyboard-handling defects that have been baked in since v0.6.8
moved the chat to append-only terminal mode. None are caused by
v0.9.0; they have been waiting for someone to bind them properly.

1. **`Ctrl+C` exits on the first press.** `_input_loop` catches the
   `KeyboardInterrupt` that prompt_toolkit raises (because
   `handle_sigint` defaults to `True`) and immediately quits. There
   is no "press Ctrl+C again to confirm" affordance and no way to
   cancel the exit. Hitting `Ctrl+C` while typing — a reflex move
   to clear a half-written message in any shell — loses the chat
   session.
2. **`Ctrl+D` on an empty buffer also exits on the first press**,
   for the symmetric reason: prompt_toolkit raises `EOFError`,
   `_input_loop` catches it, the session dies. Ctrl+D is the other
   reflex shell-exit chord and deserves the same "press again to
   confirm" treatment as Ctrl+C — anything else is an inconsistent,
   surprising UX. Added at the Phase-3 review when the user
   rejected a v0.9.1 draft that did not cover Ctrl+D.
3. **`Ctrl+Z` is swallowed.** Raw mode is on; the kernel never
   delivers `SIGTSTP`. No binding exists for `c-z`, so the keystroke
   is dropped on the floor — the user cannot background `neutrix`
   the way every Unix shell expects.
4. **Newline keys are sparse.** Only `Ctrl+J` and `Alt+Enter`
   (`escape, enter`) insert a newline. Users coming from any modern
   chat surface reach for `Shift+Enter` first, find nothing happens,
   then either accidentally submit or have to discover Ctrl+J.

Two near-tree projects already shipped good answers — Codex (Rust /
ratatui) has the most disciplined keyboard contract, and Claude
Code's chat input borrows the bash line-continuation convention so
typing `\` then `Enter` keeps the buffer open. v0.9.1 is the
smallest change that ports both.

## Goal

Match Codex's keyboard contract for the draft editor (minus the
visible hint UI — see review note below), plus add bash-/Claude-style
trailing-backslash line continuation. After v0.9.1:

1. **`Ctrl+C` and `Ctrl+D`** — pure-timer double-press, with
   **independent timers per chord**. Each chord arms and tracks
   its own `armed_at` timestamp; the two timers never reset or
   override each other. The displayed hint always names the most
   recently pressed chord so the user sees the affordance for the
   key they just hit, but cross-key presses do **not** disturb
   the other chord's clock:
   - **First press of `Ctrl+C`** arms the c-c timer (1-second
     window) AND renders a dim-gray `press Ctrl+C again to exit`
     hint directly above the input cursor.
   - **First press of empty-buffer `Ctrl+D`** arms the *separate*
     c-d timer (independent 1-second window) AND renders a
     dim-gray `press Ctrl+D again to exit` hint.
   - **Second press of the same chord** within that chord's own
     window exits: `Ctrl+C` → `KeyboardInterrupt`, `Ctrl+D` →
     `EOFError`. Both exceptions are already caught by the
     existing `_input_loop`.
   - **Cross-key presses do NOT reset the other chord's timer
     and do NOT confirm.** Pressing `Ctrl+D` while c-c is also
     armed arms c-d on its own clock and refreshes the displayed
     hint to `press Ctrl+D again to exit`, but the c-c timer
     keeps ticking on its original arming time. So `Ctrl+C` →
     `Ctrl+D` → `Ctrl+C` (all within the c-c window of the
     first press) **exits** via `KeyboardInterrupt` — the c-c
     window was never touched by the intervening `Ctrl+D`.
     Likewise `Ctrl+C` → `Ctrl+D` → `Ctrl+D` exits via
     `EOFError` (c-d's own second tap).
   - The hint uses the same dim foreground style as queued user
     messages (`fg:ansibrightblack`) — the bright-yellow palette
     of an earlier draft was rejected at Phase-3 review as
     visually too loud, contrary to the rest of the UI's muted
     hierarchy. The hint is a soft affordance, not a warning.
   - **After exactly 1 second from each arming**, that chord's
     auto-fade task fires and re-invalidates the app. The hint
     visible-or-not check naturally tracks whichever chord is
     currently within its own window; if both expire, the hint
     disappears.
   - **`Ctrl+D` only enters the quit dance when the buffer is
     empty.** When there is text in the draft, `Ctrl+D` keeps its
     default Emacs behavior (forward-delete-character) so power
     users don't lose a working chord. The buffer-empty branch is
     guarded by a `prompt_toolkit.filters.Condition` on the
     binding.
   - **Other keys are ignored** for the purpose of arming. Typing
     in between two quit presses neither cancels the arming nor
     restarts any timer; each chord's window is set by that
     chord's own most recent press alone.
2. **`Ctrl+Z`**: calls `event.app.suspend_to_background()` —
   prompt_toolkit's built-in shim that disables raw mode, sends
   `SIGTSTP` (`libc::kill(0, SIGTSTP)` equivalent), then re-applies
   raw mode on `SIGCONT`. Mirrors Codex's `tui/job_control.rs:178`
   exactly.
3. **Newline keys**: `Ctrl+J` and `Alt+Enter` (`escape, enter`)
   insert a newline at the cursor. Both work on every terminal.
   `Shift+Enter` and `Ctrl+Enter` are **not bindable in
   prompt_toolkit 3.0.x** — its `Keys` enum has no entries for them
   (`Keys.value` lists `s-tab`, `s-left`, `s-up`, …, `s-pagedown`
   but no `s-enter` or `c-enter`). In practice, **most terminals
   send `Shift+Enter` as `Ctrl+J`** (the legacy LF byte) by default,
   so binding `c-j` already covers Shift+Enter on common emulators
   (gnome-terminal, xterm, default macOS Terminal, Linux console).
   Terminals that opt into CSI-u keyboard protocol (kitty, foot,
   WezTerm in some configs) need prompt_toolkit upstream support
   for `s-enter` — out of scope; can ride along whenever
   prompt_toolkit ships it. `Ctrl+Enter` is left unbound;
   confirmed acceptable in v0.9.1 review.
4. **Backslash line continuation**: if the buffer ends with a
   literal `\` (and the cursor is at end-of-buffer), `Enter`
   deletes the trailing `\` and inserts a newline instead of
   submitting. Matches bash and Claude Code's chat input.

## Non-goals

- **No mid-stream cancellation.** Codex's `on_ctrl_c` *also*
  submits an `AppCommand::interrupt()` when a task is running. We
  defer that — without v0.9.0+ cancellation primitives in
  `Agent.stream_reply`, the first `Ctrl+C` cannot meaningfully
  interrupt an in-flight LLM call. Plumbing that through is
  v0.9.2 scope (see `docs/PRDs/v0.9.2-cancellation.md`). For
  v0.9.1, the first `Ctrl+C` / `Ctrl+D` only arms the exit
  shortcut; the stream keeps running until it finishes normally
  or the user double-presses to exit the process.
- **No `Ctrl+D` quit when the buffer is non-empty.** Forward-
  delete-character is the standard Emacs binding for that case
  and the most useful thing the chord could do mid-draft. The
  quit dance is reserved for the empty-buffer case, matching
  Codex / bash / readline conventions.
- **No `Ctrl+\` / `Ctrl+O` / paste-burst handling.** Codex's
  composer has a deep paste-burst detector for terminals that
  send paste as individual keystrokes. prompt_toolkit handles
  bracketed paste itself; we accept whatever it does.
- **No transcript format change.** The hint render, the suspend
  side-effects, and the new newline keys are all view-only.
  `ChatStore` is untouched.

## Design

### `QuitArmingState` (new, in `terminal_chat.py`)

Pure timer + a string identifying *which* chord armed it. One
predicate answers both "is any chord currently armed?" (for hint
visibility / re-arming logic) and "is THIS chord armed?" (for exit
confirmation).

```python
@dataclass
class QuitArmingState:
    """Pure-timer arming tracker for the Ctrl+C / Ctrl+D double-press.

    Each chord (``"c-c"``, ``"c-d"``) carries its own independent
    ``armed_at`` timestamp in :data:`armed_at`. Cross-key presses
    never touch the other chord's clock; the displayed hint refreshes
    to name the most recently pressed chord, but each timer keeps
    running on its own schedule. Only the same chord can confirm —
    a second press of THIS chord, within THIS chord's own window.
    """

    QUIT_WINDOW_S: ClassVar[float] = 1.0
    armed_at: dict[str, float] = field(default_factory=dict)
    last_armed_key: str | None = None  # most recent chord — drives the hint

    def within_window(self, key: str | None = None) -> bool:
        """Predicate with two modes.

        - ``key is None`` (renderer use): True iff the
          most-recently-armed chord is still within its own
          window. Drives hint visibility.
        - ``key`` set to ``"c-c"`` / ``"c-d"`` (binding use): True
          iff THAT chord's own timer is still within
          :data:`QUIT_WINDOW_S`. Cross-key arms do not affect
          this — each chord's timer is independent.
        """
        if key is None:
            return self.hint_text() is not None
        armed_at = self.armed_at.get(key, -math.inf)
        return time.monotonic() - armed_at < self.QUIT_WINDOW_S

    def arm(self, key: str) -> None:
        self.armed_at[key] = time.monotonic()
        self.last_armed_key = key

    def hint_text(self) -> str | None:
        """Renderer-facing hint string. Names the *most recently
        pressed* chord, as long as its own window is still open.

        The "fallback to the OTHER chord if its window is still
        open" branch is unreachable: if the most-recent chord has
        expired, the earlier-armed chord must also have expired
        (it was armed even further in the past). See PRD for the
        proof sketch.
        """
        if self.last_armed_key is None:
            return None
        if not self.within_window(self.last_armed_key):
            return None
        if self.last_armed_key == "c-c":
            return "press Ctrl+C again to exit"
        if self.last_armed_key == "c-d":
            return "press Ctrl+D again to exit"
        return None
```

### Bindings (rewrite of `build_draft_key_bindings`)

The `c-c` and (empty-buffer) `c-d` paths share the same arming
helper. The helper takes the chord identity AND the exit exception
as arguments — confirmation requires the chord's own timer match,
and a cross-key press falls through to arm only its own chord
(leaving the other chord's clock untouched). The chord identity
threads through to `within_window` / `arm`.

```python
def build_draft_key_bindings(quit_state: QuitArmingState):
    from prompt_toolkit.application.current import get_app
    from prompt_toolkit.filters import Condition

    bindings = KeyBindings()

    def _arm_or_exit(event, chord, exit_exception):
        if quit_state.within_window(chord):
            event.app.exit(exception=exit_exception)
            return
        quit_state.arm(chord)
        event.app.invalidate()  # paint the hint now

        # Schedule a redraw 1 s later so the hint clears on its own
        # without needing the user to press another key. The
        # background task lives on the app's task group and is
        # automatically cancelled when prompt_async returns.
        async def _fade_hint() -> None:
            await asyncio.sleep(QuitArmingState.QUIT_WINDOW_S)
            event.app.invalidate()
        event.app.create_background_task(_fade_hint())

    @bindings.add("c-c")
    def _(event) -> None:
        _arm_or_exit(event, "c-c", KeyboardInterrupt)

    # Ctrl+D enters the quit dance ONLY when the buffer is empty;
    # otherwise it falls through to prompt_toolkit's default
    # forward-delete-character. The filter is what makes the
    # fall-through work — without it the binding would always fire
    # and the user could not forward-delete inside a draft.
    @Condition
    def _buffer_is_empty() -> bool:
        return not get_app().current_buffer.text

    @bindings.add("c-d", filter=_buffer_is_empty)
    def _(event) -> None:
        _arm_or_exit(event, "c-d", EOFError)

    @bindings.add("c-z")
    def _(event) -> None:
        event.app.suspend_to_background()

    @bindings.add("enter")
    def _(event) -> None:
        buf = event.current_buffer
        # Bash / Claude-style: trailing backslash + Enter = continuation
        if buf.cursor_position == len(buf.text) and buf.text.endswith("\\"):
            buf.delete_before_cursor(count=1)
            buf.newline()
            return
        event.app.exit(result=buf.text)

    # Newline keys — only what prompt_toolkit's Keys enum recognizes.
    # See "Newline keys" goal above for why s-enter / c-enter are
    # not bindable in prompt_toolkit 3.0.x.
    @bindings.add("c-j")
    def _(event) -> None:
        event.current_buffer.newline()

    @bindings.add("escape", "enter")  # Alt+Enter
    def _(event) -> None:
        event.current_buffer.newline()

    @bindings.add("c-a")
    def _(event) -> None:
        move_buffer_to_line_start(event.current_buffer)

    @bindings.add("c-k")
    def _(event) -> None:
        delete_buffer_to_line_end(event.current_buffer)

    return bindings
```

### `PromptSession` change

Pass `handle_sigint=False` so SIGINT isn't translated to
`KeyboardInterrupt` before our binding sees the `\x03` byte. In
prompt_toolkit 3.0.x `handle_sigint` is a method arg on
`prompt` / `prompt_async`, not a `PromptSession` constructor arg —
so we pass it at call time:

```python
return PromptSession(
    self._message_supplier,
    multiline=True,
    editing_mode=EditingMode.EMACS,
    erase_when_done=True,
    placeholder=placeholder,
    prompt_continuation="",
    history=InMemoryHistory(),
    key_bindings=build_draft_key_bindings(self.quit_state),
)

# ...later, in read / read_async:
self._session.prompt(handle_sigint=False)
await self._session.prompt_async(handle_sigint=False)
```

### `DraftReader.read_async` change

No `before_key_press` subscription is needed under the pure-timer
design — the binding sees its own `Ctrl+C` events directly. Just
pass `handle_sigint=False` so SIGINT reaches the binding instead
of being translated to `KeyboardInterrupt` by prompt_toolkit's
default handler.

```python
async def read_async(self) -> str:
    if self._session is None:
        return await asyncio.to_thread(input, "")
    return await self._session.prompt_async(handle_sigint=False)
```

### `TerminalChat._above_input` change

Render a dim-gray `press Ctrl+<X> again to exit` line as the
bottom-most fragment whenever the active `DraftReader`'s
`QuitArmingState.hint_text()` is non-None. The chord-specific
string comes straight from the state object so the hint always
names the exact key the user must press to confirm — no ambiguity.
The hint sits below the queued-message and task-panel region so
it's directly adjacent to the input cursor, and shares the same
`fg:ansibrightblack` foreground style as queued messages so the
quit affordance reads as part of the dim hierarchy, not a warning.

```python
def _above_input(self):
    tasks = self.store.tasks
    queued = self.store.queued_user_messages
    reader = self.view._draft_reader
    hint = reader.quit_state.hint_text() if reader is not None else None
    if not tasks and not queued and hint is None:
        return ""
    # ... existing rendering of tasks + queued ...
    if hint is not None:
        fragments.append(("fg:ansibrightblack", f"{hint}\n"))
    # ... return FormattedText or plain str ...
```

The hint disappears via two paths:

1. **Auto-fade** — the `c-c` binding scheduled a background task
   that calls `app.invalidate()` exactly `QUIT_WINDOW_S` after
   arming. On that re-render `within_window()` returns False and
   the fragment is omitted.
2. **Any subsequent re-render** — store mutations or assistant
   output trigger `app.invalidate()` and re-evaluate
   `within_window()`. The hint can disappear early if a store
   event fires after the window has expired but before the
   scheduled task does.

Both paths converge on "hint absent once `within_window()` is
False"; the scheduled task is only needed for the **no-other-events** case.

### `_input_loop` change

Both the existing `except KeyboardInterrupt` and the existing
`except EOFError` blocks still trigger — but each now only on the
*second* (confirming) press, because our explicit bindings raise
them via `event.app.exit(exception=...)`. No code change needed
in `_input_loop`; the existing notice / return for each branch
stays as-is. The notice text on `EOFError` is whatever
`_input_loop` already prints — out of scope to redesign here.

## Acceptance

### Automated

- `tests/test_terminal_chat.py` gains:
  - `QuitArmingState` unit tests (pure-timer + independent
    per-chord timers):
    - Fresh state: `within_window()` False, `hint_text()` None.
    - After `arm("c-c")`: `within_window()` True (any chord),
      `within_window("c-c")` True, `within_window("c-d")`
      **False** (c-d's timer was never armed), `hint_text()`
      `"press Ctrl+C again to exit"`.
    - After `arm("c-d")`: symmetric.
    - **After `arm("c-c")` then `arm("c-d")`**: hint switches to
      `"press Ctrl+D again to exit"` (most recent press), but
      **`within_window("c-c")` stays True** — the c-c timer was
      not reset by the c-d arming. Both chords are independently
      armed. The proof point that cross-key presses are
      non-destructive.
    - Monkey-patched expiry: each chord's window expires only
      based on its own `armed_at`. Bumping time past
      `QUIT_WINDOW_S` of c-c's arming closes c-c but leaves
      c-d's window alone (and vice versa).
  - Backslash continuation: build a real prompt_toolkit `Buffer`
    with text `"hello\\"`, call `apply_enter_or_continuation`, and
    assert text becomes `"hello\n"` with cursor at end. Also
    assert plain `"hello"` returns False and a mid-buffer
    backslash returns False.
  - Newline / quit key registration: assert
    `build_draft_key_bindings(...)` exposes bindings for
    `Keys.ControlJ`, `(Keys.Escape, Keys.ControlM)`,
    `Keys.ControlC`, `Keys.ControlD`, and `Keys.ControlZ`.
    Iterates `bindings.bindings` and checks keys tuples. The
    `Keys.ControlD` binding additionally carries a non-trivial

    the unconditional always-True filter.
  - `_above_input` renders the hint when `within_window()` is True
    and omits it when False. Asserts both branches return the
    expected `FormattedText` fragment list (and that the visible

    `"\x03"` alone and assert the call times out (no exit). Catches
    regressions in the binding wiring.
  - **End-to-end double-Ctrl+D via real `PromptSession`**: same
    harness, feed `"\x04\x04"`, assert `prompt_async` raises
    `EOFError` within 1 s. Then feed `"\x04"` alone, assert

    feed `"\x03\x04"` and assert the call times out (no chord

  - **Ctrl+D with non-empty buffer**: in a unit test, build a

### Manual

- Launch `neutrix`. Press `Ctrl+C`. Confirm:
  - Dim yellow `press Ctrl+C again to exit` line appears
    directly above the input cursor (the hint names exactly the
    key you just pressed).
  - Do nothing for ~1.5 seconds → the hint disappears on its own
    (auto-fade), input still active.
  - Press `Ctrl+C` again — at this point the window has expired,
    so the hint reappears but neutrix does NOT exit. Press
    `Ctrl+C` once more within 1 s → clean exit.
- Press `Ctrl+C` twice quickly (within 1 s) → exit immediately
  on the second press (no need to wait for the hint to fade).
- Press `Ctrl+D` on an empty draft. Confirm:
  - The hint says `press Ctrl+D again to exit` (different
    wording from the Ctrl+C path — the hint always names exactly
    the key the user just pressed).
  - Press `Ctrl+D` again within 1 s → clean exit (no traceback).
- **Cross-key sequences refresh the hint but do NOT reset the
  other chord's timer** — both timers run independently:
  - `Ctrl+C` → hint says `press Ctrl+C again to exit`.
  - Then `Ctrl+D` (within 1 s) → hint refreshes to
    `press Ctrl+D again to exit`. neutrix does NOT exit (no
    chord has been double-tapped yet).
  - Then **`Ctrl+C` again** (still within the ORIGINAL c-c
    window of the first press) → clean exit via Ctrl+C. The
    intervening Ctrl+D did not reset the c-c clock.
  - Equivalent: `Ctrl+C` → `Ctrl+D` → **`Ctrl+D` again** within
    c-d's window → exits via Ctrl+D.
  - Symmetric the other direction: `Ctrl+D` → `Ctrl+C` →
    `Ctrl+D` again exits via Ctrl+D (c-d's original window
    still alive).
- Type some text, then `Ctrl+D`. Confirm: forward-delete-character
  fires (the character to the right of the cursor disappears);
  neutrix does NOT enter the quit dance and does NOT exit. The
  per-buffer-empty filter is the load-bearing piece here.
- Type some text, then `Ctrl+Z`. Confirm: process suspends; shell
  prompt returns; `fg` brings `neutrix` back with the typed text
  still in the draft.
- Test newline keys:
  - `Ctrl+J` → newline.
  - `Alt+Enter` → newline.
  - `Shift+Enter` → newline (works on terminals that emit
    `Shift+Enter` as `Ctrl+J`, which is the common-case default).
  - `Ctrl+Enter` is unbound by design (most terminals can't
    distinguish it from plain Enter; confirmed acceptable at
    Phase-3 review).
- Type `hello\` then `Enter`. Confirm: buffer becomes `hello\n`
  (trailing `\` consumed) and the cursor sits on the second line.
  Press `Enter` on the empty second line → submits `hello\n`.
