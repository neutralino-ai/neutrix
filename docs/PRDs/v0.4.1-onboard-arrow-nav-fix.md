# v0.4.1 — onboarding arrow-key nav (fix after Screen refactor)

## Problem

`Up` / `Down` arrow-key focus navigation worked in v0.3.1 when
`OnboardApp` was a Textual `App` with priority bindings on the root
class. After v0.4.0 refactored the body into `OnboardScreen` (so it
could be pushed onto the chat App via `/onboard`), the bindings stopped
firing — pressing `Down` did nothing and `Tab` was the only way to
move focus.

Root cause: priority bindings on a Textual `Screen` do **not** preempt
descendants the way priority bindings on the `App` do. `VerticalScroll`,
which wraps every provider section, binds `up`/`down` to `scroll_up` /
`scroll_down` with `priority=False`. Within the focus chain it still
wins over the screen's `priority=True` binding when the screen is not
the root App. Result: keys are consumed by the scroller before the
screen action ever runs.

Reproduced with a headless `pilot.press("down")` test: action does fire
when called directly (`scr.focus_next()` moves focus), but the binding
chain never invokes it.

## Goal

Restore arrow-key focus nav from any focused widget inside
`OnboardScreen`, in both entry points (first-run `OnboardApp` and
mid-chat `/onboard`).

## Non-goals

- Restoring keyboard scrolling on the surrounding `VerticalScroll`
  (PgUp/PgDn still work; mouse wheel still works).
- Rewriting Textual's binding-resolution behavior.
- Adding arrow-key nav to the chat TUI itself.

## Design

Switch from declarative bindings to widget-level `on_key` handlers on
the focusable widgets — `ModelRow` and a new `KeyInput(Input)` subclass
used for the api_key fields. Widget `on_key` runs at the start of the
focus chain (before VerticalScroll), so calling `event.stop()` reliably
prevents the scroller from acting.

```python
def _navigate_focus(widget, event):
    if event.key == "down":
        widget.screen.focus_next(); event.stop(); return True
    if event.key == "up":
        widget.screen.focus_previous(); event.stop(); return True
    return False

class ModelRow(Static):
    can_focus = True
    def on_key(self, event):
        _navigate_focus(self, event)

class KeyInput(Input):
    def on_key(self, event):
        _navigate_focus(self, event)
```

Replace the `Input(...)` in `ProviderSection.compose` with `KeyInput`.
Drop the now-dead `up` / `down` `Binding(..., priority=True)` entries
from `OnboardScreen.BINDINGS`.

Everything else — `v` / `f` / `g` / `s` / `q` / `Ctrl+C` x2 / `Esc` —
stays as-is.

## Acceptance

- Headless pilot: `await pilot.press("down")` from a `ModelRow` moves
  focus to the next focusable; `await pilot.press("up")` reverses it.
- Same from a `KeyInput` (the api_key field) — arrow moves focus to
  the next/previous focusable instead of scrolling the container.
- Behavior identical via `/onboard` from inside the chat TUI.
- `Tab` / `Shift+Tab` still work (unchanged).
- `neutrix --version` reports `0.4.1`.
- `pytest` + ruff stay green.

## Out of scope

- Vim-style `j` / `k` navigation.
- A unified Textual fix for screen-priority bindings (upstream
  behavior; outside this project's surface).
