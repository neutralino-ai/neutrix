# v0.8.2 — Colored `tool_use` / `tool_result` keyword in transcript

## Problem

Today the live transcript shows tool calls as:

```
-> TaskCreate {"subject": "Tell Joke #1", "description": "..."}
<- [tool 1] TaskCreate {...} | folded | 1 lines | ~7 tokens
```

The `->` / `<-` arrows disambiguate direction and `[tool N]` indexes the
result, but during a fast multi-tool turn (e.g. the assistant fires three
`TaskCreate`s followed by three `TaskUpdate`s while streaming prose) the
eye has nothing wider than a single arrow glyph to anchor on. Result:
users — including the project author — visually re-scan to locate
"where did the call happen, where did the result come back" instead of
locking on at a glance.

A second cost: the on-screen vocabulary doesn't match the underlying
protocol. When the user opens a `/save`'d chatstore JSON or steps
through `chatstore.py`, they see Anthropic-style `tool_use` /
`tool_result` content-block types. The terminal speaks a different
dialect for no reason.

## Goal

Make the tool-call and tool-result lines instantly identifiable in a
scrolling transcript by introducing a **colored keyword anchor** that
also matches the SDK vocabulary the user already debugs against:

```
-> tool_use    TaskCreate {"subject": "Tell Joke #1", ...}
<- tool_result [tool 1] TaskCreate {...} | folded | 1 lines | ~7 tokens
```

- `tool_use` rendered in **cyan bold**, against the existing dim line.
- `tool_result` rendered in **bright_green bold**, against the existing
  yellow line.
- Both keywords padded to width 11 (the length of `tool_result`) so the
  body's tool name aligns vertically between the two lines and across
  successive calls.

## Non-goals

- No change to fold/expand behavior (`/tool`, `/tool N` keep working).
- No change to `chatstore` save/load format, agent loop, or LLM prompt
  payload — purely a `TerminalView` rendering change.
- No change to tool descriptions, lifecycle prompts, or task-panel UI
  shipped in v0.8.0–v0.8.1.
- Not introducing a richer log-style framing (timestamps, levels). The
  keyword is a single colored noun, not a log header.

## Design

### Naming choice

`tool_use` / `tool_result`, lowercase. Matches Anthropic content-block
type strings byte-for-byte (`{"type": "tool_use", ...}` /
`{"type": "tool_result", ...}`). The same tokens appear in `/save`'d
JSON and in Python identifiers throughout `agent_loop.py`. Lowercase
reads as a *type tag*; ALL-CAPS would read as a *log level* and outshout
the assistant's markdown, which is the actual signal.

### Color choice

- `tool_use` → `"bold cyan"`. Cool hue against the surrounding dim
  prefix/suffix gives a high lightness gap; the eye locks on immediately.
- `tool_result` → `"bold bright_green"`. The result line stays yellow
  overall (existing affordance signaling "this is a folded blob you can
  expand"). `bright_green` was picked over plain `green` because
  green-on-yellow risks reading muddy — they're adjacent on the
  spectrum. Bold + the brighter shade widens the gap. Fallback if visual
  test fails: drop the yellow on the result line surround and use dim
  like the use line.

Yellow remains as the `<-` line color (existing fold affordance). Dim
remains as the `->` line color. Cyan / bright_green stay distinct from
`bold red` (errors), `dim` (system reminders), `green` (save/clear
notices) already in use.

### Alignment

Both keywords padded to width 11:

```
-> tool_use   <body>
<- tool_result <body>
```

(`tool_use` followed by 3 trailing spaces — total 11 chars; then one
separator space.) Without the pad, the body name dances 3 chars
between use and result lines, defeating the anchor purpose.

### Render sites

Four call sites in `terminal_chat.py`:

| Site | Function | Today | After |
|------|----------|-------|-------|
| Live tool_use | `_dispatch_event` (`event.kind == "tool_call"`) | `print_notice(f"-> {name} {args}")` | `view.print_tool_use(name, args)` |
| Live tool_result | `_store_and_print_tool_result` | `print_notice(record.summary, style="yellow")` | `view.print_tool_result(record)` |
| Replay tool_use on `/load` | `_render_transcript` | `print_notice(f"-> {name} {args}")` | `view.print_tool_use(name, args)` |
| `/tool` listing replay | `_cmd_tool` | `print_notice(record.summary, style="yellow")` | `view.print_tool_result(record)` |

### Data model

`ToolRecord.summary` keeps its current plain-string form (used by a
test and for any future plain-text export). A new helper
`ToolRecord.summary_parts() -> tuple[str, str, str]` returns
`(prefix, keyword, suffix)` for the colored renderer.

### View additions

```python
async def print_tool_use(self, name: str, arguments: str) -> None:
    args = compact_inline(arguments or "{}")
    text = Text.assemble(
        ("-> ", "dim"),
        ("tool_use   ", "bold cyan"),
        (f" {name} {args}", "dim"),
    )
    await self._render(lambda: self.console.print(text))

async def print_tool_result(self, record: ToolRecord) -> None:
    prefix, keyword, suffix = record.summary_parts()
    text = Text.assemble(
        (prefix, "yellow"),
        (keyword, "bold bright_green"),
        (suffix, "yellow"),
    )
    await self._render(lambda: self.console.print(text))
```

## Acceptance

1. **Test suite green.** `pytest tests/test_terminal_chat.py` passes
   after assertion strings are updated to the new format
   (`-> tool_use   list_dir ...` / `<- tool_result [tool 1] ...`).
2. **Visual check in a real terminal.** Launch `neutrix` with a
   strong model, fire a multi-tool turn (e.g. ask for several
   `TaskCreate`s), confirm:
   - Cyan `tool_use` is unmistakable against dim surroundings.
   - Bright-green `tool_result` is unmistakable against yellow
     surroundings (advisor's noted muddy-color risk).
   - Body names align vertically between use and result lines.
3. **`/save` + `/load` round-trip.** Replayed transcript shows the new
   colored format on both lines.
4. **`/tool` listing** also renders with the colored keyword.
5. User signs off on the visual before tagging.

## Out of scope

- Restyling assistant prose, system messages, task panel.
- Changing the `| folded | N lines | ~K tokens` suffix wording.
- Adding configurability (no `--no-color` flag, no theme picker) — Rich
  already honors `NO_COLOR` env var, which is enough.
