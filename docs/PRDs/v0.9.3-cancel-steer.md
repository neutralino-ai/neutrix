# v0.9.3 — Cancel as steer + ContextManager state machine

## Problem

v0.9.2 shipped cancellation with **history rollback**: on Esc, the
executor calls `agent.rollback_to(pre_turn_len)` and the user turn,
any partial assistant turn, and any orphan `tool_calls` evaporate.
The user re-prompts from a clean slate.

That's correct for "I changed my mind, start over." It is wrong for
"you were about to `rm -rf /`, instead just list the directory" —
the LLM never sees what it was about to do, so it can't be steered.
Claude Code, Codex, and Continue all keep the interrupted turn in
history and append a synthetic user "interrupted" signal so the next
prompt has context (`cc2/src/query.ts:1015, 1046, 1485, 1501`).

Picking this fix up surfaced a deeper structural issue: the
v0.9.2 `Controller` is a thin broadcaster, and the conversation
loop lives inside `Agent.stream_reply` as an async generator. With
the loop split across two layers, mutation responsibility is
ambiguous: the UI calls `store.apply(event)` on every yielded
event, `Executor.cancel` rolls back `agent.messages`,
`TerminalChat._mirror_new_agent_messages` synchronizes the store
mirror. v0.9.3 unifies all of this: the `Controller` is reshaped
into a `ContextManager` (CM) that owns `agent.messages`, drives the
LLM/tool loop as an explicit state machine, and is the *only* layer
that mutates `messages` or `ChatStore`. The cancel-steer behavior
falls out of one state transition in this design.

## Goal

1. **Steer instead of rollback.** On cancel (Esc / first Ctrl+C
   while busy), keep the interrupted turn in history and append a
   user message with content `[interrupted by user]`. The next LLM
   call sees the prior user turn, the partial assistant turn (if
   any), the orphan `tool_calls`, and the user marker — enough
   context to steer.
2. **ContextManager state machine.** Reshape v0.9.2's `Controller`
   into a `ContextManager` that owns `messages` and drives the
   conversation as an explicit state machine
   (IDLE / AWAITING_LLM / AWAITING_EXECUTOR / CANCELLING).
3. **Dismantle `Agent.stream_reply`.** The loop migrates from
   Agent into CM. The `Agent` class folds into `ContextManager`;
   its surviving responsibilities (own `messages`, hold the store
   reference, inject `<system-reminder>`s, call LLM, dispatch
   tools) all become CM responsibilities.
4. **LLM-layer pairing.** A pure function in `llm.py` ensures the
   outgoing message payload is valid (each `tool_use` paired with
   a `tool_result`) before each API call. Pure transform on a
   copy — does not mutate `messages`.
5. **stream=False for v0.9.3.** Revert v0.9.2's streaming switch.
   Re-enable streaming in a later PRD with the CC-aligned "keep
   partial text" semantic.
6. **UI subscribes to `ChatStore.changes()` for rendering.**
   `TerminalChat._mirror_new_agent_messages` and the
   AgentEvent-dispatch path are removed. UI emits user/cancel
   events to CM; CM mutates ChatStore; ChatStore.changes() wakes
   the renderer.

## Non-goals

- **Partial-assistant-text-with-streaming.** Earned-by when
  streaming re-enables in a later PRD. v0.9.3 has no partial state
  because stream=False.
- **CC's smart "skip marker when queued message follows"**
  (`signal.reason === 'interrupt'`). CM is explicitly dumb for
  v0.9.3 — always append.
- **CC's two-variant marker text**
  (`INTERRUPT_MESSAGE` vs `INTERRUPT_MESSAGE_FOR_TOOL_USE`).
  One unified `[interrupted by user]`. The orphan tool_result
  fill carries the "tool was active" context.
- **Cancel-mode taxonomy** (Codex's background vs hard kill).
  Single cancel mode.
- **Pure-compute tool cancellation.** Inherits v0.9.2's non-goal —
  `asyncio.to_thread` tools keep running on the background thread;
  their results are silently dropped when CM transitions to
  CANCELLING.
- **Live tool output / `progress` events.** Executor surface is
  minimal: `tool_started` + `tool_finished`. Earned-by when
  streaming re-enables and `run_shell` starts yielding stdout
  chunks.
- **Multi-process / RPC transport.** Service-oriented interfaces
  land; transport stays in-process.
- **The legacy Textual `tui.py` app.** Standing v0.9.x non-goal.

  mutate `messages`. User: "tool_use must match tool_result and

  restriction. so llm take responsibility."

### Split point: Synthetic tool_result content
- **CC**: two literals —
  `SYNTHETIC_TOOL_RESULT_PLACEHOLDER = '[Tool result missing due to internal error]'`
  (defensive, `messages.ts:246-247`) and `CANCEL_MESSAGE`
  (intentional stop, `messages.ts:622-631`), both with
  `is_error: true`.
- **Decision**: Alternative — conditional content in the pairing
  layer. If the history contains `[interrupted by user]` after the
  orphan, synthetic content = `[cancelled by user]`; otherwise
  `[tool result missing]`. User: "synthetic tool_result (missing)

- **Decision**: User-directed different — cancel is an *event*
  emitted by the UI to the CM. CM is the single decision-maker
  and mutator; broadcasts to LLM.stop() + Executor.cancel(), then

### Split point: Streaming + partial assistant text
- **CC**: streaming on; partial assistant content stays in history

- **Decision**: User-directed different — follow CC for the

  `OpenAIChatLLM.stream_response` reverts to `stream=False`.

### Split point: View-side dim "interrupted" notice
- **CC**: no separate UI notice — the rendered

- **Decision**: Follow CC — drop the v0.9.2 dim view-side notice.

### Split point: Idempotency of repeated cancel
- **CC**: fire-once via AbortController architecture.
- **Decision**: User-directed different — CM handles dedup. A
  second cancel event while CM is already in CANCELLING is a no-op.
  User: "controller handle that. ui just provide input to
  controller."

### Split point: Fate of `Agent.rollback_to`
- **CC**: no analog.
- **Decision**: Alternative — remove (YAGNI). The whole `Agent`
  class is dissolved into CM (see next split).

### Split point: Scope of ContextManager refactor
- **CC**: no direct analog; CC's query loop is a generator in
  `query.ts`, not a state machine.
- **Decision**: User-directed different — bundle the broader
  refactor into v0.9.3. User: "bundle it." Concretely:
  - v0.9.2's `Controller` reshapes into `ContextManager` (this is
    a substantial role expansion, not a rename — see Design).
  - `Agent.stream_reply` is dismantled; the loop becomes the CM
    state machine.
  - `Agent` class is dissolved into CM.
  - `TerminalChat._mirror_new_agent_messages` is removed; UI
    subscribes to `ChatStore.changes()` (existing since v0.7).

### Split point: Executor → CM event surface
- **CC**: per-tool `async function*` yielding `progress` events
  during execution then a final `result` (`Tool.ts`,
  `query.ts:runToolUse`).
- **Decision**: Alternative — minimal for v0.9.3:
  `tool_started(tool_call_id, tool_name, args)` and
  `tool_finished(tool_call_id, tool_name, content, ok)`. No
  `progress` events. No internal Executor state machine. User: "for
  now, keep it simple. started / finished."

### Split point: Fate of queued user messages on Esc cancel
- **CC**: bifurcated. Esc while idle → pop queued commands back to
  the editable input buffer (`useCancelRequest.ts:87-122`,
  `REPL.tsx:2166-2184`). Esc during active LLM response → leave
  queue intact, queued messages send after the cancel marker.
- **Decision**: Follow CC's Path B for both cases. Worker_loop's
  next iteration drains the queued message and dispatches it as the
  next `UserMessageEvent`, so the LLM sees the prior tool intent +
  `[interrupted by user]` marker + the queued (steering) prompt in
  one go. User: chose "send queued messages after marker (current)"
  when surfaced mid-implementation. CC's Path A (pop back to input
  draft on idle-cancel) requires non-trivial injection into
  `prompt_toolkit`'s buffer and is deferred.

### Split point: ContextManager class name
- **Decision**: User-directed different — `ContextManager` (class),
  `context_manager.py` (file), `ctx` (variable). User: "use
  ContextManager."

## Design

### Naming honesty

v0.9.2's `Controller` was a ~50-line broadcaster: `send()` wrapped
`stream_reply` in a task, `cancel()` called `llm.stop()` /
`executor.cancel()` / `task.cancel()`. v0.9.3's `ContextManager` is
that class **substantially reshaped**: it absorbs
`Agent.stream_reply`'s loop, owns `messages`, owns the system-prompt
+ system-reminder injection, runs an explicit state machine, and is
the sole mutator of `agent.messages` / `ChatStore`. This is **not a
rename**; the v0.9.2 class is a small subset of the v0.9.3 class.
CHANGELOG framing must say so.

### Layer responsibilities (post-v0.9.3)

| Layer | Owns | Touches |
|---|---|---|
| `ContextManager` | `messages`, state machine, all mutation of `messages`/`ChatStore`, system-reminder injection, queued-input buffer | called by UI; calls LLM, Executor |
| `OpenAIChatLLM` | one API call + outgoing-payload pairing (pure on a copy) | pure function on the input messages; no mutation |
| `Executor` | subprocess pool, tool dispatch, tree-kill | yields events to CM |
| `TerminalChat` (UI) | render from `ChatStore.changes()`; emit input + cancel events | calls `ctx.handle_event(...)`; reads `ChatStore` |
| `Agent` | **removed** | — |

### State machine

```
                ┌─────────────────────────────────────┐
                │                                     │
                ▼                                     │
        ╔═══════════╗                                 │
        ║   IDLE    ║◀──┐                             │
        ╚═══════════╝   │                             │
              │         │                             │
   user_message_event   │ all_finished                │
        │   │   │       │   │                         │
        │   │   ▼       │   │                         │
        │   │  drain queue   │                         │
        │   │  (also: inject system-reminder if rule fires)
        │   │   │           │                         │
        │   ▼   ▼           │                         │
        ╔══════════════╗   │                          │
        ║ AWAITING_LLM ║───┴── llm_response (no tool_calls)
        ╚══════════════╝                              │
              │                                       │
       llm_response                                   │
       (with tool_calls)                              │
              │                                       │
              ▼                                       │
        ╔═══════════════════╗                         │
        ║ AWAITING_EXECUTOR ║──── all tools done ─────▶ AWAITING_LLM
        ╚═══════════════════╝                         │
              │                                       │
              │ tool_finished events                  │
              │ append to messages                    │
              ▼                                       │

Any non-IDLE state ── cancel_event ────────────────────┐
                                                       │
                                                       ▼
                                              ╔══════════════╗
                                              ║  CANCELLING  ║
                                              ╚══════════════╝
                                                       │
                                  ┌────────────────────┤
                                  │ tell Executor.cancel()
                                  │ call LLM.stop()
                                  │ append [interrupted by user]
                                  │ (orphan tool_use unrepaired —
                                  │   LLM-layer pairing fills next send)
                                  │ task.cancel() if needed
                                  ▼
                              ╔═══════════╗
                              ║   IDLE    ║
                              ╚═══════════╝
```

### State transition semantics

**IDLE**
- Entry: drain queued user buffer; if empty, wait for events.
- Events handled:
  - `user_message_event(text)` → append `{role: "user", content: text}` to `messages`; if system-reminder rule fires (≥10 turns + actionable tasks; same algorithm as v0.8.0), inject reminder; → AWAITING_LLM.
  - `cancel_event` → no-op (dumb dedup).
  - `slot_switch_event` → switch slot; stay IDLE.
  - `clear_event` → reset messages + tasks; stay IDLE.
  - `save_event` / `load_event` → execute synchronously; stay IDLE.

**AWAITING_LLM**
- Entry: build outgoing payload from `messages` via the LLM pairing
  function; call `LLM.stream_response(payload)` (now non-streaming,
  one awaited call).
- Events handled:
  - `llm_response_event(assistant_message)` →
    - Append `assistant_message` to `messages`.
    - If `tool_calls` empty: → IDLE.
    - If `tool_calls` non-empty: → AWAITING_EXECUTOR (send all
      tool_calls to Executor).
  - `llm_error_event(error)` → append a synthetic assistant message
    `{role: "assistant", content: "[LLM error: {error}]"}` to
    `messages`; → IDLE. The next user turn proceeds normally.
  - `cancel_event` → → CANCELLING (see Cancel path below).

**AWAITING_EXECUTOR**
- Entry: pending set = `{tc.id for tc in assistant.tool_calls}`;
  dispatch all tools concurrently via Executor.
- Events handled:
  - `tool_started(tool_call_id, tool_name, args)` →
    `store.add_pending_tool_call(...)`; pending count unchanged.
  - `tool_finished(tool_call_id, tool_name, content, ok)` →
    `store.remove_pending_tool_call(tool_call_id)`; append
    `{role: "tool", tool_call_id, content}` to `messages`; remove
    from pending set; if pending set empty → AWAITING_LLM (build
    new outgoing payload including the new tool messages).
  - `cancel_event` → → CANCELLING.

**CANCELLING**
- Entry: call `Executor.cancel()` (tree-kills subprocesses; running
  tools' coroutines may still emit a final `tool_finished` with
  `content = "[cancelled by user]"` — these are appended normally
  if they arrive before CANCELLING completes).
- Entry: call `LLM.stop()` (cancels the awaiting HTTP request task
  if any; idempotent).
- Entry: append `{role: "user", content: "[interrupted by user]"}`
  to `messages`. **Orphan `tool_use` blocks in the latest
  assistant message remain unrepaired**; the LLM pairing layer
  will synthesize tool_results for them at the next API send.
- Events handled:
  - `cancel_event` → no-op (dumb dedup).
  - All other events deferred / dropped until → IDLE.
- Exit: clear pending tool-call set; → IDLE.

### LLM-layer pairing function

In `llm.py`:

```python
def _ensure_tool_result_pairing(messages: list[dict]) -> list[dict]:
    """
    Pure function. Returns a new list. Does not mutate input.

    Two passes:
    1. Dedup: for each tool_call_id, keep only the FIRST role=tool
       message with that id. Pessimistic safety net.
    2. Synthesize: for each tool_call in the latest assistant
       message with no matching tool_result, insert a synthetic
       {role: "tool", tool_call_id, content} *immediately after*
       the assistant message. Content is:
         - "[cancelled by user]" if any user message with content
           "[interrupted by user]" appears AFTER the orphan
           assistant message in the list.
         - "[tool result missing]" otherwise.
    """
```

Called once at the top of `OpenAIChatLLM.stream_response`, before
the API call. The repair affects the outgoing payload only —
`messages` (the CM-owned list) keeps its native state.

Edge cases:
- Multiple assistant messages with `tool_calls` in history. Only
  the *latest* one can have orphans (older ones were paired during
  prior rounds). Implementation scans the latest only.
- Multi-tool-call assistant message, partially paired. Synthesize
  only the unpaired ones.
- `[interrupted by user]` user message followed by a new user turn
  (e.g., the steering prompt). Only the cancel-context check is
  "is there a `[interrupted by user]` user message anywhere after
  the orphan assistant message" — the presence of subsequent user
  messages doesn't change the answer.

### Automated (`tests/`)

**`tests/test_context_manager.py`** (new):
- **State transitions, happy path.** IDLE → AWAITING_LLM →
  AWAITING_EXECUTOR → AWAITING_LLM → IDLE with a fake LLM emitting
  one assistant-with-one-tool_call then one assistant-no-tool-calls.
- **Cancel from AWAITING_LLM.** Fake LLM never resolves; cancel
  event → state goes to CANCELLING → IDLE; last message is
  `{role: "user", content: "[interrupted by user]"}`.
- **Cancel from AWAITING_EXECUTOR.** Tool dispatch suspends; cancel
  event → Executor.cancel() called → last message is the marker;
  the orphan tool_call stays in the prior assistant message
  (unrepaired in `messages`).
- **Repeated cancel is idempotent.** Two `CancelEvent`s in a row;
  marker appended once.
- **Queue drain at IDLE entry.** Two queued user messages; CM
  processes them sequentially as IDLE re-entries.
- **System-reminder injection.** ≥10 assistant turns + actionable
  tasks; CM injects the reminder before sending to LLM. (Same
  trigger as v0.8.0.)
- **`/clear` during AWAITING_LLM.** Cancel first, then clear;
  messages empty after.
- **`/save`/`/load` rejected during non-IDLE.** Returns a notice

- **LLM error path.** Fake LLM raises; CM appends
  `[LLM error: <text>]` assistant message and returns to IDLE.

### Manual

- **Mid-LLM-call cancel.** Send "write me a long story"; press Esc
  before LLM responds; next prompt "what was that?". With
  stream=False, LLM hasn't responded yet at cancel, so the
  assistant message is empty; the user marker is in history.
- **Tool cancel and steer.** Ask for `run_shell sleep 30`; Esc;
  next prompt "instead, just `echo hi`". Assert the LLM does NOT
  re-issue `sleep` — it sees the `[cancelled by user]` synthetic
  tool result + the user marker.
- **/save + /load round-trip of a cancelled session.** The marker
  reloads as a normal user turn.
- **Backwards check.** Cancel while idle (Esc with nothing in
  flight) → still no-op; queue still drains; `[interrupted by user]`
  does not appear when no cancel happened.
- **/clear during AWAITING_LLM.** Implicitly cancels first, then
  clears. Messages empty afterwards.
- **/save during AWAITING_LLM.** Rejected with a notice; chat
  state unchanged.

## Migration / risk notes

- **This is a substantial v0.9.x refactor.** Test surface churns
  noticeably (~26 tests touched). Two whole files (`agent_loop.py`,
  `controller.py`) are removed. The risk profile is closer to the
  v0.9.0 lifecycle-events bring-up than to the v0.9.1/v0.9.2
  incremental work. Implementation order in §Design is structured
  so each step has independent tests passing.
- **Next-request validity is now owned by `llm.py`.** A bug in the
  pairing function will 400 the next API call. The new pairing
  tests are the regression gate; the steering smoke test catches
  end-to-end breakage.
- **Token cost for the user.** Cancelled turns are no longer free;
  they live in the context forever (or until `/clear`). This is
  the cost of steering. A later PRD could prune cancelled turns
  older than N if it becomes painful.
- **`Agent` and `Controller` classes are gone.** External
  callers (none today) would break. Any future seam reintroduction
  (e.g., for the v0.11.0 Advisor) should be done via the CM's
  event surface, not by resurrecting the old classes.
- **`Agent.rollback_to` removed.** Its unit test goes too. If
  `/undo` is added later, the seam can be cleanly reintroduced on
  `ContextManager.messages`.
- **Streaming disabled.** v0.9.2's `stream=True` switch is reverted.
  Visible latency: full-response wait instead of incremental token
  delivery. Re-enabled in a later PRD.
- **UI rendering source switched to `ChatStore.changes()`.** The
  v0.7 store-changes machinery is exercised more heavily; any
  regression there now bites the main message flow.
