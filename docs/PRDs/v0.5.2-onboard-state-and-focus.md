# v0.5.2 — onboarding: route notifications, fix Enter flow, restore slot state, auto-assign

## Problem

Four issues after v0.5.1 user feedback:

1. **Floating notifications still appear mid-screen.** The codebase no
   longer calls `self.notify(...)`, but Textual itself can emit
   notifications (worker failures, internal warnings) that bypass the
   inline `#message` bar and show as toasts. From the user's POV the
   screen still has random popups.
2. **Enter on the api_key Input misbehaves.** Headlessly the value
   persists, but interactively the user observes the field showing the
   `EMPTY` placeholder again and focus "stays strange." Plausible
   cause: Textual fires a blur on Enter in some flows, and `on_blur`
   reads `_committed_value` *before* `on_input_submitted` updates it,
   triggering the revert path against a stale empty baseline. The user
   also expects Enter to advance focus to the next focusable (so the
   "saved" indicator is unambiguously visible and the user can
   continue).
3. **Saved `fast` / `strong` aren't reflected when re-opening
   onboarding.** `_assign` writes to YAML, but
   `OnboardScreen.__init__` always sets `self.fast_choice =
   self.strong_choice = None`. Reopening shows no `▸ fast` /
   `▸ strong` tags even though the YAML has them. The status bar
   also reads `—` for both.
4. **Slots stay unset if user only verifies and doesn't press
   `f`/`g`.** First-run users naturally think "I verified, that's
   enough." Auto-assigning the first verified model to any unset
   slot makes the path forgiving without taking control away.

## Goal

- Suppress all floating Textual toasts on `OnboardScreen`; route
  anything that calls `notify(...)` through the inline `#message` bar.
- Enter commits the api_key, advances focus to the next focusable,
  and the Input keeps its (masked) value with the green "saved"
  affordance.
- On open, load `fast` / `strong` from YAML so existing bindings show
  up as `▸ fast` / `▸ strong` tags and in the bottom status bar.
- After any verification, if `fast` is unset, auto-bind it to the
  first verified model; if `strong` is unset, auto-bind it to the
  second verified model (or the first, if only one is verified).
  Persist immediately. Never overwrite a user's explicit choice.

## Non-goals

- A general "smart slot pick" heuristic by model name (haiku → fast,
  opus → strong). First-verified is the rule.
- Re-binding slots when a previously-verified model later flips to
  `failed`.
- Changing the chat-side `/onboard` flow.
- Visual snapshot tests.

## Design

### 1. Route all notifications through the inline bar

Override `Screen.notify` on `OnboardScreen`:

```python
def notify(
    self,
    message: str | RenderableType,
    *,
    title: str = "",
    severity: str = "information",
    timeout: float | None = None,
    markup: bool = True,
) -> None:
    sev_map = {"information": "info", "warning": "warning", "error": "error"}
    self._notify(str(message), sev_map.get(severity, "info"))
```

Signature matches Textual's `Widget.notify`. Anything in the framework
or future code that calls `self.notify(...)` (or `app.notify(...)`
that bubbles) lands in our `#message` bar. No floating toasts.

### 2. Fix Enter flow on the api_key Input

Two changes inside `on_input_submitted`:

```python
async def on_input_submitted(self, event: Input.Submitted) -> None:
    provider = event.input.id.removeprefix("key-") if event.input.id else ""
    if provider not in self.provider_state:
        return
    new_key = event.value.strip()

    # Commit baseline FIRST — any blur racing from focus_next() below
    # must see the up-to-date value or it will revert to the old "".
    if isinstance(event.input, KeyInput):
        event.input._committed_value = new_key
    event.input.value = new_key

    state = self.provider_state[provider]
    if new_key != state.api_key:
        state.model_status.clear()
        for row in self.query(ModelRow):
            if row.provider == provider:
                row.status = UNKNOWN
                row._refresh()
    state.api_key = new_key
    state.key_saved = bool(new_key)
    self._set_saved_indicator(provider, state.key_saved)
    self._persist_yaml(fast=self.fast_choice, strong=self.strong_choice)
    if new_key:
        self._notify(f"saved {provider} api_key", "success")

    # Advance focus so the saved indicator is visible and the user
    # is unblocked. Has the side-effect of triggering blur on the
    # Input — but _committed_value is already current.
    self.focus_next()
```

Order: commit baseline → ensure value → persist → focus_next.

### 3. Load fast / strong from config

Inside `__init__`:

```python
self.fast_choice = self._slot_choice("fast")
self.strong_choice = self._slot_choice("strong")
```

```python
def _slot_choice(self, name: str) -> dict[str, str] | None:
    spec = self.config.slots.get(name) or {}
    p, m = spec.get("provider"), spec.get("model")
    if p and m:
        return {"provider": p, "model": m}
    return None
```

`on_mount` calls `_refresh_slot_tags()` and `_refresh_status()` after
the rows mount so the tags render on first paint.

### 4. Auto-assign first verified model

Helper called from `_verify_one` and `_verify_all` after statuses are
written:

```python
def _maybe_auto_assign(self) -> None:
    verified = [
        {"provider": r.provider, "model": r.model}
        for r in self.query(ModelRow)
        if r.status == VERIFIED
    ]
    if not verified:
        return
    changed = False
    if self.fast_choice is None:
        self.fast_choice = verified[0]
        changed = True
    if self.strong_choice is None:
        # Prefer a different model if there is one, otherwise reuse fast.
        self.strong_choice = next(
            (v for v in verified if v != self.fast_choice), self.fast_choice
        )
        changed = True
    if changed:
        self._refresh_slot_tags()
        self._refresh_status()
        self._persist_yaml(fast=self.fast_choice, strong=self.strong_choice)
        self._notify(
            f"auto-assigned fast → {self.fast_choice['model']}, "
            f"strong → {self.strong_choice['model']}",
            "info",
        )
```

Never overwrites an existing slot (the `is None` guard).

## Acceptance

- No floating toast appears anywhere on the onboarding screen, ever.
  All severity-coloured feedback lives in the bottom `#message` line.
- Typing an api_key + Enter: the Input keeps its masked dots, the
  green `saved` indicator lights up, focus moves to the next
  focusable (typically the `(all)` row of the same provider). No
  EMPTY placeholder, no stuck focus.
- Set `fast` and `strong` via `f` / `g`, `q` out, reopen `/onboard`:
  the same model rows show `▸ fast` / `▸ strong` tags; bottom bar
  reads `fast: ... strong: ...`.
- Verify a single model with no slots bound: `fast` and `strong` both
  auto-assign to that model in YAML. Verify a second model: `strong`
  flips to the second one (`fast` keeps its first auto-pick).
- After any auto-assignment, an inline info message names the new
  bindings.
- `pytest` + ruff stay green.
- `neutrix --version` reports `0.5.2`.

## Out of scope

- A modal "choose your fast/strong" picker.
- Configurable auto-assign rule (haiku → fast, etc.).
- "Re-verify all on key change" automation.
