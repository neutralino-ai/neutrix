# v0.6.6 - main chat borderless shared blocks

## Problem

The primary chat blocks still look like separate widget frames instead of a
single conversation surface:

- committed user messages, the editable user draft, the system prompt, and LLM
  responses use different framing treatments;
- bordered/title blocks waste rows and columns in a terminal UI;
- the draft `TextArea` feels especially padded because its own editor surface
  sits inside a bordered parent block.
- the system prompt should not have special spacing rules; primary blocks
  should share one gap definition;
- system and LLM response blocks need a muted gray background that is darker
  than the user block but not pure black.
- the message area needs a clear container for future adjacent sections, but
  terminal box-drawing frames pollute copy/paste output with `│` and long
  border lines.
- session/model status should occupy the bottom footer line instead of the
  Textual shortcut footer (`^l Notice`, `^p palette`).
- the top header currently shows slot/provider/model, duplicating the bottom
  status and making the title area noisy.
- the main chat should default to the strong slot when it is usable; the fast
  slot is reserved for cheaper internal/subagent-style work.
- long conversations need keyboard block navigation so users can move through
  messages without mouse interaction.
- mouse reporting should be disabled in the main TUI so dragging in the
  terminal selects/copies text instead of being captured by Textual.
- the default blue-gray panel color is not the right taste for this app; use
  a warmer Chinese ink/瓦灰-inspired palette.

## Goal

Render the three primary model-visible block types with one shared visual
system:

- system prompt;
- user message / user draft textarea;
- LLM response.

They should use the same layout, spacing, and shape. The only primary visual
differences should be role background and role text color.

Textual renders on a terminal cell grid, so CSS spacing must use whole rows
and columns. A true half-line gap is not representable in the TUI; use the
smallest non-zero terminal spacing and validate by screenshot.

Textual supports both theme tokens such as `$surface`, `$panel`, and
`$warning`, and exact CSS colors such as `#2a2824`. This pass uses exact
colors for the chat surface because the built-in `$panel` is too blue.

## Non-goals

- Switch from Textual/Python to a browser frontend.
- Change model calls, saved sessions, slash commands, or onboarding.
- Add transcript editing, prompt history, or branching.

## Design

1. Remove border and border-title framing from primary chat blocks.
2. Use one shared `.block` rule for width, margin, and minimal padding. Do not
   override spacing in system/user/assistant role classes.
3. Let `.role-system`, `.role-user`, and `.role-assistant` change only
   background and text color.
4. Keep the draft `TextArea` borderless and background-matched to the user
   block, so it reads as editable text inside the same block type.
5. Keep the existing tiny inter-block row gap for readability.
6. Keep the scrollable conversation area copy-friendly: no visible message
   section heading and no box-drawing container border.
7. Remove Textual's shortcut `Footer` from the main chat. Move the existing
   provider/model/tools/message-count status text to a plain bottom line in
   the footer's former position.
8. Remove vertical padding from primary blocks to avoid an extra empty row
   under one-line messages; keep horizontal padding for readable text inset.
9. Disable mouse reporting when launching the main app.
10. Remove the visible top `Header` from the main chat and clear the app
   title/subtitle. Keep the bottom status line as the single source of
   slot/provider/model visibility.
11. Change initial slot selection to prefer `strong` when both `fast` and
   `strong` are usable, falling back to `fast` only when strong is unavailable.
12. Add keyboard block navigation:
   - Up / Down moves focus one block through system/user/LLM/draft blocks;
   - Ctrl+Up / Ctrl+Down moves by approximately one visible page of blocks;
   - focused message blocks scroll into view.
13. Use a restrained dark Chinese-inspired palette:
   - ink background;
   - warm user surface;
   - 瓦灰/graphite system and LLM panels;
   - moon-white frame/title/subtitle/text;
   - muted amber system/tool accent.
14. Render a Textual screenshot before the changelog step so the user can judge
   the actual UI instead of CSS intent.

## Acceptance

- System prompt, user messages/draft, and LLM responses have no border line
  and no border title.
- The primary blocks share the same padding and inter-block gap.
- The system prompt has no special spacing definition.
- The user draft textarea does not add a second surface or extra frame inside
  the user block.
- A one-line draft renders with the same shared block padding as system and
  LLM blocks.
- System and LLM response blocks use a muted gray background darker than the
  user block.
- No visible `Messages` heading is rendered.
- The active provider/model/tools/message count appears on the bottom line
  where the Textual footer used to be.
- The bottom status line uses the same readable color as primary chat text.
- No visible top header/title/subtitle is rendered in the main chat or
  screenshot chrome.
- The main chat defaults to the strong slot when it is configured and usable.
- Up/Down moves focus one block; Ctrl+Up/Ctrl+Down moves by a page of blocks.
- Copying the visible message area does not include `│`, `┌`, `└`, or long
  frame lines from a container border.
- The main app launches with mouse reporting disabled.
- One-line message blocks do not have an extra empty row under their text.
- Existing Enter-submit and newline shortcuts still work.
