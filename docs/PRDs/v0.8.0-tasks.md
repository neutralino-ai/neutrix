# v0.8.0 - tasks with LLM tool integration (Claude-shaped)

## Problem

The user wants to record and track work items during a chat, and have
the LLM read and update them across turns. Today the only durable
state in a conversation is the message list. There is no first-class
notion of "things to do." Three concrete failures of the current
shape:

1. **The LLM forgets.** Across a long conversation the model has to
   re-discover from message history what items are open. It often
   misses or duplicates items.
2. **The user has no overview.** To see what was agreed to be done,
   the user must re-read the transcript. There is no `/tasks`-style
   list.
3. **Future automation is blocked.** Reminders, persistence, panel
   rendering, and (later) syncing to a mobile/web client all want
   the same backing data — which does not exist yet.

This is the third forcing function we identified during the v0.7.0
PRD review (alongside queued-message display and the heartbeat).
Promoted from v0.9.0 to v0.8.0 because of the user's stated
near-term need and because the controller refactor (now v0.9.0) is
more meaningful once tasks are already canonical store state.

The user explicitly asked that this PRD match Claude Code's design
where it's already been worked out: same tool names, same trigger
algorithm for reminders, reminders treated as persistent state in
the message history. The design below mirrors Claude Code's source
(`src/tools/Task*Tool/`, `src/utils/attachments.ts`,
`src/utils/messages.ts`) with neutrix-sized simplifications called
out explicitly.

## Goal

- Add a `Task` dataclass and a `tasks` field on `ChatStore`. Mutators:
  `add_task`, `update_task`, `remove_task`, plus a `reset` that clears
  them.
- Expose three LLM-callable tools using **Claude Code's exact names**:
  - `TaskCreate(subject, description?)` — append a new task.
  - `TaskUpdate(taskId, status?, subject?, description?)` — change
    one task's status / fields. `status="deleted"` is the special
    value that removes the task (matches Claude).
  - `TaskList()` — read-only, returns the current task list.
- Inject a **task reminder** into the message history when both:
  - at least `TURNS_SINCE_WRITE = 10` assistant turns have passed
    since the LLM last called `TaskCreate` or `TaskUpdate`,
  - at least `TURNS_BETWEEN_REMINDERS = 10` assistant turns have
    passed since the last reminder was injected.
  Numbers match Claude Code's `TODO_REMINDER_CONFIG` exactly.
- The reminder is a **user-role message** with `isMeta=True`-style
  content wrapped in `<system-reminder>…</system-reminder>` tags,
  matching the format produced by Claude's `wrapInSystemReminder`.
  Body lists actionable items (`pending` / `in_progress`).
- The reminder is **part of the message history**. It is appended to
  `agent.messages` (between the last assistant turn and the next user
  turn) and persists naturally through `transcript.save` /
  `transcript.load`. Future trigger checks scan backwards through
  the same message history to find the most recent reminder.
- Add a `/tasks` slash command that prints the current task list to
  scrollback on demand. Matches the `/status` pattern from v0.7.0.
- Extend `transcript.py` to persist tasks too (the message history
  already persists, so reminders ride along automatically): bump
  on-disk format to version 2 by adding a `tasks` key. Version 1
  files load with `tasks = []`.

## Non-goals

- The full Claude Code task surface: `TaskGet`, `TaskStop`,
  `TaskOutput`, `activeForm`, `owner`, `blockedBy`, `metadata`,
  dependencies. v0.8.0 ships the three core verbs and the minimal
  fields. Extras can be added one at a time later if they pull weight.
- Always-visible task panel in the UI. Visible-everywhere rendering
  belongs with v0.10.0 (TUIView purify); v0.8.0 ships data + tools +
  reminders + on-demand `/tasks`.
- Time-based reminders, alarms, scheduling.
- Hierarchical tasks / subtasks.
- Cross-session shared task store.
- HTTP / REST API for tasks. v0.11+ territory.
- Refactoring `agent_loop.py` into a pure controller (v0.9.0).
- Refactoring `tui.py` / `terminal_chat.py` into pure readers
  (v0.10.0).
- Cancelling a turn rolling back task mutations the LLM made during
  it. Mutations are immediate.

### Boundary types

```python
# store.py additions
from typing import Literal

TaskStatus = Literal["pending", "in_progress", "completed"]

@dataclass(frozen=True)
class Task:
    id: str                                          # sequential, "1", "2", ...
    subject: str                                     # brief title
    description: str = ""                            # optional detail
    status: TaskStatus = "pending"
    created_at: datetime = field(default_factory=datetime.now)
    updated_at: datetime = field(default_factory=datetime.now)
```

IDs are monotonic positive integers serialized as strings, scoped to
one `ChatStore` instance. `add_task("…")` produces `id = str(N+1)`
where N is the highest existing id (or 0 if empty). After `/load`,
the next-id counter resumes from the maximum loaded id. This keeps
ids stable across save/load and lets the user (and the LLM) refer
to a task by a short number in conversation.

Status values match Claude exactly (`pending`, `in_progress`,
`completed`). The string `"deleted"` is not a status — it is a
special action carried by `TaskUpdate`'s `status` argument, mapped
to `remove_task` rather than `update_task`. This matches Claude
Code's `TaskUpdateStatusSchema().or(z.literal('deleted'))` shape.

### `ChatStore` additions

```python
class ChatStore:
    # existing fields ...

    @property
    def tasks(self) -> tuple[Task, ...]: ...

    def add_task(self, subject: str, description: str = "") -> Task:
        """Append a new pending task and return it."""

    def update_task(
        self,
        task_id: str,
        *,
        status: TaskStatus | None = None,
        subject: str | None = None,
        description: str | None = None,
    ) -> Task | None:
        """Update one or more fields of an existing task. Returns the
        new record, or None if the id is unknown."""

    def remove_task(self, task_id: str) -> Task | None:
        """Remove the matching task; return the removed record, or
        None if the id is unknown."""

    def replace_tasks(self, tasks: Iterable[Task]) -> None:
        """Used by transcript.load to seed the list. Resets the
        next-id counter to max(loaded ids) + 1."""
```

Every mutator notifies subscribers via the existing `_notify()` so
`changes()` consumers redraw.

### Tools in `tools.py`

Three new entries in `BUILTIN_TOOLS`, names matching Claude exactly:

- **`TaskCreate`** — schema `{ subject: str, description?: str }`.
  Calls `store.add_task(subject, description)`. Returns
  `"ok, created task {id}: {subject}"`.

- **`TaskUpdate`** — schema
  `{ taskId: str, status?: enum["pending","in_progress","completed","deleted"], subject?: str, description?: str }`.
  - If `status == "deleted"`: calls `store.remove_task(taskId)`;
    returns `"ok, deleted task {id}: {subject}"` or
    `"task {id} not found"`.
  - Otherwise: calls `store.update_task(taskId, **fields)`;
    returns `"ok, task {id} updated: {summary of changed fields}"`
    or `"task {id} not found"`.

- **`TaskList`** — schema `{}` (no arguments). Returns a JSON-encoded
  list of `{id, subject, status, description}` for every task. Used
  when the LLM explicitly wants the full state (not just the
  actionable-subset reminder).

Tool dispatch passes the store via a new keyword argument:

```python
def dispatch(
    name: str,
    arguments: str,
    *,
    store: ChatStore | None = None,
) -> str: ...
```

Pre-existing tools ignore `store`. The three Task tools require it
and return an explicit error string when `store is None`.
`agent_loop.Agent` is the only caller that passes it in v0.8.0.

### Reminder injection — matching Claude's algorithm

Before each LLM call, `Agent.stream_reply` runs the reminder check:

```python
TURNS_SINCE_WRITE = 10
TURNS_BETWEEN_REMINDERS = 10
TASK_REMINDER_TAG = "<system-reminder>"   # opening marker for scanning

def maybe_inject_task_reminder(
    messages: list[dict],
    store: ChatStore,
) -> dict | None:
    """If both turn-thresholds are met and at least one actionable
    task exists, return a meta user message containing the reminder;
    otherwise None. The caller appends the returned dict to
    `messages` so the reminder is part of the persistent history."""
    turns_since_write = _assistant_turns_since_task_management(messages)
    turns_since_reminder = _assistant_turns_since_reminder(messages)
    if (
        turns_since_write < TURNS_SINCE_WRITE
        or turns_since_reminder < TURNS_BETWEEN_REMINDERS
    ):
        return None
    actionable = [
        t for t in store.tasks
        if t.status in ("pending", "in_progress")
    ]
    if not actionable:
        return None
    body = _build_task_reminder_body(store.tasks)
    return {
        "role": "user",
        "content": f"<system-reminder>\n{body}\n</system-reminder>",
    }
```

Where:

```python
def _assistant_turns_since_task_management(messages: list[dict]) -> int:
    """Count assistant turns scanning backwards until we find an
    assistant message containing a tool_call for TaskCreate or
    TaskUpdate. Returns len(messages) if no such tool call exists."""

def _assistant_turns_since_reminder(messages: list[dict]) -> int:
    """Count assistant turns scanning backwards until we find a
    user message whose content starts with `<system-reminder>` and
    contains 'Here are the existing tasks:'. Returns len(messages)
    if no prior reminder exists."""

def _build_task_reminder_body(tasks: tuple[Task, ...]) -> str:
    actionable = [t for t in tasks if t.status in ("pending", "in_progress")]
    lines = [
        "The task tools haven't been used recently. If you're "
        "working on tasks that would benefit from tracking progress, "
        "consider using TaskCreate to add new tasks and TaskUpdate "
        "to update task status (set to in_progress when starting, "
        "completed when done). Also consider cleaning up the task "
        "list if it has become stale. Only use these if relevant to "
        "the current work. This is just a gentle reminder - ignore "
        "if not applicable. Make sure that you NEVER mention this "
        "reminder to the user.",
    ]
    if actionable:
        lines.append("")
        lines.append("Here are the existing tasks:")
        lines.append("")
        for t in actionable:
            lines.append(f"#{t.id}. [{t.status}] {t.subject}")
    return "\n".join(lines)
```

The reminder is **a user-role message** (matches Claude's
`isMeta: true` user message with `<system-reminder>` wrapping), not
a system-role message. Three reasons:

1. Direct compatibility with how Claude formats it — copy a known-good
   shape.
2. OpenAI-compatible providers handle user messages uniformly; some
   providers misroute mid-conversation system messages.
3. The XML wrapper unambiguously labels the content as "out-of-band
   instruction, not from the user" for any LLM.

The reminder lives in `agent.messages`. It is persisted with the
rest of the messages via `transcript.save`. Future reminder
trigger checks scan backwards through these messages.

### Persistence: transcript format v2

`transcript.py` bumps `TRANSCRIPT_VERSION` to 2 and writes:

```json
{
  "version": 2,
  "saved_at": "...",
  "provider": "...",
  "model": "...",
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "what's 2+2"},
    {"role": "assistant", "content": "4"},
    {"role": "user", "content": "<system-reminder>\n...\n</system-reminder>"},
    {"role": "user", "content": "..."}
  ],
  "tasks": [
    {
      "id": "1",
      "subject": "refactor onion",
      "description": "",
      "status": "pending",
      "created_at": "2026-05-24T12:34:56",
      "updated_at": "2026-05-24T12:34:56"
    }
  ]
}
```

Reminders are not a separate key — they are user messages already in
`messages`. Round-tripping preserves them because messages
round-trip unchanged.

`transcript.load` accepts version 1 (no `tasks` key → empty list)
and version 2. Save always writes v2.

If a future v3 ever lands, `load` raises `ValueError` for
versions > 2 with a "this neutrix expects version 2" message,
matching the v0.6.x → v0.7.0 pattern.

### `/tasks` command

`terminal_chat.py` (and `tui.py`) gain `_cmd_tasks`:

- With no args: prints the current task list to scrollback, one line
  per task, formatted as `#{id} [{status}] {subject}` in dim style.
  Empty list: prints `"no tasks"`.
- Out of scope for v0.8.0: editing via the CLI. The LLM tools are the
  canonical write path.

Add `/tasks` to the `/help` listing right under `/status`.

## Acceptance

### Automated (the bulk of the gate)

- `tests/test_store.py`: add/update/remove tasks round-trip; `id`
  monotonicity; `update_task` and `remove_task` return `None` on
  unknown id; `reset` clears tasks and the id counter;
  `replace_tasks` seeds the counter from max id.
- `tests/test_transcript.py`:
  - Save → load round-trips tasks including all timestamp fields.
  - A v0.7.x v1 file loads with `tasks == ()`.
  - Saving emits `version == 2`.
  - A reminder message in the messages list round-trips unchanged
    (it's an ordinary user message).
- `tests/test_tools.py` (NEW) or extension of `tests/test_smoke.py`:
  - `dispatch("TaskCreate", '{"subject": "x"}', store=store)`
    mutates the store and returns `"ok, created task 1: x"`.
  - `dispatch("TaskUpdate", '{"taskId":"1","status":"in_progress"}', store=...)`
    mutates; unknown id returns the "not found" string.
  - `dispatch("TaskUpdate", '{"taskId":"1","status":"deleted"}', ...)`
    removes the task.
  - `dispatch("TaskList", "{}", store=...)` returns JSON-encoded
    list.
  - Calling any Task tool without `store` returns the explicit
    error string.
- `tests/test_agent_loop.py` (or new): construct messages that
  simulate 0/5/10/15 assistant turns since the last
  TaskCreate/TaskUpdate and since the last reminder. Verify the
  reminder is appended **only** when both counts ≥ 10 and at least
  one task is actionable.
- `tests/test_terminal_chat.py`:
  - `/tasks` with no tasks prints `"no tasks"`.
  - `/tasks` with seeded tasks prints the expected lines.
  - A fake agent that emits a `tool_call` for `TaskCreate` populates
    the store; subsequent `/tasks` shows the new item.

### Manual (genuinely-human judgment only)

1. Launch `neutrix` with your own provider. Type: "Track these as
   tasks: refactor onion, ship v0.8.0, write v0.9.0 PRD." The LLM
   should emit three `TaskCreate` tool calls. Confirm `/tasks`
   shows three items, each `pending`, ids 1-2-3.
2. Continue: "mark the first as in progress." Confirm `/tasks`
   shows id 1 as `in_progress`.
3. Continue with at least 10 more LLM turns of any kind that don't
   touch tasks. On the next turn, the reminder should silently
   appear in `agent.messages` (visible in `/save` output) and the
   LLM should not mention it to the user.
4. Continue: "I finished the first one." LLM should emit
   `TaskUpdate(taskId="1", status="completed")`. Confirm.
5. `/save` the conversation. `/quit`. Restart with
   `neutrix --load PATH`. Run `/tasks`. Confirm all three are
   restored with the correct statuses. Confirm the saved JSON
   contains the reminder message in `messages` and the task list in
   `tasks`.

## Open questions (resolve during implementation)

- A reminder injected then immediately answered by the LLM with a
  `TaskUpdate` call resets the "turns since last management"
  counter, but the reminder message stays in history. That's
  fine — it's the intended audit trail. Confirm during testing
  that this doesn't surprise anyone.
- Should `TaskList` also be triggered automatically when the user
  types `/tasks`? Default: no — `/tasks` reads directly from the
  store, doesn't go through the LLM. The LLM uses `TaskList` only
  when it wants to "remind itself" of the full state.
- Should `TaskUpdate` with `status="deleted"` also be exposed as a
  separate `TaskDelete` tool for clarity? Default: no, follow
  Claude exactly — one update tool with a special status value.
- The reminder is wrapped in `<system-reminder>` tags. Some
  providers (non-Claude) might not understand the tag; the model
  may treat it as part of the user message. Acceptable risk for
  v0.8.0 — the surrounding "this is a reminder, never mention it"
  text inside the tags handles the semantics regardless.
