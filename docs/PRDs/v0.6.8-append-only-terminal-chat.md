# v0.6.8 - append-only terminal chat

## Problem

The main chat currently runs as a fullscreen Textual app. That makes the
conversation area occupy exactly one terminal screen, so long transcripts need
an internal scroll region and short conversations still consume the whole
screen.

This feels unlike Codex and Claude terminal chats, where output is appended to
normal terminal scrollback. The terminal should own transcript height: a short
conversation should be short, and a long conversation should naturally extend
past one screen without an app scrollbar.

Tool results are also too noisy. Even with truncation, a directory listing or
large file result can flood the visible transcript. The model still needs the
full result, but the human-visible transcript should default to a compact
summary.

## Goal

Replace the main chat launch path with an append-only terminal renderer:

- no fullscreen chat app;
- no internal message scrollbar;
- normal terminal scrollback for long conversations;
- normal terminal prompt height for short conversations;
- a real multiline draft editor at the bottom;
- status attached to the draft editor instead of printed into the transcript;
- normal editing keys in the draft editor, including `Ctrl+A` and `Ctrl+K`;
- submitted drafts are erased from the editor area and re-rendered as styled
  user transcript content;
- the draft editor remains active while the assistant is working, so the user
  can type the next prompt without waiting;
- prompts submitted while the assistant is working are queued and run in order;
- blocking tools such as `run_shell` do not block the UI event loop;
- operational logs do not write directly into the interactive terminal surface;
- system, user, and assistant messages are distinguished by color instead of
  textual role labels such as `assistant:`;
- folded one-line tool results by default;
- exact line counts and cheap approximate token counts on folded tool results.

## Non-goals

- Replace the onboarding UI. First-run and `/onboard` may continue to use the
  existing Textual onboarding screen.
- Remove the existing Textual main chat class in this release. It can remain
  available for tests and future comparison while the CLI default changes.
- Add a tokenizer dependency. Approximate token counts should be cheap and
  deterministic.
- Hide full tool results from the model transcript. Only the human rendering is
  folded.
- Print a user-turn label such as `you>` before every prompt.
- Render assistant responses with a literal `assistant:` heading.
- Block the input editor while the assistant is working.
- Add tree-style flow editing in this release. Future branch/fork editing needs
  a DAG session model and a spatial UI; the terminal should remain a simple
  active-path interface for now.
- Build a web graph UI in this release. The design should not block one later.

## Design

1. Add a `TerminalChat` controller that owns terminal rendering and command
   handling.
2. Change `cli.py` to launch `TerminalChat.run()` instead of `NeutrixApp.run()`
   for the main chat path.
3. Keep `Agent.stream_reply()` unchanged. It still appends full user,
   assistant, and tool messages to the model-visible transcript.
4. Render user input through a bottom draft editor backed by `prompt_toolkit`
   when available. The draft editor has no `you>` label; the cursor and
   placeholder are enough.
5. Keep provider/model/tool/message status in the draft editor's bottom toolbar
   rather than writing it after every assistant/tool block.
6. Use explicit draft-editor key bindings for common terminal editing keys:
   `Ctrl+A` moves to the beginning of the current line and `Ctrl+K` deletes to
   the end of the current line.
7. Submit with `Enter`; insert a newline with `Ctrl+J` or `Alt+Enter`.
8. In stale environments without `prompt_toolkit`, fall back to plain `input("")`
   without adding a `you>` prompt.
9. Configure the draft editor to erase submitted text, then append the submitted
   user turn as styled transcript output.
10. Split terminal chat into a small controller/view structure:
    - the controller owns queueing, command dispatch, model/tool workers, and
      agent state mutation;
    - the view owns prompt reading, status toolbar text, and transcript output.
11. Keep one prompt reader active while model/tool work runs. Submitted normal
    prompts are appended to a FIFO queue; slash commands execute immediately
    when safe, or print an explicit message if they require an idle agent.
12. Use prompt-toolkit stdout patching when available so model/tool output is
    written above the active composer instead of corrupting the draft.
13. Tool dispatch runs off the UI event loop. The controller receives
    `tool_call` / `tool_result` events and the view renders those events; tool
    implementation logs and subprocess internals do not write directly to the
    terminal.
14. Configure chat logging to write to a file instead of stderr before the
    interactive chat starts.
15. Render assistant content as appended terminal output without an
    `assistant:` label. Markdown rendering can
   continue using Rich when enabled; `--no-markdown` prints plain text.
16. Render system, user, and assistant transcript content with distinct styles.
17. Render every tool call as a compact one-line call record.
18. Render every tool result as one folded line:

   ```text
   <- [tool 1] list_dir {"path":"."} | folded | 37 lines | ~210 tokens
   ```

19. Store folded tool results in memory and expose `/tool N` to print one full
   result on demand. `/tool` lists recent folded results.
20. Count lines exactly from `splitlines()`, treating an empty result as zero
   lines. Approximate tokens by counting non-whitespace word chunks.
21. Keep existing slash commands where practical: `/help`, `/fast`, `/strong`,
    `/model`, `/onboard`, `/save`, `/load`, `/clear`, `/tools`, and `/quit`.
22. On `/onboard`, launch the existing onboarding app, reload config after it
    exits, and keep the current slot unchanged until the user runs `/fast` or
    `/strong`.
23. Keep future branch editing out of the terminal layout decision. The terminal
    remains the active-path chat surface; a later local web UI can own visual
    branch/tree editing once the session model supports it.

## Acceptance

- Running `neutrix` no longer calls `NeutrixApp.run()` for the main chat.
- The main chat writes to ordinary terminal output and returns to a normal
  draft editor for each user turn.
- Long output relies on terminal scrollback instead of a Textual
  `VerticalScroll`.
- Short conversations do not reserve a full terminal screen.
- The input prompt does not show a `you>` label.
- Submitted user text does not remain as unstyled prompt text.
- Submitted user text is appended as styled user transcript content.
- The user can type a new prompt while the assistant is working.
- A prompt submitted while the assistant is working is queued and sent after the
  active response finishes.
- Model/tool output does not corrupt the active draft editor.
- Long-running shell tools do not block typing in the draft editor.
- `loguru` operational lines such as shell command logs do not appear in the
  chat transcript.
- Status is not printed after every assistant/tool block.
- Status stays attached to the bottom draft editor when `prompt_toolkit` is
  available.
- `Ctrl+A` and `Ctrl+K` work inside the draft editor.
- `Enter` submits; `Ctrl+J` or `Alt+Enter` inserts a draft newline.
- Assistant replies do not render an `assistant:` heading.
- System, user, and assistant content have distinct transcript styles.
- Tool results render as one folded summary line by default.
- Folded tool result summaries include the tool name, arguments, exact line
  count, and approximate token count.
- `/tool N` prints the full stored result for a folded tool result.
- `/tool` lists stored folded result summaries.
- Full tool result content still remains in `Agent.messages` for follow-up
  model calls.
- `/save`, `/load`, `/clear`, `/tools on`, `/tools off`, `/fast`, `/strong`,
  and `/model` continue to work in terminal mode.
- `/onboard` still opens the existing onboarding flow and reloads config after
  exit.
- The implementation does not introduce a fullscreen branch/tree UI or make the
  terminal layout depend on future flow editing.
- Existing non-UI agent-loop, LLM, config, session, and tool tests pass.
