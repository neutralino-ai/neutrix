# v0.8.1 — persistent task panel, in-band continuation nudge, folded reminders, placeholder polish

## Problem

After v0.8.0 shipped tasks as first-class store state, four gaps are
visible in everyday use:

1. **Tasks are invisible by default.** They live in the store but only
   render on demand via `/tasks`. The user has to remember to ask, and
   during execution they cannot see what is pending vs in-progress.
   This breaks the "tasks are a first-class signal" intent that
   motivated v0.8.0.

2. **The agent stops after planning.** When the user gives a request
   the LLM answers with `TaskCreate(...)`, the tool returns
   `"ok, created task 1: ..."`, and the model writes a short
   acknowledgement and ends its turn. The loop exits. The user has to
   type "ok continue" to get the LLM to actually work on what it just
   planned. Planning should hand off to doing, not back to the human.

3. **The v0.8.0 reminder is either invisible or leaks raw.** The
   10-turn `<system-reminder>` injection lives in `agent.messages` and
   never renders in the live transcript, but it *does* surface — as a
   plain user-text block — after `/save → /load`. The user sees nothing
   when it actually fires, then sees a wall-of-text reminder on the
   next reload. Both halves are wrong: live, the user has no idea the
   harness nudged the LLM; on replay, the LLM-addressed body is
   rendered as if the user wrote it.

4. **Placeholder text reads as foreground.** The current prompt_toolkit
   placeholder `"Message the assistant  (/help for commands)"` renders
   in the default style, which on the user's terminal reads close to
   white. It should look like the hint it is — dim gray, visually
   subordinate to typed text.

## Goal

1. **Always-visible task panel above the input**, matching Claude
   Code's `TaskListV2` design: show every task (pending, in_progress,
   completed) when any exist, hide when none, cap at a small number of
   rows with a `"… +N pending, M done"` overflow line.

2. **V2-shape tool descriptions** — port Claude Code's V2
   `TaskCreate` / `TaskUpdate` / `TaskList` tool prompts verbatim into
   the descriptions our LLM sees on every turn. The lifecycle cues
   live in the *tool schema* (where Claude Code itself puts them),
   not in result-text suffixes. Specifically:
   `TaskCreate` carries the *"When you start working on a task —
   mark it as in_progress BEFORE beginning work"* clause; `TaskUpdate`
   carries *"After resolving, call TaskList to find your next task"*
   and *"Always mark your assigned tasks as resolved when you finish
   them"*; `TaskList` carries *"Prefer working on tasks in ID order"*.
   The tool result strings themselves stay the bare V2 acks (`"Task #N
   created successfully: subject"`, `"Updated task #N status"`, etc.)
   — Claude Code's V2 tools do not append nudges to results.

3. **Render the v0.8.0 reminder as a folded dim notice.** Keep the
   trigger algorithm, body text, and persistence format exactly as
   shipped in v0.8.0 — matching Claude Code's
   `getTaskReminderTurnCounts` mechanism, in which the controller
   checks before every LLM call and injects a `<system-reminder>` only
   when both *≥10 assistant turns since the last `TaskCreate` /
   `TaskUpdate`* and *≥10 turns since the last reminder* (and ≥1
   actionable task). What changes is purely renderer-side: detect the
   `<system-reminder>` shape and render a single dim one-line notice
   in both the live transcript and `/load` replay, instead of leaking
   the full body or staying silent. The folded form fits the principle
   *"everything visible, just by default folded or not"* and avoids
   duplicating the task list (already shown by the panel from Goal 1).

4. **Dim placeholder.** Convert the input draft placeholder to
   prompt_toolkit `FormattedText` with `fg:ansibrightblack`, matching
   the dim style already used for queued user messages so the input
   region has a single visual hierarchy.

## Non-goals

- **A separate `Advisor` actor in v0.8.1.** The user's third-actor
  reasoning ("LLM, executor, and a third instance interacting with
  the controller: the advisor") is architecturally correct, and is
  the v0.11.0 PRD (`v0.11.0-smart-advisor.md`). Spinning up the actor
  now pulls the v0.9.0 controller refactor forward and bundles two
  scopes that are cleaner shipped separately. For v0.8.1 the v0.8.0
  controller-side polling logic stays exactly as it is and serves as
  the cheap safety net. v0.11.0 replaces it with judgment-driven
  injections.
- **Deleting the v0.8.0 polling mechanism.** Considered and rejected:
  Claude Code keeps the same polling check; removing it between
  v0.8.1 and v0.11.0 would leave a window where the LLM has only the
  in-band nudge and no stale-state safety net. Keep parity with CC.
- **Pulling forward the rest of v0.10.0.** The TaskPanel is the one
  v0.10.0 item that ships now; the broader TUIView-purify refactor
  (`_tool_records` → store, `FakeView` swap test, boundary check) stays
  in v0.10.0.
- **The v0.9.0 controller refactor and heartbeat indicator.**
- **A `/reminder N` expand command.** The folded notice plus the
  task panel cover the information; the full body is still in
  `/save`'d files for debugging. When the v0.11.0 Advisor lands and
  emits judged content worth expanding inline, design the expand
  affordance with it.
- **A `/stop` command for halting auto-continue.** Ctrl+C is the
  existing escape hatch. If the in-band nudge proves too eager and
  the model loops, address it then.
- **Configurable panel cap, row count, or styling.** One good default.
- **Recently-completed TTL highlighting.** Claude's
  `RECENT_COMPLETED_TTL_MS = 30_000` boosts just-finished tasks in the
  display sort. Useful, but not load-bearing for v0.8.1 — defer.

## Design

### Architecture: where does "advice" live across releases?

The user's framing — *"besides LLM and executor, there is another
instance interacting with the controller: the advisor"* — is correct
and is the v0.11.0 PRD (`docs/PRDs/v0.11.0-smart-advisor.md`). The
v0.8.1 release does not add that actor; it keeps the v0.8.0
controller-side reminder logic intact and only changes how the
reminder is rendered.

The arc across releases:

| Kind of advice | v0.8.0 (shipped) | v0.8.1 (this) | v0.11.0 (future Advisor) |
|---|---|---|---|
| "You just created/updated a task — go do it" | none | **V2 tool descriptions** carry the lifecycle cues (*"Mark it as in_progress BEFORE beginning work"*, *"After resolving, call TaskList to find your next task"*). Lifted verbatim from Claude Code's V2 task tool prompts. Result strings stay the bare V2 acks. | Same descriptions + smart Advisor judging the conversation. |
| "You haven't touched the task tools in 10 turns" | Templated `<system-reminder>` user message injected by the controller. Invisible live; leaks as a plain user block via `/save → /load`. | **Trigger unchanged, body unchanged, persistence unchanged.** Renderer now shows a folded dim "↳ system-reminder · injected …" notice the moment it lands and on replay, instead of leaking on `/load` or staying silent live. | Replaced by judgment-driven Advisor injections — content-aware revisions ("task #3 looks done based on the last commit — mark completed?"). Uses a different tag so its renderer treatment doesn't collide with the folded v0.8.x form. |
| "I've reviewed your task list and conversation and here's a revised plan" | none | none | New. The Advisor runs its own (small) LLM call, judges the gap between current tasks and recent work, and emits structured revisions. |

Why render the v0.8.0 reminder *folded* rather than either fully
expanded or fully suppressed?

- **Fully expanded** duplicates the task panel content from Goal 1.
  The reminder body lists every actionable task; the panel above the
  input already shows them. No new information; just noise.
- **Fully suppressed** loses the user's audit trail. They no longer
  see when the harness nudges the LLM. For a system whose principle
  is *"everything visible, folded or not,"* silent injections are a
  leak in the other direction.
- **Folded** captures the *event* ("a system-reminder was injected
  because the task tools went idle for 10 turns") without re-rendering
  the *data* (the task list, already on screen). One dim line is
  enough.

The v0.11.0 Advisor will use a different tag (proposed `<advisor>` or
`<system-suggestion>`) so its judged, expanded-by-default content
doesn't collide with the v0.8.x folded-by-default shape. The
`_is_task_reminder` predicate already in `agent_loop.py` is pinned to
the v0.8.0 wording and will continue to recognize only the legacy
shape.

### 1. Task panel above the input

`TerminalChat._queued_message` is the single supplier of "what gets
drawn between the transcript and the input cursor". Rename to
`_above_input` and have it return a `FormattedText` that stacks (top →
bottom):

```
  system reminder: task list injected (1 done, 1 in progress, 1 todo)  ← only when present
  ◼ #2 [in_progress] refactor onion                           ← bright/accent
  ◻ #3 [pending] write v0.8.1 PRD                             ← normal
  ✓ #1 [completed] ship v0.8.0                                ← dim green
  … +2 pending, 1 done                                        ← only if truncated
  › queued user message                                       ← existing queue line
> _                                                           ← input cursor
```

(The reminder folded notice is rendered in the transcript flow when
the reminder fires, not in the `_above_input` region — listed in this
sketch only to show the full visual stack the user perceives.)

**Hide entirely when there are no tasks.** Matches Claude
(`TaskListV2.tsx` line 89-91: `if (tasks.length === 0) { return null }`).

**Cap at `MAX_PANEL_ROWS = 5`** task lines. When `len(tasks) > 5`,
keep the top 5 by Claude's priority (in_progress → pending →
completed, with id ascending inside each group) and add a `… +N
pending, M done` line in dim. The `+30s recently-completed boost` is
not implemented in v0.8.1.

**Icons + colors** (loosely mirroring Claude's `getTaskIcon`):

| Status | Icon | Color |
|---|---|---|
| `in_progress` | `◼` (U+25FC BLACK MEDIUM SQUARE) | `fg:ansicyan bold` |
| `pending` | `◻` (U+25FB WHITE MEDIUM SQUARE) | default |
| `completed` | `✓` | `fg:ansigreen` + dim |

A renderer helper `_format_task_panel(tasks: tuple[Task, ...]) ->
list[tuple[str, str]]` converts the store's task tuple into the list
of `(style, text)` fragments the panel needs. Pure function so it can
be unit-tested without a TUI.

**Subscriber wiring.** `_above_input` is already re-evaluated on every
`prompt_toolkit` redraw, and `_invalidation_watcher` already triggers
a redraw on any `ChatStore` mutation. So the panel updates live the
moment `add_task` / `update_task` fires — no new wiring needed.

### 2. V2-shape tool descriptions

**Diagnosis from manual testing of v0.8.0 + an initial v0.8.1 draft:**
the LLM doesn't auto-start work after `TaskCreate` and (worse) will
sometimes claim *"I cancelled them"* without firing any
`TaskUpdate(deleted)` call. Investigation against the cc2 source
revealed the root cause: Claude Code has two task systems gated by
`isTodoV2Enabled()` — **V1** (`TodoWriteTool`, single tool, whole-list
replacement) and **V2** (`TaskCreate`/`TaskUpdate`/`TaskList`,
multi-tool, granular). Our v0.8.0 architecture mirrors **V2**. The
initial v0.8.1 draft lifted *"Please proceed with the current tasks
if applicable"* from V1's `TodoWriteTool.mapToolResultToToolResultBlockParam`
— but **V2's TaskCreate/TaskUpdate carry no such nudge** (verified in
cc2 source and in a captured real Claude Code payload at
`dr.sai-juno/dr.sai-nano/notebooks/note.json`). What V2 does instead
is put the lifecycle guidance in the **tool prompt** sent to the LLM
on every turn.

`tools.py` change:

```python
# Result strings — bare V2 acks, no nudge.
# TaskCreate: f"Task #{task.id} created successfully: {task.subject}"
# TaskUpdate (status/subject/description change):
#            f"Updated task #{updated.id} {', '.join(changed_fields)}"
# TaskUpdate (status="deleted"): f"Updated task #{taskId} deleted"
# TaskUpdate (not found):        f"Task #{taskId} not found"

# Tool descriptions — Claude Code V2 prompts, dropped agent-swarm-only
# branches. The full strings live as module-level constants
# `_TASK_CREATE_DESCRIPTION`, `_TASK_UPDATE_DESCRIPTION`,
# `_TASK_LIST_DESCRIPTION` in tools.py.
#
# Load-bearing clauses (Claude Code verbatim):
#   TaskCreate: "When you start working on a task - Mark it as
#               in_progress BEFORE beginning work"
#               "After completing a task - Mark it as completed and
#               add any new follow-up tasks discovered during
#               implementation"
#   TaskUpdate: "IMPORTANT: Always mark your assigned tasks as
#               resolved when you finish them"
#               "After resolving, call TaskList to find your next task"
#   TaskList:   "Prefer working on tasks in ID order (lowest ID first)"
```

No changes to `agent_loop.py`. The existing tool-continuation `while
True` loop already calls the LLM again with the tool result appended;
what was missing was the lifecycle teaching in the tool schema. With
the V2 prompts present, the LLM sees on every turn the same cues
Claude Code itself uses to drive auto-continuation and reduce
"claimed but didn't call the tool" hallucinations.

**Why this works without a separate Advisor:**

- The LLM is shaped by every turn's tool schema — the V2 prompts are
  always-present context, not a one-shot nudge.
- The `IMPORTANT:` and `BEFORE beginning work` capitalization in
  Claude's prompts is deliberate — these are the LLM-attention
  signals that drive the auto-start behavior.
- If the LLM still ends the turn after a `TaskCreate`, the panel
  still shows the pending tasks and the v0.8.0 10-turn reminder
  mechanism (still alive, now folded-visible) will reactivate work
  eventually.

**What about the V1 nudge wording?** It exists in `TodoWriteTool`
because V1 has no per-task lifecycle — it's one whole-list-replace
tool, so V1 puts the cue in the result text. V2 puts the cue in the
prompt because V2 has per-task lifecycle and the cue is about *how*
to navigate that lifecycle. Mixing them (V2 architecture + V1 nudge)
gave the LLM contradictory shaping and was the proximate cause of
the observed misbehavior.

### 3. Render the v0.8.0 reminder as a folded dim notice

`agent_loop._maybe_inject_task_reminder` and all its supporting
helpers (`build_task_reminder`, `assistant_turns_since_task_management`,
`assistant_turns_since_reminder`, `_message_calls_task_management`,
`_build_task_reminder_body`, plus the `TURNS_*` / `TASK_REMINDER_*`
constants) remain **unchanged**. Trigger algorithm, persistence
format, message shape: all identical to v0.8.0. The `_is_task_reminder`
predicate stays in `agent_loop.py` and gets reused by the renderers.

Two render paths to update.

**`_render_transcript` (replay on `/load` and after `/clear`).** When
iterating `agent.messages`, detect any `role == "user"` entry whose
`content` matches `agent_loop._is_task_reminder`. Instead of rendering
it as a plain user block via `view.print_user`, emit a single dim
one-line notice with live counts of the task list at render time:

```
system reminder: task list injected (1 done, 1 in progress, 1 todo)
```

Counts come from the current `store.tasks` snapshot at the moment of
rendering, not from re-parsing the reminder body. The "todo" bucket
matches `status == "pending"` (CC convention in user-facing strings).
Style: `dim`. Adjacent real user/assistant messages render normally —
only the reminder gets the folded treatment. A helper
`_format_reminder_notice(tasks) -> str` keeps the count-formatting
out of both render sites.

**`_mirror_new_agent_messages` (live mirror after each
`_send_message`).** Today the mirror walks any newly-appended agent
messages to copy them into the store, but it does not render anything
to the view (the reminder appended by `_maybe_inject_task_reminder`
before the LLM stream begins is therefore invisible in the live
transcript). Add a render hook: when the mirror encounters a new
user message matching `_is_task_reminder`, call `view.print_notice`
with the same dim summary so the user sees the reminder land in real
time, in order, between turns.

Single predicate, two call sites, same dim notice. The full reminder
body is still in `agent.messages` and still persists through `/save`
— the LLM sees it unchanged; the user just sees the folded summary.

The notice text deliberately surfaces *what the LLM is being reminded
about* (live `done / in_progress / todo` counts) rather than echoing
the templated body or restating the 10-turn-idle trigger. The user
gets a single-glance read of the task-state delta the reminder is
nudging the LLM toward, without re-reading the full list already on
screen in the panel above the input.

When v0.11.0 lands, its Advisor injections will use a different tag
(`<advisor>` or similar) and get their own renderer treatment —
likely expanded by default, since the body will carry actual advice
rather than redundant state.

### 4. Placeholder color

Change `DraftReader._build_session` placeholder from a bare string to
prompt_toolkit `FormattedText`:

```python
from prompt_toolkit.formatted_text import FormattedText
PLACEHOLDER = FormattedText([
    ("fg:ansibrightblack", "Message the assistant  (/help for commands)"),
])
# ...
return PromptSession(..., placeholder=PLACEHOLDER, ...)
```

`fg:ansibrightblack` is the same dim/gray style already used for
queued user messages and the proposed task panel. The input region
ends up with one consistent dim hierarchy: dim hint, dim queue lines,
dim or accented task panel, normal typed text.

### Decisions locked at PRD sign-off

- **Panel max rows: 5.** Matches Claude Code's `TaskListV2` default;
  overflow line absorbs the rest. Drop to 4 only if manual test shows
  the queue gets crowded out under heavy multitasking.
- **In-progress icon set: Unicode `◼ ◻ ✓`.** Crisp in most monospace
  fonts. No ASCII fallback shipped in v0.8.1 — revisit only if a real
  user terminal misrenders.
- **Completed task styling: dim + `✓`, no strikethrough.**
  prompt_toolkit's strikethrough varies by terminal and the dim green
  checkmark already reads as "done."
- **`TaskUpdate(status="deleted")` does NOT carry the nudge.**
  Deletion ends that item's lifecycle; appending a generic "proceed
  with current tasks" after a delete would push the LLM to pivot work
  unnecessarily.
- **Folded reminder notice wording:**
  `"system reminder: task list injected (N done, N in progress, N todo)"`,
  in dim. Counts are recomputed from `store.tasks` at render time so
  the notice always reflects the live state, even when re-rendered on
  `/load` after deletions or status changes. "todo" maps to
  `status == "pending"`.
