# v0.9.5 — Visibility parity: show what the LLM sees

## Problem

The transcript today hides several channels that the LLM receives
every round:

1. **Tool schemas.** `agent_loop.Agent` passes `get_schemas()` to
   `chat.complete(... tools=tools ...)` every round. The user
   never sees what tool definitions the LLM is being offered. A
   wrong / stale / missing schema is invisible until it manifests
   as broken tool-calling.
2. **System prompt.** `DEFAULT_SYSTEM_PROMPT` sits at
   `agent.messages[0]`. `terminal_chat.TerminalChat._iterate_messages`
   walks the message list with `if role == "user"` / `"assistant"` /
   `"tool"` branches — `"system"` falls through. The user has no
   in-app way to see what instructions the LLM is operating under.
3. **`<system-reminder>` injections.** The v0.8.x task-reminder loop
   appends a `role: "user"` message whose entire content is wrapped
   in `<system-reminder>...</system-reminder>`. The renderer treats
   it as a normal user turn, so the user *sees* the text but cannot
   tell that they didn't type it. Parity is met on bytes; not on
   provenance.
4. **Tool results.** Today rendered as a one-line fold-summary
   (`tool_result | folded | …`) with no way to expand. The LLM sees
   the full text; the user sees a summary with no escape hatch.
   That's a latent break of the rule the moment anyone wants to
   debug.

The invariant we want to encode is **visibility parity**: every
byte the LLM receives in a round appears in the user-visible
transcript for that round, differing only by fold/expand — never
by omission, summarization, or truncation. v0.9.5 brings the
existing channels into compliance, labels injected turns, adds a
fold/expand affordance, and lays down an automated invariant test.

## Goal

1. Render the **system prompt** at the top of the transcript, in a
   folded block (one-line summary; expand on demand).
2. Render the **tool schemas** as a folded block whenever the
   `tools=` bundle is sent (initially: once at session start;
   re-render if the schema set changes).
3. Render **`<system-reminder>`** injections with a distinct
   prefix and style — visually separated from typed user turns.
4. Provide a **fold/expand affordance** so any long block can be
   expanded inline. Default-folded: system prompt, tool schemas,
   tool results above a byte threshold, reminders.
5. Lay down a normative **`.claude/rules/visibility-parity.md`**
   so future PRDs don't quietly reintroduce hidden channels.
6. Add an **invariant test** that asserts, for every recorded LLM
   round, the set of channels sent to the LLM is a subset of the
   set of channels rendered.

## Non-goals

- **Editing** the system prompt or tool schemas from the UI.
  Read-only renders.
- **Per-message fold/expand state persistence** across restarts.
  Folded blocks reset each session.
- **Truncation of what's sent to the LLM.** If we ever decide to
  send the LLM less than what we render (the opposite asymmetry),
  that's a different design question, out of scope here.
- **Visibility parity for non-LLM-bound channels** — debug logs
  (`logger.debug`), internal store state (`store._llm_active`),
  lifecycle events. Those aren't sent to the LLM.
- **The legacy `tui.py` Textual app.** Same non-goal as the rest
  of v0.9.x — only `terminal_chat` is in scope.

## Design

### 1. The normative rule file

Create `.claude/rules/visibility-parity.md` with this body
(reviewable wording — fine to amend during accept-gate):

```markdown
# LLM-view and user-view differ only by fold, never by content

## Rule

Every byte the LLM receives in a round must appear in the
user-visible transcript for that round. LLM-view and user-view
differ by **fold/expand only** — never by omission, summarization,
or truncation.

Channels in scope:
- System prompt
- Tool schemas (the `tools=` parameter)
- All `messages[*]` content (user, assistant, tool)
- Injected `<system-reminder>` turns
- Any future input channel (RAG snippets, cached prefixes,
  multimodal blobs, etc.)

## Why

Without parity, a user reading the transcript cannot reproduce or
audit what the LLM actually saw. Bugs that only manifest with the
hidden context — wrong tool schema, stale system prompt,
accidental reminder injection — become invisible. The transcript
stops being a trustworthy record and becomes an editorial summary.

This invariant also constrains a recurring temptation in agent
loops: "the LLM needs this, but it's noisy for the user, so let's
hide it." That trade always favours the developer over the user.
Folding is the right answer — same data, less visual weight, one
keystroke to expand.

## How to apply

1. **Single source of truth.** The renderer iterates from the
   same `(system, messages, tools)` bundle passed to
   `chat.complete(...)`. No parallel walk over a partial view.
2. **One render hook per LLM-bound channel.** Adding a new input
   channel to the LLM in a PR must add the matching renderer hook
   in the same PR.
3. **Folding, not omission.** Long content renders in a folded
   block with byte-count and an expand affordance.
4. **Label injected turns.** `<system-reminder>` and any other
   loop-injected turn renders with a distinct prefix/style.
5. **Invariant test.** A test asserts that for every recorded
   LLM round, the set of channels sent to the LLM is a subset of
   the set of channels rendered.

## Out of scope

- Layout choices (color, prefix string, fold heuristics) — those
  are PRD-level design.
- Cosmetic asymmetries (renderer prefixes `<- ` to tool results;
  LLM sees raw JSON) — presentation, not content.
- `--verbose` / debug modes that show *more* than the LLM saw —
  fine; parity is a lower bound, not a ceiling.
```

### 2. Single source-of-truth bundle

Today `Agent.stream_reply` builds `(system, messages, tools)`
inline as positional/keyword arguments to `chat.complete`, and
`terminal_chat._iterate_messages` walks `agent.messages` in a
parallel pass. Two pipelines means asymmetry is just a missing
branch away.

Introduce a small read-only view at the agent_loop ↔ terminal_chat
boundary so both sides refer to the same object:

```python
@dataclass(frozen=True)
class LLMRoundBundle:
    system: str
    messages: tuple[dict[str, Any], ...]
    tools: tuple[dict[str, Any], ...] | None
```

`Agent.stream_reply` constructs this once per round, emits an
`AgentEvent("llm_round_start", data=bundle)`, and the renderer
subscribes. Storage stays where it is (`agent.messages` is still
the mutable list); the bundle is a frozen snapshot taken at the
moment the LLM call is made.

### 3. System-prompt block

At session start, the view prints:

```
[system]   prompt · folded · 1.4 KB
```

Folded by default. Expand via the keybinding from §6. Re-renders
only if the prompt text changes (rare; mostly fixed for the
session).

### 4. Tool-schemas block

Whenever `tools` is non-None and changed (initially: once at
session start), the view prints:

```
[tools]    8 schemas · folded · 6.2 KB
```

Expand reveals one line per schema (`name — one-line description`).
Nested expand on a schema reveals its full JSON.

### 5. `<system-reminder>` labelling

`terminal_chat` detects `role: "user"` messages whose content is
wrapped in `<system-reminder>...</system-reminder>` and renders
with a `[reminder]` prefix and a dim style, distinct from the
normal `>` user prompt:

```
[reminder] task tools haven't been used recently · folded
```

Detection: a single regex on the message content. No change to
how reminders are stored — only to how they render.

### 6. Fold/expand affordance

Add a keybinding `Ctrl+E` that expands/collapses the most recent
foldable block. Default-folded:

- system prompt
- tool schemas
- tool results > 200 B
- system-reminder injections

Default-expanded: chat turns, tool results ≤ 200 B, tool_use
lines (already short).

The fold heuristic and keybinding choice are best-fit picks; ride
v0.9.1's keyboard owner if there's a conflict.

### 7. Invariant test

`tests/test_visibility_parity.py`:

- Drive a fake LLM through one round via the existing
  `agent_loop` test scaffolding.
- Record (a) the `LLMRoundBundle` snapshot, (b) the sequence of
  `view.print_*` calls made during that round.
- Assert: every non-empty channel in the bundle has at least one
  matching render call. Channel inventory: `system`, each
  message by index, `tools` (presence, count matches).
- Scenarios covered: bare round (system + 1 user message),
  round with tools enabled, round with `<system-reminder>`
  injection, round with a tool call + tool result.

## Acceptance

### Automated

- `tests/test_visibility_parity.py` passes (new file, scenarios above).
- `tests/test_terminal_chat.py`, `tests/test_agent_loop.py`,
  `tests/test_chatstore.py` pass unchanged.
- `ruff check .` clean.
- `.claude/rules/visibility-parity.md` exists with the wording
  above (or an amended-and-accepted variant).

### Manual

- Launch `neutrix`. At session start the transcript opens with
  `[system]   prompt · folded · N B` and (if tools are enabled)
  `[tools]    M schemas · folded · K B`.
- Press `Ctrl+E` on the system block → full prompt expands; press
  again → re-folds.
- Press `Ctrl+E` on the tools block → schema list expands; nested
  expand on a single schema reveals JSON.
- Send a message that triggers a tool. Tool result renders folded
  if > 200 B. `Ctrl+E` expands.
- Run the session long enough to trigger a `<system-reminder>`
  injection. The reminder renders with `[reminder]` prefix and dim
  style, visually distinct from typed user turns.
- No transcript content is *visible to the LLM* (by reading
  `agent.messages` + the bundle) that does *not* appear somewhere
  on screen, folded or expanded.

## Out of scope (revisit later)

- Editing the system prompt or tool schemas from the UI.
- A `/dump` slash command that prints the entire round bundle to
  a file for offline inspection. Useful but separable — v0.10.x
  or a dedicated debugging PRD.
- Per-block fold state persisted across restarts.
- Parity for retrieval/cache snippets if/when we add them. The
  rule covers them; the implementation lands when those channels
  exist.
