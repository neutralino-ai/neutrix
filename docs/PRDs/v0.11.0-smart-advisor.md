# v0.11.0 — smart Advisor (third actor with LLM-driven task judgment)

> **Status: skeleton.** Detail solidifies after v0.8.1 (visible task
> panel + in-band nudges), v0.9.0 (controller refactor + heartbeat),
> and v0.10.0 (TUIView purify + FakeView swap test) land. The Advisor
> is the architectural payoff: a third actor with its own LLM call,
> watching the store, revising the task list with judgment instead of
> mechanical bookkeeping.

## Problem

After v0.8.1 + v0.9.0 + v0.10.0 the system has:

- a clean **controller** that runs LLM turns and dispatches tools,
- a pure **executor** path for tool dispatch,
- the **user** typing into the controller via the input queue,
- a **store** that is the canonical log of messages, queue, pending
  tools, tasks, and heartbeat state,
- a **TUIView** that is a pure reader of the store.

But the only "advice" the LLM gets between turns is mechanical:

1. **In-band tool-result nudges** (`TaskCreate` / `TaskUpdate` results
   end with *"please proceed with the current tasks if applicable"*) —
   shipped in v0.8.1. Works, but the same string regardless of state.
2. **The 10-turn stale-state reminder** (`<system-reminder>` user
   message, rendered as a folded one-line dim notice in v0.8.1) —
   shipped in v0.8.0. Fires on a counter, body is templated from the
   current task list with no judgment about whether the items are
   still relevant.

Both are dumb. They cannot:

- detect that *task #3 is effectively done* based on the work the LLM
  has been doing for the last 5 turns (and nudge `TaskUpdate(status=
  "completed")`),
- spot that *the conversation has drifted away from task #5* and that
  the task is stale (and propose deletion or revision),
- notice that *the user just asked for something new* that should be
  added as a task before the LLM forgets,
- aggregate across multiple chat turns to summarize *"you've done X,
  Y, Z this session; here's what's left"* in a way that's useful for
  both the LLM (context) and the user (situational awareness).

The user articulated this in PRD review for v0.8.1:

> "the advisor will be upgraded to be smart => not just mechanically
> record task update etc. but revise."

That upgrade is v0.11.0.

## Goal

Introduce a third actor, `Advisor`, that lives next to the controller
in the runtime topology:

```
                  ┌──────────────┐
   user input  →  │              │  → tool dispatch  →  Executor
                  │  Controller  │
   LLM stream  ←  │              │  ← LLM API        ←  LLM
                  └──────────────┘
                         ↑
                         │ injects pseudo-user messages
                         │ ("here's a revised task plan")
                         │
                  ┌──────────────┐
                  │   Advisor    │ ← own LLM call (cheap slot)
                  └──────────────┘
                         ↑
                         │ subscribes to ChatStore.changes()
                         │ reads tasks + recent messages
                         │
                  ┌──────────────┐
                  │  ChatStore   │
                  └──────────────┘
```

Concrete deliverables:

- A new `neutrix.advisor` module owning the `Advisor` class:
  subscribes to `ChatStore.changes()`, runs its own LLM calls on a
  configurable trigger, and either (a) directly mutates the store via
  the existing Task tools, or (b) injects a `<system-reminder>`-style
  pseudo-user message into the next controller turn (the v0.8.0
  mechanism, repurposed for *judged* content), or both.
- A second reminder shape with **its own renderer treatment** —
  chose a distinct tag (proposal: `<advisor>` or
  `<system-suggestion>`) so the v0.8.1 renderer can tell judged
  Advisor output apart from the v0.8.0 mechanical reminder. v0.8.1
  renders the v0.8.0 shape as a folded one-line dim notice (because
  the always-visible task panel already shows the data the body
  duplicates). The Advisor's content carries actual advice, not
  state echo, so its default rendering is **expanded inline** — the
  user reads it, the LLM acts on it. Same "everything visible,
  folded or not" principle, different folding default per content
  type.
- A `/advise` slash command for on-demand triggering.
- Trigger configuration in `~/.config/neutrix/config.yaml`: cadence
  (every N controller turns), idle threshold (after M seconds of no
  user input), and which slot the Advisor uses (default: `fast`).

## Non-goals

- **Replacing the in-band tool-result nudge from v0.8.1.** The cheap
  nudge keeps working and is the right tool for the immediate
  "I just created a task, now do it" case. The Advisor handles
  cross-turn judgment, not single-tool-call cues.
- **Replacing the v0.8.0 10-turn stale-state reminder.** Keep it as
  the cheap safety net. The Advisor *augments* it with judged
  reminders that carry more signal than "you haven't called the tool
  in a while". The two can coexist; over time, if the Advisor is
  good enough, the mechanical reminder can be removed in a later
  release.
- **A general-purpose subagent framework.** The Advisor is one
  specific actor for task-list judgment, not a generic
  spawn-subagents-from-the-controller surface. Subagents are a v1.0+
  scope question.
- **Cross-session learning.** The Advisor sees the current session
  only. Memory across sessions is a separate concern (and a separate
  PRD if/when it arrives).
- **Writing to anything other than the store.** The Advisor calls
  Task tools or injects messages — it does not edit files, run
  shells, or call the user's tools. Its blast radius is the task list
  and the message history.
- **Streaming output.** The Advisor's LLM call is small,
  background, and final-only. No streaming UI needed.

## Design (skeleton — finalize after v0.10.0)

### Trigger policy

Three triggers, OR'd together; first to fire runs the Advisor:

1. **Periodic.** After every N user turns (default N = 5). Cheap
   amortized cost; catches drift.
2. **Idle.** After M seconds of no user input (default M = 60). Lets
   the Advisor weigh in during natural pauses.
3. **Explicit.** `/advise` command. User-driven.

A run lock prevents re-entry: if an Advisor call is in flight when
another trigger fires, the second one is dropped.

### Per-run context

The Advisor's LLM call sees:

- the system prompt for the Advisor role (separate from the main
  agent's system prompt; emphasizes "review and suggest, don't
  execute"),
- the current task list (full, including completed),
- the last K user/assistant turn pairs (K = 6 default, cap on token
  budget),
- the available Task tools (`TaskCreate`, `TaskUpdate`, `TaskList`).

It does **not** see the full conversation, tool results, or the
v0.8.0 stale reminders — keep the Advisor's context small and
focused.

### Output: two channels

The Advisor's turn can produce zero or more of:

1. **Direct store mutations** via the Task tools — same dispatch path
   the main agent uses (with `store=` passed through). Example: the
   Advisor decides task #3 is done and emits `TaskUpdate(taskId="3",
   status="completed")`.
2. **A user-visible suggestion message** — pseudo-user message
   wrapped in a distinct tag (`<advisor>…</advisor>` or
   `<system-suggestion>…</system-suggestion>`, decided during
   implementation). The renderer treats this differently from the
   v0.8.0 `<system-reminder>`: render as a dim, expandable notice
   in the live transcript, prefixed with `↳ advisor:`. The next
   controller LLM call sees it in `messages` just like a real user
   message, so the main LLM can react to the suggestion.

The two channels are independent. A single Advisor run might just
mutate the store (silently keep tasks in sync) or just emit a
suggestion (no state change, just a note for the user and the LLM).

### Naming: why "Advisor"

The user named it. The name captures the role precisely: it
**observes** without controlling, **suggests** without executing
arbitrary code, and **defers** to the controller for any action that
touches the world. Alternatives considered and rejected:

- `Curator` — too narrow, only fits the task-list-pruning case.
- `Supervisor` — implies authority over the controller.
- `Critic` — adversarial framing; we want collaboration.
- `Coach` — too informal.
- `Strategist` — too plan-heavy.

The CC analog is `tasks/DreamTask/` (background memory
consolidator). Same architectural shape; different domain.

### Why this needs v0.9.0 + v0.10.0 first

- **v0.9.0 controller refactor**: gives the Advisor a clean place to
  *inject* messages. Today the input path is hardcoded into
  `TerminalChat._worker_loop` via an `asyncio.Queue[str]`. After the
  controller refactor, the controller exposes a structured input
  surface (likely `controller.submit_user_message(...)` or
  equivalent) that the Advisor can call without poking TUI internals.
- **v0.10.0 TUIView purify**: gives the Advisor confidence that
  store mutations are the only way state changes. If view code still
  has side-channel writes, the Advisor's "I'll just mutate the
  store" model has holes.
- **v0.8.1 visible task panel**: gives the user a way to see Advisor
  edits land in real time. Without the panel, store mutations are
  invisible, and the user has no feedback loop for whether the
  Advisor is making good decisions.

### Cost / budget

Conservative defaults:
- Use the `fast` slot, not `strong`. The Advisor is doing pattern
  matching, not deep reasoning.
- Cap per-run input tokens (K=6 turn pairs, full task list).
- Hard cap on Advisor calls per session (default: 20). After cap,
  Advisor stops auto-triggering; user can still invoke via `/advise`.
- Log per-run cost to `~/.cache/neutrix/neutrix.log`; surface session
  total via `/status` (extend the existing line).

## Acceptance (sketch)

- Automated:
  - `tests/test_advisor.py`: unit-test the trigger policy (periodic
    counter, idle threshold, run-lock against re-entry).
  - Mock-LLM tests for the Advisor's prompt building (right context
    in, right tools available) and for its output handling (store
    mutations via dispatch, suggestion message wrapping).
  - Renderer test: `<advisor>` user messages render as an
    expanded-by-default notice with the full advice body;
    v0.8.0 `<system-reminder>` user messages still render as the
    v0.8.1 folded one-liner (no regression in the legacy shape).
- Manual:
  - Multi-turn session where the LLM clearly finishes a task without
    calling `TaskUpdate(completed)`. After ≤2 follow-up turns the
    Advisor should fire and mark it completed; the panel updates
    live.
  - Idle for ≥60s mid-session with stale tasks: an `↳ advisor:`
    notice appears in the transcript suggesting cleanup.
  - `/advise` works on demand; produces a suggestion within a few
    seconds.
  - v0.8.0's "haven't been used" reminder still renders as the
    v0.8.1 folded one-liner when it fires (no regression from
    v0.8.1).
- Human gate before tag.

## Open questions (resolve before implementing)

- **Tag name for visible advisor injections.** `<advisor>` is short
  and unambiguous; `<system-suggestion>` mirrors the existing
  `<system-reminder>` shape. Pick during design review.
- **Should the Advisor's tool calls render in the transcript?** It
  uses the same Task tools as the main agent; surfacing them as
  normal tool blocks would inflate the chat. Probably fold them the
  way v0.8.1 folds the legacy reminder — one dim notice per call
  (`↳ advisor: marked #3 completed`) — so the user has an audit
  trail without the visual weight of full tool blocks.
- **Does the Advisor have its own message history?** Or is each run
  context-free (fresh system prompt + current store + last K turns
  from the main session)? Default proposal: context-free. Simpler;
  cheaper; less surface area for bugs. Reconsider if early use shows
  the Advisor "forgets" useful framing between runs.
- **What happens when the user is mid-typing and the Advisor injects
  a message?** Race against the input queue. Likely answer: Advisor
  acquires the same lock the user input does, so injection happens
  between turns, never mid-turn. Confirm during implementation.
- **Conflict resolution**: what if the Advisor and the main LLM make
  conflicting Task mutations in the same turn? Last-write-wins on
  the store is the cheap answer; a real conflict-merge is out of
  scope for v0.11.0.
- **Does the Advisor know about itself?** I.e. should its system
  prompt say "don't suggest creating yourself as a task" or similar
  guardrails? Probably yes; phrase carefully during implementation.

## Out of scope (revisit in later releases)

- Multi-advisor (one for tasks, one for context, one for memory).
  v0.11.0 ships the task Advisor only.
- User-configurable Advisor system prompt. Hard-coded for v0.11.0;
  expose later if usage shows people want to customize tone or
  domain.
- Advisor-emitted streaming UI ("thinking…" indicator while the
  Advisor's LLM call is in flight). The heartbeat from v0.9.0 can
  cover this without new UI work.
