# v1.5.1 — interactive prompt can't hang the turn

> **Status: finalized.** Fixes a v1.4.8 regression, diagnosed live (py-spy + `ss`)
> and **advisor-confirmed**. User-directed: "the agent should handle it, not the
> user." Splits: [docs/splits/v1.5.1-prompt-no-hang.html](../splits/v1.5.1-prompt-no-hang.html).

## Problem

v1.4.8 turned the permission-`ask` verdict from **block-with-notice** (the turn
continued) into an **interactive prompt that parks the turn awaiting
`_pending_answer` with no timeout and no escape but answering**. The user's
routine commands (`rm -rf build`, `git push --force`) trip `auto`-mode's
destructive-Bash heuristic → a prompt → and if it goes unanswered (the indicator
was a dim "answering:" that was easy to miss amid build output), **the turn hangs
forever**: idle event loop, no network in flight, and the v1.4.9 watchdog never
fires (it only runs during `AWAITING_LLM`). Live evidence: the stuck process had
**no `ESTAB` connection** (no LLM request), CPU idle, and `_ask_user` awaiting a
Future with no timeout.

## Goal

An interactive prompt (AskUserQuestion **and** permission-`ask`) must **never**
park the turn indefinitely. Unanswered → degrade to the safe non-interactive
behavior; the agent handles it without the user needing Esc or a config tweak.

## Non-goals

- Removing the interactive prompt (the user *wanted* the v1.4.8 upgrade — keep it,
  just make it non-fatal).
- Auto-retry of dropped LLM connections (that was a misdiagnosis of this hang —
  discarded; the connection was a leaked idle pool socket, not the cause).
- The session-resume feature (slid to v1.5.2; `session_store.py` already built).

## Claude Code reference

CC's prompts can be aborted (Esc → `onAbort` → cancel the turn) and are never
silent. neutrix matches *the never-park guarantee* via a timeout→safe-fallback,
reusing the existing "no interactive consumer" branch.

### Split point: prompt timeout → safe fallback
- **Decision**: `_ask_user` awaits `asyncio.wait_for(_pending_answer, PROMPT_TIMEOUT_S)`;
  on `TimeoutError` returns `None`. The Executor's existing `answer is None` branch
  → permission-`ask` block-with-notice (turn continues), AskUserQuestion
  not-available. Never parks.

### Split point: prominent indicator + Esc
- **Decision**: `_above_input` shows `▶ answer needed: <header> — type a number, or
  Esc to skip` (bold reverse), not a dim "answering:". A required answer is not an
  idle affordance. Esc still raises `CancelledError` (a real cancel, unwinds the turn).

### Split point: cross-task safety
- **Decision**: on `wait_for` timeout the Future is cancelled (done); the input
  loop's `not pending.done()` guard skips it, and `finally` nulls the pending state
  — a late keystroke is a normal message. No new state.

### Split point: should `auto` still prompt for `rm -rf`/`--force`?
- **Decision**: yes (user wanted the interactive upgrade) — but it's now *safe*
  (can't hang). Not reverting to silent block.

## Design

### `src/neutrix/terminal_chat.py`
- `PROMPT_TIMEOUT_S = 300.0` (generous; you have time to answer, a walked-away
  session can't zombie).
- `_ask_user(spec) -> Answer | None`: wrap the per-question await in
  `asyncio.wait_for(self._pending_answer, PROMPT_TIMEOUT_S)`; on `TimeoutError`,
  print a notice and `return None` (the `finally` clears `_pending_question`/
  `_pending_answer`). `CancelledError` (Esc) still propagates (real cancel).
- The pending indicator becomes `▶ answer needed: <header> — type a number, or Esc
  to skip` in `fg:ansiyellow bold reverse`.

## Acceptance

- **Automated**: `_ask_user` with no answer + a short `PROMPT_TIMEOUT_S` returns
  `None` and clears pending state; the existing round-trip (answered) + cancel +
  `_input_loop` routing tests stay green; the Executor `answer is None` →
  block-with-notice / not-available tests stay green (the fallback target). Full
  suite green; ruff clean.
- **Manual**: a `rm -rf build` in `auto` mode shows a prominent `▶ answer needed`
  line; answering `2` allows-always, `3` denies, Esc skips; **walking away no
  longer hangs** — after `PROMPT_TIMEOUT_S` the turn continues with the command
  blocked.
- Ship `v1.5.1`.

## Out of scope

- Auto-retry of dropped connections, session-resume (v1.5.2), single-in-progress
  task enforcement (a separate small follow-up the user flagged).
