# v1.6.1 — `anthropic/*` streaming (Anthropic SSE inbound parse)

## Problem

Switching the active slot to `anthropic/claude-opus-4-7` (IHEP gateway) produces
**no output and appears stuck**. Evidence from the gateway's request log: the
upstream response is Anthropic Messages shape
(`{"id":"msg_…","content":[{"type":"text","text":"…"}]}`), `streaming: true`,
`x-stainless-read-timeout: 300.0`.

**Root cause (confirmed empirically against the live gateway, 2026-05-31):** the
IHEP gateway streams the **Anthropic Messages SSE protocol** for `anthropic/*`
models on the same `/chat/completions` endpoint —
`event: message_start / content_block_start / content_block_delta
{delta:{type:"text_delta",text}} / content_block_stop / message_delta / message_stop`.
neutrix's `llm.py` adapts the **outbound** payload for that gateway
(`_outbound_prompt` hoists `system`, maps `tool`→`user` via
`_uses_anthropic_messages_gateway()`) but its **inbound** `stream_response` loop
only reads `choices[0].delta.content`. The `AsyncOpenAI` SDK parses the Anthropic
SSE into chunks with **empty `choices`** (the event survives in
`chunk.model_extra`), so `_first_choice()` returns `None` every chunk → no text
captured → `content=None`, no streamed `token` events → the no-progress watchdog
can sit until the 300 s `llm_timeout_s` → user-visible "stuck, nothing output."
`deepseek-ai/*`, `zhipu/*`, `openai/*` work because the gateway returns proper
OpenAI chunks for them.

**Bug #2 (linked, surfaced mid-diagnosis, user-confirmed):** the empty
`{"role":"assistant","content":null}` turns Bug #1 leaves in history then **poison
subsequent requests on strict backends** — `openai/gpt-5.5` and the anthropic
gateway return an EMPTY reply (tokens=0, no error) when such a turn (null/empty
content AND **no `tool_calls`**) is anywhere in the conversation; `deepseek-ai/*`
tolerates it. Confirmed empirically (synthetic histories + replay of the user's
real session through `OpenAIChatLLM.stream_response`) and by the user ("now
gpt5.5 is ok … anthropic return empty message and that make things broken"). This
is why `gpt-5.5` "also stopped working": the opus empties polluted the active
session. Bug #2 is independent of #1 (any empty-turn source — e.g. a no-text
cancellation — triggers it) but #1 is its dominant cause here. Note: a
null-content assistant **with** `tool_calls` is the normal text-free-tool-call
shape (~60-73 % of agentic-session assistant turns) and is **not** the bug.

This is a P0 live regression on the user's `strong`-tier reasoning model during
real OMILREC work.

## Goal

1. `anthropic/*` slots stream text and use tools exactly like the
   OpenAI-compatible slots: tokens render live, tool calls dispatch, the turn
   completes normally (Bug #1).
2. An empty assistant turn (null/empty content, no `tool_calls`) — from any
   source — never silently breaks the next request; existing sessions already
   carrying such turns become usable again without starting over (Bug #2).

## Non-goals

- **Token-usage / cost capture** — deferred to v1.7.0 (cost/timing). The events
  carry usage but there is no consumer yet (YAGNI).
- **Changing the outbound payload** (`_outbound_prompt` / `extra_body.system`).
  It works; a bug-fix patch must not perturb it. **Known follow-up:** for the
  anthropic gateway, `_outbound_prompt` drops `tool_calls` and emits `content:""`
  for tool-call turns — opus tolerates this today (verified: a multi-turn
  tool conversation completes correctly), so full OpenAI-`tool_calls` →
  Anthropic `tool_use`/`tool_result`-block translation is **deferred**, not part
  of this patch.
- **Adding the `anthropic` SDK** or a second HTTP transport.
- **Thinking / reasoning-block display** — `claude-opus-4-7` may emit thinking
  blocks; surfacing them is a separate future feature, not this unbreak.

## Claude Code reference

See `docs/splits/v1.6.1-anthropic-stream.html` for the full survey (7 split
points, all agent-decided under the user's "fix it" directive — the diagnosis was
confirmed empirically and the core approach is forced).

CC consumes the Anthropic Messages stream **natively** — `updateUsage()` reads
cumulative usage off `message_start`/`message_delta`
(`services/api/claude.ts:2916-2987`); text arrives via `content_block_delta`.
neutrix mirrors those **semantics** without a native client.

### Split point: where/how to fix the inbound parse
- **CC**: native Anthropic stream consumer.
- **Decision**: Alternative — parse the Anthropic event off
  `chunk.model_extra` inside the existing SDK loop. Empirically the SDK already
  transports the SSE and the full event survives in `model_extra`, so no new
  dependency, no second transport, and `stop()`/watchdog stay unchanged.
  Coaxing OpenAI shape (D) is **ruled out empirically** — the gateway returns
  Anthropic SSE even for a pure-OpenAI request.

### Split point: shape detection — per-chunk vs config-predicate
- **Decision**: per-chunk — use the OpenAI `choices[0].delta` when present, else
  parse `chunk.model_extra["type"]` as an Anthropic event. Robust to provider
  misconfiguration; a pure-OpenAI backend never carries `model_extra["type"]`.

### Split point: tool-use accumulation
- **Decision**: map Anthropic `tool_use` blocks
  (`content_block_start{content_block:{type,id,name}}` +
  `input_json_delta.partial_json`) into the existing `tool_calls_by_index`
  accumulator; reuse `_finalize_tool_calls`. No ContextManager change.

### Split point: text vs tool block disambiguation
- **Decision**: a per-stream `block_kind[index]` map set on `content_block_start`
  routes each `content_block_delta` (`text_delta` → token; `input_json_delta` →
  tool arguments).

### Split point: `finish_reason` mapping
- **Decision**: map `message_delta.delta.stop_reason` → OpenAI `finish_reason`
  (`end_turn`/`stop_sequence`/`max_tokens` → `stop`, `tool_use` → `tool_calls`).

### Split point: token-usage capture
- **Decision**: Alternative — defer to v1.7.0. v1.6.1 is a minimal unbreak.

### Split point: outbound `extra_body.system` hoist
- **Decision**: leave untouched (out of scope for the patch).

### Split point: empty-turn (null-content, no-tool_calls assistant) repair — Bug #2
- **Decision**: outbound repair via a pure `_repair_empty_assistants` transform
  that replaces null/empty content with a `[no response]` placeholder for
  assistant messages with no `tool_calls`. **Placeholder, not drop** — drop
  creates consecutive user turns that strict backends reject; placeholder
  preserves role alternation and is API-valid everywhere. **Outbound, not
  commit-time-only** — it must heal the user's existing poisoned session, not
  just prevent new empties. Sits beside `_ensure_tool_result_pairing` (same
  outbound payload-validity class; accepted `visibility-parity` asymmetry).

## Design

All changes are confined to `src/neutrix/llm.py`.

1. **`stream_response` loop** — for each chunk, first try the OpenAI path
   (`_first_choice` → `delta.content` / `delta.tool_calls`, unchanged). If no
   OpenAI choice is present, read `event = chunk.model_extra` (or the chunk's
   extra attrs) and, when `event.get("type")` is an Anthropic event, route to a
   new pure helper `_apply_anthropic_event(event, state)`.

2. **`_apply_anthropic_event(event, state)`** — a pure folder over per-stream
   state (`content_parts`, `tool_calls_by_index`, `block_kind`, `finish_reason`):
   - `content_block_start` → `block_kind[index] = content_block.type`; if
     `tool_use`, seed `tool_calls_by_index[index] = {"id", "type":"function",
     "function":{"name","arguments":""}}` from the block's `id`/`name`.
   - `content_block_delta` →
     - `text_delta`: append `delta.text`; return it as a token to yield.
     - `input_json_delta`: append `delta.partial_json` to the indexed tool call's
       `function.arguments`.
   - `message_delta` → record mapped `finish_reason` from `delta.stop_reason`.
   - `message_start` / `content_block_stop` / `ping` / `message_stop` → no-op for
     content (usage ignored in v1.6.1).
   The helper returns any text delta so the loop can `yield LLMEvent("token",
   text)` (which bumps `last_progress_at` → no false timeout) — keeping the
   single-yield-point structure of the existing loop.

3. **finish_reason map** — module-level `_ANTHROPIC_STOP_REASON` dict
   (`end_turn`/`stop_sequence` → `stop`, `max_tokens` → `length`, `tool_use` →
   `tool_calls`).

4. **`_ensure_sdk_compliant(messages)` (Bug #2, user-directed)** — a single
   outbound payload-compliance umbrella that `stream_response` routes through
   before `_outbound_prompt`, composing two pure steps: (1)
   `_ensure_tool_result_pairing` (existing) and (2) `_repair_empty_assistants` —
   for each assistant message with **no `tool_calls`** whose content is
   `None`/empty/whitespace, set content to `EMPTY_ASSISTANT_PLACEHOLDER`
   (`"[no response]"`). Messages with `tool_calls` (text-free tool calls) and
   non-empty content are untouched. Pure — does not mutate `messages`. Runs
   **pre-translation** (on the canonical OpenAI-shaped message): empirically the
   poison is `content:null` specifically (gpt-5.5 returns empty; `""` is
   tolerated, and the anthropic `_outbound_prompt` already coerces `null→""`), so
   filtering post-translation would only touch tolerated `""` blocks and would
   mislabel anthropic tool-call turns (tool_calls already dropped) as
   `[no response]`. Pre-translation can still see `tool_calls`, so only genuine
   empties are repaired. The user's directive: "if message sent has empty block,
   filter it, in `ensure_sdk_compliant` — just like tool_use-result pairing."

Cancellation (`stop()` closing `_active_stream`), the timeout watchdog, and the
terminal `LLMEvent("assistant", LLMResponse(...))` assembly are unchanged.

## Acceptance

1. **Unit (recorded events):** feed a recorded Anthropic SSE event sequence
   (text-only, and text+tool_use) through the parser; assert the assembled
   assistant message has the joined text and the correctly-keyed `tool_calls`
   with valid JSON arguments, and that `token` events were yielded for each
   text delta.
2. **Regression:** the existing OpenAI-chunk path (deepseek/glm fixtures) still
   produces identical output — per-chunk detection never misroutes an OpenAI
   chunk.
3. **Empty-turn repair (Bug #2):** `_repair_empty_assistants` replaces
   null/empty/whitespace content with `[no response]` for no-tool_calls
   assistants; leaves null-content-WITH-tool_calls and non-empty content
   untouched; pure (input list unmutated).
4. **Live smoke:** against the IHEP gateway, `anthropic/claude-opus-4-7` returns
   visible streamed text within seconds (no 300 s stall) and a tool-using turn
   dispatches the tool; and a conversation seeded with an empty assistant turn no
   longer returns empty on `openai/gpt-5.5`.
5. `ruff` clean; full `pytest` green.
