# v1.4.8 — AskUserQuestion (interactive)

> **Status: finalized.** v2-band release #6. Decided autonomously under the
> `build-to-v2.0.0` /goal; **user-asked "also add AskUserQuestion tool".**
> Concurrency-bearing splits advisor-gated before push. Splits:
> [docs/splits/v1.4.8-ask-user-question.html](../splits/v1.4.8-ask-user-question.html).

## Problem

A coding agent that "completely replaces Claude Code" must be able to *stop and
ask the human* mid-turn — "which of these approaches?", "this is destructive,
proceed?" — instead of guessing or refusing. neutrix has no channel for the
model to solicit a structured answer from the user during a turn. Relatedly,
v1.4.0's permission `ask` verdict is a dead end: it blocks with a notice because
there was no interactive prompt to route it through.

## Goal

1. An `AskUserQuestion` tool (CC's schema: 1–4 questions, 2–4 options each,
   `multiSelect`, implicit "Other" free-text) that **pauses the turn**, surfaces
   the question(s) to the user, and returns the chosen answer(s) as the tool
   result so the next LLM round sees them.
2. A single async **`ask_user` port**, held by the **ContextManager** (not the
   Executor), used by BOTH the AskUserQuestion tool AND the permission `ask`
   verdict — the latter upgraded from block-with-notice to a real
   yes / always / no prompt. The Executor stays a pure event leaf: it *yields* a
   `needs_user_input` event and receives the answer via the generator's
   `.asend()`, so it never calls the UI (UI→CM→Executor layering intact).
3. The turn pauses and resumes **without a worker thread** (the tool does no
   compute) and **without growing the 4-state machine**.
4. Cancel (Esc) while a question is pending aborts the whole turn, reusing the
   existing cancel path and the existing orphan-tool_call repair.
5. Degrade safely when there is no interactive terminal (tests, piped stdin,
   inside a subagent): fail fast and visibly, never deadlock.

## Non-goals

- Arrow-key / full-screen selector UI (numbered in-flow input instead — split #2).
- The `preview` side-by-side field (a terminal can't render it — split #8).
- A 5th ContextManager state (orthogonal flag — split #4).
- Remote/headless answering of questions over a transport (CC has a remote
  bridge; out of scope until the v2 background/scheduled work).

## Claude Code reference

See [docs/splits/v1.4.8-ask-user-question.html](../splits/v1.4.8-ask-user-question.html)
for the full survey. CC files: `tools/AskUserQuestionTool/AskUserQuestionTool.tsx`
(schema/return), `hooks/toolPermission/handlers/interactiveHandler.ts`
(`canUseTool`, `onAbort`), `FallbackPermissionRequest.tsx` (yes/always/no),
`useCancelRequest.ts` (Esc aborts the turn).

### Split point: channel (where the port lives)
- **CC**: tool execution `await`s an injected `canUseTool` handler in one event loop.
- **Decision**: **Alternative** — a **bidirectional async generator**: `dispatch_all`
  does `answer = yield ToolEvent("needs_user_input", {spec})`; the **CM** owns the
  `ask_user` port and drives it with `gen.asend(await self.ask_user(spec))` (or
  `asend(None)` when non-interactive). CC's port-on-the-runner shape would make the
  Executor call *up* into the UI — a layering leak; this keeps the Executor a pure
  event leaf. Not an `asyncio.Queue` MQ: we want lockstep request-response with
  backpressure, not a decoupled buffer. *(Reworked mid-implementation after the
  user flagged the leak: "did you do full layered separation?" / "async for +
  yield … MQ?")*

### Split point: selector UI
- **CC**: arrow-key `Select`/`SelectMulti` (Ink).
- **Decision**: **Alternative** — numbered options in scrollback + single-line
  answer (number / comma-list / Other text). neutrix is append-only + one-line
  input; a full-screen selector fights the architecture. Semantics preserved.

### Split point: render target
- **CC**: question + answer in the transcript.
- **Decision**: **Follow CC** — durable scrollback (question block on ask; answer
  via the existing `role:tool` render path), not the ephemeral `_above_input`.

### Split point: input routing while pending
- **CC**: no state-enum analog.
- **Decision**: **No CC analog / minimal-state** — `_pending_question` + an
  `asyncio.Future` on `TerminalChat`; `_input_loop` resolves it instead of
  enqueuing. The 4-state machine is unchanged (the turn is still
  `AWAITING_EXECUTOR`). Cross-task on one loop, not cross-thread.

### Split point: cancel while pending
- **CC**: Esc → `onAbort` → cancel tool use + abort turn.
- **Decision**: **Follow CC** — `_drive_task.cancel()` raises `CancelledError`
  at `await ask_user`, unwinds like a cancelled tool; the orphan AskUserQuestion
  tool_call is repaired by `llm.py`'s existing pairing layer. Only addition:
  clear `_pending_question` in `ask_user`'s `finally`.

### Split point: permission `ask` unification
- **CC**: separate components, shared `Select` widget + abort channel.
- **Decision**: **Alternative** — share the whole `ask_user` channel. `ask`
  verdict → yes/always/no prompt; "always" appends an `allow` rule to the
  session policy. Falls back to v1.4.0 block-with-notice when the port is None.

### Split point: headless / subagent fallback
- **CC**: always interactive; subagents exclude interactive prompts.
- **Decision**: **No CC analog** — port None → AskUserQuestion returns a tool
  error, permission `ask` stays block-with-notice; AskUserQuestion excluded from
  `subagent_tool_names()`. Never deadlock a humanless turn.

### Split point: schema + encoding
- **CC**: 1–4 Q, 2–4 opts, header ≤12, multiSelect joined ", ", implicit Other.
- **Decision**: **Follow CC** (drop `preview`). Sequential ask; answers keyed by
  question text.

### Split point: answer parsing (Other detection / empty)
- **Decision**: **No CC analog** — all-in-range comma-ints → option labels;
  else → Other verbatim; empty → re-prompt; Esc → abort.

## Design

### `src/neutrix/tools.py`
- `AskUserQuestion` added to `BUILTIN_TOOLS` with the CC schema (questions[],
  each `{question, header, options[{label,description}], multiSelect}`).
- `_ask_user_question(args, *, executor)` — **validates** the schema (1–4 Q, 2–4
  opts, unique text, header ≤12), then, because the tool needs the async port,
  is **intercepted in `dispatch_all`** rather than executed in-thread (the
  in-thread function only validates + returns a sentinel telling dispatch to
  route; simpler: dispatch_all special-cases the name before `to_thread`).
- `subagent_tool_names()` also subtracts `"AskUserQuestion"`.

### `src/neutrix/permissions.py`
- `format_permission_question(tool_name, args, verdict)` → a `QuestionSpec`
  (yes / always / no) for the `ask` path. `apply_always_rule(policy, tool_name)`
  → a new `PermissionPolicy` with an `allow` rule appended (session-scoped).

### `src/neutrix/executor.py` (pure event leaf — no UI reference)
- `dispatch_all` is a **bidirectional async generator**. (a) `ask` verdict →
  `answer = yield _needs_input(permission_question(name,args,verdict))`; if `answer
  is None` → block-notice (no interactive consumer); else map yes→run /
  always→`self.policy = apply_always_rule(…)` + run / no→`USER_DENIED`. (b)
  `name == "AskUserQuestion"` → validate; `answer = yield _needs_input(spec)`; if
  `answer is None` → not-available; else yield `format_answers_result(answer)`.
  The Executor builds specs and interprets answers (pure functions) but holds no
  port and never touches the UI. No `to_thread` on these paths.
- `AskUserPort = Callable[[QuestionSpec], Awaitable[Answer]]` lives in `prompts.py`
  (TYPE_CHECKING-only) and is held by the **CM**.

### `src/neutrix/context_manager.py` (owns the port, drives the generator)
- `ask_user: AskUserPort | None = None` field, injected by the UI.
- `_dispatch_tools` drives `dispatch_all` with `gen.asend(send)` in a loop: on
  `needs_user_input` → `send = await self.ask_user(spec)` (or `None`); on
  `tool_finished` → append the `role:tool` message (CM stays sole mutator). A
  `finally: await gen.aclose()` unwinds a generator parked at a `needs_user_input`
  yield on cancel.

### `prompts.py` data
- `QuestionSpec` / `Answer` dataclasses (pure) in a new `src/neutrix/prompts.py`
  (SRP — schema + parsing, importable by tools/executor/CM/terminal_chat without
  cycles).

### `src/neutrix/prompts.py` (new)
- `QuestionSpec` (questions list), `Answer` (per-question label(s)/Other text),
  `parse_question_spec(args_json)` (validation → spec or ValueError),
  `parse_answer_line(line, options, multiSelect)` (split #9 rule),
  `render_question(question, idx, total)` (the numbered scrollback block),
  `format_answers_result(spec, answers)` (the tool-result string, CC-keyed).

### `src/neutrix/terminal_chat.py`
- `_pending_question: QuestionSpec | None`, `_pending_answer: asyncio.Future | None`,
  `_pending_options`/index for the in-progress question.
- `async def _ask_user(spec) -> Answer` (wired as `ctx.ask_user`): for each
  question — print the numbered block to scrollback, set `_pending_question` +
  fresh Future, `await` it, parse, collect; `finally` clear `_pending_question`.
- `_input_loop`: if `_pending_answer` is pending, route the line to
  `parse_answer_line` → resolve the Future (or re-prompt on empty); else the
  existing enqueue path.
- `_above_input`: a one-line "answering: <header>" indicator only while pending
  (minimal, not an "ugly hint" — it is state, like `working`).

### `src/neutrix/cli.py`
- Wire `ctx.ask_user = chat._ask_user` after construction (the port is None
  everywhere else — tests, subagents — by default).

## Acceptance

- **Automated**:
  - `prompts.py`: schema validation (bounds, uniqueness, header width);
    `parse_answer_line` (single int, comma multi, Other text, out-of-range→Other,
    empty→sentinel); `format_answers_result` matches CC keying + ", " join.
  - `executor` (bidirectional generator): driving with a responder, AskUserQuestion
    resolves to the formatted answer; `asend(None)` (no consumer) returns the
    not-available error; `ask` verdict + "always" runs the tool AND `executor.policy`
    gained the rule; `asend(None)` → block-notice (v1.4.0 unchanged). Layering
    guard: `Executor` has no `ask_user` attribute.
  - `context_manager`: `_dispatch_tools` with `ctx.ask_user` set relays an
    AskUserQuestion to a tool result; with `ask_user=None` → not-available; an
    "always" permission answer updates `executor.policy`.
  - `terminal_chat`: `_ask_user` round-trips through `_input_loop` (drive a fake
    input); empty line re-prompts; AskUserQuestion **not** in `subagent_tool_names()`.
  - Cancel: a pending question + `ctx.cancel()` aborts the turn, `_pending_question`
    is cleared, and the orphan tool_call is repaired on next send (pairing-layer
    test stays green).
  - Full suite green; ruff clean.
- **Manual**: ask the model a question → it calls AskUserQuestion → numbered
  options print → typing `2` returns "option 2" to the model → it continues;
  typing free text returns Other; Esc mid-question aborts; a destructive Bash in
  `auto` mode now prompts yes/always/no instead of silently blocking.
- Ship `v1.4.8`.

## Out of scope

- Arrow-key selector, `preview` field, remote answering, a 5th state.
