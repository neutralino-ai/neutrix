# v0.9.5 — LLM error surface (stalled / timeout / inline error)

> Two-layer feedback when the LLM API is slow or hung. ~50 s of
> silence in `AWAITING_LLM` flips the heartbeat to a stalled
> palette + `● LLM (stalled)` label (UI only, no abort). At
> ~300 s a background watchdog fires the v0.9.3 cancel
> infrastructure with `reason='timeout'`; the transcript records
> `[LLM timeout after 300s]` and state returns to IDLE. Forward-
> compatible with v0.10.1 streaming (same `last_progress_at`
> field, same watchdog).

## Post-gate revisions (v0.9.5 acceptance, 2026-05-28)

The first acceptance pass rejected the original threshold values as
too aggressive for slow hosted reasoning models (deepseek-v4-pro).
The following deltas were taken as a **code-only tiny tweak** (user
direction; splits #1/#5 Decision fields carry the same note). The
split-point *discussion* below is preserved as the durable design
record; these bullets are the authoritative shipped values:

- **Hard timeout default 60 s → 300 s** (split #5). Matches CC's
  local-model tier; leaves headroom for slow reasoning replies while
  still aborting well inside the SDK's 600 s read-timeout. Per-slot
  `llm_timeout_s` override unchanged.
- **Stall threshold: fixed 5 s → derived `max(10, llm_timeout_s / 6)`**
  (split #1), ≈50 s at the 300 s default. Single knob — raising the
  per-slot timeout pushes the hint out with it, so it no longer
  flickers on healthy-but-slow calls.
- **No auto-retry after a watchdog timeout — confirmed** (extends
  split #10). Transient 429/5xx are still retried by the SDK *inside*
  the envelope; the watchdog cancel surfaces `[LLM timeout after Ns]`
  and the user resends. Matches CC (timeout = surfaced error).
- **Bundled polish: heartbeat refresh 10 Hz → 120 Hz** (480 frames,
  same 4 s period) for a visually smooth fade. Not part of the LLM
  error surface; rides this release per forward-only versioning.

## Problem

After v0.9.4:

- `OpenAIChatLLM.stream_response` at
  `src/neutrix/llm.py:185` wraps
  `self._client.chat.completions.create(...)` in
  `asyncio.ensure_future` and awaits. The `AsyncOpenAI(...)`
  construction at `llm.py:183` passes no `timeout=`, so the
  openai Python SDK's ~600 s default applies.
- `stream=False` (per v0.9.3) means one big await; no per-chunk
  progress signal.
- The v0.9.4 heartbeat correctly shows `● LLM` while
  `ContextManager` is in `AWAITING_LLM`, but the breathing dot
  doesn't distinguish "LLM is computing" from "LLM API silently
  hung." For up to 10 minutes the user sees the same affordance.
- LLM errors *do* surface via the
  `except Exception` catch at
  `src/neutrix/context_manager.py:361` which writes
  `[LLM error: <err>]` — but only after the SDK finally times
  out, which on hosted APIs can be 10 minutes of staring at a
  breathing dot.

User-observed during v0.9.4 dogfooding (2026-05-27):
> *"seems LLM is dead and CM is waiting for it forever. so user
> don't know what happened. and user can only esc to interrupt."*

## Goal

Two layers, matching CC's pattern adapted for our non-streaming
state and our v0.9.3 cancel infrastructure:

1. **UI hint at ~50 s** (derived `max(10, llm_timeout_s/6)`) —
   heartbeat palette swaps to a red gradient and the label becomes
   `● LLM (stalled)`. Does NOT abort; just tells the user
   "something looks slow."
2. **Hard abort at ~300 s** — a background watchdog calls
   `cm.cancel(reason='timeout')`. The cancel broadcast flows
   through v0.9.3's `llm.stop()` machinery; the CancelledError
   catch in `_drive` reads the `reason`, appends a fresh
   assistant message `[LLM timeout after 300s]`, returns CM to
   IDLE, and `logger.error` records the event.

Per-slot tunable via a new `llm_timeout_s` YAML field (default
300.0; see Post-gate revisions for why 300 not 60).

## Non-goals

- **No streaming-idle timeout.** That layer comes back with
  v0.10.1 (streaming-steer), which will bump the same
  `last_progress_at` field on each chunk arrival and let the
  same watchdog handle "no chunk in 90 s." v0.9.5 lays the
  field and the watchdog; v0.10.1 reuses them.
- **No custom retry policy.** The openai SDK's default
  `max_retries=2` for 429 / 5xx / 529 stays on; the retries
  happen *inside* our 300 s envelope (split #10).
- **No new lifecycle event.** The stall is renderer-side math
  on `cm.last_progress_at`; the abort flows through the
  existing v0.9.3 cancel path with a new `reason` flag. No
  reducer entry needed.
- **No env-var configurability.** Per-slot YAML
  (`llm_timeout_s`) is the configuration surface. `env_var` is
  a defensible follow-up if usage shows the per-session knob
  earns its keep.
- **No stall hint or timeout in `AWAITING_EXECUTOR`.** Tool
  latency has its own budget and its own concept of "stuck"; a
  unified affordance would create false positives.
- **No reshape of `OpenAIChatLLM`.** Timeout enforcement lives
  in `ContextManager` (the watchdog task) and uses the existing
  `OpenAIChatLLM.stop()` — the LLM class stays as-is.
- **No legacy `tui.py` work.** Standing non-goal across v0.9.x.

## Claude Code reference

See `docs/splits/v0.9.5-llm-error-surface.html` for the full
12-split-point survey.

### Split point: Stall threshold value (5 s → derived, post-gate)
- **CC**: hardcoded 3 s of no streaming tokens
  (`cc2/src/components/Spinner/useStalledAnimation.ts:42`).
- **Decision**: **Alternative — originally 5 s hardcoded; revised
  at acceptance (2026-05-28) to derived `max(10, llm_timeout_s/6)`
  ≈ 50 s at the 300 s default.** CC's 3 s is calibrated for
  streaming; our non-streaming one-big-await is much slower, and a
  fixed small threshold flickered on healthy slow models. Deriving
  the stall from the hard timeout makes it a single knob. See
  Post-gate revisions.

### Split point: Stall hint visual (both color + label)
- **CC**: color shift only (spinner intensity ramp); verb text
  unchanged (`useStalledAnimation.ts:42-44`).
- **Decision**: **Alternative — both.** Palette swaps amber→red
  AND label `● LLM` becomes `● LLM (stalled)`. CC ships
  color-only because its busy verb text would be noisy to swap;
  our static `LLM` label has no such constraint.

### Split point: Stall hint smoothing (pulse on red palette)
- **CC**: linear ramp over 2 s after the 3 s threshold
  (`useStalledAnimation.ts:42-44`).
- **Decision**: **Alternative — pulse on red palette.** Tick
  lookup switches from `HEARTBEAT_BRIGHTNESS_CYCLE` to a parallel
  `HEARTBEAT_STALLED_CYCLE`. Matches v0.9.4's Steve-Jobs-mode
  decision (the breathing rhythm continues; only the palette
  changes). No fade scaffolding.

### Split point: Stall detection mechanism (renderer-side math on `cm.last_progress_at`)
- **CC**: `Date.now() - lastTokenAt` checked every render on
  a 50 ms animation clock; React effect
  (`useStalledAnimation.ts:34-44`).
- **Decision**: **Alternative — renderer-side math, with a
  forward-looking field name.** Add
  `ContextManager.last_progress_at: float | None`, set to
  `time.monotonic()` on `AWAITING_LLM` entry, cleared on exit.
  `format_heartbeat` reads it and computes
  `monotonic() - last_progress_at > stall_threshold` (the
  threshold is derived from the slot timeout post-gate). The field is named
  `last_progress_at` (not `awaiting_llm_since`) because v0.10.1
  streaming will bump it on each chunk arrival; same field, same
  math, semantics evolve. User instruction: *"better align with
  future stream design \[…\] think like an Architect"*
  (2026-05-27).

### Split point: Hard timeout duration (60 s → 300 s default, post-gate)
- **CC**: layered 90 s streaming-idle
  (`cc2/src/services/api/claude.ts:1877`) / 120 s remote
  non-streaming (`claude.ts:807-810`) / 300 s local / 600 s
  HTTP (`cc2/src/services/api/client.ts:144`).
- **Decision**: **Alternative — originally 60 s; revised at
  acceptance (2026-05-28) to 300 s default, per-slot override.**
  60 s was too aggressive for slow hosted reasoning models
  (deepseek-v4-pro); 300 s matches CC's local-model tier and still
  aborts inside the SDK's 600 s read-timeout. Slow models tune via
  the per-slot YAML field (split #6). See Post-gate revisions.

### Split point: Configurability surface (YAML per-slot)
- **CC**: env vars only — `API_TIMEOUT_MS`,
  `CLAUDE_STREAM_IDLE_TIMEOUT_MS`,
  `CLAUDE_ENABLE_STREAM_WATCHDOG` (`client.ts:144`,
  `claude.ts:1877`). No settings.json keys for timeouts.
- **Decision**: **Alternative — YAML per-slot, no env var.**
  Add `llm_timeout_s: float | None` to the slot YAML schema;
  default 300.0 when absent. Per-slot is the natural home — a
  slow local model gets a different timeout than a hosted-API
  slot. Slot-schema migration is additive (existing configs
  fall back to default). Diverges from CC.

### Split point: Hard timeout enforcement path (background watchdog → cancel)
- **CC**: SDK client constructed with `timeout: 600_000`
  (`client.ts:144`) plus an explicit `setTimeout` watchdog on
  the stream (`claude.ts:1877`). Both bubble up to the same
  error renderer.
- **Decision**: **Alternative — background watchdog calling
  `cm.cancel(reason='timeout')`.** On `AWAITING_LLM` entry,
  spawn `asyncio.create_task(self._llm_timeout_watchdog())`
  that sleeps `slot.llm_timeout_s` (default 300.0) and then
  invokes the same cancel infrastructure as user-Esc, carrying
  `reason='timeout'`. Watchdog is cancelled on `AWAITING_LLM`
  exit (success, error, or user-cancel). Forward-compatible
  with v0.10.1: streaming will bump `last_progress_at` on each
  chunk, and the watchdog will read "no progress for 60 s"
  rather than "60 s since entry" — same machinery, different
  bump policy. User instruction: align with future stream
  design (2026-05-27).

### Split point: Error message wording (`[LLM timeout after Ns]`)
- **CC**: literal `"Request timed out"` as the synthetic
  `AssistantMessage` content
  (`cc2/src/services/api/errors.ts:169`).
- **Decision**: **Alternative —
  `[LLM timeout after 300s]`.** Bracketed-system-message shape
  preserves the visual category established by
  `LLM_ERROR_PREFIX = "[LLM error: "`
  (`context_manager.py:87`), but distinguishes "timed out" from
  "errored" and states the duration. No recovery tutorial in the
  body — users find the YAML knob via `--help` or the slot
  schema docs; the message is glanceable.

### Split point: Cancel vs timeout transcript distinction (`reason` flag)
- **CC**: same render path (synthetic `AssistantMessage` with
  `isApiErrorMessage: true`); different content text;
  different error classification for analytics
  (`errors.ts:967-977`).
- **Decision**: **Alternative — `reason` flag on
  `CancelEvent`/`cm.cancel()`.** Both user-Esc and the
  timeout watchdog flow through `cm.cancel()` and
  `llm.stop()`, but carry
  `reason: Literal['user', 'timeout']`. CM stores
  `self.cancel_reason`; the CancelledError catch in `_drive`
  reads it: `user` → v0.9.3's `[interrupted by user]` marker
  attached to the prior in-flight message; `timeout` → fresh
  assistant message `[LLM timeout after 300s]` (the LLM never
  spoke, so there's nothing to attach the marker to).
  Single broadcast path; receiving-side distinction.
  Reshape consequence of the split #7 decision.

### Split point: Auto-retry (SDK default — 2 retries)
- **CC**: 429 / 5xx / 529 with exponential backoff + jitter +
  `Retry-After`, up to `maxRetries`
  (`cc2/src/services/api/withRetry.ts:55,543-547,696`).
- **Decision**: **Follow CC, via SDK default.** Leave
  `AsyncOpenAI.max_retries=2` (the SDK's own default). Because
  timeout enforcement is our external watchdog (not the SDK's
  `timeout=`), SDK retries happen *inside* our 300 s envelope: a
  fast 5xx-retry cycle finishes in <15 s. Only true silent
  hangs consume the full envelope. Clean composition: SDK
  handles transient HTTP failures; our watchdog handles silent
  hangs.

### Split point: Stall hint during `AWAITING_EXECUTOR` (suppress)
- **CC**:
  `isStalled = timeSinceLastToken > 3000 && !hasActiveTools` —
  suppressed during tool execution
  (`useStalledAnimation.ts:42`).
- **Decision**: **Follow CC — suppress.** `last_progress_at`
  is set only on `AWAITING_LLM` entry; `AWAITING_EXECUTOR` and
  `CANCELLING` use the existing palettes and labels. Tool
  stall is a future problem with its own design.

### Split point: Logging policy (silent stall, ERROR timeout)
- **CC**: stalls log nothing; timeouts go through `logError`
  at error level (`claude.ts:2334`, `errors.ts:169`).
- **Decision**: **Follow CC.** No log line on stall hint
  entry/exit (UI noise). On watchdog fire:
  `logger.error("LLM call timed out after {}s", elapsed)` per
  the project's loguru convention.

### Flow — AWAITING_LLM entry / exit

In `ContextManager._drive` at `context_manager.py:352`, on
state transition to `AWAITING_LLM`:

```python
self.state = State.AWAITING_LLM
self.last_progress_at = time.monotonic()
self._llm_timeout_task = asyncio.create_task(
    self._llm_timeout_watchdog()
)
self._maybe_inject_system_reminder()
try:
    assistant_msg = await self._call_llm()
except asyncio.CancelledError:
    self._consume_cancel()
    self._finalize_cancel()
    return
except Exception as exc:
    # existing LLM_ERROR_PREFIX path
    ...
finally:
    self._cancel_watchdog()
    self.last_progress_at = None
```

```python
async def _llm_timeout_watchdog(self) -> None:
    try:
        await asyncio.sleep(self.slot.llm_timeout_s)
    except asyncio.CancelledError:
        return
    elapsed = self.slot.llm_timeout_s
    logger.error("LLM call timed out after {}s", elapsed)
    self.cancel(reason="timeout")

def _cancel_watchdog(self) -> None:
    task = self._llm_timeout_task
    if task is not None and not task.done():
        task.cancel()
    self._llm_timeout_task = None
```

### Flow — cancel with reason

`cm.cancel()` at `context_manager.py:249` gains a `reason`
parameter; `_do_cancel` stores it on the CM:

```python
def cancel(self, *, reason: Literal["user", "timeout"] = "user") -> bool:
    if self.state in (State.IDLE, State.CANCELLING):
        return False
    self.cancel_reason = reason
    self._do_cancel()
    return True
```

`CancelEvent` gets an optional `reason: str = "user"` field; the
watchdog emits `CancelEvent(reason="timeout")` or calls
`cm.cancel(reason="timeout")` directly (the latter is cleaner — no
event-loop round-trip).

### Flow — finalize cancel by reason

After the cancel propagates and `_drive`'s
`except asyncio.CancelledError` catch runs:

```python
def _finalize_cancel(self) -> None:
    if self.cancel_reason == "timeout":
        elapsed = int(self.slot.llm_timeout_s)
        self._append_assistant_message({
            "role": "assistant",
            "content": f"[LLM timeout after {elapsed}s]",
        })
    # 'user' reason: v0.9.3 [interrupted by user] marker is
    # already attached to the prior in-flight message by the
    # existing cancel-as-steer path; no extra append.
    self.cancel_reason = "user"  # reset for next turn
```

### Renderer — heartbeat reads `last_progress_at`

`format_heartbeat` at `src/neutrix/terminal_chat.py:220` signature
extends to take CM directly (or a small struct). Simpler form:

```python
def format_heartbeat(
    state: State,
    store: ChatStore,
    tick: int,
    *,
    last_progress_at: float | None = None,
    stall_threshold_s: float = HEARTBEAT_STALL_FLOOR_S,  # caller passes
    # stall_threshold_for(slot.llm_timeout_s) = max(10, timeout/6)
) -> list[tuple[str, str]]:
    if state == State.IDLE:
        return []
    is_stalled = (
        state == State.AWAITING_LLM
        and last_progress_at is not None
        and (time.monotonic() - last_progress_at) > stall_threshold_s
    )
    if state == State.AWAITING_LLM:
        label = "LLM (stalled)" if is_stalled else "LLM"
    elif state == State.AWAITING_EXECUTOR:
        pending = store.pending_tool_calls
        label = f"tool: {pending[0].name}" if pending else "tool"
    elif state == State.CANCELLING:
        label = "cancelling…"
    else:
        label = state.value
    cycle = (
        HEARTBEAT_STALLED_CYCLE
        if is_stalled
        else HEARTBEAT_BRIGHTNESS_CYCLE
    )
    glyph_style = cycle[tick % HEARTBEAT_CYCLE_FRAMES]
    return [
        (glyph_style, f"{HEARTBEAT_GLYPH} "),
        (HEARTBEAT_LABEL_STYLE, f"{label}\n"),
    ]
```

The caller at `terminal_chat.py:747` passes
`last_progress_at=cm.last_progress_at`. The default keyword arg
keeps existing tests at `tests/test_heartbeat_format.py`
working (they pass no `last_progress_at`, the heartbeat behaves
exactly as v0.9.4).

`HEARTBEAT_STALLED_CYCLE` is a sibling table of
`HEARTBEAT_BRIGHTNESS_CYCLE` with the red gradient. Same length,
same tick semantics — only the truecolor anchors change.

## Acceptance

### Manual (user-runnable)

1. **Stall hint fires.** Set `slot.llm_timeout_s: 60.0` in the
   active slot — the derived stall threshold is then
   `max(10, 60/6) = 10 s`, and the hard timeout sits 50 s further
   out. Send a prompt that's known-slow (or pause the server).
   Within ~10 s the heartbeat glyph palette shifts to red and the
   label reads `● LLM (stalled)`. When the response arrives (before
   the 60 s timeout), both revert to the normal palette + `● LLM`.
   (At the 300 s default the stall would not show until ~50 s — too
   long to eyeball, hence the 60 s slot for this check.)
2. **Hard timeout fires.** Set `slot.llm_timeout_s: 5.0`.
   Disconnect network (or `iptables` block the API host). Send
   a prompt. Within ~5 s of the configured timeout fires:
   - transcript appends `[LLM timeout after 5s]` as an
     assistant message;
   - CM returns to IDLE (input area becomes editable);
   - log contains `LLM call timed out after 5s` at ERROR.
3. **Cancel still distinct.** With normal `llm_timeout_s: 300.0`
   and a healthy network, send a long prompt, press Esc within
   the first second. The v0.9.3 `[interrupted by user]` marker
   appears (NOT the timeout marker); state returns to IDLE.
4. **Tools unaffected.** Run a long tool call (e.g.
   `run_shell` of a 30 s `sleep`). The heartbeat shows
   `● tool: run_shell` for the full duration; no stalled
   palette, no abort.

### Unit (added under `tests/`)

- `tests/test_heartbeat_format.py` — extend to assert that
  `format_heartbeat(AWAITING_LLM, store, tick,
  last_progress_at=t0, stall_threshold_s=…)` returns
  stalled-palette fragments when `monotonic() - t0 >
  stall_threshold_s`, and normal-palette fragments otherwise; plus
  `stall_threshold_for(timeout)` returns `max(10, timeout/6)`.
  Pass `last_progress_at=None` to verify v0.9.4 behavior is
  unchanged.
- `tests/test_context_manager_timeout.py` (new) — drive a CM
  with a fake LLM whose `stream_response` blocks indefinitely.
  Set `slot.llm_timeout_s = 0.1`. After
  `await asyncio.sleep(0.2)`, assert: CM state == IDLE; last
  appended message has `content == "[LLM timeout after 0s]"`;
  `cancel_reason` reset to `"user"`.
- `tests/test_context_manager_timeout.py` — separately assert
  that user-Esc during AWAITING_LLM (via
  `cm.cancel(reason="user")`) does NOT append a timeout
  marker, and the watchdog is properly cancelled (no late-fire
  marker after the cancel).
- `tests/test_slot_config.py` (extend) — assert
  `llm_timeout_s` defaults to 300.0 when absent and reads
  per-slot value when present.

### Out of scope for v0.9.5 acceptance

- Streaming-idle timeout behavior (v0.10.1).
- Env-var configurability (not part of this PRD).
- Stall hint in `AWAITING_EXECUTOR` (not part of this PRD).
