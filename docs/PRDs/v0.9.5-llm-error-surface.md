# v0.9.5 — LLM error surface (stalled / timeout / error)

> **Status: skeleton.** Inserted ahead of the previously-planned
> v0.9.5 (emergency-compact, now v0.9.6) after the v0.9.4 user
> gate surfaced a daily-blocking bug: when the LLM API hangs
> silently, the heartbeat keeps breathing but neither neutrix
> nor the LLM is making progress, and the user has no signal
> until the SDK's ~10-minute default timeout fires. Direction
> approved by the user via AskUserQuestion: match CC's two-layer
> pattern — **5 s of LLM-call silence → heartbeat color shift +
> `● LLM (stalled)` label (UI hint, no abort); 60 s → hard abort
> with inline assistant error message**. Detail solidifies once
> the splits at `docs/splits/v0.9.5-llm-error-surface.html` are
> drafted and gated.

## Problem

After v0.9.4:

- `OpenAIChatLLM.stream_response` calls `AsyncOpenAI(...)` with
  no explicit timeout — falls back to the SDK default of ~10
  minutes.
- `stream=False` (per v0.9.3) means one big await; no
  per-chunk progress signal.
- The heartbeat correctly shows `● LLM` while the
  ``ContextManager`` is in ``AWAITING_LLM``, but the breathing
  dot doesn't distinguish "LLM is computing" from "LLM API
  silently hung" — for up to 10 minutes the user sees the same
  affordance regardless.
- Errors *do* surface via the existing `LLM_ERROR_PREFIX` catch
  in `_drive` (`context_manager.py:362`), but only after the SDK
  finally times out.

User-observed during v0.9.4 dogfooding (2026-05-27):
> *"seems LLM is dead and CM is waiting for it forever. so user
> don't know what happened. and user can only esc to interrupt."*

## Goal

Two-layer feedback, matching CC's pattern but adapted for our
non-streaming v0.9.x state:

1. **UI hint** at ~5 s of LLM-call silence — heartbeat glyph
   color shifts (red tint or similar) and the label changes to
   `● LLM (stalled)`. Does NOT abort the request; just tells the
   user "something looks slow."
2. **Hard timeout** at ~60 s (configurable) — abort the LLM
   call, append an inline `assistant` error message in the
   transcript, return CM state to IDLE.

## Non-goals

- **No streaming-idle timeout.** That layer needs the streaming
  path back; covered by v0.10.1 (streaming-steer).
- **No SDK-level auto-retry.** Keep the abort path minimal for
  v0.9.5; auto-retry can be a small follow-up if the error rate
  warrants it.
- **No new event kinds.** The CM state machine is enough.
- **No reconfiguration of `OpenAIChatLLM`.** Timeout is set on
  the SDK client at construction.

## Claude Code reference

See `docs/splits/v0.9.5-llm-error-surface.html` (to be written).

Background from the v0.9.4 reject investigation:

- CC's HTTP request timeout: 600 s, configurable via
  `API_TIMEOUT_MS` (`cc2/src/services/api/client.ts:144`).
- CC's streaming-idle timeout: 90 s of no SSE bytes →
  `APIConnectionTimeoutError` (`cc2/src/services/api/claude.ts:1877`).
- CC's stalled UI: 3 s of no tokens → spinner turns red, fading
  intensity over 2 s; UI-only, doesn't abort
  (`cc2/src/components/Spinner/useStalledAnimation.ts:42`).
- CC's auto-retry: SDK `maxRetries` for 429 / 529 / 5xx / auth.
- CC's error surface: inline assistant API error message
  (`cc2/src/utils/messages.ts:435`).
