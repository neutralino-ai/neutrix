# v0.9.0 — lifecycle events + store reducer

## Problem

`Agent.stream_reply` (`src/neutrix/agent_loop.py`) currently emits six
`AgentEvent` kinds: `token`, `assistant`, `tool_call`, `tool_result`,
`done`, `error`. These tell a renderer **what content arrived**; they do
not tell observers **when an LLM round started or ended**, and the only
side effect on `ChatStore` happens via direct calls from the view —
`terminal_chat._handle_event` calls `store.add_pending_tool_call` /
`store.remove_pending_tool_call` itself.

Two near-term features both need a structured "what is currently in
flight" signal that lives on `ChatStore`:

- **v0.9.1 heartbeat** wants to pulse next to `"LLM …"` while a
  request is streaming and next to `"tool: {name} …"` while a tool
  runs. The store only knows about `pending_tool_calls`; it has no
  notion of "an LLM request is in progress."
- **v0.10.0 FakeView swap-test** wants the renderer to be a *reader* of
  the store, not a writer. Today `terminal_chat._handle_event` writes
  to the store mid-event; a `FakeView` that only reads can't reproduce
  that behavior. Moving the mutations off the view path is what makes
  the swap test possible.

v0.9.0 is the smallest change that unlocks both: add two LLM lifecycle
events, move pending-tool-call mutations off the view, and put both on
the store via a single reducer entry point.

## Goal

1. Extend the `AgentEvent` union with two new kinds:
   `llm_request_start` and `llm_request_end`. Existing kinds
   (`token`, `assistant`, `tool_call`, `tool_result`, `done`, `error`)
   stay byte-compatible.
2. `Agent.stream_reply` emits exactly one `llm_request_start` before
   each `llm.stream_response(...)` call and exactly one
   `llm_request_end` after that call returns (or raises). One pair
   per LLM round — multi-round tool-loop turns emit multiple pairs.
3. Add one new field on `ChatStore`: `llm_active: bool` (read-only
   public property, default `False`). Mutated only by the reducer.
4. Add a single reducer entry point `ChatStore.apply(event) -> None`
   that maps:
   - `llm_request_start` → set `llm_active = True`, notify.
   - `llm_request_end` → set `llm_active = False`, notify.
   - `tool_call` → `add_pending_tool_call(name, arguments)`.
   - `tool_result` → `remove_pending_tool_call(name)`.
   - `token`, `assistant`, `done`, `error` → no store change.
5. Update `TerminalChat._send_message` / `_handle_event` so every
   event from `agent.stream_reply` first goes through
   `self.store.apply(event)`, then through the existing rendering
   switch. Direct calls to `add_pending_tool_call` /
   `remove_pending_tool_call` in the view are deleted.
6. Public API name and class stays: `Agent`, `agent_loop.py`,
   `stream_reply`. No rename in this release — defer until a real
   second consumer exists.

## Non-goals

- **No UI change.** Heartbeat ships in v0.9.1; this release must be
  invisible to a user staring at the terminal chat.
- **No rename.** `agent_loop.py` keeps its name. `Agent` keeps its
  name. `stream_reply` keeps its signature. Earlier roadmap notes that
  referenced a "controller" rename are deferred.
- **No new public surface on `Agent`.** Same constructor, same
  `stream_reply(user_text) -> AsyncIterator[AgentEvent]`.
- **No cancellation.** The new lifecycle events make cancellation easier
  to wire later; this release does not implement Ctrl-C cancel, tool
  abort, or timeout escalation.
- **No parallel tool execution.** Today tool calls within one assistant
  turn run sequentially; the reducer maps one `tool_call` to one
  pending entry and one `tool_result` to one removal. Parallelism is
  out of scope.
- **No second event-payload schema.** `tool_call` / `tool_result`
  payloads stay exactly as today (`{"name": str, "arguments": str}` and
  `{"name": str, "result": str}`).
- **No `/save` / `/load` format change.** `llm_active` is in-memory
  only — it is *always* `False` at the moment the chatstore is written,
  so the on-disk format does not need a new field.

## Design

### `AgentEvent` additions

In `src/neutrix/agent_loop.py`, the existing dataclass:

```python
@dataclass(frozen=True)
class AgentEvent:
    kind: str  # "token" | "tool_call" | "tool_result" | "done" | "error"
    data: Any = None
```

becomes (docstring updated, no schema change):

```python
@dataclass(frozen=True)
class AgentEvent:
    kind: str
    # Kinds:
    #   "llm_request_start" — data=None
    #   "token"             — data=str (partial assistant text)
    #   "assistant"         — data=str (full assistant text, when not streamed)
    #   "tool_call"         — data={"name": str, "arguments": str}
    #   "tool_result"       — data={"name": str, "result": str}
    #   "llm_request_end"   — data=None
    #   "done"              — data=None
    #   "error"             — data=str
    data: Any = None
```

The new kinds carry `data=None`. No payload — the heartbeat will read
`store.llm_active` and `store.pending_tool_calls` (which carries the
tool name) when it needs labels.

### Emission in `Agent.stream_reply`

Current loop (`agent_loop.py` lines 84–142, simplified) is:

```
while True:
    try:
        async for event in llm.stream_response(...): forward to caller
        finalize assistant_msg, append to messages
        yield each tool_call → dispatch → yield each tool_result
        if no tool_calls: yield done; return
    except Exception:
        yield error; return
```

New loop keeps the **outer** `try / except` (so a tool-dispatch failure
or a malformed assistant message still surfaces as an `error` event
and terminates the generator) and adds an **inner** `try / except`
around the LLM stream that emits `llm_request_end` on both the normal
exit and the exception-reraise path. Crucially, **no `yield` inside a
`finally`** — yielding during `.aclose()` would raise
`RuntimeError("async generator ignored GeneratorExit")` and break the
existing `TerminalChat.run_async` worker-cancel path on Ctrl-C
(PEP 525). `GeneratorExit` is not a subclass of `Exception`, so the
explicit-except pattern below skips the `end` yield during
cancellation and lets the generator unwind cleanly:

```python
while True:
    try:
        yield AgentEvent("llm_request_start")
        try:
            async for event in self.llm.stream_response(...):
                ...  # token / assistant forwarding unchanged
            # finalize assistant_msg
        except Exception:
            yield AgentEvent("llm_request_end")
            raise
        yield AgentEvent("llm_request_end")
        # tool_call / dispatch / tool_result loop unchanged
        ...
    except Exception as exc:
        error = compact_error(exc)
        logger.warning("agent loop failed: {}", error)
        yield AgentEvent("error", error)
        return
```

The `end` event fires before the existing `error` event so an observer
can synchronously clear `llm_active` before reacting to the error —
and tool-dispatch exceptions still go through the same outer `except`.
Cancellation (`GeneratorExit` from `.aclose()`) is **not** caught by
either `except Exception`, so the generator exits cleanly without
emitting `end` during the cancellation window. The reducer treats a
missing `end` after cancellation as observers' problem — in practice
the `TerminalChat` shuts down its `_invalidation_watcher` at the same
time the worker is cancelled, so a stale `llm_active = True` is
unobservable.

### `ChatStore` additions

In `src/neutrix/store.py`:

```python
class ChatStore:
    def __init__(self) -> None:
        ...
        self._llm_active: bool = False

    @property
    def llm_active(self) -> bool:
        return self._llm_active

    def apply(self, event: Any) -> None:
        """Mutate the store from an :class:`AgentEvent`.

        The single entry point used by renderers that consume an
        ``Agent.stream_reply`` event stream. Only the kinds listed
        below have an effect; other kinds are no-ops.
        """
        kind = getattr(event, "kind", None)
        data = getattr(event, "data", None)
        if kind == "llm_request_start":
            self._llm_active = True
            self._notify()
        elif kind == "llm_request_end":
            self._llm_active = False
            self._notify()
        elif kind == "tool_call":
            name = str((data or {}).get("name") or "")
            arguments = str((data or {}).get("arguments") or "")
            self.add_pending_tool_call(name, arguments)
        elif kind == "tool_result":
            name = str((data or {}).get("name") or "")
            self.remove_pending_tool_call(name)
        # token / assistant / done / error: no store change.
```

`apply` accepts `event: Any` to avoid a circular import (`store` would
have to import `AgentEvent` from `agent_loop`, which already imports
`store`). It reads `event.kind` and `event.data` reflectively. The
intent is documented in the docstring; the cost of a structural type is
acceptable because there is exactly one producer (`Agent.stream_reply`).

`reset()` also clears `self._llm_active = False`.

### Renderer change

In `src/neutrix/terminal_chat.py`:

`_send_message` becomes:

```python
async for event in self.agent.stream_reply(text):
    self.store.apply(event)
    assistant_started = await self._handle_event(event, assistant_started)
```

`_handle_event` no longer calls `self.store.add_pending_tool_call` or
`self.store.remove_pending_tool_call`. It only renders.

`_pop_tool_arguments` is **removed**. Because `store.apply(event)`
runs first and removes the pending entry for `tool_result`, the
renderer can no longer look up the original `arguments` string from
`pending_tool_calls` after the reducer fires. The renderer owns a
short-lived per-`_send_message` dict, `tool_arg_cache: dict[str,
list[str]]`, populated on `tool_call` (append arguments under the tool
name) and popped FIFO on `tool_result` (pop the head, fall back to
`"{}"` on miss). The dict is discarded at the end of `_send_message`.

This keeps the store API stable (`remove_pending_tool_call` is not
renamed; nothing else changes shape), keeps the reducer-then-renderer
ordering uniform across all event kinds, and isolates the renderer's
short-lived state from the canonical store.

### Save / load

`neutrix.transcript` already serializes only the message list and the
task list. `llm_active` is transient runtime state; it is `False` at
serialize time (we only `/save` between turns) and at load time. No
schema change. Existing `tests/test_transcript*.py` pass unchanged.

### Why `Agent` is not renamed

The original v0.9.0 skeleton called for `agent_loop.py → controller.py`
and `Agent → Controller`. Two reasons we defer:

1. The release-workflow rule wants one user-readable CHANGELOG line per
   tag. A rename is a mass diff with no observable behavior change.
2. There is still only one consumer of `Agent` (`TerminalChat`). A
   "controller" name earns its keep when a second consumer (FakeView,
   smart Advisor) materializes. v0.10 / v0.11 reference the
   "controller" only as a label for the role `Agent` plays — the rename
   can ride along with the FakeView swap-test in v0.10.0 if it still
   looks useful then.

## Acceptance

### Automated

New tests added to existing files (avoids file-explosion; keeps the
PRD's gate names tied to the actual file layout):

- `tests/test_agent_loop.py` gains lifecycle-event cases:
  - With a mock LLM that emits two tokens and no tool calls,
    `stream_reply` yields the sequence (in order):
    `llm_request_start`, `token`, `token`, `llm_request_end`, `done`.
    The `assistant` event is asserted iff `not rendered_tokens` — i.e.
    only when the LLM client returns a final-message-only response
    with no streamed tokens (matches `agent_loop.py` line 109).
  - With a mock LLM that emits one tool call followed by a tokens-only
    second round, `stream_reply` yields **two** `llm_request_start` /
    `llm_request_end` pairs, with the `tool_call` and `tool_result`
    between them.
  - When the mock LLM raises mid-stream, exactly one
    `llm_request_end` is emitted (via the inner `finally`) before the
    `error` event.
  - When `dispatch(...)` raises (synthesized via a tool whose `func`
    raises), the outer `try / except` still catches it and the
    generator emits `error` then terminates — regression gate that
    proves the outer try/except is intact (not collapsed into the
    inner LLM-stream guard).
  - **Cancellation gate.** A mock LLM that suspends mid-stream
    (`await asyncio.Event().wait()`) followed by an explicit
    `.aclose()` on the `stream_reply` iterator must exit cleanly
    with no `RuntimeError` and no `llm_request_end` emitted during
    `.aclose()`. Covers `TerminalChat.run_async`'s worker-cancel
    path on Ctrl-C.
- `tests/test_store.py` gains a reducer section:
  - `apply(AgentEvent("llm_request_start"))` flips `llm_active` to
    `True`; `apply(AgentEvent("llm_request_end"))` flips it back; a
    subscriber on `changes()` is notified for each.
  - `apply(AgentEvent("tool_call", {"name": "x", "arguments": "{}"}))`
    appends one pending call; `apply(AgentEvent("tool_result",
    {"name": "x", "result": "ok"}))` removes it. Final
    `pending_tool_calls` is empty.
  - `apply` on `token`, `assistant`, `done`, `error` produces no
    `_notify` and no field change.
  - `reset()` clears `llm_active`.
- `tests/test_terminal_chat.py`, `tests/test_transcript.py`,
  `tests/test_tools.py`, `tests/test_tui.py`, `tests/test_smoke.py`,
  `tests/test_onboard.py`, `tests/test_llm.py`: all pass without
  modification — the renderer's user-visible output is unchanged.
- `ruff check` clean.

## Out of scope (revisit later)

- Cancellation (Ctrl-C, tool timeout escalation). Earliest v0.10.x.
- Parallel tool execution.
- Renaming `agent_loop.py` / `Agent`. Can ride v0.10.0 (purify view) if
  still warranted.
- A separate event-bus library. Async generators remain the wire.
- Surfacing per-event timing (latency, token rate) on the store.
- Per-tool "executor active" semantics distinct from the
  `pending_tool_calls` list (e.g. for parallel tool execution).
