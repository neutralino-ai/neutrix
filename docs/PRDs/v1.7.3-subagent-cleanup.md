# v1.7.3 — sub-agent HTTP-client cleanup (no more "Event loop is closed")

## Problem

Every time the **`Agent` tool** runs a sub-agent, an unhandled error is printed
after it finishes:

```
Unhandled exception in event loop:
  ... httpx/_client.py aclose → http_proxy.py aclose → http11.py aclose
  ... anyio _asyncio.py: self._transport.close()
  ... asyncio/base_events.py: raise RuntimeError('Event loop is closed')
Exception Event loop is closed
```

**Confirmed root cause.** The sub-agent is dispatched on its own **new event
loop** via `asyncio.run(run_subagent(...))` on a worker thread
(`tools.py`, ~`:647`), with its **own `AsyncOpenAI` client** built at
`tools.py:641` (`llm = OpenAIChatLLM(slot)`). neutrix **never closes that
client** — there is no `aclose()` on the client anywhere. So when the sub-agent
finishes and `asyncio.run()` returns, that loop closes, but httpx's connection
pool still holds **proxied keep-alive connections** (the `http_proxy.py` frames =
the user's local proxy). Those connections tear down later and schedule their
close (`loop.call_soon`) on the now-dead loop → `RuntimeError: Event loop is
closed`, surfaced by asyncio's default handler. It is the **only** place in
neutrix that closes a loop mid-session, so a mid-run "Event loop is closed" is
unambiguously a sub-agent loop tearing down.

It's **cosmetic** (fires *after* the sub-agent returned its result; the session
continues) but noisy, recurs once per sub-agent, and is worse on a flaky proxy.

## Goal

Eliminate the error by **matching the httpx connection pool's lifetime to the
loop it's bound to** — close the sub-agent's client inside its own loop before
`asyncio.run()` returns.

## Non-goals

- **Not re-architecting sub-agents onto the main loop** (CC's model). neutrix
  deliberately isolates sub-agents on their own loop/thread for clean
  cancellation (`subagent.py`); that stays.
- **Not disabling keep-alive.** Keep-alive is a latency win and CC embraces it;
  the `aclose` fix removes the need to disable it.
- **Not changing sub-agent semantics / cancellation** — teardown hygiene only.

## Claude Code reference

See `docs/splits/v1.7.3-subagent-cleanup.html` (3 split points, agent-decided
under "decide yourself", CC-surveyed + advisor-reviewed).

### Split point: fix shape
- **CC**: one process-wide event loop; sub-agents (`utils/forkedAgent.ts`) are
  async tasks on it, cancelled via `AbortController` — no per-sub-agent loop or
  client, so CC is structurally immune.
- **Decision**: Alternative (match semantics, not runtime shape) — **`aclose()`
  the sub-agent client inside its own loop** (`run_subagent` `finally`), not a
  single-loop re-architecture (which would undo neutrix's deliberate cancellation
  isolation for a cosmetic error).

### Split point: keep-alive
- **CC**: embraces keep-alive — a global shared pool, pre-warmed
  (`utils/apiPreconnect.ts`); skips preconnect only under a proxy/mTLS.
- **Decision**: Follow CC — **keep keep-alive**; the `aclose` fix makes lingering
  connections moot, so there's no need to trade the latency win for
  `max_keepalive_connections=0`.

### Split point: main-session client teardown
- **Decision**: No-CC-analog (CC's pool is process-lifetime) — also `aclose` the
  main client on shutdown, since the session loop closes at exit and would emit
  the same error once.

## Design

1. **`OpenAIChatLLM.aclose()`** (`llm.py`): `async def aclose(self) -> None:` that
   `await self._client.close()` (closes the underlying httpx pool), best-effort
   (swallow/log exceptions so teardown never raises). Idempotent-safe.
2. **`run_subagent`** (`subagent.py`): call `await llm.aclose()` in a `finally`
   so it runs inside the sub-agent's `asyncio.run()` loop on every exit path
   (success, error, cancel) — before the loop closes.
3. **Main client** (`terminal_chat.run_async`): `await self.ctx.llm.aclose()` in
   the shutdown `finally` (after the worker/renderer tasks are cancelled), so the
   session's own client closes before the main loop ends.
4. **Keep-alive unchanged** — the `AsyncOpenAI` client keeps its default pool.

## Acceptance

1. `OpenAIChatLLM.aclose()` exists and closes the underlying client (a spy/fake
   client records `close()` was awaited); it never raises even if the client's
   close fails (best-effort).
2. `run_subagent` calls `llm.aclose()` in its `finally` on normal completion and
   when the LLM call errors (the CM turns the error into an `[LLM error]` turn and
   returns, so the finally still runs) — unit with a spy LLM. **Note:** the CM
   *consumes* cancellation internally (v0.9.2 — `_drive` catches `CancelledError`
   and `_consume_cancel()`s it), so a raw `CancelledError` does **not** reach
   `run_subagent`'s finally; the finally therefore runs on a normal path in every
   realistic case (this is *why* the aclose is reliable, not a gap).
3. `terminal_chat.run_async` aclose-es the main client on shutdown (unit/inspection).
4. **Live check (run, 2026-05-31):** a real sub-agent via `asyncio.run(run_subagent)`
   on a thread (the exact bug path, with `HTTP_PROXY` present) completes with **no
   "Event loop is closed"** printed afterward — confirmed via captured stderr +
   `sys.unraisablehook`. **Caveat (honest):** a *before/after* proof could not be
   forced — with `aclose` neutralized (old behavior), a minimal single-sub-agent
   harness **also** produced no error, because the original is intermittent +
   accumulation/timing-dependent (idle proxied keep-alive connections across a
   multi-agent session). So the live run proves **no regression + the close is
   wired**, not reproduction-and-cure. The fix rests on being
   **correct-by-construction** — closing the pool inside its own loop is the
   standard remedy for an async client bound to a short-lived loop — plus the unit
   coverage below.
5. `ruff` clean; full `pytest` green.
