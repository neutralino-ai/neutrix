# v0.6.2 - main visible block list

## Problem

The main chat UI still reads like a composer wrapped by a log instead of
a clear list of messages sent to the language model:

- the system prompt is displayed inside the composer, not as a first
  conversation block;
- slash-command feedback is rendered into the same block list even
  though it is not sent to the model;
- the input is visually separate from the message list, so it does not
  feel like the next user block being drafted;
- role styling is present, but the distinction between system prompt,
  user input, and assistant response should be more explicit.

This makes `/onboard` especially confusing: the command is app control,
not a model-visible message, so it should not appear in the block list.

## Goal

Make the main surface a list of model-visible blocks:

- show the active system prompt as the first system block;
- show committed user turns, assistant turns, and tool results as blocks;
- show the editable input as the current draft user block in the same
  visual list;
- keep app-only slash commands and their feedback outside the
  model-visible block list;
- keep role styling distinct enough to identify system prompt, user
  draft/user turns, and assistant responses at a glance.

## Non-goals

- Change what is sent to the model.
- Add transcript editing, branching, avatars, or attachments.
- Redesign onboarding.
- Add command history or a separate command palette.

## Design

Keep the change inside `tui.py`:

1. Replace the separate system prompt display with a rendered
   `Message(role="system")` mounted into the main block list.
2. Render all existing `agent.messages` with non-empty content,
   including the system prompt.
3. Move the composer into the block list and style it as a draft user
   block with a role label and editable `Input`.
4. Treat slash commands as app control:
   - do not mount the command text as a user block;
   - do not mount command output as system blocks;
   - write command feedback to the existing status/notice area instead.
5. Keep streaming assistant output as assistant blocks and tool events
   as tool blocks, because those messages become part of the model
   conversation.

## Acceptance

- Launching the app shows a system prompt block and an editable draft
  user block.
- The input lives inside the block list, styled like a draft user
  message.
- Submitting normal text creates a user block, streams an assistant
  block, and preserves distinct role styling.
- Typing `/onboard` opens onboarding without adding `/onboard` or
  onboarding feedback to the model-visible block list.
- App-only command feedback is visible outside the block list.
- Existing onboarding secret isolation remains intact.
