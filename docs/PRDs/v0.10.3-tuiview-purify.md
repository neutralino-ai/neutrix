## Problem

After v0.9.x + v0.10.0 the agent loop emits lifecycle events with
a cancellation-safe stream, the `ContextManager` owns the state
machine and conversation-loop mutations, the store carries
canonical "what's in flight" state, the keyboard handles two-tap
exit and suspend + Esc cancellation, the heartbeat renders the
live state, streaming is back, and visibility parity has landed.
`terminal_chat.py` is *mostly* a reader of the store but still
hosts:

- inline command-dispatch logic that decides whether `/save` /
  `/load` happen at all,
- transcript-rendering decisions (which lines fold, which messages
  to skip on load, how to print the assistant's streaming tokens),
- a few residual writes that bypass the store (e.g. `_tool_records`
  for the folded-result tray).

This is the last layer of the onion that isn't on the right side of
the boundary. As long as it stays, every new feature risks
re-tangling the renderer with state — exactly what v0.7.0 set out to
prevent. And the v0.11.0 Advisor's design assumes the store is the
only mutator: if view code still writes side-channel state, the
Advisor's "I'll just mutate the store" model has holes.

## Goal

- TUIView (the `terminal_chat.py` module) becomes a pure reader of
  `ChatStore`. No state mutation inside view code; all writes go
  through `ContextManager` events or through the input-queue
  hand-off.
- Move the remaining residual state into the store — specifically
  `_tool_records` (the folded tool-result tray, currently a
  `list[ToolRecord]` private attribute on `TerminalChat` at
  `src/neutrix/terminal_chat.py:525`), as
  `store.folded_tool_results: tuple[ToolRecord, ...]`. The
  `ToolRecord` dataclass (defined at
  `src/neutrix/terminal_chat.py:137` with fields
  `index: int, name: str, arguments: str, result: str`) moves to
  `src/neutrix/store.py` — pure data, no rendering logic, so it
  belongs next to the rest of the canonical state.
- Add a **`FakeView`** in `tests/`: a ~50-line stdout renderer that
  reads `ChatStore` and prints conversation, queue, heartbeat, and
  tasks. `tests/test_fake_view.py` drives a synthetic controller
  event stream and asserts the printed output matches a captured
  baseline. Passing this test is the architectural proof that the
  view layer is genuinely swappable — and the contract that lets
  the v0.11.0 Advisor mutate the store with confidence that no
  hidden view-side writes will race or conflict.

## Non-goals

- **Web frontend.** Post-v1.0 question. The FakeView swap test
  stays in `tests/` as architectural hygiene; it is not a product
  surface in v1.0.
- **Multi-session UI / transcript browser.**
- **Cross-session task aggregation.**
- **Persistent visible heartbeat history** (e.g. recent latencies).
- **Refactoring `onboard.py`.** Continues to stay out of scope for
  the v0.7-arc.
- **TaskPanel changes.** v0.8.1 shipped the panel; v0.10.1 does
  not modify its behavior. (If the panel currently writes
  side-channel state outside the store — to be audited — those
  writes are in scope as part of "remove residual writes.")
- **Removing the heartbeat ticker as a side-channel of the
  renderer.** The v0.9.4 ticker is a *coroutine that calls
  `app.invalidate()`*, not a state mutator — it stays as-is.

## Design (skeleton — finalize after v0.10.0)

- Audit `src/neutrix/terminal_chat.py` for state writes that
  bypass the store. From the v0.9.3 source the known sites are:
  - `self._tool_records.append(record)` at line 759
    (in `_finalize_tool_record`) — moves to a
    `store.add_folded_tool_result(record)` mutator emitted by the
    `ContextManager` `ToolEvent("tool_finished", …)` reducer
    branch.
  - `self._tool_records.clear()` at lines 873 and 885
    (on `/clear` and on `/load`) — folded into the existing
    `ChatStore.reset()` / `ChatStore.replace_*()` mutators that
    `/clear` and `/load` already call.
  - `self._cancel_hint` private state — acceptable. Pure UI hint,
    not LLM-visible, not subscribed to by any other actor; stays
    as a private view-side attribute.
  Each remaining write either moves to (a) an input-queue
  hand-off (user actions), (b) a `ContextManager`-emitted store
  mutation (LLM- / tool-driven), or (c) a `ChatStore` mutator
  called from the reducer.
- Move `_tool_records` (folded-tool tray) onto the store as
  `folded_tool_results: tuple[ToolRecord, ...]`. `/tool [N]` reads
  from there.
- Add `tests/fake_view.py`: a `FakeView` class with a
  `render(store) -> str` method and a `subscribe(store)` async
  loop. Captures full snapshots after each store change.
- Add `tests/test_fake_view.py`: drives a sequence of synthetic
  events through a `ChatStore` and asserts that `FakeView` prints
  the expected baseline at each step. Scenarios include: empty
  store; user-only turn; assistant streaming reply; tool round
  (call + result); `<system-reminder>` injection; cancel-as-steer
  with `[interrupted by user]`.
- Boundary check: a CI grep (or import-linter rule) that asserts
  `terminal_chat.py` imports only from `store` (plus
  prompt_toolkit / stdlib). No imports from `context_manager`,
  `llm`, `tools`, `executor`.

## Acceptance (sketch)

- Manual:
  - With v0.8.0–v0.10.0 features intact, all existing transcript
    behaviors render identically (no visible regression).
  - `/tool [N]` still expands folded tool results from the new
    `store.folded_tool_results` location.
  - No blinks, no orphan state.
- Automated:
  - `tests/test_fake_view.py` runs the swap test and asserts on
    captured output across the scenario set. This is the
    **view-is-swappable** gate.
  - Boundary check passes: `terminal_chat.py` imports only the
    allowed modules.
  - All existing test suites pass unchanged.
- Human gate before tag.

## Open questions (to resolve before implementing)

- **What does the `FakeView` baseline look like** — exact bytes,
  or structured assertion (set of expected substrings)? Exact
  bytes is brittle; structured is friendlier. Decide during
  implementation.
- **`folded_tool_results` semantics on `/clear`** — wipe along
  with messages, or keep across clears? Default proposal: wipe
  (matches the v0.7.0 "store is the conversation" model).
- **Boundary check enforcement** — review-only, CI grep, or a
  proper import-linter rule? Default proposal: a small CI grep
  step in the existing test job; promote to import-linter if
  someone routes around it.
