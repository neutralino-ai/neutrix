# v1.6.2 — tool_use execution timeout (harden Bash)

## Problem

A `tool_use` can wedge the whole turn. The `Bash` tool **already** has a
`timeout` schema parameter (integer seconds, default 30; `src/neutrix/tools.py:404-457`,
`:757-761`), but its implementation has four concrete defects:

1. **Latent deadlock (the likely "stuck").** On timeout, `_run_shell` calls
   `proc.kill()` — SIGKILL to the **direct child only**, not the process group —
   then blocks on a bare `proc.communicate()` with **no timeout**
   (`tools.py:437-438`). `_run_shell` launches with `start_new_session=True`, so a
   command that spawns a grandchild (a backgrounded subprocess, a daemon, a shell
   pipeline) leaves that grandchild holding the stdout pipe open; the second
   `communicate()` then blocks **forever** even though the timeout already
   "fired." The turn never returns.
2. **Default 30 s is too short.** Ordinary dev commands — `pytest`, `pip install`,
   a build, `git clone` — routinely exceed 30 s. The LLM then misreads the timeout
   as a real failure and thrashes (re-runs, "fixes" nonexistent bugs).
3. **No maximum cap.** The LLM can pass `timeout: 999999` and hang the turn ~11
   days; only a user-cancel or the LLM-stall watchdog recovers it.
4. **Partial output discarded.** On timeout the tool returns only
   `"ERROR: command timed out after {timeout}s"` (`tools.py:443-444`), throwing
   away the stdout/stderr captured before the kill — often the most diagnostic
   signal (where did it hang).

This surfaced as a live "tool_use gets stuck, nothing output" during real OMILREC
work, in the same session as the v1.6.1 streaming bug.

## Goal

1. A timed-out `Bash` command **always** returns control to the turn — the
   timeout path can never re-block (defect 1).
2. The default timeout fits real dev work; a runaway LLM-supplied value can't hang
   the turn (defects 2, 3).
3. The model sees the partial output captured before the kill, with a clear
   timeout notice (defect 4).

## Non-goals

- **A global per-tool-call timeout over ALL tools.** User-directed: scope is
  Bash-only, mirroring CC (Split #1). The pure-compute tools (Glob, Grep-python,
  Read, Edit, Write) run in `asyncio.to_thread`, and Python threads can't be
  force-killed — a backstop could only *abandon* a hung thread, leaking it. The
  user judged that hang rare enough to skip (Split #7). Reopen only if real
  OMILREC use surfaces such a hang.
- **Millisecond units.** Keep seconds — neutrix already ships seconds; switching
  breaks the existing schema for no user benefit (Split #2).
- **Auto-background / `run_in_background` on timeout.** Background Bash is an
  already-deferred post-v2 subsystem (Split #8).
- **Touching the cancel path** (`Executor.cancel` / `_tree_kill` wiring) or any
  other tool. The fix reuses `_tree_kill` but does not change it.
- **The Grep-rg fixed 30 s timeout** (`tools.py:354`) — `subprocess.run(timeout=)`
  already kills its child correctly; out of scope.

## Claude Code reference

See `docs/splits/v1.6.2-tool-timeout.html` for the full survey (8 split points).
CC exposes `timeout` as an LLM-settable schema param on the **Bash tool only**
(`tools/BashTool/BashTool.tsx:229`, ms, default 120 000 / max 600 000,
env-overridable), enforces it SIGTERM→tree-kill-SIGKILL after a 5 s grace
(`utils/ShellCommand.ts:135-140,323-343`), and **keeps partial output** with a
`"Command timed out after <duration>"` stderr notice (`ShellCommand.ts:323-327`).
Other tools have fixed internal timeouts; there is **no global tool-dispatch
timeout wrapper**.

### Split point: timeout scope
- **CC**: Bash-only schema param; no global wrapper (`BashTool.tsx:229`).
- **Decision**: Follow CC — **user-directed** (AskUserQuestion 2026-05-31, "Bash-only
  (mirror CC)" chosen over the agent's global-backstop recommendation). Harden the
  existing Bash `timeout`; leave pure-compute tools unprotected. Fits "keep it
  simple, otherwise we never ship."

### Split point: units (seconds vs ms)
- **CC**: milliseconds.
- **Decision**: Alternative — keep **seconds**; neutrix already ships seconds,
  switching breaks the schema for no benefit.

### Split point: default timeout
- **CC**: 120 s (`utils/timeouts.ts`).
- **Decision**: Follow CC — raise 30 s → **120 s**, env-overridable via
  `NEUTRIX_BASH_DEFAULT_TIMEOUT_S` ([[feedback_llm_timeout_calibration]]).

### Split point: maximum cap
- **CC**: 600 s hard cap, env-overridable.
- **Decision**: Follow CC — **cap at 600 s** (env `NEUTRIX_BASH_MAX_TIMEOUT_S`),
  clamp a larger LLM-supplied value with a note in the result.

### Split point: kill scope on timeout
- **CC**: SIGTERM → tree-kill SIGKILL after a grace (`ShellCommand.ts`).
- **Decision**: Follow CC semantics — route the timeout path through the existing
  `_tree_kill` (`executor.py:185-212`, `killpg` SIGTERM→SIGKILL); bound the
  post-kill output drain so the path can never re-block. Fixes defect 1.

### Split point: partial output on timeout
- **CC**: keeps partial stdout/stderr + notice (`ShellCommand.ts:323-327`).
- **Decision**: Follow CC — return the partial output with the timeout notice;
  keep the `ERROR:` prefix so the executor's `ok=False` still propagates
  (`executor.py:173`).

### Split point: non-killable pure-compute (thread) tools
- **CC**: no analog (async file I/O).
- **Decision**: No CC analog / consequence of Bash-only scope — **leave
  unprotected**; accepted risk documented (a pathological regex / huge `rglob` can
  still hang a turn; only user-cancel recovers it).

### Split point: auto-background / `run_in_background`
- **CC**: auto-backgrounds on timeout; `run_in_background` param.
- **Decision**: Alternative — **defer** (background is a post-v2 subsystem).

## Design

All changes are confined to `src/neutrix/tools.py` (the `_run_shell` function, the
`Bash` tool schema, and new module constants).

1. **Configurable bounds (module constants, env-overridable at import).**
   ```python
   _BASH_DEFAULT_TIMEOUT_S = _env_int("NEUTRIX_BASH_DEFAULT_TIMEOUT_S", 120)
   _BASH_MAX_TIMEOUT_S = max(_env_int("NEUTRIX_BASH_MAX_TIMEOUT_S", 600),
                             _BASH_DEFAULT_TIMEOUT_S)
   ```
   `_env_int(name, fallback)` parses a positive int or returns the fallback
   (mirrors CC's `getDefaultBashTimeoutMs`; `max(...)` guarantees `max ≥ default`).

2. **`_run_shell` timeout resolution.** Signature `timeout: int | None = None`;
   when the LLM omits it, resolve to `_BASH_DEFAULT_TIMEOUT_S`. Clamp the effective
   value to `[1, _BASH_MAX_TIMEOUT_S]`; if the requested value was clamped, record
   a one-line notice for the result so the LLM learns the ceiling.

3. **Deadlock-free timeout enforcement.** Replace `proc.kill()` + bare
   `communicate()` with:
   ```python
   try:
       out, err = proc.communicate(timeout=effective)
   except subprocess.TimeoutExpired:
       _tree_kill(proc)                      # killpg SIGTERM→SIGKILL (group is dead)
       try:
           out, err = proc.communicate(timeout=_DRAIN_S)   # buffered output; returns fast
       except subprocess.TimeoutExpired:
           out, err = "", ""                 # defensive: never re-block
       timed_out = True
   ```
   `_tree_kill` is imported lazily inside the except branch
   (`from .executor import _tree_kill`) to avoid an import cycle
   (`executor` imports `dispatch` from `tools`); the import cost is irrelevant on
   the rare timeout path. The post-kill drain is bounded by a small `_DRAIN_S`
   (≈2 s) so even a pathological fd state cannot re-block.

4. **Partial output + notice on timeout.** Return the `ERROR:`-prefixed timeout
   line (so `ok=False` propagates), followed by any captured `stdout:` / `stderr:`
   blocks and the clamp notice if present:
   ```
   ERROR: command timed out after 120s
   stdout:
   …partial…
   ```

5. **Schema description.** Update the `Bash` tool's `timeout` property to
   `default = _BASH_DEFAULT_TIMEOUT_S` and description
   `"Timeout in seconds (default {default}, max {max})."` so the LLM sees the real
   bounds.

Cancellation (`Executor.cancel` → `_tree_kill` on the cancel path), the
`[cancelled by user]` branch, and `start_new_session=True` are unchanged.

## Acceptance

1. **Unit — clamp & default:** `_run_shell` with no `timeout` uses 120 s; with
   `timeout=999999` clamps to 600 s and the result carries the clamp notice; with a
   normal value runs unchanged. (Bounds read from the module constants / env.)
2. **Unit — deadlock-free timeout:** a command that times out **and leaves a
   grandchild holding stdout** (e.g. `sleep 0.1; (sleep 30 &) ; sleep 30`) returns
   within ~`timeout + _DRAIN_S`, not forever — the regression test for defect 1.
   The result is `ERROR:`-prefixed and includes any partial output.
3. **Unit — partial output:** a command that prints then hangs
   (`echo hi; sleep 30` with `timeout=1`) returns the `hi` partial stdout alongside
   the timeout notice.
4. **Regression:** a fast command still returns `exit_code/stdout/stderr` exactly
   as before; the user-cancel path and `[cancelled by user]` are unaffected;
   `ok=False` still derives from the `ERROR:` prefix.
5. **Live smoke:** in a real session, a `Bash` command that exceeds the default no
   longer hangs the turn — it returns a timeout result with partial output, and the
   turn continues.
6. `ruff` clean; full `pytest` green.
