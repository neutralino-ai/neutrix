neutrix roadmap

A terminal agentic chat client for OpenAI-compatible LLM APIs.

This page is the user-visible map of where the project is going. Per CLAUDE.md, every PRD acceptance updates this page before implementation starts. Entries link to their PRDs under docs/PRDs/.

v1.0 destination: a TUI Claude-Code-shaped agentic client — cancel-as-steer, lifecycle-driven store, visibility parity, smart Advisor, conversation compaction, subagent dispatch, rewind / branch. Sealed surface at v1.0. Markdown rendering, cost tracking, web frontend, MCP, skills, hooks are post-v1.0 (v1.x).

Ordered by version, newest first; the priority tag marks urgency: P0 blocks daily use right now; P1 important for the v1.0 lock; P2 nice to have pre-v1.0; P3 deferred to post-v1.0 (v1.x).

A band of minor releases building to a v2.0.0 lock, mirroring how v0.10.x built to v1.0.0. Goal (user-directed 2026-05-30): a practical, reliable terminal coding agent for real work — fast and autonomous enough to speed up OMILREC — not a complete Claude Code clone. "Keep it simple, otherwise we never ship." Parity features that don't serve that are cut (see Descoped below).
Core principle: .claude/-compatible (read-only). neutrix auto-discovers CC's config — CLAUDE.md, .claude/skills·commands·agents, .claude/settings.json (permissions) — so it drops into a CC project and inherits the config. It never writes into ~/.claude (user-directed): neutrix-owned state (session logs, …) lives under ~/.cache/neutrix/; ccusage-compatible JSONL emission is opt-in (v1.8.0).

v1.6.2 Hotfix — Bash tool_use timeout (deadlock + calibration) The Bash tool already had a timeout schema param, but its timeout path could hang the turn forever: proc.kill() killed only the direct child, then a bare communicate() blocked on a leaked grandchild holding the stdout pipe. Now the whole process group is tree-killed (reusing _tree_kill) with a bounded drain — it can never re-block. Default raised 30 s → 120 s (30 s spuriously killed pytest/builds), capped at 600 s with a clamp note, partial output kept. v0.9.5 LLM error surface (stalled / timeout / inline error) User-reported during v0.9.4 dogfooding: when the LLM API hangs silently, the heartbeat keeps breathing but neither neutrix nor the LLM is making progress, and the user has no signal until the SDK's ~10-minute default timeout fires.
