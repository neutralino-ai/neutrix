neutrix вҖ” architecture (how it works now)

The how-it-works companion to roadmap.html (which is where the project is going). This is how the current v2-surface code is wired, as of v1.7.x.

Code references are symbol-based (module.py вҖә Symbol), not line numbers вҖ” line numbers rot, symbols survive edits. The module docstrings remain the authoritative low-level detail; this is the map.

The shape in one sentence

A user-facing UI and a headless conversation engine, joined by one typed store: ui вҶ” store вҶ” CM вҶ” { LLM, Executor, Advisor }. But the real graph is hub-and-spoke, not a chain: the UI sends typed events straight to the engine and separately subscribes to the store's change notifier to render; the engine is the only writer of conversation state into the store; the Executor writes tool results back into it too.

store
store.py вҖә ChatStore вҖ” the single source of truth (typed MessageRecords, pending stream text, pending tool calls, tasks, the queued-input panel). ChatStore.changes() is the one async notifier the UI subscribes to; every mutation calls _notify() (and v1.7.2's ChatStore.notify() pokes it without a mutation, for compaction liveness).
UI вҶ’ CM
The UI sends typed events via context_manager.py вҖә ContextManager.handle_event() вҖ” UserMessageEvent, CancelEvent, SlotSwitchEvent, ClearEvent, ReplaceHistoryEvent. This is a direct call, distinct from the render path.
UI вҶҗ store
terminal_chat.py вҖә TerminalChat._render_watcher() iterates store.changes() and renders new records append-only; the same loop persists them (_persist_new_records / _persist_new_usage).
CM вҶ’ engine
Per round the CM calls OpenAIChatLLM.stream_response(), dispatches tools via Executor.dispatch_all(), writes records to the store, and вҖ” if attached вҖ” feeds CostLedger.record() (observe-only; the loop never depends on it, keeping the CM headless for the v3 RL-env goal).
Executor вҶ’ store
The Executor holds executor.store so tool results (and Task* tools running on its dispatch thread) land in the same store.
Advisor
terminal_chat.py вҖә TerminalChat._maybe_run_advisor() consults a third actor at turn-end on a separate fast-slot client (never the shared ctx.llm), while the CM is IDLE.

The CM state machine

One driver loop, ContextManager._drive(), walks LLM rounds until the conversation settles:

  IDLE в”Җв”Җuser_messageв”Җв”Җв–ә AWAITING_LLM в”Җв”Җassistant, no tool_callsв”Җв”Җв–ә IDLE
                              в”Ӯ  assistant WITH tool_calls
                              в–ј
                       AWAITING_EXECUTOR в”Җв”Җall tools doneв”Җв”Җв–ә AWAITING_LLM
                              в–І в”Ӯ
            (any state) в”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җ CANCELLING в”Җв”Җв–ә IDLE   (short-circuit)

Each round: _call_llm() streams a turn (a no-progress watchdog cancels a genuinely stalled round; a prompt-too-long error triggers one compaction-and-retry), the assistant message is appended, and if it carries tool_calls they dispatch before looping. Threshold compaction runs once at the top of a turn (_maybe_auto_compact()); both it and the manual path set _compacting so the heartbeat shows a live "Compacting" indicator (v1.7.2).

Four functions, end to end

1 В· cancel (keep-partial + append-marker)

  1. Esc in the input area fires the draft key binding's cancel_hook вҶ’ TerminalChat.try_cancel_current_stream().
  2. вҶ’ ContextManager.cancel() emits a CancelEvent; handle_event() routes it to _do_cancel().
  3. _do_cancel(): Executor.cancel() tree-kills any running subprocess; OpenAIChatLLM.stop() closes the active stream; the in-flight assistant text in store.pending_assistant_text is kept (committed as a real turn); then llm.py вҖә INTERRUPTED_BY_USER_MARKER is appended as its own turn. State вҶ’ CANCELLING, which _drive() short-circuits. The CM consumes the CancelledError so the awaiting handler returns normally (v0.9.2/v0.9.3 append-marker semantics, not history rollback).

2 В· rewind (/rewind, round-safe)

  1. /rewind [N] вҶ’ TerminalChat._cmd_rewind() вҶ’ ContextManager.rewind_to(message_index).
  2. _safe_rewind_index() snaps the cut to a tool-round boundary вҖ” a rewind never bisects a round, leaving a dangling assistant with unanswered tool_calls or an orphan role:tool.
  3. History is rebuilt through the same ReplaceHistoryEvent seam that /load and resume use. Dropped user prompts are recallable with Up/Down in the input (recallable_user_turns() / the draft key bindings). (v0.9.7)

3 В· save / load (two distinct mechanisms)

  1. Explicit single-file export: /save [PATH] / /load PATH вҶ’ transcript.py вҖә save() / load() вҖ” one portable JSON file.
  2. Automatic resume: every settled turn is appended as a JSONL line under ~/.cache/neutrix/sessions/<cwd>/<uuid>.jsonl (neutrix's own cache, never ~/.claude) by session_store.py вҖә SessionWriter. --continue loads the most recent, /resume a picker; both restore via session_store.load_session() вҶ’ ReplaceHistoryEvent, and rebuild the cost ledger via CostLedger.from_jsonl() (v1.7.0).
  3. Both restore paths share the ReplaceHistoryEvent seam вҖ” the same one rewind uses.

4 В· goal (/goal autonomous loop)

  1. /goal <text> вҶ’ TerminalChat._cmd_goal() sets _goal and puts a _GOAL_KICK sentinel on the worker queue to unblock the idle loop.
  2. After each turn settles, _run_goal_continuations() drives the next turn autonomously until done / step cap (GOAL_MAX_STEPS) / interrupt; the live region shows в—Һ goal В· step N/MAX.
  3. A user message typed mid-goal sets _goal_interrupt вҖ” the loop releases on its next check (the user is taking over). In-memory only; not resumed across sessions. (v1.6.0)
